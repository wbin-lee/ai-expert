# 1. Optimal Control의 두 관점

## 1.1 문제 정의

Deterministic system:

\[
s_{t+1}=f(s_t,a_t),\quad \max \sum_t r(s_t,a_t)
\]

stochastic이 아니므로 \(\mathbb{E}\)가 필요 없다.

## 1.2 두 가지 뷰포인트

| 관점 | 최적화 변수 | 결과 형태 | 특징 |
|---|---|---|---|
| **Dynamic Programming** | policy \(\pi\) | state → action 매핑 | global optimal, **closed-loop**, scalability 약함 |
| **Trajectory Optimization** | action 시퀀스 \(a_0,...,a_{T-1}\) | 시간축 trajectory | local optimal, **open-loop**, scalable |

> ⭐ Deterministic이라서 trajectory optimization이 가능. Stochastic이면 state perturbation 때문에 미리 계획한 action sequence는 더 이상 optimal하지 않다.

---

# 2. Dynamic Programming 방식

## 2.1 Bellman Equation (Deterministic)

\[
V_t^*(s) = \max_a\left[\,r(s,a) + V_{t+1}^*(f(s,a))\,\right]
\]

boundary condition (terminal reward)부터 역순으로 계산. action도 함께 정해진다.

## 2.2 Closed-Loop / Feedback Control 연결

- Policy가 state를 보고 action을 결정 → **feedback loop** 형성.
- Disturbance가 state에 들어와도 그에 맞는 action으로 자동 대응.

## 2.3 한계

- **Curse of Dimensionality** — state·action 공간이 커지면 DP를 exact하게 풀기 어려움.
- 고차원 문제에서는 사용 불가.

---

# 3. Trajectory Optimization

## 3.1 문제 형태

\[
\max_{s_{0:T},a_{0:T-1}}\sum_t r(s_t,a_t)\quad \text{s.t.}\quad s_{t+1}=f(s_t,a_t)
\]

constraint optimization을 numerical solver (Interior Point Method 등)로 푼다.

## 3.2 결과 — Action Trajectory

\(a_t^*\)는 시간 \(t\)에 취할 action을 미리 다 계산한 것. 현재 state를 관찰하지 않고 시간 따라 미리 정해진 action을 내뱉는다.

## 3.3 단점

| 단점 | 설명 |
|---|---|
| Local optimal | 비선형 시스템에서 boundary condition 의존 |
| **Open-loop** | 중간에 state perturbation이 들어오면 더 이상 optimal하지 않음 |
| Initial state 의존 | \(s_0\)가 바뀌면 전체 문제를 다시 풀어야 함 |
| Infinite horizon | optimization variable이 무한 차원 → 어려움 |

> ⭐ DP는 글로벌·closed-loop지만 비싸고, Trajectory Optimization은 빠르지만 perturbation에 약하다.

# 4. System Identification — 전통적 모델 베이스 방식

## 4.1 절차

1. Base policy로 trajectory data 수집.
2. 도메인 지식으로 모델 구조 \(f_\theta(s,a)\) 정의 (예: 유체역학, 열역학, 로봇 dynamics).
3. **Least Square Fitting**으로 파라미터 \(\theta\) 학습:
   \[
   \min_\theta \sum_i \|f_\theta(s_i,a_i)-s'_i\|^2
   \]
4. 학습된 모델로 model-based control (DP 또는 controller synthesis).

산업계에서 수십 년 ~ 100년 표준이었던 방법.

## 4.2 한계 — Stationary 가정

환경이 변하지 않는다는 가정에 의존. 자율주행처럼 **non-stationary** 환경에서는 매번 재설계가 필요해 부담이 크다.

## 4.3 Online System Identification

추가 데이터를 모을 때마다 시스템 식별을 다시 돌려 환경 변화에 적응하는 변형. 환경 변화 속도가 느릴 때 적용 가능.

> ⭐ 단점: control 부분은 여전히 open-loop trajectory optimization → 컨트롤 측면에선 **unreliable**.

# 5. Model Predictive Control (MPC)

## 5.1 핵심 아이디어

> ⭐ ⭐ ⭐ **Trajectory Optimization을 매 시점 반복하고, 그 결과 중 첫 번째 action만 쓴다.**

알고리즘:

1. 현재 state \(s_t\)를 받음.
2. \(t\)부터 \(t+H-1\)까지의 action sequence를 trajectory optimization으로 계산.
3. **첫 번째 action만 적용**.
4. 새 state 관찰 후 1로 돌아감.

## 5.2 왜 Closed-Loop이 되는가

매 시점 현재 state를 받아 다시 최적화하므로 state perturbation에 대응 가능. 결과적으로 policy처럼 (state → action) 매핑이 작동한다.

## 5.3 Receding Horizon Control

고려하는 시간 윈도우 \(H\)는 고정. 시간이 흐르면 윈도우가 한 칸씩 미끄러진다. 그래서 MPC를 **Receding Horizon Control**이라고도 부른다.

## 5.4 Explicit MPC

오프라인으로 모든 state에 대한 optimization을 미리 풀어 lookup table 형태로 저장. 온라인에서는 lookup으로 즉시 action 반환 가능.

- 선형 시스템에서 강력.
- nanosecond 수준 sample time까지 처리 가능.

## 5.5 적용 범위

샘플 타임이 짧아지는 순서:

| 어플리케이션 | 샘플 타임 |
|---|---|
| HVAC (빌딩 공조) | 500 ms |
| Vehicle Steering / Traction Control | 짧음 |
| Converter Control | 더 짧음 |
| **CPU Thermal Management** | nanosecond |

> ⭐ 과거에는 process engineering처럼 느린 시스템에만 쓰였지만, 컴퓨팅 향상과 실시간 최적화 발전으로 점점 빠른 시스템에까지 확대.

# 6. Model-Based RL with Neural Network

## 6.1 단순 결합의 한계

NN으로 글로벌 모델 \(f_\theta\)를 학습하고 MPC에 넣으면:

- NN의 비선형성 때문에 MPC 최적화가 더 어려워짐.
- 학습 안 된 영역에서 모델 부정확 → MPC가 이상한 action 결정.
- task의 모델이 policy보다 훨씬 복잡할 때 학습 자체가 어려움 (예: 로봇 손가락).

## 6.2 Local Linearization — 핵심 해결책

> ⭐ ⭐ 글로벌 NN 모델을 포기하고 **로컬 선형화 모델**을 쓴다.

\[
s_{t+1} \approx A_t (s_t - \bar s) + B_t(a_t - \bar a) + \bar s
\]

\(A_t, B_t\)는 현재 동작점 근처에서 Jacobian. Linear regression으로 학습.

장점:

- 모델이 단순 → MPC가 가벼움.
- **Linear MPC** 영역으로 들어가 Explicit MPC 활용 가능 → 온라인 컴퓨테이션 비용 ↓.

## 6.3 알고리즘

```
1. Base policy로 데이터 모음
2. while not converged:
     - 현재 state·action 근처에서 Linear Regression으로 (A, B) 추정
     - Inner loop: K번
         - Linear MPC로 다음 action 계산
         - 실행 후 데이터 추가
     - K가 너무 크면 안 됨: 로컬 선형 모델은 멀리 가면 부정확
```

> ⭐ **K (inner loop 길이) 를 작게 가져가야 한다** — 로컬 선형화는 오래 쓰면 invalid.

# 7. 알고리즘 선택 가이드라인 (Sergey Levine)

> ⭐ 강의 마무리 — 실무에서 어떤 RL을 골라야 할지의 결정 트리.

```
시뮬레이터가 있는가?
├─ No
│  ├─ 정말 기다리기 어렵다 → Model-Based RL (도메인 지식으로 모델 구조 잡고 파라미터 학습)
│  └─ 어느 정도 기다릴 수 있다 → Off-Policy (DDPG / SAC) — 실물 try-and-error 가능
└─ Yes
   ├─ 시뮬레이션 비용 거의 0 (병렬 2,000개 가능) → On-Policy (TRPO / PPO)
   └─ 시뮬레이션 비용이 어느 정도 든다 → Off-Policy (DDPG / SAC)
```

핵심 메시지:

| 환경 | 추천 알고리즘 |
|---|---|
| 시뮬레이터 풍부, 빠름 | TRPO / PPO |
| 시뮬레이터 보통 / 실물 직접 학습 | DDPG / SAC |
| 시뮬레이터 없음, 기다리기 어려움 | Model-Based RL (도메인 지식 활용) |

# 9. SAC 실습 정리

## 9.1 Entropy 직관 다지기

두 policy 비교:

- Policy 1: action 1과 action 2를 거의 균등하게 선택 → **높은 entropy**.
- Policy 2: action 1에 거의 100% → **낮은 entropy**.

> ⭐ entropy term이 reward와 함께 더해지면 policy가 너무 한쪽으로 쏠리지 않게 된다 → exploration.

## 9.2 Soft Policy Evaluation

advantage 관점:

\[
A(s,a) = Q(s,a) - V(s)
\]

Maximum Entropy 버전에서는 V 안에 entropy term이 자연스럽게 포함되어, advantage가 양수인 액션의 확률을 그만큼 키우는 식으로 학습된다.

## 9.3 α 값에 따른 Policy 형태

α가 작아지면 policy는 greedy해지고, 커지면 exploratory해진다.

| α | 행동 |
|---|---|
| 0.1 | 거의 greedy (action 1만) |
| 1.0 | 균형 |
| 2.0 | 매우 exploratory |

## 9.4 Critic 설계 디테일

- **Twin Q-Networks**: \(Q_1, Q_2\) 두 개 두고 **min을 사용**해 overestimation 방지 (TD3 트릭).
- Target은:
  \[
  y = r + \gamma \mathrm{mask} \cdot \left(\min(Q_1^-, Q_2^-) - \alpha \log\pi\right)
  \]
- mask: terminal state면 다음 value를 더하지 않게.

## 9.5 Actor 설계 — Reparametrization 구현

- Gaussian policy: \(\mu_\theta(s), \sigma_\theta(s)\) 두 head.
- \(\log\sigma\)를 출력 (range는 [-20, 2]로 clip).
- `rsample()` 사용 — gradient를 action을 통해 흘려보내기 위함.
- 액션을 \([-1, 1]\)로 매핑하기 위해 `tanh`를 적용하고, 그에 따른 **log-prob 보정**:
  \[
  \log\pi(a|s) = \log\mathcal{N}(u|\mu,\sigma) - \sum_i\log(1 - \tanh^2(u_i))
  \]

## 9.6 Auto Temperature Tuning (α 학습)

원본 SAC의 약점인 α 민감성을 해결한 후속 논문.

\[
L(\alpha) = -\alpha\left(\log\pi(a|s) + \bar H\right)
\]

- 학습할 변수는 \(\log\alpha\) (α는 양수 제약).
- Target entropy: \(\bar H = -|\text{action dim}|\) — action dim이 클수록 자연스럽게 entropy가 음수 방향으로 커지므로 target도 같이 음수로 크게.

## 9.7 Ant 환경 실습

- obs dim = 105, action dim = 8 → 고차원 문제.
- Stable-Baselines3 활용 가능.

# 10. Model-Based RL 실습 — Random Shooting과 Cross-Entropy Method

## 10.1 NN으로 모델 학습

\[
\hat s' = f_\theta(s,a),\quad L = \|\hat s' - s'\|^2
\]

## 10.2 Random Shooting (가장 단순한 MPC)

```
1. N개의 action sequence를 랜덤하게 샘플
2. 각 sequence에 대해 학습된 모델로 trajectory 예측
3. 누적 reward로 평가
4. 가장 좋은 sequence의 첫 번째 action만 실행
```

## 10.3 Cross-Entropy Method (CEM)

> ⭐ Random Shooting의 학습 버전.

```
mu, sigma 초기화
while not converged:
    1. p(a) = N(mu, sigma)에서 N개 action sequence 샘플
    2. 모델로 trajectory 예측 후 reward로 평가
    3. 상위 M개 (Elite Sample) 의 통계로 mu, sigma 업데이트
```

학습이 진행됨에 따라 action sequence 분포가 좋은 영역으로 수렴.

## 10.4 Dyna-Style 모델 활용

DQN/SAC 같은 Off-Policy 알고리즘과 결합 시 **imagined rollout**으로 sample efficiency 추가 향상:

```
실제 환경 1 step → Q 업데이트
+ 학습된 모델로 상상 rollout N step → Q 추가 업데이트
```

> ⭐ 어려운 환경에서 RL을 실제로 돌릴 때 꼭 고려해야 할 트릭.

## 10.5 World Model

최근 핫한 World Model 계열도 본질은:

- latent space에서 미래를 예측.
- 상상한 trajectory를 가지고 policy 업데이트.

→ 결국 같은 원리: **모델로 sample efficiency 향상**.

## 10.6 MPC의 Prediction Horizon vs Control Horizon

실무 MPC에는 두 hyperparameter가 있다.

| 파라미터 | 의미 |
|---|---|
| **Prediction Horizon (H)** | 미래를 얼마나 멀리 예측할지 |
| **Control Horizon** | 그 중에 실제로 최적화할 action sequence 길이 (보통 3~5) |

H는 길게 가져가도 되지만 Control Horizon은 짧게: 너무 길면 모델 오차 누적 + 최적화 비용 폭증.

# 11. 오후 핵심 정리

## 11.1 Model-Based RL의 위치

- Model-Free는 검증성·sample efficiency가 약하다.
- Classical은 비-stationary 환경에 약하다.
- **Model-Based RL**이 둘 사이의 가교 역할.

## 11.2 MPC의 본질

> ⭐ MPC = **Trajectory Optimization을 매 시점 반복 + 첫 action만 사용** → open-loop를 closed-loop로 회복.

## 11.3 NN 모델 + MPC

- 글로벌 NN 모델은 MPC와 결합이 어렵다 → **Local Linearization**으로 가볍게.
- Linear MPC + Explicit MPC로 온라인 비용 최소화.

## 11.4 SAC 실습에서 기억할 것

| 항목 | 핵심 |
|---|---|
| Twin Q-net + min | overestimation 방지 |
| Target에 entropy term 포함 | maximum entropy 정의의 자연스러운 결과 |
| Reparametrization Trick | stochastic actor의 gradient variance 감소 |
| α 자동 조절 | target entropy = -action dim |

## 11.5 Model-Based 실습에서 기억할 것

- **Random Shooting**: 가장 쉬운 baseline.
- **CEM**: 분포를 학습하며 점점 좋은 action sequence로 수렴.
- **Dyna-style**: 학습한 모델로 imagined rollout을 만들어 sample efficiency 향상.

## 11.6 알고리즘 선택 결정 트리 (최종)

```
시뮬레이터 있음?
├─ Yes
│  ├─ 매우 빠름 → TRPO / PPO
│  └─ 비용 있음 → DDPG / SAC
└─ No
   ├─ 기다리기 어려움 → Model-Based RL
   └─ 가능 → DDPG / SAC
```

## 11.7 시험 대비 (강사 직접 안내)

- **모든 문제 서술형**, 답안에 수식 불필요.
- 마지막 두 문제: TRPO vs PPO 비교 + Model-Based RL vs Model-Free RL 차이.
- 큰 그림과 직관, 알고리즘이 적합한 상황만 정리하면 충분히 합격선.
