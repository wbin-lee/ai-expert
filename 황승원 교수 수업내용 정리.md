# 1. 이 수업은 어디서 시작했는가

황승원 교수의 수업은 처음부터 ChatGPT를 바로 뜯어보는 방식으로 시작하지 않았다. 오히려 질문은 반대였다.

```text
LLM이 없던 시대에는 NLP를 어떻게 했을까?
```

이 질문이 중요하다. 지금은 ChatGPT 하나가 번역, 요약, 질의응답, 코드 생성, 검색 보조, 에이전트 행동까지 다 하는 것처럼 보인다. 하지만 그 모든 능력은 어느 날 갑자기 하늘에서 떨어진 것이 아니다. 그 밑에는 오랫동안 쌓인 NLP의 고전적인 문제들이 있다.

예를 들어 스팸 메일을 걸러내려면 먼저 "이 문서가 스팸인가 아닌가"를 분류해야 한다. 리뷰를 읽고 긍정인지 부정인지 판단하려면 감정 분석이 필요하다. 문장에서 사람 이름, 회사 이름, 장소 이름을 찾으려면 Named Entity Recognition이 필요하다. "그 사람", "그 회사"가 실제 세상의 누구를 가리키는지 알려면 grounding이 필요하다. 검색 결과에서 답을 찾으려면 검색, 랭킹, RAG가 필요하다.

수업은 이 큰 그림을 따라갔다.

```text
규칙 기반 NLP
→ 확률 기반 NLP
→ 언어모델
→ 임베딩
→ 검색
→ RAG
→ 지식그래프
→ 다국어 NLP
→ 코드 생성
→ 포스트트레이닝
→ 에이전트와 강화학습
→ SQL과 MCTS
```

이 흐름을 이해하면, 각 주제가 따로 흩어진 키워드가 아니라 하나의 발전사로 보인다.

# 2. 자연어의 첫 번째 문제: 같은 뜻이 너무 많은 모양으로 나타난다

NLP의 첫 난관은 아주 단순하다. 사람은 같은 뜻을 너무 다양한 겉모습으로 표현한다.

교수는 이것을 surface form 문제로 강조했다.

```text
woodchuck
woodchucks
Woodchuck
WOODCHUCKS
```

사람에게는 모두 거의 같은 단어처럼 보이지만, 컴퓨터에게는 기본적으로 다른 문자열이다. 그래서 처음 NLP는 규칙에서 시작했다.

정규표현식은 이런 규칙 기반 접근의 대표 도구이다.

```text
[Tt]he
colou?r
5+
5*
```

정규표현식은 특정 패턴을 찾는 데 강력하다. 하지만 곧 한계가 나온다. `[Tt]he`는 `the`와 `The`를 찾을 수 있지만 `THE`는 놓친다. 반대로 `there`, `other` 안의 `the`까지 잘못 잡을 수 있다.

여기서 바로 평가 문제가 나온다.

```text
False Positive:
찾으면 안 되는 것을 찾았다.

False Negative:
찾아야 하는 것을 못 찾았다.
```

범죄자 검거에서는 false positive, 즉 무고한 사람을 잡는 것이 치명적일 수 있다. 반면 감염병 검사에서는 false negative, 즉 아픈 사람을 놓치는 것이 더 위험할 수 있다. 그래서 어떤 에러가 더 중요한지는 항상 문제에 따라 달라진다.

이 균형을 보기 위해 precision, recall, F1이 등장한다.

```text
Precision:
내가 찾았다고 한 것 중 실제로 맞은 비율

Recall:
실제 정답 중 내가 찾아낸 비율
```

F1은 둘의 조화평균이다.

$$ F1 = 2 \cdot \frac{Precision \cdot Recall}{Precision + Recall} $$

조화평균을 쓰는 이유는 한쪽만 높고 다른 한쪽이 낮은 모델을 강하게 벌주기 위해서다. precision만 챙기려면 아무것도 찾지 않으면 된다. recall만 챙기려면 모든 것을 다 찾았다고 하면 된다. 둘 다 쓸모없다. 좋은 모델은 필요한 것을 많이 찾되, 쓸데없는 것도 적게 가져와야 한다.

# 3. "거의 맞았다"를 계산하기: Edit Distance

정규표현식은 맞거나 틀리거나 둘 중 하나이다. 하지만 자연어에는 "거의 맞았다"가 중요하다.

예를 들어 `graffe`라는 단어를 보면 사람은 "아마 graph를 치려다 오타가 난 것 같다"고 생각한다. 이것을 수학적으로 표현하는 방법이 edit distance이다.

Edit distance는 한 문자열을 다른 문자열로 바꾸기 위해 필요한 최소 편집 횟수이다.

기본 연산은 세 가지이다.

```text
Insertion:
글자 하나를 삽입한다.

Deletion:
글자 하나를 삭제한다.

Substitution:
글자 하나를 다른 글자로 바꾼다.
```

두 문자열 사이에는 가능한 편집 경로가 무수히 많다. 다 지웠다가 다시 써도 된다. 하지만 우리가 원하는 것은 가장 짧은 경로이다.

이때 naive하게 모든 경로를 펼치면 지수적으로 폭발한다. 그래서 dynamic programming을 쓴다.

$$ D[i][j] = \min(D[i-1][j]+1, D[i][j-1]+1, D[i-1][j-1]+sub\_cost) $$

이 점화식은 세 가지 선택지를 비교한다.

```text
S1의 마지막 글자를 지운다.
S2의 마지막 글자를 삽입한다.
마지막 글자를 맞추거나 치환한다.
```

이 아이디어는 뒤에서 Viterbi, Beam Search, MCTS까지 계속 반복된다. 큰 문제를 한 번에 다 보지 않고, 작은 부분 문제의 최적해를 쌓아간다.

# 4. 언어를 쪼개는 일: Tokenization, Normalization, BPE

언어모델을 만들려면 먼저 텍스트를 단위로 쪼개야 한다. 이 단위가 token이다.

가장 단순한 방법은 공백으로 나누는 것이다.

```text
I love you
→ I / love / you
```

하지만 금방 문제가 생긴다.

```text
dog
dog.
I'd
U.S.A.
아버지가방에들어가신다
```

영어는 그나마 공백이 있지만, 중국어, 일본어, 태국어처럼 공백이 뚜렷하지 않은 언어도 있다. 한국어도 조사와 어미 때문에 단어 경계가 단순하지 않다.

그래서 normalization이 필요하다.

```text
US / U.S. / USA / U.S.A. → 미국
cats / cat's / cats' → cat
am / is / are / was → be
```

여기서 lemmatization과 stemming이 갈린다.

Lemmatization은 사전과 언어학 지식을 이용해 정확한 원형을 찾는다.

```text
running → run
geese → goose
is → be
```

Stemming은 규칙으로 대충 자른다.

```text
studies → studi
relational → relate
cats → cat
```

Lemmatization은 정확하지만 비싸고, stemming은 싸지만 거칠다.

그 다음 등장하는 것이 BPE이다. BPE는 현대 LLM 토크나이저의 핵심 아이디어다.

BPE는 언어학자가 단어를 가르쳐주지 않는다. 데이터를 보고 자주 함께 나오는 글자쌍을 합친다.

```text
처음에는 글자 단위
→ 자주 붙는 쌍을 merge
→ 다시 자주 붙는 쌍을 merge
→ subword vocabulary 생성
```

예를 들어 `low`, `lower`, `newest`, `widest` 같은 단어를 보면, BPE는 자주 함께 나오는 조각을 찾아 `low`, `er`, `new` 같은 subword를 만들어낸다. 이것은 언어에 덜 의존적이다. 7000개 언어마다 tokenizer를 따로 설계하는 대신, 데이터 통계로 token 단위를 학습한다.

BPE의 두 단계는 꼭 구분해야 한다.

```text
Learn:
코퍼스에서 merge rule을 학습한다.

Apply:
새 텍스트에 학습된 merge rule을 적용한다.
```

# 5. 언어모델의 탄생: 문장에 확률을 주는 함수

수업에서 언어모델의 정의는 명확했다.

```text
언어모델은 임의의 텍스트에 확률을 할당하는 함수이다.
```

수식으로 쓰면 다음과 같다.

$$ LM(w_1,w_2,\dots,w_N) = P(w_1,w_2,\dots,w_N) $$

좋은 언어모델은 자연스러운 문장에 높은 확률을 주고, 이상한 문장에 낮은 확률을 준다.

이 능력은 여러 문제에 쓰인다.

```text
스펠체크:
minuet보다 minute가 문맥상 자연스러운가?

음성인식:
I saw a van이 I saw a ban보다 자연스러운가?

번역:
high wind가 large wind보다 자연스러운가?
```

초기 언어모델은 count-based였다. 코퍼스에서 단어가 얼마나 자주 나왔는지 세고, 그 빈도로 확률을 추정했다.

Unigram은 단어 하나만 본다. Bigram은 직전 단어 하나를 본다. Trigram은 직전 두 단어를 본다.

$$ P(w_i|w_1,\dots,w_{i-1}) \approx P(w_i|w_{i-1}) $$

이것이 Markov assumption이다. 긴 과거를 다 기억하지 않고, 최근 몇 개만 보겠다는 근사다.

하지만 N-gram은 금방 sparsity 문제를 만난다. vocabulary가 3만 개라면 bigram table은 9억 칸이다. 실제로 관측되는 조합은 극히 일부이다. 대부분은 0이다.

0 확률은 치명적이다. 문장 안에 한 번도 못 본 bigram이 하나라도 있으면 전체 확률이 0이 된다.

그래서 smoothing을 한다.

대표적으로 Laplace smoothing은 모든 count에 1을 더한다.

$$ P(w_i|w_{i-1}) = \frac{Count(w_{i-1},w_i)+1}{Count(w_{i-1})+V} $$

이것은 "한 번도 못 봤다고 절대 불가능하다고 말하지 말자"는 태도이다. 교수는 이것을 자기 경험에만 매몰된 "꼰대"와 비교했다. 좋은 모델은 본 적 없는 일에도 작은 가능성을 남겨둔다.

# 6. 베이즈 정리와 Noisy Channel: 오타를 고치는 방식

스펠체크는 언어모델의 아름다운 응용이다.

관찰한 것은 noisy word이다.

```text
x = graff
```

알고 싶은 것은 원래 쓰려던 단어이다.

```text
w = graph?
```

우리는 $P(w|x)$를 알고 싶다. 하지만 직접 구하기 어렵다. 그래서 Bayes rule로 뒤집는다.

$$ \hat{w} = \arg\max_w P(w|x) = \arg\max_w P(x|w)P(w) $$

여기서 $P(w)$는 언어모델이다. 즉, 단어 자체가 얼마나 자주 자연스럽게 쓰이는지 나타낸다. $P(x|w)$는 noise model이다. 원래 $w$를 쓰려 했는데 실수로 $x$가 나올 확률이다.

Peter Norvig의 유명한 spell corrector는 이 아이디어를 아주 짧은 코드로 구현했다.

```text
1. edit distance 1 또는 2 안의 후보를 만든다.
2. 코퍼스에 있는 후보만 남긴다.
3. 언어모델 확률이 가장 높은 후보를 고른다.
```

여기서 언어모델이 좋아지면 스펠체커도 좋아진다. Bigram을 쓰면 문맥까지 볼 수 있다.

```text
versatile acress
```

이 경우 단어 자체 빈도만 보면 `across`가 유리할 수 있지만, 문맥을 보면 `actress`가 더 자연스럽다.

이것이 수업의 반복되는 메시지다.

```text
좋은 언어모델은 여러 NLP 문제를 한꺼번에 좋게 만든다.
```

# 7. 분류 문제: Naive Bayes와 평가

문서 분류도 베이즈로 풀 수 있다.

우리가 원하는 것은 문서 $d$가 클래스 $c$일 확률이다.

$$ P(c|d) \propto P(c)P(d|c) $$

문서 $d$를 단어들의 bag으로 보면 다음처럼 근사한다.

$$ P(c|d) \propto P(c)\prod_i P(w_i|c) $$

이것이 Naive Bayes이다. naive하다는 말은 단어들이 클래스가 주어졌을 때 독립이라고 가정하기 때문이다. 실제로는 독립이 아니지만, 이 단순한 가정이 놀랍게 잘 작동한다.

직관은 폴더 비유로 이해하면 쉽다.

```text
스팸 폴더에는 스팸 문서들이 모여 있다.
햄 폴더에는 정상 문서들이 모여 있다.

각 폴더마다 unigram 언어모델을 만든다.
새 문서가 어느 폴더 언어모델에서 더 그럴듯한지 비교한다.
```

Naive Bayes의 장점은 모든 것이 확률이라는 점이다. 그래서 텍스트가 아닌 feature도 곱해 넣기 쉽다.

```text
이메일에 이미지가 있는가?
보낸 사람이 지도교수인가?
제목에 긴급이라는 단어가 있는가?
```

분류 모델을 평가할 때는 accuracy만 보면 안 된다. 특히 클래스 불균형이 있으면 accuracy는 쉽게 속인다. 희귀병 검사에서 모두 정상이라고 답해도 99% accuracy가 나올 수 있다.

그래서 precision, recall, F1, 그리고 micro/macro average를 함께 봐야 한다.

Micro average는 모든 클래스의 TP, FP, FN을 합쳐서 전체적으로 본다. Macro average는 클래스별 성능을 먼저 계산한 뒤 평균낸다. Macro는 소수 클래스의 약점을 드러낸다.

# 8. Perplexity, Backoff, Interpolation: 언어모델을 평가하고 보완하기

언어모델 자체를 평가하는 대표 지표는 perplexity이다.

$$ PP(W) = P(w_1,w_2,\dots,w_N)^{-1/N} $$

log로 쓰면 다음과 같다.

$$ PP(W) = \exp(-\frac{1}{N}\sum_i \log P(w_i|context)) $$

Perplexity는 "다음 단어를 고를 때 평균적으로 몇 갈래에서 헷갈리는가"로 이해할 수 있다.

랜덤하게 0부터 9까지 숫자를 균등하게 뽑는 모델의 perplexity는 10이다. 좋은 언어모델일수록 실제 문장에 대해 perplexity가 낮다.

N-gram에서는 backoff와 interpolation도 중요하다.

Trigram이 있으면 문맥을 더 잘 볼 수 있지만, sparse하다. Bigram은 덜 sparse하고, unigram은 더 안정적이다.

Backoff는 상위 모델에서 못 찾으면 하위 모델로 내려간다.

```text
trigram count가 있으면 trigram
없으면 bigram
없으면 unigram
```

Interpolation은 여러 모델을 섞는다.

$$ P = \lambda_3P_{trigram} + \lambda_2P_{bigram} + \lambda_1P_{unigram} $$

여기서 $\lambda$는 validation set으로 조정한다. 이 관점은 훗날 여러 retriever, 여러 generator, 여러 후보를 섞는 방법으로 계속 확장된다.

# 9. 희소한 세계에서 조밀한 세계로: Word2Vec과 임베딩

Count-based 모델의 한계는 명확하다.

```text
buy와 purchase가 비슷하다는 것을 모른다.
멀리 떨어진 단어 관계를 잘 못 본다.
본 적 없는 조합에는 약하다.
```

그래서 dense embedding이 등장한다.

Sparse representation은 vocabulary 크기만큼 차원이 크고 대부분이 0이다. Dense representation은 수백 차원의 실수 벡터이다. 중요한 차이는 의미가 벡터의 위치에 담긴다는 점이다.

Word2Vec은 distributional semantics를 따른다.

```text
비슷한 문맥에서 등장하는 단어는 비슷한 의미를 가진다.
```

CBOW는 주변 단어로 가운데 단어를 맞힌다.

```text
context → target
```

Skip-gram은 가운데 단어로 주변 단어를 맞힌다.

```text
target → context
```

둘 다 결국 비슷한 문맥에 등장하는 단어들을 가까운 벡터로 만든다.

이렇게 학습된 벡터에서는 유명한 산술이 가능해진다.

```text
king - man + woman ≈ queen
Paris - France + Japan ≈ Tokyo
```

임베딩은 단어 의미뿐 아니라 사회적 bias도 담는다. 어떤 코퍼스에서 "doctor"가 남성과, "nurse"가 여성과 자주 연결되어 있으면 벡터에도 그 편향이 들어간다. 그래서 debiasing이나 retrofitting 같은 후처리가 필요해진다.

---

# 10. Contextual Embedding과 Transformer: 단어는 문맥 속에서 달라진다

Word2Vec은 단어마다 하나의 벡터를 준다. 하지만 같은 단어도 문맥에 따라 의미가 바뀐다.

```text
Broadway play
Yankees won the play
```

둘 다 `play`지만 하나는 연극, 하나는 경기 플레이이다.

ELMo와 BERT는 이 문제를 해결했다. 단어 하나만 넣으면 벡터를 주는 것이 아니라, 문장 전체 문맥을 함께 보고 벡터를 만든다.

BERT는 Transformer를 사용한다. 교수는 Transformer를 soft memory로 설명했다.

일반 데이터베이스는 query와 key가 정확히 맞아야 value를 가져온다.

```text
Query = 황승원의 연봉
Key가 정확히 있으면 Value 반환
없으면 실패
```

Transformer는 다르다. query와 key의 similarity를 계산하고, 여러 value를 가중평균한다.

```text
0.7 × Value1 + 0.2 × Value2 + 0.1 × Value3
```

즉, hard match가 아니라 soft match이다. 이것이 dense retrieval, RAG, ColBERT의 기반으로 이어진다.

---

# 11. 검색의 고전: Sparse Retrieval, TF-IDF, BM25

검색은 NLP에서 오래된 핵심 문제이다. 사용자는 짧은 query를 던지고, 시스템은 관련 문서를 찾아야 한다.

가장 고전적인 표현은 term-document matrix이다.

```text
행: 단어
열: 문서
값: 그 단어가 그 문서에 등장한 횟수
```

이 행렬은 거의 0이다. 그래서 sparse representation이다.

Boolean retrieval은 단어가 있느냐 없느냐만 본다. 하지만 결과가 너무 많거나 아예 없을 수 있다. 그래서 ranked retrieval이 필요하다.

TF는 문서 안에서 단어가 얼마나 자주 나왔는지 본다.

하지만 200번 나온 문서가 50번 나온 문서보다 4배 더 관련 있다고 말할 수는 없다. 그래서 log를 씌운다.

IDF는 단어가 얼마나 희귀한지 본다.

$$ IDF(t) = \log \frac{N}{DF(t)} $$

`the`처럼 모든 문서에 나오는 단어는 변별력이 없다. 반대로 특정 문서에만 나오는 전문 용어는 검색에 중요하다.

TF-IDF는 둘을 곱한다.

```text
문서 안에서는 자주 나오고,
전체 문서에서는 흔하지 않은 단어를 중요하게 본다.
```

문서와 query의 유사도는 cosine similarity로 계산한다. cosine은 벡터의 길이가 아니라 방향을 본다. 같은 문서를 두 번 복사하면 길이는 커지지만 의미는 같다. 그래서 검색에서는 각도가 더 적절하다.

BM25는 TF-IDF의 강력한 진화형이다.

BM25는 두 가지를 보정한다.

```text
TF saturation:
단어가 많이 나와도 점수가 무한히 커지지 않게 한다.

Length normalization:
긴 문서가 단지 길다는 이유로 유리해지지 않게 한다.
```

BM25는 단순해 보이지만 검색 논문에서 빠지지 않는 강력한 baseline이다.

---

# 12. Inverted Index: Sparse 검색이 빠른 이유

검색이 빠른 이유는 모든 문서를 매번 다 읽지 않기 때문이다.

Inverted index는 책 뒤의 색인과 같다.

```text
term → [(doc_id, count), (doc_id, count), ...]
```

query가 들어오면 query term이 등장한 문서들만 후보로 본다. 전체 문서를 훑지 않는다.

이 자료구조는 그래프의 sparse adjacency matrix와도 비슷하다. 웹 전체 PageRank를 계산할 때도 10억 × 10억 행렬을 dense하게 저장하지 않는다. 실제로 연결된 edge만 저장한다.

이 sparse 구조 덕분에 BM25 같은 검색은 GPU 없이도 매우 빠르다. 그래서 BERT 시대에도 sparse retrieval은 죽지 않았다.

---

# 13. Dense Retrieval, Bi-Encoder, Cross-Encoder, ColBERT

Sparse retrieval은 정확한 단어 매치에 강하다. 하지만 `murder`와 `assassination`처럼 의미는 비슷하지만 단어가 다르면 놓친다.

Dense retrieval은 문서를 벡터로 만든다. 의미가 비슷하면 단어가 달라도 가까운 벡터가 된다.

Bi-Encoder는 query와 document를 각각 따로 인코딩한다.

```text
q_vec = Encoder(query)
d_vec = Encoder(document)
score = cosine(q_vec, d_vec)
```

장점은 document vector를 미리 계산해둘 수 있다는 점이다. 검색 시에는 query만 인코딩하면 된다.

Cross-Encoder는 query와 document를 함께 넣는다.

```text
Encoder([query, document])
```

두 텍스트가 서로 attention할 수 있으므로 성능은 좋다. 하지만 모든 문서와 query를 매번 붙여 넣어야 하므로 매우 비싸다.

그래서 실무에서는 multi-phase retrieval을 쓴다.

```text
BM25 또는 Bi-Encoder로 Top 1000 후보
→ Cross-Encoder로 re-ranking
→ Top 10 반환
```

ColBERT는 중간 지점이다. query와 document를 따로 인코딩하지만, 하나의 벡터로 압축하지 않고 token별 vector를 유지한다. query token마다 document token 중 가장 잘 맞는 것을 찾고 합산한다.

```text
Query token별 best document token match
→ max
→ sum
```

이것이 late interaction이다. sparse의 효율과 dense의 soft match를 어느 정도 함께 가져간다.

---

# 14. RAG: LLM의 외부 기억

LLM은 pretraining 시점의 지식을 파라미터에 담고 있다. 하지만 회사 내부 문서, 최신 뉴스, 반도체 기밀, 개인 메일은 모델 파라미터에 없다.

그래서 RAG가 필요하다.

```text
1. 사용자의 질문을 받는다.
2. 검색기로 관련 문서를 찾는다.
3. 찾은 문서를 prompt context에 넣는다.
4. LLM이 그 문서를 근거로 답한다.
```

RAG의 핵심 장점은 hallucination 감소와 citation이다. 답변이 틀릴 수는 있지만, 적어도 사용자는 "어떤 문서를 근거로 답했는지" 확인할 수 있다.

하지만 RAG도 함정이 많다.

```text
Retriever가 관련 문서를 못 찾을 수 있다.
찾아도 generator가 못 쓸 수 있다.
중간에 있는 정보는 lost in the middle로 무시될 수 있다.
마지막에 있는 정보는 recency bias로 과대반영될 수 있다.
```

검색 엔진도 사람용으로 최적화된 것과 LLM용으로 최적화된 것이 다를 수 있다. 사람은 1등 결과를 클릭하지만, LLM은 prompt의 앞과 뒤를 더 잘 보고 중간을 놓칠 수 있다.

그래서 RAG는 단순히 top-K 문서를 붙이는 문제가 아니다. chunking, overlap, re-ranking, summarization, multi-turn search, query augmentation이 모두 중요하다.

---

# 15. RankGPT와 Listwise Re-ranking

검색 결과를 다시 정렬하는 방법도 발전했다.

Pointwise re-ranking은 query-document 한 쌍마다 점수를 매긴다.

Pairwise re-ranking은 두 문서를 비교한다.

Listwise re-ranking은 여러 문서를 한꺼번에 보고 순서를 매긴다.

사람도 발표 하나만 보고 90점이라고 하는 것보다, 여러 발표를 같이 보고 순위를 매기는 것이 더 쉽다. RankGPT는 이 직관을 LLM에 적용한다.

하지만 listwise에는 한계가 있다.

```text
너무 많은 문서를 한 번에 넣을 수 없다.
입력 순서에 따라 결과가 달라진다.
중간 문서를 놓칠 수 있다.
```

그래서 sliding window나 여러 순서로 반복해 noise를 평균내는 방식이 필요하다.

---

# 16. Knowledge Graph: 텍스트를 구조로 바꾸기

검색과 RAG가 문서 chunk를 다룬다면, Knowledge Graph는 텍스트 안의 구조를 꺼낸다.

문장은 단순한 단어 나열이 아니다. 그 안에는 사람, 장소, 조직, 사건, 관계가 있다.

예를 들어 다음 문장을 보자.

```text
승원 is teaching at SNU.
```

여기서 `승원`은 사람이고, `SNU`는 조직 또는 장소이며, `teaching at`은 관계이다.

그래프로 바꾸면 다음과 같다.

```text
[승원] --teaching_at--> [SNU]
```

이것을 일반화하면 relation extraction이다. 텍스트에서 head, relation, tail을 추출한다.

```text
(Head, Relation, Tail)
```

트리플을 많이 모으면 knowledge graph가 된다.

Named Entity Recognition은 이 과정의 시작이다. 먼저 문장에서 entity를 찾아야 한다. 그리고 그 entity가 실제 세상의 무엇을 가리키는지 grounding하거나 wikification해야 한다.

NER에서는 다단어 entity가 중요하다. `Andrew Wilkie`가 한 사람인지, 두 사람인지 구분해야 한다. 이를 위해 IOB encoding을 쓴다.

```text
B-PERSON I-PERSON O B-ORG
```

B는 시작, I는 내부, O는 entity 바깥이다. 이 방식은 NER뿐 아니라 span extraction, 중국어 분절, QA 답 위치 찾기에도 쓰인다.

---

# 17. POS Tagging과 HMM: 숨겨진 품사를 찾기

품사는 문장의 구조를 이해하는 기본 단서다. entity는 대체로 명사구이고, relation은 대체로 동사나 전치사구에서 나온다.

POS tagging은 각 단어에 품사를 붙이는 작업이다.

문제는 같은 단어가 여러 품사로 쓰인다는 것이다.

```text
back:
명사, 동사, 형용사, 부사

lead:
동사 lead / 명사 lead
```

HMM은 이 문제를 확률적으로 푼다.

HMM에는 두 종류의 확률이 있다.

```text
Transition probability:
이전 품사에서 다음 품사로 넘어갈 확률

Emission probability:
어떤 품사일 때 특정 단어가 나올 확률
```

품사는 숨겨져 있고, 단어만 관찰된다. 그래서 Hidden Markov Model이다.

가장 좋은 품사 시퀀스를 찾는 방법은 Viterbi algorithm이다.

$$ V(j,s) = \max_{s'}[V(j-1,s')A(s',s)]B(s,w_j) $$

이 식은 "j번째 위치에서 품사 s로 끝나는 최선의 경로"를 구한다. backpointer를 저장해두면 마지막에 최적 경로를 복원할 수 있다.

Viterbi는 DP이다. Greedy보다 느리지만 optimal을 보장한다. Beam Search는 그 중간이다. 이 decoding 아이디어는 LLM generation에서도 그대로 다시 등장한다.

---

# 18. Relation Extraction: 규칙, 지도학습, 부트스트래핑

Relation extraction은 "얘네 무슨 사이야?"라는 질문이다.

가장 고전적인 방식은 rule-based이다. Hearst Pattern이 대표적이다.

```text
NP such as NP
NP including NP
NP and other NP
```

이 패턴은 is-a 관계를 뽑는다.

```text
Korean universities such as Seoul National University
→ SNU is-a Korean university
```

하지만 세상의 관계는 is-a만 있는 것이 아니다. founder, located_in, works_for, acquired_by 같은 관계가 많다. 그래서 supervised relation extraction이 등장한다.

중요한 feature는 다음과 같다.

```text
두 entity 사이에 있는 단어
entity type
POS
주변 단어
parse relation
```

예를 들어 `Steve Jobs, co-founder of Apple`에서는 `co-founder`가 결정적 단서다.

데이터를 많이 만들기 어렵기 때문에 semi-supervised 방식도 나왔다. Snowball은 seed pattern이나 seed triple에서 시작해 패턴과 triple을 서로 키운다.

```text
Seed pattern → triple 추출
triple → 새 pattern 발견
새 pattern → 더 많은 triple 추출
반복
```

이 방식은 눈덩이처럼 커질 수 있다. 하지만 노이즈가 한 번 들어오면 오염도 눈덩이처럼 커진다. 그래서 verification이 중요하다.

---

# 19. Knowledge Graph Embedding과 Graph RAG

Knowledge Graph는 symbolic하다. 노드 이름이 같으면 같고, 다르면 다르다. 하지만 현실에서는 surface form이 다르다.

```text
Bill Gates
William Henry Gates III
William Gates
```

이를 부드럽게 다루기 위해 knowledge graph embedding이 등장한다.

TransE의 핵심은 간단하다.

$$ head + relation \approx tail $$

예를 들어 다음 관계가 있다고 하자.

```text
(France, capital, Paris)
```

그러면 벡터 공간에서 다음이 성립하도록 학습한다.

```text
France vector + capital vector ≈ Paris vector
```

학습은 positive triple과 negative triple을 비교한다. positive는 실제 KB에 있는 관계이고, negative는 말이 안 되는 관계이다.

Margin loss는 positive가 negative보다 margin만큼 더 좋게 만들려 한다.

Graph RAG는 이 그래프를 RAG에 활용한다.

Vanilla RAG는 chunk를 검색한다. Graph RAG는 entity와 relation을 뽑아 그래프를 만들고, community detection을 한 뒤, 커뮤니티별로 요약하거나 질의한다.

Graph RAG가 특히 강한 질문은 전체 코퍼스나 multi-hop 질문이다.

```text
이 25GB 문서 전체에서 인물들의 공통점은?
팀쿡과 픽사는 어떤 관계로 이어지는가?
어떤 약과 음식이 multi-hop으로 위험하게 연결되는가?
```

한 문단 안에 모든 정보가 없으면 일반 RAG는 약하다. 그래프는 관계를 따라 여러 hop을 이동할 수 있다.

---

# 20. 다국어 NLP: 언어 사이의 별자리 맞추기

다국어 NLP에서 중요한 문제는 언어 간 자원 격차다.

영어는 high-resource이다. 한국어는 medium-resource에 가깝다. 많은 부족 언어는 low-resource이다. 문제는 low-resource 언어일수록 bilingual annotator도 부족하다는 점이다. 그래서 격차가 더 커진다.

Cross-lingual NLP는 강한 영어 모델을 다른 언어에 활용하려는 접근이다.

핵심 아이디어는 embedding alignment이다.

교수는 이것을 별자리 비유로 설명했다.

```text
영어 임베딩 공간과 한국어 임베딩 공간은 서로 다른 하늘이다.
하지만 같은 세상을 보고 만든 별자리라면,
회전하거나 뒤집으면 어느 정도 맞출 수 있다.
```

수학적으로는 source embedding에 행렬을 곱해 target embedding과 맞춘다.

Orthogonal Procrustes는 회전과 대칭만 허용한다.

$$ \min_{\Omega}\|A\Omega - B\|_F \quad \text{subject to } \Omega^T\Omega = I $$

Orthogonal 제약은 벡터 길이, 거리, 각도를 보존한다. 즉, source 언어 내부의 의미 관계를 망가뜨리지 않으면서 target 언어와 맞춘다.

이 문제는 SVD로 닫힌 해를 구할 수 있다. gradient descent가 필요 없다.

Seed dictionary가 조금 있으면 supervised alignment가 가능하다. seed가 없어도 숫자, 고유명사, 글로벌 브랜드 같은 자연 seed가 있어서 unsupervised alignment가 되기도 한다.

어족이 가까울수록 잘 된다. 영어와 스페인어는 잘 맞고, 영어와 핀란드어는 어렵다. 한국어와 일본어는 어순과 문화적 공유가 있어 상대적으로 가능성이 있다.

현대 다국어 LLM은 이 alignment를 임베딩 레이어 안에서 암묵적으로 수행한다.

---

# 21. Multilingual RAG: 질문과 문서의 언어가 달라도 찾기

다국어 RAG는 질문과 문서의 언어가 다를 때 필요하다.

```text
질문: 한국어
문서: 영어
답변: 한국어
```

Sparse BM25는 lexical match 기반이라 한국어 질문으로 영어 문서를 찾기 어렵다. Dense multilingual embedding은 `전통음식`과 `traditional food`를 비슷한 공간에 둘 수 있어 cross-lingual retrieval이 가능하다.

대안은 query translation이다.

```text
한국어 query → 영어로 번역 → 영어 BM25
```

이 방식은 간단하고 강력하지만 번역 품질과 용어 선택에 의존한다.

검색 평가는 Recall@K와 MRR을 쓴다.

Recall@K는 top K 안에 relevant document가 하나라도 있는지 본다. MRR은 가장 먼저 나온 relevant document의 rank를 본다.

$$ MRR = \frac{1}{rank} $$

1등이면 1점, 2등이면 1/2점, 3등이면 1/3점이다.

# 22. Code Generation: 자연어와 코드의 차이

수업은 NLP에서 코드 생성으로 넘어갔다. 이 전환은 자연스럽다. 코드 생성도 결국 번역이다.

```text
자연어 요구사항 → Python 코드
Python 코드 → Rust 코드
입출력 예시 → 함수
스크린샷 → UI 코드
```

하지만 코드와 자연어에는 결정적 차이가 있다.

자연어는 문법이 조금 틀려도 통한다. 코드는 세미콜론 하나, indentation 하나가 틀려도 실행되지 않을 수 있다.

대신 코드는 실행할 수 있다. 이것이 큰 장점이다.

자연어 번역은 reference와 얼마나 비슷한지를 BLEU 같은 지표로 평가했다. 하지만 코드는 reference와 다르게 생겨도 같은 일을 할 수 있다.

```text
for loop
while loop
built-in sum
재귀
```

모두 같은 결과를 낼 수 있다.

그래서 코드 평가는 execution-based metric이 중요하다. 테스트를 돌려서 통과하면 맞은 것이다.

Pass@K도 이 맥락에서 나온다. 모델이 K개의 코드를 생성했을 때 하나라도 테스트를 통과하면 성공으로 본다.

$$ Pass@K = 1 - \frac{\binom{n-c}{K}}{\binom{n}{K}} $$

여기서 $n$은 생성한 전체 후보 수, $c$는 맞은 후보 수이다.

Pass@K는 "한 번에 맞히는 능력"과 "여러 번 시도하면 맞는 후보를 하나쯤 내는 능력"을 구분한다.

# 23. Verifier, Iterative Refinement, SWE-bench

코드 생성의 강점은 verifier가 있다는 것이다.

```text
코드 생성
→ 테스트 실행
→ 실패 로그 확인
→ 수정
→ 다시 테스트
```

이것은 사람이 개발하는 방식과 닮았다. 한 번에 완벽한 코드를 쓰는 사람은 드물다. 실행하고, 에러를 보고, 고친다.

Iterative refinement는 blind retry보다 효율적이다. 처음부터 다시 생성하는 대신, 이전 코드와 에러 메시지를 보고 수정한다.

하지만 한계도 있다.

```text
context가 계속 길어진다.
A를 고치다 B를 망칠 수 있다.
3~4번 이후에는 개선이 saturate된다.
```

SWE-bench는 이 흐름을 실제 개발자 업무에 가깝게 만든다. 단일 함수가 아니라 repository 전체에서 issue를 읽고, 관련 파일을 찾고, 수정하고, 테스트를 통과해야 한다.

이때도 retrieval이 중요하다. 모든 파일을 LLM context에 넣을 수 없기 때문에 issue와 관련된 파일을 찾아야 한다.

# 24. NL to SQL: 자연어를 데이터베이스 질의로 바꾸기

SQL 생성은 코드 생성의 특수한 형태이다. 사용자는 자연어로 묻는다.

```text
재즈 장르에서 한 곡 이상 가진 아티스트 이름을 알려줘.
```

모델은 database schema를 보고 SQL을 만들어야 한다.

문제는 DB schema가 자연스럽지 않다는 점이다. 컬럼명은 만든 사람 편의에 따라 축약되거나 도메인 용어로 되어 있다.

그래서 semantic layer가 필요하다.

```text
테이블 설명
컬럼 설명
약어 설명
값의 의미
```

SQL 평가는 string match가 아니라 execution accuracy가 중요하다. 같은 결과를 내는 SQL은 여러 개일 수 있다.

```text
MAX 사용
ORDER BY DESC LIMIT 1
ORDER BY ASC 후 마지막 선택
```

표현은 달라도 결과가 같으면 맞는 것으로 본다.

Chase-SQL은 이 문제를 워크플로우로 푼다.

```text
Value Retriever
→ 여러 Generator
→ Query Fixer
→ Pairwise Selection
```

Generator도 하나가 아니라 여러 방식으로 만든다.

```text
DC-CoT:
문제를 작은 sub-question으로 나눈다.

QP-CoT:
어떤 테이블, 어떤 operation, 어떤 결과인지 계획한다.

OS:
현재 schema로 few-shot 예시를 직접 만든다.
```

이것은 "LLM 하나에게 한 방에 맡기지 말고, 문제를 좋은 단위로 쪼개 조립하라"는 수업 전체의 메시지와 이어진다.

# 25. Open vs Closed LLM과 Post-training

모델은 공개 정도에 따라 나뉜다.

```text
Closed:
API로만 사용. 내부는 블랙박스.

Open Weight:
weight는 공개하지만 데이터와 학습 과정은 비공개.

Open Source:
데이터, 코드, 레시피까지 공개.
```

교수는 이를 밀키트 비유로 설명했다.

Open source는 재료, 소스 만드는 법, 조리법을 모두 주는 것이다. Open weight는 완성된 소스를 주는 밀키트다. 양념을 조금 더할 수는 있지만, 알레르기 있는 재료가 들어갔는지 알기 어렵다.

Pretraining만으로는 부족하다. 모델을 실제로 쓸 수 있게 만드는 핵심은 post-training이다.

Post-training에는 여러 단계가 있다.

```text
Continual pretraining:
도메인 문서를 더 넣어 next-token prediction을 계속한다.

Instruction tuning / SFT:
사용자 instruction과 정답 output 쌍을 학습한다.

Alignment:
도움이 되고 안전한 답을 하도록 RLHF, DPO 등을 적용한다.
```

Continual learning에서는 catastrophic forgetting이 문제다. 새 지식을 가르치면 예전 능력을 잊을 수 있다.

이를 막기 위해 parameter-efficient tuning이 등장한다.

Adapter는 본체는 그대로 두고 작은 어댑터를 끼우는 방식이다.

LoRA는 low-rank update를 학습하는 방식이다.

전체 모델을 다 바꾸지 않으므로 비용이 줄고, 기존 지식을 덜 망가뜨린다.

# 26. Instruction Tuning, Auto-IF, Persona, Alignment

Instruction tuning은 모델에게 "시키는 말"을 이해시키는 과정이다.

기존 NLP 데이터셋도 instruction 형식으로 바꿀 수 있다.

```text
Instruction: 다음 문장을 POS tagging 하시오.
Input: I am a student.
Output: PRON VERB DET NOUN
```

사람이 직접 만들면 비싸다. 그래서 LLM as teacher가 등장한다. 큰 모델이 instruction-response pair를 만들고, 필터링한 뒤 작은 모델을 학습시킨다. 이것이 distillation이다.

Auto-IF는 기계적으로 검증 가능한 instruction을 대량 생성한다.

```text
3문장으로 답하라.
JSON 형식으로 답하라.
마지막 단어가 autumn이어야 한다.
특정 단어를 포함하지 말라.
```

Python 한 줄로 verifier를 만들 수 있는 constraint는 무제한 데이터 생성에 적합하다.

Persona도 constraint의 일종이다.

```text
너는 5살 아이를 가르치는 수학 튜터야.
너는 임원에게 보고하는 컨설턴트야.
너는 Mr. Robot의 Elliot Anderson 톤으로 말해.
```

Alignment는 모델이 위험한 답을 하지 않도록 만드는 과정이다. pretraining은 "다음 토큰"을 잘 맞히게 할 뿐이다. 위험한 정보를 알고 있으면 그대로 말할 수 있다. 그래서 helpfulness와 safety를 별도 목표로 둔다.

RLHF는 사람의 선호를 reward model로 학습하고, 그 reward를 높이는 방향으로 모델을 조정한다. RLAIF는 사람 대신 LLM judge를 쓰는 방향이다.

# 27. Tools와 Skills: 에이전트를 만드는 단위

에이전트 수업에서는 tool과 skill을 구분했다.

Tool은 외부 기능을 단발적으로 호출하는 것이다.

```text
search_web
internal_doc_search
execute_sql
```

Skill은 재사용 가능한 절차이다. 여러 tool을 내부적으로 쓸 수 있고, 언제 실행할지 trigger와 output schema를 가진다.

```text
미팅 노트에서 action item 추출
이메일 요약 후 답장 초안 작성
사내 정책 문서 검색 후 근거 포함 답변
```

Skill에는 다음 요소가 필요하다.

```text
Trigger:
언제 이 skill을 쓸지

Instruction:
무엇을 해야 하는지

Avoid:
무엇을 피해야 하는지

Output schema:
어떤 형식으로 결과를 내야 하는지
```

에이전트에서 중요한 것은 한 번 생성하고 끝내지 않는 것이다. validation loop가 필요하다. 모델이 지어내지 말라고 해도 지어낼 수 있으므로, 한 번 더 원문과 비교해 충실한지 확인해야 한다.

# 28. DeepSeek, MoE, 그리고 RL로 Reasoning을 가르치기

DeepSeek R1은 수업 후반의 중요한 전환점이었다. 핵심은 "SFT 없이도 RL만으로 reasoning이 가능한가?"라는 질문이다.

R1-Zero는 출시 모델이라기보다 thought experiment에 가깝다. Supervised reasoning data 없이도 reward만으로 reasoning behavior가 생길 수 있는지를 보려는 시도였다.

DeepSeek-V3는 큰 base model이고, MoE를 사용한다.

MoE는 Mixture of Experts이다. 모든 토큰이 모든 파라미터를 쓰는 대신, router가 일부 expert만 선택한다.

```text
Router
→ Top-1 또는 Top-2 expert 선택
→ 선택된 expert만 계산
```

뇌의 영역처럼 수학, 언어, 코드, 논리 기능을 일부 분리하는 비유로 이해할 수 있다.

RL은 강아지 훈련 비유로 설명됐다.

```text
좋은 행동 → 간식
나쁜 행동 → 벌
```

LLM도 policy로 볼 수 있다.

```text
State:
지금까지 생성한 토큰들

Action:
다음 토큰 또는 tool call

Reward:
정답 여부, 테스트 통과, 사람 선호, 포맷 만족
```

문제는 reward가 늦게 온다는 점이다. 코드 생성은 마지막에 테스트를 돌려야 reward가 나온다. 번역도 문장 전체를 만든 뒤 BLEU가 나온다. 이것이 sparse reward와 long-horizon 문제다.

Credit assignment는 "마지막 결과가 좋거나 나빴을 때, 중간의 어떤 선택 때문이었는가"를 판단하는 문제이다. 이것은 어렵다.

# 29. PPO, GRPO, DPO

PPO는 LLM alignment에서 많이 쓰인 방법이다. 세 가지 모델이 등장한다.

```text
Reference model:
원래 모델. 너무 멀리 벗어나지 않게 기준 역할.

Reward model:
답변이 얼마나 좋은지 점수화.

Value model:
현재 상태에서 앞으로 받을 reward를 예측.
```

Reference KL constraint는 모델이 reward만 보고 지나치게 변하지 않도록 막는다.

GRPO는 PPO에서 value model을 제거한다. 여러 샘플을 만들고, 그 평균 reward를 기준으로 advantage를 잡는다.

```text
평균보다 잘했으면 positive advantage
평균보다 못했으면 negative advantage
```

Value model을 따로 학습하는 것이 어렵고 방해될 수 있으므로, 이를 없앤 것이 핵심이다.

DPO는 preference pair를 직접 활용한다.

점수 모델을 만들지 않고도 "A가 B보다 좋다"는 비교 데이터로 policy를 조정한다.

옷 고르기 비유처럼, 절대 점수를 매기기는 어렵지만 두 개 중 무엇이 더 나은지는 비교적 쉽다.

# 30. Reward Shaping과 Reward Hacking

Sparse reward는 학습하기 어렵다. 그래서 중간 reward를 주고 싶어진다. 이것이 reward shaping이다.

자동차 시뮬레이터에서 도로 중앙을 잘 달리면 +0.1, 바깥으로 나가면 -0.1을 줄 수 있다.

하지만 shaped reward를 잘못 주면 reward hacking이 생긴다.

```text
책상에 3시간 앉아 있으면 보상
→ 공부하지 않고 앉아만 있음
→ 시험 점수는 나빠짐
```

즉, 중간 보상이 진짜 목표를 바꿔버릴 수 있다.

Potential-based reward shaping은 이 문제를 피하려는 이론적 방법이다.

$$ F(s,a,s') = \gamma\Phi(s') - \Phi(s) $$

state의 potential 차이를 reward로 주면 optimal policy를 보존할 수 있다. 중간 신호를 주되 최종 목표는 바꾸지 않는다는 아이디어다.

하지만 실제 NLP에서는 prefix BLEU shaping이 다양성을 억제하는 등 부작용도 있었다. 그래서 continuation 기반 평가나 MCTS 같은 inference-time search로 이어진다.

# 31. MCTS와 Alpha-SQL

MCTS는 AlphaGo의 핵심 아이디어다.

가능한 모든 바둑 수를 끝까지 다 보는 것은 불가능하다. 대신 트리를 조금씩 확장하면서 유망한 경로를 더 많이 탐색한다.

MCTS의 4단계는 다음이다.

```text
Select:
어떤 자식 노드로 갈지 선택한다.

Expand:
새 노드를 만든다.

Simulate:
끝까지 또는 일정 깊이까지 진행해 reward를 본다.

Backprop:
reward를 부모 노드들로 전파한다.
```

선택에는 UCB1을 쓴다.

$$ UCB1 = \bar{R}_n + c\sqrt{\frac{\ln N_{parent}}{N_n}} $$

앞 항은 exploitation이다. 지금까지 잘했던 선택을 더 한다. 뒤 항은 exploration이다. 덜 가본 선택지도 시도한다.

Alpha-SQL은 이 MCTS를 SQL 생성에 적용한다.

```text
Node:
partial reasoning step 또는 partial SQL

Edge:
다음 SQL 변형 action

Reward:
완성된 SQL의 consistency나 실행 결과
```

여기서도 중요한 메시지는 같다. LLM을 마법처럼 한 번 호출하고 끝내는 것이 아니라, 선택지를 만들고, 평가하고, 다시 탐색하는 구조를 설계해야 한다.

# 32. 7일 수업을 하나로 잇는 결론

이 수업은 겉으로 보면 주제가 많다.

```text
정규표현식
Edit Distance
BPE
N-gram
Naive Bayes

Knowledge Graph

```

하지만 하나의 질문으로 묶을 수 있다.

```text
언어라는 애매하고 다양한 표면을,
컴퓨터가 계산 가능한 구조와 확률로 바꾸고,
그 구조를 이용해 더 잘 검색하고,
더 잘 생성하고,
더 잘 행동하게 하려면 어떻게 해야 하는가?
```

처음에는 정규표현식으로 문자열을 찾았다. 그 다음 edit distance로 "얼마나 비슷한지"를 계산했다. BPE로 언어를 subword 단위로 쪼갰다. N-gram과 Naive Bayes로 확률을 계산했다. Word2Vec과 BERT로 의미를 벡터에 담았다. TF-IDF와 BM25로 검색했고, dense retrieval과 ColBERT로 의미 검색을 더했다. RAG로 LLM에 외부 기억을 붙였다. Knowledge Graph로 문서 속 관계를 구조화했다. 다국어 임베딩으로 언어 사이를 맞췄다. 코드 생성에서는 실행 가능한 verifier를 얻었다. SQL에서는 reasoning을 workflow로 나누었다. 마지막에는 RL과 MCTS로 LLM의 행동을 reward와 search로 개선하는 단계까지 왔다.

결국 이 수업은 "LLM을 잘 쓰는 법"만 가르친 것이 아니다. LLM 이전의 NLP가 어떤 문제를 붙잡고 씨름했는지, 그리고 그 문제들이 지금도 RAG, 에이전트, 코드 생성, post-training, reasoning에서 어떻게 되살아나는지를 보여준 수업이다.

# 33. 시험과 복습을 위한 핵심 문장

정규표현식은 surface form을 다루는 가장 단순한 규칙 기반 방법이지만 false positive와 false negative trade-off가 있다.

F1은 precision과 recall의 조화평균이며, 한쪽만 높은 모델을 강하게 벌준다.

Edit distance는 한 문자열을 다른 문자열로 바꾸는 최소 편집 비용이며, dynamic programming으로 효율적으로 계산한다.

BPE는 데이터에서 자주 함께 등장하는 symbol pair를 반복적으로 merge하는 subword tokenizer이다.

언어모델은 텍스트에 확률을 부여하는 함수이며, N-gram은 Markov assumption으로 긴 문맥을 짧게 근사한다.

Smoothing은 관측되지 않은 사건에 0이 아닌 확률을 주어 overfitting을 줄인다.

Noisy channel spell correction은 $P(w|x)$를 Bayes rule로 $P(x|w)P(w)$로 바꿔 푼다.

Naive Bayes는 클래스가 주어졌을 때 단어들이 독립이라고 가정하고 문서 분류를 수행한다.

Perplexity는 언어모델이 다음 단어를 얼마나 헷갈리는지 나타내며 낮을수록 좋다.

Word2Vec은 distributional semantics, 즉 비슷한 문맥에 등장하는 단어는 비슷한 의미라는 가정으로 embedding을 학습한다.

Contextual embedding은 같은 단어라도 문맥에 따라 다른 벡터를 만든다.

TF-IDF는 문서 안에서는 자주 나오고 전체 문서에서는 드문 단어를 중요하게 본다.

BM25는 TF saturation과 length normalization으로 TF-IDF를 개선한 강력한 sparse retrieval baseline이다.

Bi-Encoder는 빠르고, Cross-Encoder는 정확하며, ColBERT는 token-level late interaction으로 둘 사이 균형을 잡는다.

RAG는 외부 문서를 검색해 LLM prompt에 넣어 hallucination을 줄이고 citation을 가능하게 한다.

Knowledge Graph는 entity와 relation을 triple로 구조화한 그래프이다.

TransE는 $head + relation \approx tail$이 되도록 entity와 relation을 embedding한다.

Orthogonal Procrustes는 source embedding을 회전/대칭 변환으로 target embedding에 맞추며, 내부 거리와 각도를 보존한다.

코드 생성 평가는 BLEU보다 execution-based metric이 중요하며, Pass@K는 여러 후보 중 하나라도 통과할 확률을 본다.

Instruction tuning은 inference 형식의 input-output pair를 학습시키는 post-training 단계이다.

Adapter와 LoRA는 전체 모델을 바꾸지 않고 일부 파라미터만 학습하는 PEFT 방법이다.

Tool은 단발 호출이고, Skill은 재사용 가능한 절차와 schema를 가진다.

LLM은 policy로 볼 수 있으며, 다음 토큰이나 tool call은 action, 정답이나 테스트 통과는 reward가 될 수 있다.

PPO는 reference, reward, value model을 사용하고, GRPO는 value model을 제거해 여러 샘플 평균으로 advantage를 잡는다.

DPO는 절대 reward 대신 preference pair로 policy를 직접 조정한다.

Reward shaping은 중간 신호를 주지만 잘못 설계하면 reward hacking을 만들 수 있다.

MCTS는 select, expand, simulate, backprop 단계를 통해 exploration과 exploitation을 균형 있게 수행한다.
