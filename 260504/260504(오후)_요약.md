## 1. ⭐ LLM Output 평가 — Conventional vs LLM-as-Judge

### 1.1 Task의 두 가지 구분
- **명확한 답 (Clear Answer)**:
  - 수학 문제 → Final Answer parsing (정규표현식, split).
  - 코드 문제 → **Unit Test 통과율 (Pass@K)** — 예: HumanEval (2023).
- **Open-ended Answer**:
  - 시 작성, Application Form 채우기 등.
  - **BLEU / ROUGE Score**: 답변과 비교.
  - **BERTScore**: Semantic Similarity.
  - **Human Evaluation**: 자연스러움 / 정보성 1~5점 (Amazon MTurk, 설문).

### 1.2 Conventional 평가의 한계
- **Parsing Issue**: LLM이 원하는 포맷으로 출력 안 함 → Error Case 多.
- **Unit Issue**: 1km vs 1000m, 같은 답이지만 코드로 비교 어려움.
- **Open-ended는 의미 비교 불가능** + Gold Answer 자체가 마음에 안 들 때 多.
- **Human Evaluation은 비싸고 느림**.

### 1.3 ⭐ LLM-as-Judge (LLMS Approach)
- 컨셉: "요새 LLM 말 잘 알아먹는데 그냥 얘로 평가하자".
- 시초 — **G-Eval (2023, GPT-Eval)**:
  - Task Instruction + Evaluation Criteria (Coherence 등) + Evaluation Step.
  - Auto-CoT로 1~5점 부여.
- ⭐ 반발 → 정당화 논리: **"평가 Task ≠ 생성 Task"**.
  - LLM은 Task별 성능 차이 多 — GPT-4가 생성은 못해도 평가는 잘할 수 있음.
  - "기준 두고 평가"가 "직접 답 적기"보다 쉬움.

### 1.4 ⭐ Pointwise vs Pairwise Evaluation
- **Pointwise**: 특정 Criteria에 대해 1~5점 스케일.
- **Pairwise**: Model A, B 두 개 답변 중 어느 게 더 나은가.
- ⭐ **Pairwise가 더 컨시스턴트하다** (평균적으로) — 더 알려져 있음.
- 그러나 요새는 Pointwise 총합도 많이 활용.

### 1.5 ⭐ LLM Judge의 Bias 3가지
| Bias | 설명 |
|------|------|
| **Position Bias** | A/B 위치에 따라 점수 영향. ICL을 A 쪽에 몰아넣으면 마지막도 A 선택 가능성 ↑ (테크닉으로 활용되기도) |
| **Verbosity (Flumosity) Bias** | 길게 답변하는 쪽 점수 ↑ |
| **Self-Enhancement Bias** | 같은 모델 family 답변 선호. 라마가 라마 평가하면 토큰 확률 ↑로 인한 편향. 거의 어쩔 수 없음 |
- ⭐ Position / Verbosity는 거의 해결, **Self-Enhancement는 여전히 문제** → **다른 모델 family로 평가하는 것이 좋음**.

### 1.6 실습 (LLMEAG 노트북, Grok API)
- **Task 1.1**: 수학 답변 Parsing 시나리오 → Judge Prompt 설계 → 인간 라벨과 정확도 비교 (~95%).
- **Task 3.2**: Open-ended (CoT 영향 설명) → Pointwise judge, criteria=correctness, 1~5점.
- ⭐ **반드시 사람과의 일치율(%) / Correlation 보고**해야 함 — 논문/실무 모두 표준.

## 2. ⭐ AI Agent 컨셉

### 2.1 RL 용어로서의 Agent
- 원래 RL 용어: **Environment - Sensors(Percept) - Actuators(Action)** 전체를 Agent로 통칭.
- **AI Agent / LLM Agent** = 그 판단을 LLM으로 수행.
- RL = **Decision Making의 과학** (CS, AI, 심리학, 경제학, neuroscience 모두 적용).
- 흐름: Observation → Action → Reward → 반복.

### 2.2 ⭐ LLM Agent의 두 가지 관점
- **LLM First View**: 챗봇에서 출발하여 Agent로 확장 (예: RAG Agent, Counselor Agent).
- **Agent First View**: 원래부터 AI Agent 영역에서 LLM을 도입 (강의자 랩 — Inbody Agent, Computer Agent, Web Agent, Code Agent).

### 2.3 ⭐ LLM Agent Key Concept 3가지
| 컨셉 | 의미 |
|------|------|
| **Reasoning** | Language를 통한 추론 능력 |
| **Memory** | 과거 사용 기록 / 환경 액션 히스토리 / 사용자 personalization |
| **Planning** | Task 수행 위해 부분으로 잘라서 가져오는 능력 (예: 보드마카 가져오기 = 다가가기 → 집기 → 돌아오기 → 놓기) |
- ⭐ **Planning은 Agent에서 처음 등장하는 개념**. Reasoning은 단일 turn 논리, Planning은 multi-step task 분해.

## 3. ⭐⭐ Workflow vs Agent — 시험 핵심 ⭐

> 강사 직접 명시: **"오늘 시험 문제를 방금 확인했는데, 시험 범위에서 나올 만한 시험 문제는 이거고요. 그 외에는 없습니다."**

### 3.1 정의
| 구분 | 정의 | 특징 |
|------|------|------|
| **Workflow** | **Predefined code path** — 사람의 개입으로 흐름이 잡힌 시스템 | Predictability ↑ / Consistency ↑ |
| **Agent** | 스스로 own process를 다이나믹하게 결정 — Tool/API 키만 주고 autonomous하게 넘김 | 유연성 ↑ |

### 3.2 ⭐ 핵심 차이 정리 (시험 답안용)
- **Workflow**: 어떤 식으로 task가 진행되는지 강제성 있음 (예: "먼저 코드 체크 → 다음 단계").
- **Agent**: 강제성 X. 줄 수 있는 도구만 주고 알아서 처리.

### 3.3 ⭐ 실용 가이드
- **대부분의 애플리케이션 / 사업은 Workflow로 충분 + 더 잘 됨**.
- Agent는 좀 더 미래지향적 / 연구 영역 / Generalist Agent 관점 (OpenAI 등).

## 4. ⭐ Agent Building Block — Augmented LLM

### 4.1 컨셉
- Transformer가 attention + FFN block을 쌓듯이, **Agent도 Building Block 존재**.
- **Augmented LLM**: LLM이 Retriever + Tool + Memory와 결합된 형태.
  - Tool calling, Memory read/write, Query Retrieval 모두 가능.

### 4.2 Tool의 종류 (Environment에 따라 다름)
- **Inbody Agent (강의자 연구)**: navigate / pick / place / open / close / explore (시뮬레이터에서 골드로 동작).
- **Web Agent (강의자 연구)**: fill(234, "Sony WH-1000XM4") / click(번호) — HTML 박스에 ID 부여.
- **Program-based**: code interpreter, database 등.
- ⭐ "어떤 Tool을 주느냐가 Agent의 성격을 결정".

## 5. ⭐ Workflow 5가지 패턴 (Anthropic 분류)

### 5.1 Prompt Chaining
- 어려운 Task → LLM 콜 여러 번, **Gate**로 충분한 답인지 확인 → 다음 단계.
- 예: 문서 outline 작성 → 체크리스트 4개 → 하나씩 작성 / 확인.

### 5.2 Routing
- Customer Service 비유 — 중앙 한 명이 분류 → 마케팅 / 법무 / 일반에 라우팅.
- 각각의 Specialized Prompt LLM에 전달.

### 5.3 Parallelization
- 같은 Task 여러 번 → Aggregator로 모음.
- 대표: **Majority Vote** (= Self-Consistency), 다른 관점의 토론.

### 5.4 Orchestrator-Workers
- 매니저(Orchestrator) + 하위 Workers.
- ⭐ 클로드 코드의 멀티 에이전트 구조와 유사. 현재 실용적으로 가장 많이 활용.

### 5.5 Evaluator-Optimizer
- Solution을 평가하는 애 + Rejected Feedback Loop.
- 한 콜 안에서 Evaluator-Optimizer가 안쪽에서 돌 수도 있음.

⭐ **5개가 결합된 형태도 가능** — 어떤 Agent 연구든 결국 이 5가지로 귀결됨.

## 6. ⭐ Autonomous Agent — Environment Feedback의 중요성

### 6.1 핵심 특징
- 복잡한 인풋 이해 + Reasoning + Planning + ⭐ **Environment Interaction**.
- ⭐ **Feedback (Ground Truth)을 환경에서 매 step 받아야 함**:
  - 틀렸다 / 맞았다 / 에러 발생 여부.
  - 제대로 못 받으면 헛돌게 됨.
- 강의자 강조: **"Agent 설계의 시작은 Environment에서 어떤 Feedback을 받을 것인가"** — 가장 큰 성능 영향 요소.

### 6.2 ⭐ Context Engineering vs Harness Engineering
- 예전 가정: "LLM의 Reasoning / Planning 능력이 부족하다" (능력 의심).
- ⭐ **현재 가정**: "정보가 충분하다는 전제 하에 LLM은 풀 수 있다 — 우리가 정보를 충분히 주지 못하는 것 뿐".
- **Context Engineering**: Task 수행에 필요한 정보를 명확하게 입력으로 줌. Prompt에 들어가는 정보를 깔끔히 다룸.
- **Prompt Engineering과의 차이**: Agent는 Environment에서 정보가 너무 많이 들어와서 **단순 prompt 변경 이상의 엔지니어링이 필요**.
- **Harness Engineering**: Environment에서 무엇을 받느냐까지 건드리는 컨셉.

## 7. ⭐ ReAct — Agent의 기본 Prompt Format

### 7.1 LLM Agent Basic Framework
- **Controller** (LLM 머리) / **Toolset** / **Environment** / **Perceiver**.

### 7.2 ⭐ ReAct (2022~2023)
- **Triplet**: **Thought → Action → Observation** 무한 반복.
- **Thought**: CoT처럼 추론 먼저.
- **Action**: 주어진 Tool Set에서 실행 (예: Search[Apple Remote]).
- **Observation**: Action의 환경 출력 받기.

### 7.3 작동 원리
1. Thought + Action까지 LLM이 1번 생성 → 멈춤.
2. Action을 환경에서 실제 실행 → Observation 받음.
3. Observation을 prompt에 augment → 다음 iteration.
4. Task 끝날 때까지 반복.

### 7.4 Long Horizon
- 각 state에서 action으로 다음 state로 넘어감 → state sequence 길어짐 → 시간 多.
- 이 sequential 길이를 **Long Horizon**이라 함.

### 7.5 실습 (Day 5 — Agent and Tools)
- **PoT 복습**: Python 코드로 수학 문제 풀이 → 코드 인터프리터에서 실행.
- **ReAct + Calculator + Python Interpreter**:
  - Available Tools 명시 → ReAct 포맷 정의 → parse_action 함수로 Action 파싱 → 툴 실행 → Observation 반환.
  - LLaMA 3.1 8B로 실행 시 처음엔 포맷 어겨 에러 → 반복 후 자동 교정 (action input ** 포맷).
  - ⭐ **Environment Feedback이 살아있으면 모델이 스스로 수정**.

## 8. ⭐ Tool Usage — 학습 패러다임 발전사

### 8.1 발전의 일반 패턴
1. **사람 골드 데이터로 능력 증명**.
2. **Synthetic Data Augmentation 연구**.
3. **대용량 확장 연구**.
- ⭐ AI 연구의 일반적 사고 흐름 — 어떤 능력이든 동일.

### 8.2 ⭐ Imitation Learning (시초)
- 로봇 분야에서 시작 — 사람 손을 따라하게 해서 강화학습.
- Tool Usage에도 적용: 사람이 Tool 사용 방식을 annotate → 모델이 Fine-tuning으로 imitate.

### 8.3 ⭐ WebGPT (OpenAI)
- 인터페이스 두고 사람이 어떻게 사용하는지 6,000개 annotation.
- SFT + RL로 학습 → 좋은 성능.

### 8.4 ⭐ Toolformer (Meta) — Self-Supervised
- **Self-supervised manner** 데이터 augmentation.
- Text completion 중 API call (QA, Wikipedia search, Calculator, Calendar) 삽입.
- 절차: API 콜 미친 듯 샘플링 → execution → 말 안 되는 것 필터링 → 모델 출력 형태에 합성으로 붙여넣기.

### 8.5 ⭐ ToolLLM (2023, 중국) — 대용량 확장
- **16k+ API**로 데이터셋 대규모화.
- 절차: API 컬렉션 → 두 개씩 묶어 Instruction 생성 → API로 풀이 → 필터링.
- SFT로 Tool Calling 가능 모델 산출.

## 9. ⭐ MCP (Model Context Protocol)

### 9.1 등장 배경
- 회사마다 다른 Tool Calling 방식 (OpenAI tool_calls / Anthropic tool_use / 모델별 토큰 삽입).
- ⭐ Anthropic이 통일하자 — **Type-C 비유** (마이크로C, 애플, 타입C → 타입C로 통일).
- "어떤 학문에 표준이 정해지는 순간을 실시간으로 본 흥미로운 사건".

### 9.2 ⭐ MCP 아키텍처 — 3요소
| 요소 | 역할 |
|------|------|
| **Host** | AI 애플리케이션 자체 (Cursor, Claude Code) |
| **Server** | 회사가 자기 API/Tool 올림 |
| **Client** | Host 안에서 각 Server와 **1:1 매칭**된 커넥션 |
- 하나의 Host 안에 여러 Client → 각각 Server (파일시스템 / DB / Slack / Google Calendar / Drive)와 연결.

### 9.3 두 가지 레이어
- **Protocol Layer (Inner)**: JSON-RPC 2.0 기반, 데이터 자체 관리.
- **Transport Layer (Outer)**: 실제 연결, 메시지 송수신, Authentication.
- 흐름: Transport에서 Initialize → Data Layer로 데이터 송수신 → 다시 Transport로.

### 9.4 ⭐ 통합 전 vs 후
```
[통합 전]
OpenAI: tool_calls / type:function / function:{name, augment}
Anthropic: tool_use / function name / input
모델별: 채팅 템플릿 토큰 안에 직접 삽입
[통합 후 — MCP]
JSON-RPC 2.0:
  id: ...
  method: tool_call
  params: {name, arguments...}
```

### 9.5 ⭐ MCP의 의의 — 5가지 장점
- 가장 중요: **표준 확보**.
- 다양한 모델에서 같은 Tool **Reuse**.
- 회사들이 API를 MCP 서버로 올림 → Agent / Service 확장 가속.
- ⭐ OpenAI도 결국 Anthropic의 MCP 수용.

### 9.6 실습 (FastMCP)
- 별도 터미널에서 MCP 서버 동작 (Python `fastmcp`).
- Tool 예시: Open-Meteo (무료, API 키 X) — 도시별 기상 정보.
- ReAct Agent (LangGraph) + MCP Tools → 동시 multi-tool 호출 가능 (서울/도쿄/파리 비교).
- ⭐ 단계 흐름: Tool calling 직접 prompt 설계 (수동) → LangChain `bind_tools` (자동) → MCP (표준화).

### 9.7 Tool 개수에 대한 강사 경험
- ⭐ **10개 이내 권장** — 그 이상은 노이즈로 성능 하락.
- 대안: Retriever로 Tool 검색 후 그때그때 컨텍스트에 삽입 (연구 주제).

## 10. ⭐ Q&A — 강사 인사이트

### 10.1 Skills vs MCP
- Skills 등장으로 MCP 중요도 떨어졌다 ❌
- Skills = Tool 활용법 / MCP = 표준 프로토콜 → **별개 차원**.
- Skills 내부도 MCP로 연결될 수 있음.

### 10.2 Cursor / Claude Code / Codex 비교
- **Cursor**: IDE 기반 (VSCode-like), 빠른 연구 코드용. 강의자 선호.
- **Claude Code**: CLI, 검증 / 유닛테스트 빡센 서비스용.
- **Codex**: 또 다른 옵션.
- ⭐ "본인 개발 스타일에 따라 다름" — 연구는 Cursor, 서비스는 Claude Code.

### 10.3 ⭐ LLM에 연구 주제 던지기
- "기가 막힌 주제" 받기 어려움 — 컨텍스트 충분히 주면 사람도 생각할 수준.
- ⭐ **놓치는 게 없는가, 사고 프로세스 다듬기**에 활용.

### 10.4 ⭐ 로봇의 한계
- 로코모션(걷기), 단순 물건 집기 → OK.
- 복잡한 명령 ("저기 컵 가져와서 옆에 놓아줘") → ❌ — 정해진 코스만 가능.
- 콘센트 꽂기조차 힘 조절 어려움 → 부숨.
- LLM 도입이 더딘 이유: Policy 출력 속도 (1초에 100~1000번) — LLM 너무 느림.

### 10.5 ⭐ VLA (Vision-Language-Action) 모델
- LLM처럼 Vision + Language + Action 통합 모델.
- **서부 (실리콘밸리)**: VLA 잘 될 거다.
- **동부 (MIT, Boston Dynamics)**: VLA는 돈 빨아먹는 fraud다.
- 강의자 의견: **VLA는 속도 느려서 다른 아키텍처로 진화해야 할 것**.

### 10.6 네이버 Clova 비화
- 지식인 데이터 多 → 좋을 거라 기대 → 데이터 퀄리티 개판 → 성능 좋지 않음.
- ⭐ **"LLM은 결국 데이터 퀄리티 싸움"**.

### 10.7 학계의 한숨
- 데이터 잘 만들어 학습 = 너무 뻔하니 리뷰어가 싫어함 → Fancy Method 강요.
- 그러나 Fancy Method가 실제 성능 영향 적음.
- 진짜 잘하는 사람들은 다 창업 → 학계 리뷰어 퀄리티 ↓.

### 10.8 ⭐ Agent Safety / Security
- 최근 Anthropic 논문 — 에이전트 20일 동안 그대로 두니 **개판 (다 공개, 다 접근, 민감 정보 무시)**.
- 원인: Task Completion만을 위해 학습 → 민감 정보 / 위험 경로 지각능력 ↓.
- 현실적 해결: **개발 단으로 경로 차단 / Access 막기** (모델 신뢰 X).

### 10.9 ⭐ World Model 실용 관점
- World Model의 본질 = **도메인에 대한 이해도**.
- "이 버튼 클릭하면 다음 화면은 이렇다"를 잘 표현하는 것.
- 강의자 리뷰 우수 논문: **웹사이트별 정보 트리 → 유저 task에 따라 트리 서치 → 경로를 prompt로 줌** → 점수 大 ↑.
- ⭐ Fancy 용어가 아니라 **"정보를 어떻게 명확하게 줄 것인가"가 본질**.

### 10.10 IR (Information Retrieval)
- 다음 주 황성웅 교수님 IR 강의 예정.
- 강의자: "IR 하지 마, RAG는 몰라도 IR은 하지 마" — 옛날부터 있던 학문, "꼰대" 多.

## 11. ⭐ 한 페이지 요약

### 11.1 ⭐⭐ 시험 핵심 (Workflow vs Agent)

| 구분 | 정의 | 특징 |
|------|------|------|
| **Workflow** | **Predefined code path** — 사람이 흐름 강제 | Predictability / Consistency ↑ |
| **Agent** | 스스로 own process를 **dynamic**하게 결정 | 유연성 ↑ — Tool만 주고 autonomous |

⭐ 강사 어록 (직접): **"workflow는 predefined code path, agent는 dynamically own process를 다 하는 애들. 그 외에는 없습니다."**

### 11.2 오후 핵심 포인트

| 주제 | 핵심 |
|------|------|
| **LLM Output 평가** | Conventional (BLEU/ROUGE/BERT/Human) → LLM-as-Judge (G-Eval) |
| **Pointwise vs Pairwise** | Pairwise가 더 컨시스턴트 |
| **Judge Bias** | Position / Verbosity / Self-Enhancement (다른 family로 평가) |
| **AI Agent (RL 용어)** | Environment - Sensors - Actuators |
| **LLM Agent 3 Concept** | Reasoning / Memory / Planning |
| **Workflow vs Agent** | ⭐ predefined vs dynamic — **시험 출제** |
| **Building Block** | Augmented LLM (Retriever + Tool + Memory) |
| **Workflow 5패턴** | Prompt Chaining / Routing / Parallelization / Orchestrator-Workers / Evaluator-Optimizer |
| **Autonomous Agent** | Environment Feedback이 핵심 |
| **Context vs Harness Eng.** | Context = prompt의 정보 / Harness = environment까지 |
| **ReAct** | Thought → Action → Observation 반복, **Long Horizon** |
| **Tool Usage 발전** | Imitation (WebGPT) → Self-Supervised (Toolformer) → 대용량 (ToolLLM 16k) |
| **MCP 아키텍처** | Host - Client(1:1) - Server / Protocol + Transport Layer |
| **MCP 의의** | 표준 — Type-C 비유, OpenAI도 수용 |
| **Tool 개수** | 10개 이내 권장 |

### 11.3 전체 구조 도식
```
[LLM Output 평가]
Conventional (Parsing / BLEU / Human)
  → LLM-as-Judge (G-Eval, 2023)
      → Pointwise / Pairwise + Bias (Position / Verbosity / Self-Enhancement)

[AI Agent 진입]
RL의 Agent (Environment-Sensors-Actuators)
  → LLM Agent (Reasoning / Memory / Planning)

[⭐⭐ Workflow vs Agent — 시험]
Workflow (predefined code path)  vs  Agent (dynamic own process)

[Agent 내부 구조]
Building Block: Augmented LLM (LLM + Tool + Memory + Retriever)
  → Workflow 5 패턴
      Prompt Chaining / Routing / Parallelization / Orchestrator-Workers / Evaluator-Optimizer

[Autonomous Agent의 핵심]
Environment Feedback ← (Context Engineering / Harness Engineering)

[Agent 동작 포맷]
ReAct: Thought → Action → Observation × 반복 (Long Horizon)

[Tool Usage 발전]
Imitation Learning (WebGPT, 사람 6k) →
Self-Supervised (Toolformer, Meta) →
대용량 (ToolLLM, 16k API)

[표준화]
MCP (Anthropic) — Host / Client / Server, JSON-RPC 2.0
  Tool ≤ 10개 권장
```

### 11.4 비유 / 직관 모음

| 개념 | 비유 / 직관 |
|------|-----------|
| **LLM-as-Judge 정당화** | "평가 Task ≠ 생성 Task — 풀기보다 평가가 쉽다" |
| **Self-Enhancement Bias** | "라마가 라마 평가하면 토큰 확률이 더 높게 나오니 자기 답변을 선호" |
| **Workflow vs Agent** | "워크플로 = 매뉴얼대로 / Agent = 알아서" |
| **Augmented LLM** | "Transformer의 attention block처럼 Agent의 building block" |
| **Orchestrator-Workers** | "Claude Code의 매니저-하위 에이전트 구조" |
| **Environment Feedback** | "Feedback 없으면 헛돌아 — Agent 설계의 시작점" |
| **Context Engineering** | "Prompt만 건드는 게 아니라 환경 정보 전체를 다룬다" |
| **ReAct Long Horizon** | "state 시퀀스가 길어지면 시간 폭증" |
| **MCP의 의미** | "마이크로C / 애플 / 타입C → 타입C로 통일" |
| **Tool 10개 한계** | "그 이상은 노이즈, 성능 ↓" |
| **로봇 한계** | "콘센트 꽂다가 부순다 — 힘 조절 못함" |
| **VLA의 한계** | "동부 = 사기, 서부 = 미래. 강의자: 속도 느려 진화해야" |
| **데이터 퀄리티** | "네이버 지식인 많아도 퀄리티 떨어지면 성능 X" |
| **World Model 본질** | "Fancy 용어 아니라 도메인 정보를 명확히 주는 것" |
| **Agent Safety** | "Task Completion만 학습돼서 민감 정보 무시 — 개발 단에서 차단" |

### 11.5 강사 어록
- "workflow는 predefined code path, agent는 dynamically own process를 다 하는 애들. **그 외에는 없습니다**." (시험 직접 명시)
- "Task 자체가 다르다고 생각하시면 됩니다 — LLM은 Task별 성능 차이가 심하다."
- "대부분의 애플리케이션은 Workflow로 충분히 다 잘 됩니다."
- "Agent 설계의 시작은 Environment에서 어떤 Feedback을 받을 것인가다."
- "정보가 충분하다는 전제 하에 LLM은 풀 수 있다 — 우리가 정보를 충분히 주지 못하는 것 뿐."
- "어떤 학문에 표준이 정해지는 순간을 실시간으로 본 흥미로운 사건이었다." (MCP)
- "10개를 넘기는 순간 노이즈가 세진다."
- "로봇은 갈 길이 멀다 — 콘센트 꽂는 것조차 힘 조절 어렵다."
- "LLM 연구는 결국 데이터 퀄리티 싸움이다."
- "진짜 잘하는 사람들은 거의 다 창업한다."
- "Agent는 Task Completion만을 위해 학습돼서 Safety 관련 지각능력이 떨어진다."
- "Fancy 용어보다 도메인 정보를 명확하게 주는 것이 본질이다."

### 11.6 오전 + 오후 통합 — 시험 출제 가능성 정리

| 영역 | 시험 비중 | 핵심 |
|------|----------|------|
| RNN / Seq2Seq / Attention / Transformer | ⭐⭐ 高 | 차원 / 파라미터 / Bottleneck |
| 언어모델 / ICL / IT / RLHF / DPO | ⭐⭐ 高 | 3축 분석, KL Penalty |
| CoT 계열 (Few-shot / Zero-shot / Self-Consistency / Plan-and-Solve) | ⭐ 中 | 차이와 동기 |
| **Workflow vs Agent** | ⭐⭐⭐ **확정** (강사 명시) | predefined vs dynamic |
| LLM-as-Judge / Bias | ⭐ 中 | 개념적 이해 |
| ReAct / MCP / Tool Usage | 低 | 교양 수준 |
