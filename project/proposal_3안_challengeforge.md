## 0. 한 줄 요약

> 사무국이 보유한 **과거 행사 자료**(챌린지 문안·루브릭·심사 코멘트)로 **EventEmbed Lite(유사 행사 검색)** · **ChallengeGen / JudgeAssist Lite(SFT·vLLM)** 를 학습하고, 행사 기획·운영자용 **MCP 2개**(`draft_challenge`, `draft_judge_note`)만 제공한다. 행사 플랫폼 개발·참가자 매칭·DSDN 연동은 **Phase 2+**.

**MVP 성공 정의**: 다음 행사 1건에 대해 — (1) 챌린지 초안을 AI 생성 → 기획자 HITL 확정, (2) 제출물 5건에 심사 코멘트 초안 생성 → 심사위원 HITL. **10월 행사 또는 모의 운영 1회 완료.**

## 2. 배경 (왜 이 과제인가)

### 2.1 사무국 행사 업무

| 행사 유형 | 기획·운영 pain |
|-----------|----------------|
| **해커톤** | 챌린지 문안·평가 기준 **매번 제로부터** 작성 |
| **이너소스 공모전** | 과거 우수작·루브릭 **재활용 어려움** |
| **AI4SE 개발자 챌린지** | 2026 전략(BP 발굴)과 직결, **주제·난이도 설계** 부담 |
| **세미나** | (Phase 2) 주제·초록 초안 |
| **Best Reviewer 등 시상** | (Phase 2) 본 MVP 범위 외 |

### 2.2 왜 MVP인가

| 현황 | 문제 |
|------|------|
| 행사마다 기획·심사에 **수백 시간** | 사무국 인력 병목 |
| 우수 챌린지·심사 사례가 **파일·메일에만** 존재 | 집단 지성 미자산화 |
| AI4SE 전략: 챌린지로 BP 발굴 | **기획 품질**이 결과 좌우 |
| AI-Expert: RAG·SFT·서빙 | 행사 텍스트 코퍼스는 **법무·접근 용이** (사무국 내부) |

**핵심 문제**: 행사는 사무국 **핵심 업무**이나, 기획·심사는 여전히 수작업. AI는 **초안 생성 + 과거 사례 검색**에만 쓰고, 확정은 **HITL(기획자·심사위원)** 이 담당해야 한다.

## 3. MVP 범위 — 기획 1개 · 심사 1개 · MCP 2개

```
행사 기획자 / 심사위원
        │
        ▼
   Claude Code (Bedrock) 또는 사무국 CLI
        │
        ├── MCP: draft_challenge     ──►  EventEmbed RAG + ChallengeGen Lite
        │
        └── MCP: draft_judge_note    ──►  EventEmbed RAG + JudgeAssist Lite
                │
                ▼
           👤 HITL (기획 확정 · 심사 확정)
```

### 3.1 Ingest Lite

| MVP | Phase 2+ |
|-----|----------|
| 과거 행사 **5~10회** 아카이브 | 전 행사·세미나·시상 통합 |
| 챌린지 문안 + 루브릭 + (익명) 심사 코멘트 | 제출물 전문·코드·발표자료 |
| JSONL **200~500건** | 실시간 ingest |

**데이터 예**: `{(theme, event_type) → challenge_doc}`, `{(submission_summary, rubric) → judge_comment}`

### 3.2 EventEmbed Lite (내부, MCP 노출 없음)

| MVP | Phase 2+ |
|-----|----------|
| (행사주제, 챌린지문) 쌍 Contrastive FT | 제출물 유사도·표절 탐지 |
| `draft_*` 호출 시 **유사 과거 행사 3건** 자동 주입 | 단독 `search_events` MCP |

### 3.3 ChallengeGen Lite — **기획**

| MVP | Phase 2+ |
|-----|----------|
| 입력: `event_type`, `theme`, `constraints` | 세미나 초록·타임테이블 초안 |
| 출력: **챌린지 문안 + 루브릭 초안** (마크다운) | 참가자 가이드·FAQ |
| LoRA SFT + vLLM | DPO(기획자 수정 반영) |

### 3.4 JudgeAssist Lite — **운영·심사**

| MVP | Phase 2+ |
|-----|----------|
| 입력: `submission_summary`, `rubric` | 코드·데모 자동 요약 |
| 출력: **심사 코멘트 초안** (항목별 점수 근거) | Best Reviewer 후보 근거 |
| LoRA SFT + vLLM | 다심사위원 캘리브레이션 |

### 3.5 HITL (필수)

```
AI: 챌린지·루브릭·심사 코멘트 초안
  → 기획자/심사위원: 수정·승인·공개
  → (Phase 2) 우수작 → DSDN BP 초안 (2안 연동)
```

**MVP에서 AI가 하지 않는 것**: 최종 순위 결정, 참가자 선정, 행사 플랫폼, 상금·시상 판단.

## 4. 시스템 개요 (MVP)

```
 행사 아카이브 (5~10회)
       │
       ▼
 [Ingest 스크립트] ──► JSONL
       │
       ├──────────────────┐
       ▼                  ▼
 EventEmbed Lite    ChallengeGen Lite
 (GPU FT)            JudgeAssist Lite (GPU LoRA)
       │                  │
       └────────┬─────────┘
                ▼
         MCP Server (2 tools)
                │
       ┌────────┴────────┐
       ▼                 ▼
  사무국 기획자 CLI    Claude Code (선택)
```

**행사 전용 UI·참가 신청 시스템 없음** — 기획자·심사위원 **도구**만 제공.
