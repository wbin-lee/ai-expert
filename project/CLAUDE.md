## 4. 현업 과제 제안 (2026-05-08 부서장 통화 반영)

> **컨텍스트**: 부서장(누리 파트장)과의 통화에서 팀장님이 지속 요구해 온 두 가지 축을 가이드로 받음. 기존에 검토하던 "코드 리뷰 에이전트 RLHF" 제안은 부서장 가이드와 직접 정합도가 낮아 메인에서 제외하고, 사용자의 기존 운영 업무(코드 리뷰 에이전트 운영, ETL 파이프라인)는 데이터·인사이트 공급원으로만 활용.

### 방향 A (메인 후보, 도전적): Claude Code 사내 도입 가속화 + AIDLC 워크플로우 도구
- **배경**:
  - Claude Code 사내 오픈 — **5월 5,000명 → 6월 10,000명** 규모 확대 예정.
  - 단순 도구 배포로는 정착·활용도 한계. 사용자가 Claude Code를 효과적으로 쓰도록 만드는 지원 도구·방법론이 팀장님의 핵심 요구.
  - **AIDLC** (AI-Driven Lifecycle, 부서 내 기존 논의 컨셉) — AI 기반 SW 개발 워크플로우/플레이북을 어떻게 실제 도구로 풀어낼지가 과제.
- **과제 후보**:
  1. **사내 컨벤션·스택 특화 Claude Code 활용 가이드 자동 생성기** — 프로젝트 코드베이스를 분석해 CLAUDE.md, slash command, hook, MCP 설정을 자동 추천/생성. 신규 사용자 정착 가속.
  2. **AIDLC 워크플로우 어시스턴트** — 요구사항→설계→구현→리뷰 단계별 Claude Code 프롬프트/스킬을 부서별 SDLC에 맞춰 자동 추천하는 메타 에이전트.
  3. **Claude Code 활용도 분석 + 개인화 추천 대시보드** — 익명 사용 로그 기반 활용 패턴 분석, 미사용 기능 코칭, 이탈 위험 사용자 알림.
  4. **외부-사내 하이브리드 에이전트 오케스트레이션 (Big-Little LLM Routing)** — 외부 Claude Code의 토큰 제약과 사내 AI Agent의 성능 격차를 동시에 해결하는 라우팅 솔루션. *2026-05-11 사용자 제안.*
     - **핵심 아이디어**: 외부 Claude Code = 고수준 의사결정(계획·아키텍처·코드 리뷰 판단), 사내 AI Agent = 저수준 실행(파일 편집, 보일러플레이트, 단순 lookup). 토큰 효율과 성능을 동시 확보.
     - **주요 설계 결정**:
       - *고수준/저수준 경계 정의*: 정적 규칙 vs 학습된 라우터(난이도·불확실성 점수). 후자가 논문성·도전성 상위.
       - *오케스트레이션 토폴로지*: (a) Plan-Execute (외부=plan, 사내=execute) / (b) Cascade (사내 먼저, 실패 시 외부로 escalate) / (c) Tool-as-Subagent (사내 모델을 MCP 서버 또는 Claude Code subagent로 노출). (c)가 Claude Code 기존 추상화와 가장 호환적.
       - *품질 보증*: 외부 Claude Code의 sample-based 검증 + 사용자 피드백 → RLHF/reward modeling으로 자연 확장 (양인순 교수 RL 영역과 연결).
     - **주요 리스크**:
       - 라우팅 자체가 외부 토큰을 소비해 절약 효과가 상쇄될 수 있음 → 라우터를 사내 모델 또는 규칙+학습 하이브리드로.
       - 사내 모델 API/제어권 확보 전제 → 5월 중 사내 모델팀 컨택이 마일스톤 필수 항목.
     - **후보 1~3과의 관계**: 본 후보는 상위 프레임. 후보 1(가이드 자동 생성)·2(AIDLC 어시스턴트)는 본 라우터 위의 application layer로, 후보 3(활용도 분석)은 라우팅 정책 학습용 데이터 소스로 재배치 가능.
- **평가 매핑**:
  - 도전성·혁신성: 신규 도구 + 신규 메트릭 정의 + 사용자 행동 모델링 결합 → 상위. 특히 후보 4는 LLM cascade/routing 학계 흐름과 직결되어 논문성 추가 가점.
  - 과제 난이도: 데이터·LLM·UX 통합 → high.
  - 경영 기여도·중요도: 5,000~10,000명 규모 사용자 경험에 직접 반영, KPI(활용률·생산성) 직결. 후보 4는 추가로 **토큰 비용 절감**이라는 정량 KPI 확보.
  - 교육 이해도·활용도: 황승원(Agentic/Code Model) + 여진영(Prompting/RAG/Distillation) + 양인순(RL) 강의 내용 직접 활용.

### 방향 B (백업·병행, 실용적): DSA 2.0 커뮤니티 AI 기능 — AI 위키 자동 생성 중심
- **배경**:
  - DSA(사내 SW 커뮤니티) 2.0 개발 진행 중. AI 기능 통합은 무조건 들어가는 요건.
  - 부서장 강조 포인트: "있는 것들을 잘 해서 서비스화를 어떻게 잘 하느냐" — 기술 난이도보다 **통합·UX 완성도·도메인 특화**가 평가 포인트.
- **요구되는 AI 기능 (부서장 언급)**:
  - AI 자동 답변 (Q&A bot)
  - 콘텐츠 큐레이션
  - AI 글 작성/교정 어시스턴트
  - **스킬 마켓플레이스** (DSA 2.0 통합 검토 중)
  - **AI 위키 자동 생성** — 흩어진 Q&A·지식 콘텐츠를 정제된 위키 페이지로 자동 변환. 현재는 튜토리얼을 사람이 수동 작성. 오픈소스 사례 다수 → baseline 확보 용이.

- **평가 매핑**:
  - 난이도: 방향 A 대비 낮음 → 단독 메인보다는 병행/백업 적합.

  - 혁신성: 상대적으로 낮음 → 도메인 특화 기능과 평가 메트릭 부각 필요.
