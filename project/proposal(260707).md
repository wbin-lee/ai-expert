## 0. 한 줄 요약

> 사내 **Claude Code Enterprise** 사용 로그·대시보드(AWS 게이트웨이 연동)를 기반으로, **우수·저토큰 고효율 활용 패턴**을 학습·검색하고 **활용·예산 효율 코칭**을 제안하는 **ClaudeCoach**를 구축한다. 1인당 월 **~$180 종량제 한도** 제약 하에서 “잘 쓰는 법”과 “한도 안에서 효율적으로 쓰는 법”을 함께 지원한다. 학습은 AI 포털 GPU(Embed·SFT), 운영은 DS LLM·게이트웨이로 분리한다.

## 2. 문제 정의

1. **도구만 배포**하면 부서·개인별 활용 편차가 크고, 무엇이 “잘 쓰는 것”인지 **조직 차원에서 설명·전파**하기 어렵다.  
2. Enterprise **월 ~$180 한도** 하에서 조기 소진·과다 컨텍스트·재시도 누적 등 **비효율 사용**이 발생하나, 개인별 **예산 효율 코칭**은 없다.  
3. 기존 대시보드는 **정형 지표** 중심이고, *「내 상황·남은 예산에 맞는 쓰는 법」* 을 주는 **코칭·예시 추천**은 부족하다.  
4. Claude Code(Bedrock)는 범용 성능이 좋아 **파인튜닝만으로 이기기 어렵** — 차별화는 **사내 로그·BP·비용 효율 패턴**에 있다.  
5. 교육 과제는 **Foundation → 후처리 → 평가 → 서빙 연동** 증명이 필요하다.

## 3. 제안: ClaudeCoach — Claude 활용 코칭 엔진 (MVP)

### 3.1 무엇을 만드는가

```
[기존] Claude 사용 로그 DB + 대시보드 + AWS 게이트웨이
              │
              ▼
[신규] ClaudeCoach
  ① Ingest   — 로그 익명화·세션·태스크 정제·토큰/비용 메타·품질·효율 라벨
  ② PatternEmbed — 우수·저토큰 고효율 패턴 임베딩 (Contrastive FT)
  ③ CoachModel   — (질문+맥락+예산상태 → 활용·예산효율 코칭) LoRA SFT
  ④ BudgetCoach — 월 한도 대비 사용률·저토큰 우수사례·컨텍스트 축소 권고
  ⑤ Serve    — DS LLM API + 검색·어댑터 (게이트웨이/MCP)
              │
              ▼
[사용자] 「이렇게 쓰려는데 조언해줘」 → 우수 예시·팁 + (선택) 예산 효율 조언
[운영자] 부서별 활용·한도사용률·우수패턴 대시보드 (기존 UI 고도화)
```

### 3.2 MVP 범위 (10월까지)

| 포함 | 제외 (Phase 2+) |
|------|-----------------|
| 익명 Claude Code **사용 로그** 수집·정제 파이프라인 | 전사 모든 AI 도구 통합 |
| **우수 세션** 라벨링 v0 (품질 + **토큰 대비 효율**, 규칙+수동) | 완전 자동 품질·효율 판정 |
| **토큰·월 누적 비용** 메타 수집·집계 (익명) | 개인별 실시간 과금 연동 |
| **PatternEmbed** — (태스크, 우수·저토큰 맥락) 학습 | Text-to-SQL 동적 대시보드 전체 |
| **CoachModel** — 활용·**예산 효율** 코칭 SFT | DSDN Q&A 봇 본체 (2안 심화) |
| **BudgetCoach** — 한도 70/90% 구간 알림, 저토큰 사례 검색 (규칙→학습) | 자동 태스크 라우팅(외부↔사내 LLM) |

### 3.3 사용자 시나리오 (MVP)

**시나리오 A — 개발자 코칭**

> 사용자: 「Claude로 레거시 모듈 리팩토링 하려는데 컨텍스트를 어떻게 넣어야 할까?」  
> ClaudeCoach: 사내 유사 태스크 **우수 세션 3건** 요약 + 권장 프롬프트·MCP 사용 패턴 제안 (출처: 익명 로그 ID)

**시나리오 B — 예산 효율 코칭**

> 사용자: 「레거시 분석하려는데 이번 달 예산 70% 썼어요. 어떻게 하면 될까?」  
> ClaudeCoach: 유사 태스크 **저토큰 우수 세션** 3건 + 컨텍스트 축소·작업 분할 순서 + 예상 토큰 구간 안내

**시나리오 C — 운영자·파트장**

> 운영자: 대시보드에서 **부서 A vs B** 활용률·**월 한도 사용률**·우수 패턴 수 비교 → 코칭 대상 부서 선정

## 4. 모델·인프라 — 교육 vs 운영 분리

| 레이어 | 기술 | 인프라 | 교육·평가 |
|--------|------|--------|-----------|
| **학습** | PatternEmbed (Contrastive), CoachModel (LoRA SFT) | **AI 포털 GPU** (~10 A100 mix, 교육·PoC) | ✅ RAG·SFT |
| **평가** | Hit@k, 코칭 적합도(설문), A/B vs DS 베이스, (탐색) 동일 태스크 토큰 절감 | 오프라인 벤치 + 파일럿 | ✅ |
| **운영 서빙** | DS LLM API + 검색 컨텍스트 (+ 어댑터 merge 협의) | **DS·AWS 게이트웨이** | 서비스 연동 증명 |
| **추론 보조** | Bedrock Claude (복잡 설명 escalate) | 기존 Claude Code | — |

> **원칙**: 사무국이 **GPU 운영·vLLM 상시 서빙** 하지 않음. **학습은 GPU, 서비스는 DS·게이트웨이** — 7/3 통화·6/25 피드백 공통 반영.

## 5. 베이스 모델 선정 조사 (SFT·임베딩)

SFT를 본격 적용하기 전, **7월 Phase 0** 에서 후보 모델을 비교·확정한다. (7/3 통화: 파운데이션 모델 선정·파인튜닝 필요성 합의)

### 5.1 조사가 필요한 이유

| 이유 | 설명 |
|------|------|
| **GPU 제약** | AI 포털 ~10 A100 — 7B full FT 불가, **LoRA 대상·크기** 선정 필수 |
| **과제 특성** | 코드 생성이 아니라 **코칭·설명·사례 요약** — Code LM 필수 아님 |
| **서빙 연동** | 학습 모델이 **DS LLM API·어댑터 merge** 와 호환되어야 Phase 3 서비스 가능 |
| **교육** | 모델 선정·벤치마크·근거 제시가 **이해도·완성도** 평가에 포함 |

### 5.2 이중 트랙 — 임베딩 vs 생성

| 트랙 | 역할 | 후보 (1차 리스트) | 학습 방식 |
|------|------|-------------------|-----------|
| **A. PatternEmbed** | 우수 사용 패턴 검색 | `bge-m3`, `multilingual-e5-large`, `gte-multilingual-base` | Contrastive FT (경량) |
| **B. CoachModel** | 코칭·사례 설명 생성 | `Qwen2.5-7B-Instruct`, `Qwen2.5-3B-Instruct`, `Gemma-2-9b-it`, `Llama-3.1-8B-Instruct` | LoRA SFT |

> Code 전용 LM(StarCoder, CodeLlama)은 **코칭·자연어 설명** 비중이 높아 1차 후보에서 **제외**. 파일럿 결과상 코드 스니펫 인용 비중이 크면 Phase 2에서 재평가.

### 5.3 선정 기준 (스코어카드)

| 기준 | 가중 | 측정 방법 |
|------|------|-----------|
| **코칭 품질** (골드 30쌍) | 높음 | 전문가 5점, BLEU/ROUGE 보조 |
| **한·영 혼용** | 높음 | 사내 로그 샘플 실제 분포 |
| **LoRA 학습 안정성** | 중간 | loss 수렴, OOM 여부 (7B QLoRA) |
| **추론 latency** | 중간 | DS·게이트웨이 배포 시 예상 지연 |
| **라이선스·사내 배포** | 필수 | IT/법무·DS 배포 정책 |
| **DS 베이스와 어댑터 호환** | 필수 | DS 팀 확인 (merge·API 스펙) |
| **vs DS 베이스 only** | 높음 | 동일 프롬프트 A/B — **차별이 없으면 SFT 불필요** 판단 |

### 5.4 조사 절차 (7월)

```
Step 1  DS LLM API 스펙·베이스 모델 확인 (어댑터 탑재 가능 여부)
Step 2  코칭 골드셋 30쌍(활용·예산효율 포함) + 패턴 검색 쿼리 30개 확보
Step 3  임베딩 후보 2~3개 → Hit@3 오프라인 비교 (FT 전·후)
Step 4  생성 후보 2~3개 → LoRA SFT 1 epoch PoC (동일 데이터·하이퍼파라미터)
Step 5  스코어카드 작성 → 《베이스 모델 선정 보고서》 (7월 말)
Step 6  8월 본 학습은 확정 모델 1종씩만 진행
```
