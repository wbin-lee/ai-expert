## 1. 배경 및 문제 정의

| 현황 | 문제 |
|------|------|
| Claude Code 사내 오픈 규모 급증 (5월 5K → 6월 10K) | 도구만 배포하면 **활용 편차·이탈** 발생 |
| 프로젝트·부서마다 스택·컨벤션 상이 | 사용자가 **무엇을 컨텍스트에 넣고, 어떤 규칙으로 써야 하는지** 모름 |
| CLAUDE.md·skill·hook 수동 작성 | 품질 편차 큼, 작성 부담으로 **미설정 비율 높음** |
| 팀장님 지속 요구: "Claude Code를 잘 쓰게 하는 도구" | **온보딩·설정 자동화**가 가장 직접적인 대응 수단 |

**핵심 문제**: 개발자가 Claude Code를 열었을 때, 해당 repo/부서에 맞는 **컨텍스트·워크플로우 설정이 즉시 갖춰지지 않아** 생산성 이득이 지연된다.

## 2. 제안: RepoScout — 가이드 자동 생성기

### 2.1 핵심 아이디어

대상 저장소(또는 모노레포 서브패스)를 입력하면, **4단계 파이프라인**으로 Claude Code 설정 산출물을 생성한다.

```
코드베이스 스캔 → 컨벤션·스택 추출 → (선택) 사내 RAG 보강 → 산출물 생성·검증
```

### 2.2 생성 산출물

| 산출물 | 내용 | 효과 |
|--------|------|------|
| **CLAUDE.md** | 빌드/테스트 명령, 디렉터리 구조, 코딩 컨벤션, 금지 사항 | 에이전트 컨텍스트 즉시 정렬 |
| **Slash commands** | 자주 쓰는 작업(리뷰 요청, 테스트 실행, PR 설명 생성 등) | 반복 프롬프트 표준화 |
| **Hooks** | 커밋 전 lint, 민감정보 차단 등 | 안전·품질 게이트 |
| **MCP 추천** | 스택에 맞는 MCP 서버·설정 초안 | 도구 연동 장벽 제거 |

### 2.3 차별화 (범용 스캐너 대비)

- **사내 컨벤션 RAG**: 코드 리뷰·DSA 지식 코퍼스에서 부서별 규칙·BP 인용
- **검증 루프**: 생성 후 `npm test` / lint 등 **결정론적 검증**으로 환각 최소화
- **활용도 연계**: 생성 설정 사용 후 로그와 연결 → 품질·채택률 측정 (후보 3과 연동)

## 3. 시스템 개요

```
┌─────────────┐    ┌──────────────────┐    ┌─────────────────────┐
│  대상 repo  │───▶│  Analyzer        │───▶│  Generator (LLM)    │
│  (로컬/원격) │    │  스택·구조·패턴   │    │  CLAUDE.md·skill 등  │
└─────────────┘    └────────┬─────────┘    └──────────┬──────────┘
                            │ (선택)                     │
                            ▼                            ▼
                   ┌──────────────────┐         ┌─────────────────┐
                   │  사내 RAG        │         │  Validator      │
                   │  리뷰·위키 코퍼스 │         │  lint/test/형식  │
                   └──────────────────┘         └─────────────────┘
```

**MVP 구현**: CLI (`reposcout init ./my-project`) + Claude Code plugin/skill 형태 배포. 사내 모델 API **불필요** — 외부 Claude Code만으로 Phase 1 가능.

| 리스크 | 대응 |
