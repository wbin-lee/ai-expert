## 1. ⭐⭐ RL 마무리 — Continuation 평가 (오전 이어서)

### 1.1 ⭐⭐ 액션 단위 재정리 — 시험 출제 포인트
- ⭐ ⭐ ⭐ RL에서 액션 단위는 ⭐ **토큰만이 아님**:
  - ⭐ 토큰 단위.
  - ⭐ 툴 콜 (Tool Call) 단위.
  - ⭐ 긴 답변 / 트래젝토리 전체.
- ⭐ ⭐ MDP 포뮬레이션은 ⭐ **시퀀셜 디시전 메이킹**이므로 단위는 자유.

### 1.2 ⭐ Search Aron 토이 재요약
- ⭐ Question 3종 (A 명백 / B 명백 / Ambiguous).
- ⭐ Observation = Binary vector (A 플래그 / B 플래그 / Ambiguous 플래그 / 검색 완료 플래그).
- ⭐ ⭐ Outcome reward (+1 정답 / -1 오답 / -0.15 검색) → ⭐ **빠르게 Optimal로 수렴**.
- ⭐ 어떤 정보 검색해야 하는지 ⭐ **항상 슈퍼비전 주기 어려움** → ⭐ Outcome reward만으로도 학습.

### 1.3 ⭐⭐⭐ Reward Shaping의 부정적 결과 — 시험 출제 포인트
- ⭐ ⭐ ⭐ Outcome reward + 잘 만든 shaped reward면 ⭐ 좋아 보이는데 ⭐ **너무 인위적 shaped reward는 ⭐ 방해**.
- ⭐ ⭐ Prefix BLEU shaping → 옛날 결과 ⭐ **마지널한 게인**.
- ⭐ ⭐ ⭐ ⭐ 이유: ⭐ **초반에 다른 토큰 만드는 게 크게 페널티** → 모델 ⭐ **위축 / 다양성 ⭐ 억제**.

### 1.4 ⭐⭐⭐ 대안 — Continuation 기반 평가 — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ 가장 확실한 방법: ⭐ **prefix에서 ⭐ continuation을 여러 번 만들어 봄**.
- ⭐ ⭐ 괜찮은 prefix → 이어서 좋은 것이 ⭐ 더 자주.
- ⭐ ⭐ 비용 절감: ⭐ **Beam Search 트리에서 ⭐ 이미 공짜로 만든 continuation 활용** (prefix tree).
- ⭐ ⭐ ⭐ ⭐ 결론: 이게 ⭐ **MCTS** / Inference-time search와 ⭐ **연결**.

## 3. ⭐⭐⭐ Execution Accuracy — 시험 출제 포인트

### 3.1 ⭐⭐⭐ Surface Form 문제
- ⭐ ⭐ ⭐ NLP의 ⭐ **고질적인 문제** — 같은 의미인데 ⭐ **여러 표현**.
- ⭐ ⭐ "United States" vs "USA" — 다양.

### 3.2 ⭐⭐⭐ SQL의 Surface Form
- ⭐ ⭐ ⭐ 논리적으로 같은 거여도 ⭐ **SQL 쿼리 자체는 다를 수 있음**.
- ⭐ ⭐ ⭐ 예시: ⭐ **MAX 구하기**:
  - ⭐ ⭐ **Ascending 정렬 → 맨 뒤**.
  - ⭐ ⭐ **Descending 정렬 → 맨 앞**.
  - ⭐ ⭐ 둘 다 ⭐ 논리적 equivalent 지만 쿼리 다름.

### 3.3 ⭐⭐⭐ Execution Accuracy — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ **String 1대1 매치가 아니라 ⭐ 직접 돌려본 결과 같으면 OK**.
- ⭐ ⭐ ⭐ 가장 ⭐ 실용적 메트릭.
- ⭐ ⭐ **한계**: ⭐ 결과가 binary면 ⭐ **로직 엉뚱해도 우연히 맞을 수 있음**.
- ⭐ → 완벽하진 않지만 ⭐ 제일 실용적.

### 3.4 ⭐ 실습 데이터
- ⭐ Easy / Medium / Hard 구분.
- ⭐ Easy: "DB에 아티스트 몇 명?" — ⭐ 단순 SQL 한 줄.
- ⭐ ⭐ Medium+: ⭐ **Multi-hop reasoning**:
  - "이름이 LCDC인 아티스트의 앨범 타이틀 리스트".
  - "재즈 장르에서 ⭐ at least one track인 아티스트 이름".
- ⭐ ⭐ 실제: ⭐ **병원 DB에서 의사 쿼리**:
  - ⭐ ⭐ ⭐ "환자가 ⭐ **가장 최근에 받은 수술에서 처방받은 약**" — 수술 기록 추출 → 최근 날짜 → 처방약 리스트.

### 3.5 ⭐ 베이스라인 — One-shot
- ⭐ LLM에 ⭐ Question + Schema 줘서 답 한 번 받음.
- ⭐ ⭐ ⭐ 문제: ⭐ **중간 한 번 삐끗하면 끝**.
- ⭐ 가장 최근 수술이 뭔지 헷갈리면 ⭐ **그 뒤 reasoning 다 틀림**.
- ⭐ ⭐ 단순한 예제는 ⭐ GPT-4 mini로 ⭐ 100점 — 실세계는 반타작이면 잘하는 편.

## 4. ⭐⭐⭐ Chase-SQL (ICLR 2025) — 시험 출제 포인트

### 4.1 ⭐⭐⭐ 큰 그림
- ⭐ ⭐ ⭐ **워크플로우 디자인을 잘하자**의 ⭐ **잘 구현된 예시**.
- ⭐ ⭐ ⭐ 한꺼풀 벗기면 ⭐ **다 LLM 딸깍** — 그 딸깍을 ⭐ **좋은 유닛으로 패키징해 조립**.
- ⭐ ⭐ ⭐ 이름 = ⭐ "Multi-Path Reasoning + Preference Optimized + Candidate Selection".

### 4.2 ⭐⭐ 전체 파이프라인
1. ⭐ Value Retriever — DB 터미놀로지 검색.
2. ⭐ ⭐ ⭐ **3개 Generator** (다양한 방법):
   - DC-CoT.
   - QP-CoT.
   - OS.
3. ⭐ Query Fixer — 에러 트레이스 보고 수정.
4. ⭐ ⭐ Pairwise Selection — 최선의 쿼리 선택.

### 4.3 ⭐⭐ Value Retriever
- ⭐ ⭐ DB는 자기들 ⭐ **터미놀로지** 많이 씀.
- ⭐ ⭐ ⭐ 예시: 병원 DB의 ⭐ **PO = 경구투여** (Per Os 라틴어 약어).
- ⭐ ⭐ ⭐ 예시: 유저는 ⭐ "석유 1갤런 얼마"라고 묻는데 컬럼은 ⭐ `unit_price` — ⭐ 모를 수 있음.
- ⭐ ⭐ 임베딩 + 다양한 retrieval 방식 섞어서 ⭐ **관련 키워드 매핑**.

## 5. ⭐⭐⭐ Generator 1 — DC-CoT (Divide and Conquer)

### 5.1 ⭐ 인사이트
- ⭐ ⭐ ⭐ 어려운 질문은 ⭐ **여러 개 작은 질문이 합쳐진 것**.
- ⭐ ⭐ ⭐ 예: "재즈 장르에서 한 곡 이상인 아티스트" = 여러 sub-question.

### 5.2 ⭐⭐ 워크플로우
1. ⭐ ⭐ 복잡한 질문을 ⭐ **Sub-question으로 분해**.
2. ⭐ 각 Sub-question에 대해 ⭐ **작은 SQL** 생성.
3. ⭐ ⭐ Sub-answer 모으고 ⭐ **합쳐서 최종 SQL**.

### 5.3 ⭐⭐ GraphRAG와 유사 — 시험 출제 포인트
- ⭐ ⭐ ⭐ 이전 실습 GraphRAG: ⭐ **커뮤니티 단위로 잘라 → 커뮤니티별 쿼리 답 → 합침**.
- ⭐ ⭐ 같은 패턴: ⭐ 복잡한 쿼리 ⭐ **나눠서 합치기**.

### 5.4 ⭐ 실행 예시
- 평균 아티스트당 앨범 수보다 많은 아티스트 이름:
  1. ⭐ 평균 앨범 수 구함.
  2. ⭐ 각 아티스트 앨범 수 구함.
  3. ⭐ 평균 이상 필터.
  4. ⭐ 이름 컬럼 추출.

## 6. ⭐⭐⭐ Generator 2 — QP-CoT (Query Execution Plan)

### 6.1 ⭐ 인사이트
- ⭐ ⭐ 질문 자체를 나누는 게 아니라 ⭐ **Reasoning 단계를 명확히 3가지로 분리**.

### 6.2 ⭐⭐ 단계 — 시험 출제 포인트
1. ⭐ ⭐ ⭐ **어떤 테이블에서 가져올지** 찾기.
2. ⭐ ⭐ ⭐ **어떤 오퍼레이션** (JOIN, GROUP BY 등) 할지 찾기.
3. ⭐ ⭐ ⭐ **결과 어떻게** 정리할지 생각.

### 6.3 ⭐⭐ Inductive Bias — 시험 출제 포인트
- ⭐ ⭐ ⭐ 논문에서 ⭐ **있어 보이게 말할 때**: ⭐ "Inductive Bias 준다".
- ⭐ ⭐ 추론 시 ⭐ 강조하는 부분에 ⭐ **바이어스 두고 해봐라**.
- ⭐ ⭐ Step-by-step 디자인 = ⭐ Chain-of-Thought.

### 6.4 ⭐ 실행 예시
- ⭐ SELECT FROM → JOIN + GROUP BY + HAVING COUNT 식 step 분리.

## 8. ⭐⭐⭐ Query Fixer + Pairwise Selection

### 8.1 ⭐ Query Fixer
- ⭐ 이전 실습 (코드 fixer) 같은 패턴.
- ⭐ ⭐ ⭐ **에러 트레이스를 input으로 다시 줘서 고치기**.
- ⭐ 안 돌아가는 SQL을 잡음 (오답인 건 잡기 어렵).

### 8.2 ⭐⭐⭐ Pairwise Selection — 시험 출제 포인트
- ⭐ ⭐ ⭐ 3개 후보 중 ⭐ **둘씩 비교 → Tournament**.
- ⭐ ⭐ ⭐ 둘 다 같은 결과 → 캐리오버 (선택 X).
- ⭐ ⭐ 다른 결과 → ⭐ **LLM 호출**로 어떤 게 더 맞는지 점수.

### 8.3 ⭐ 논문 vs 실습 vs 공식 코드
- ⭐ 논문: ⭐ **SFT로 페어와이즈 컴페리즌 모델** 학습.
- ⭐ 실습: ⭐ SFT 대신 ⭐ **LLM 프롬프팅**으로 대체.
- ⭐ ⭐ ⭐ ⭐ **공식 코드는 ⭐ 룰베이스** (JOIN 서브스트링 / 길이 등 점수) — ⭐ **논문과 공식 임플리멘테이션 불일치** 사례.

### 8.4 ⭐ 왜 페어와이즈인가 (질문)
- ⭐ ⭐ ⭐ 그냥 ⭐ **디자인 초이스**.
- ⭐ ⭐ 3개씩 비교하면 ⭐ "왜 3개죠?" 리뷰어 질문.
- ⭐ ⭐ ⭐ 페어와이즈가 ⭐ **제일 미니멀** → 디폴트.

### 8.5 ⭐ 머저리티 보트 한계
- ⭐ ⭐ 일반 QA는 ⭐ **String 비교 머저리티 보트** 흔히 사용.
- ⭐ ⭐ ⭐ SQL은 ⭐ **Surface form 다름** → ⭐ 실행 결과 비교 머저리티가 ⭐ 베이스라인.
- ⭐ ⭐ 모델이 ⭐ 잘못된 bias면 ⭐ **잘못된 답 많이 뱉을 수도** → 더 발전된 방법 필요.

## 9. ⭐⭐⭐ MCTS — Monte Carlo Tree Search

### 9.1 ⭐⭐⭐ MCTS 기본 — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ **알파고의 성공 비결**.
- ⭐ ⭐ ⭐ 트리에 expand하며 여러 상태 탐색 → ⭐ Leaf node에서 reward 확인 → ⭐ 위로 ⭐ 전파.
- ⭐ ⭐ 가장 reward 최대화 경로 따라 ⭐ Rollout.

### 9.2 ⭐⭐ 바둑 비유 — 시험 출제 포인트
- ⭐ ⭐ State = 바둑판 (16x16 등).
- ⭐ Action = 돌 하나 놓기.
- ⭐ ⭐ ⭐ 가능 경우의 수 = ⭐ **우주 원자 수 (10³⁰⁺) 보다 많음**.
- ⭐ Reward = 끝에 1 / 0 / -1.
- ⭐ ⭐ ⭐ 무한 리소스면 ⭐ **다 둬보면 되는데** 실제론 불가능.

### 9.3 ⭐⭐⭐ Monte Carlo + Cut-off — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ 알파고 제로: ⭐ **1600번 정도** 깊이 내려가다 ⭐ **뉴럴 모델로 reward 받음**.
- ⭐ ⭐ 거기서 받은 reward를 ⭐ 위로 전파.

## 10. ⭐⭐⭐ Alpha-SQL (ICML 2025) — MCTS for SQL

### 10.1 ⭐⭐⭐ 컨셉 — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ **Node = Partial Reasoning Step** (SQL 상태).
- ⭐ ⭐ ⭐ ⭐ **Edge = Action** (SQL 변형 오퍼레이션).
- ⭐ ⭐ Leaf Node = ⭐ 완성된 SQL 또는 ⭐ 곧 샘플링 가능한 상태.

### 10.2 ⭐⭐⭐ Self-Supervised Reward — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ SQL이 ⭐ **정답인지 알 길이 없음** (알면 그냥 서브밋).

- ⭐ ⭐ ⭐ 트릭:
  1. ⭐ 완성 시점에 ⭐ **샘플링 여러 번** (Temperature ≠ 0).
  2. ⭐ ⭐ 일관되게 같은 결과 = ⭐ Confidence 높음.
  3. ⭐ ⭐ 그걸 ⭐ **reward**로 활용.
- ⭐ ⭐ ⭐ 예: ⭐ **5번 중 3번 일치 → reward 0.6**.

### 10.3 ⭐ 액션 단위 — 시험 출제 포인트

## 11. ⭐⭐⭐ MCTS 4단계 — 시험 출제 포인트

### 11.1 ⭐⭐ 4단계 — 시험 출제 포인트
| 단계 | 의미 |
|------|------|
| **Select** | 트리에서 다음 어떤 자식 노드로 갈지 ⭐ UCB1로 선택 |
| **Expand** | 리프 노드에서 다음 state 생성 → 트리 확장 |
| **Simulate** | 완결될 때까지 SQL 만들어 reward 관찰 |
| **Backprop** | reward 위로 전파 (모든 부모 노드까지 누적) |

### 11.2 ⭐⭐⭐ UCB1 식 — 시험 출제 포인트
```
UCB1 = R̄_n + c × √(ln N_parent / N_n)
```
- ⭐ ⭐ ⭐ ⭐ 앞 항 = ⭐ **Exploitation** (지금까지 평균 reward).
- ⭐ ⭐ ⭐ ⭐ 뒤 항 = ⭐ **Exploration** (적게 방문된 노드 가중).

### 11.3 ⭐⭐ Exploration vs Exploitation — 시험 출제 포인트
- ⭐ ⭐ ⭐ Exploration: ⭐ **잘 안 가본 데 가서 ⭐ 이익 얻기**.
- ⭐ ⭐ ⭐ Exploitation: ⭐ **이미 잘 알려진 reward 좋은 액션**.
- ⭐ ⭐ ⭐ 바둑 비유: ⭐ **단기 이익 (4집)**만 추구 → ⭐ 장기 불리.

### 11.4 ⭐ Log + Sqrt 이유
- ⭐ 로그 없으면 ⭐ N_parent 너무 강해 → ⭐ 많이 방문된 부모의 자식만 선택.
- ⭐ ⭐ 로그 + sqrt로 ⭐ **레귤러라이즈**.

### 11.5 ⭐⭐ Backpropagation — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ **뉴럴넷의 Backprop과 컨셉 유사하지만 수식 무관**.
- ⭐ ⭐ 그냥 ⭐ **카운트**:
  - Visit count +1.
  - Value sum += reward.
- ⭐ ⭐ 루트까지 ⭐ 부모 따라 올라감.
- ⭐ ⭐ 평균값은 ⭐ Value sum / Visit count.

### 11.6 ⭐ 강사 메시지 — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ ⭐ LLM을 ⭐ **마법의 엔진**으로 쓰면 → ⭐ 윗단에서 최적화 선택지 ⭐ **좁아짐**.
- ⭐ ⭐ ⭐ MCTS는 그 ⭐ **얼마 없는 선택지 중 하나** → 요즘 각광받음.
- ⭐ 다른 랭귀지 에이전트에도 등장 → 알아두면 유익.

- ⭐ ⭐ ⭐ ⭐ ⭐ ⭐ 일종의 ⭐ **공짜 Hard Negative**.

## 26. ⭐⭐ Bi-encoder vs Cross-encoder vs ColBERT — 시험 출제 포인트

### 26.1 ⭐⭐ Bi-encoder
- ⭐ ⭐ 쿼리 ⭐ **벡터 1개** / 다큐먼트 ⭐ **벡터 1개**.
- ⭐ ⭐ ⭐ 서로 ⭐ **못 봄** → 가벼움.

### 26.2 ⭐⭐ Cross-encoder
- ⭐ ⭐ ⭐ ⭐ **둘 다 input** → 모든 토큰 ⭐ 같이 봄.
- ⭐ ⭐ ⭐ 헤비 → ⭐ 수백만 문서엔 ⭐ **불가능**.
- ⭐ ⭐ ⭐ ⭐ 보통 ⭐ **Bi-encoder로 100~200개 후보 → Cross-encoder로 ⭐ Re-ranking**.

### 26.3 ⭐⭐⭐ ColBERT — 시험 출제 포인트
- ⭐ ⭐ ⭐ ⭐ Bi와 Cross의 ⭐ **중간 스펙트럼**.
- ⭐ ⭐ ⭐ 쿼리 인코더 → ⭐ **토큰 레벨 벡터 다수**.
- ⭐ ⭐ ⭐ 다큐먼트 인코더 → ⭐ **토큰 레벨 벡터 다수**.
- ⭐ ⭐ ⭐ 토큰 간 ⭐ Similarity 구해 ⭐ Pooling → 최종 스코어.
- ⭐ ⭐ ⭐ ⭐ **Multi-vector Dense** = ColBERT의 별칭.
- ⭐ ⭐ Bi보다 비싸지만 ⭐ 성능 ↑.

### 27.3 ⭐ 비유 / 직관 모음

| 개념 | 비유 / 직관 |
|------|-----------|
| **Continuation 대안** | 만들다 만 prefix에서 여러 번 이어 생성 — 평가 신뢰성 ↑ |
| **Beam Search 공짜 활용** | 이미 만든 prefix tree의 갈래를 그대로 평가에 활용 |
| **Surface Form** | "United States" vs "USA" — 같은 의미 다른 표현 |
| **MAX 두 방식** | Asc 정렬 마지막 / Desc 정렬 첫번째 — 같은 결과 |
| **테이블 = 설계자 마음** | LLM이 pretraining으로 배울 수 없는 컬럼명 / 추약 |
| **병원 PO** | Per Os 라틴어 약어 — DB 터미놀로지 모르면 쿼리 불가 |
| **DC-CoT = GraphRAG** | 큰 거 쪼개서 풀고 합치기 |
| **QP-CoT = Inductive Bias** | "이렇게 단계를 따라가 봐" 강조 |
| **OS Few-shot 자체 생성** | 학습 데이터 없이 (Q, SQL) 페어 만들어 예시로 쓰기 |
| **Pairwise = 페어가 미니멀** | 3개 비교하면 리뷰어가 "왜 3개?" 묻는다 |
| **MCTS = 알파고 비결** | 트리에 expand → leaf reward → 위로 전파 |
| **바둑 우주 원자** | 가능 경우 ≈ 10³⁰⁺ — 다 못 둠 |
| **알파고 1600 cut** | 그 깊이까지 가서 뉴럴 백업으로 reward |
| **마법 엔진 위 최적화** | LLM을 마법 엔진처럼 쓰면 윗단 선택지 좁음 → MCTS |
| **UCB1 = 모험 + 안정** | Exploitation 평균 + Exploration 안 가본 곳 |
| **바둑 4집 단기 이익** | Exploration 없이 단기만 추구하면 장기 불리 |
| **Backprop ≠ Neural Backprop** | 컨셉 유사 / 수식 무관 — 그냥 카운트 |
| **Lemmatization vs Stemming** | 사전 + 형태소 vs 도끼로 자르기 |
| **BPE = 형태소 학습** | running / runner의 공통 부분을 임베딩 |
| **Add-One Smoothing** | 분자 +1 분모 +V — 0 방지 |
| **자서전 = Credit Assignment** | 일생 보고 의미 부여, 사실 correlation |
| **DPR In-Batch Negative** | 같은 배치 다른 쿼리 Positive를 Negative로 (공짜 Hard) |
| **Bi vs Cross vs ColBERT** | 1 벡터 vs 모든 토큰 같이 보기 vs 토큰 레벨 페어 |
| **Cross-encoder Re-rank** | Bi로 100개 가져오고 Cross로 1~100등 정렬 |
| **LoRA = 돈 절약** | Forgetting 막기와는 다른 목적 |

### 27.4 ⭐ 강사 어록
- "Prefix BLEU로 Reward Shaping을 시도해도 ⭐ 다양성이 억제돼서 ⭐ 마지널한 게인밖에 안 나왔다."
- "가장 확실한 대안은 그냥 ⭐ continuation을 여러 번 해보는 것이고, Beam Search 트리에서 ⭐ 공짜로 활용 가능 — 이게 ⭐ MCTS와 연결된다."
- "DB 테이블 구조는 ⭐ 만든 사람 편의에 의한 설계라 ⭐ LLM Pre-training으로는 배울 수 없다."
- "병원 DB에서 PO는 ⭐ Per Os 라틴어 약어로 ⭐ 경구투여라는 뜻인데, 이런 터미놀로지를 모르면 쿼리를 날릴 수가 없다."
- "MAX는 ⭐ 어센딩 정렬 맨 뒤로 구할 수도, ⭐ 디센딩 정렬 맨 앞으로 구할 수도 있고 ⭐ 논리적으로 같지만 SQL은 다르다 — 그래서 Execution Accuracy를 본다."
- "한 번에 답을 받는 베이스라인은 ⭐ 중간 한 번 삐끗하면 끝이라, 가장 최근 수술이 뭔지 헷갈리면 그 뒤 reasoning이 다 틀린다."
- "AI 워크플로우를 잘 디자인하라는 말은 ⭐ 밥 먹으면 배부르다 하는 거랑 똑같이 당연한데, ⭐ 그래서 어떻게 하느냐가 핵심이다."
- "Chase-SQL은 한 꺼풀 벗기면 ⭐ LLM에 이거 해라 저거 해라 똑같은데 ⭐ 워크플로우 디자인이 좋은 참고가 된다."
- "DC-CoT는 ⭐ GraphRAG에서 커뮤니티 단위로 잘라 합쳤던 것과 비슷하다."
- "QP-CoT처럼 ⭐ 단계를 강조해 시키는 걸 논문에서는 ⭐ Inductive Bias라고 있어 보이게 말한다."
- "OS는 ⭐ Few-shot이 더 잘하니까 ⭐ 예시를 자기가 직접 만든다 — Schema 주고 (Q, SQL) 페어를 ⭐ 인스턴스화."
- "페어와이즈가 ⭐ 디자인 초이스인 이유는 3개씩 비교하면 ⭐ 리뷰어들이 '왜 3개죠?' 라고 물어볼 거니까 ⭐ 미니멀하게 페어와이즈."
- "MCTS는 ⭐ 알파고 성공의 비결이고 — ⭐ 트리 expand → leaf reward → 위로 전파."
- "바둑 가능 경우의 수가 ⭐ 우주 원자 수보다 많아서 다 둘 수는 없고 ⭐ 1600번 정도 깊이에서 ⭐ 뉴럴 모델로 백업한다."
- "Self-Supervised Reward는 ⭐ 일관성이 곧 reward — ⭐ 5번 샘플링 중 3번 같으면 0.6."
- "꼭 토큰을 액션으로 봐야 할 이유는 없다 — ⭐ SELECT FROM WHERE 같은 ⭐ 절도 액션 단위가 될 수 있다."
- "UCB1의 ⭐ Exploration과 Exploitation의 ⭐ 조화가 ⭐ 근본적 딜레마."
- "MCTS의 Backpropagation은 ⭐ 뉴럴넷의 그것과 ⭐ 수식적으로는 거의 상관없는 단순 카운트다."
- "LLM을 ⭐ 마법의 엔진으로 쓰면 윗단에서 ⭐ 최적화 선택지가 좁은데 ⭐ MCTS가 그 얼마 없는 선택지 중 하나다."
- "출제는 ⭐ 이론에서만 기본이고 ⭐ 실습 코드는 안 나오지만 ⭐ 실습에서 ⭐ 이론적으로 언급된 내용은 ⭐ 출제될 수 있다."
- "BPE는 ⭐ 학교 중간고사 단골 출제이고, ⭐ 코퍼스가 작아도 ⭐ Merge 순서를 ⭐ 눈으로 추적할 수 있어야 한다."
- "Micro vs Macro Averaging은 ⭐ 알면 맞추고 모르면 틀리는 문제인데 ⭐ 서울대 학생들 중에서도 ⭐ 꽤 틀린다."
- "Distance 쓰면 ⭐ 같은 문서를 ⭐ 복사 붙여넣기만 해도 ⭐ 거리가 멀어지니까 ⭐ 코사인을 써야 한다."
- "Isomorphism은 ⭐ 왕이 어떤 언어에서든 ⭐ 여왕과 가깝고 ⭐ 책보다는 멀다는 가정."
- "LoRA / Adapter는 ⭐ Continual Learning과 ⭐ 모티베이션이 다르다 — 이건 ⭐ 돈 절약."
- "GRPO 식의 ⭐ Clipping과 ⭐ KL Divergence는 ⭐ Policy가 너무 급격 변하지 않도록 ⭐ 제약하는 역할."
- "DPR의 ⭐ In-Batch Negative ⭐ 빨간색 표시 빠뜨려서 ⭐ 별표 붙이는데, ⭐ 같은 배치 다른 쿼리의 Positive를 ⭐ Negative로 쓰는 게 핵심 트릭."
- "Bi-encoder로 100개 가져오고 ⭐ Cross-encoder로 1~100등 ⭐ Re-ranking이 보통 패턴이고, ⭐ ColBERT는 ⭐ 그 중간 스펙트럼."
