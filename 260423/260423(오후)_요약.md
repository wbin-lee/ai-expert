## 2. NLP 태스크 카탈로그

### 2.1 분류 vs 생성 (난이도 색상)
- **녹색 (쉬움, 일찍 상품화)**: Spam Detection, Sentiment, POS Tagging
- **노란색 (조금 어려움)**: NER, Coreference Resolution, Word Sense Disambiguation, Dependency Parsing
- **빨간색 (생성, 옛날엔 어려웠음)**: QA, Paraphrase, Summarization, Translation, Dialogue
- 지금은 분류든 생성이든 모두 잘 됨.

### 2.2 POS Tagging (Part-of-Speech)
**같은 단어가 다른 품사로 쓰일 수 있다**
- "Sleep" — 동사("아이디어가 잔다") vs 명사("Did you have a good sleep?")
- "Lead" — 동사 lead /li:d/ 인도하다 vs 명사 lead /led/ 납

품사 태깅: 입력 텍스트 → 단어별 품사 태그 (Adj/N/V/Adv 등). 후속 태스크(추가 질문, 발음, 번역)의 기반.

### 2.3 ⭐ Named Entity Recognition (NER)과 Grounding
**🔥 중요 개념: Grounding (그라운딩)**
- "Einstein met with UN officials in Princeton"
- Einstein, UN, Princeton은 **세상에 실재하는 엔티티에 연결(grounded)** 되어야 진정한 이해.
- AI가 Einstein이라고 들으면 **위키피디아 페이지가 걸려있는 것처럼** 배경지식을 끌어와야 다음 질문(독일 출신? 오펜하이머와 관계?)에 대답 가능.
- 물리 AI(Physical AI)에서 더욱 중요. "Einstein한테 이메일 보내" 시 정확히 누구인지 그라운딩이 되어야 행동 가능.

### 2.4 Sentiment Analysis
오전에 다룸. 리뷰 → 긍정/부정. "샌프란시스코 최고의 스팸" = 좋음, "20분 기다리게 한 웨이터" = 나쁨.

### 2.5 Coreference Resolution (대명사 해석)
- "Carter said to Mubarak that **he** shouldn't run again."
- "he"가 Carter인지 Mubarak인지 모호.
- 두 가지 해석 가능: (a) Carter가 Mubarak에게 출마하지 말라 했다, (b) Carter가 Mubarak에게 자기는 다시 안 나가겠다 했다.

### 2.6 Word Sense Disambiguation (다의어 해소)
- "mouse" = 쥐 vs 컴퓨터 마우스
- 주변 단어로 판단: "mouse with battery" → 컴퓨터 마우스(쥐에 배터리 필요 X).

### 2.7 Dependency Parsing
문장 구조 (주어/동사/목적어/수식). 명령 수행 시 필수: "음료수를 저기에 갖다 줘" → 동사·목적어·위치를 정확히 파싱해야 실행 가능.

### 2.8 ⭐ Paraphrase: Abstractive vs Extractive
**🔥 시험에 나올 수 있는 분류**
| 방식 | 설명 | 예시 |
|------|------|------|
| **Extractive** | 원문에서 일부를 그대로 뽑아옴 | 영어 논문에서 각 문단 첫 문장만 모아 요약 |
| **Abstractive** | 새로운 표현으로 다시 씀 | "XYZ acquired ABC" → "ABC was taken over by XYZ" (겹치는 단어 X) |

- **Machine Translation**도 광의의 paraphrase (언어 변경 제약 추가).
- **Summarization**도 광의의 paraphrase (길이 축소 제약 추가).
- Abstractive가 평가도 어렵고 모델도 어려움. 최근 잘 됨.

### 2.9 요새도 어려운 NLP 문제
1. **Neologism (신조어)** — 트위터 시대 이후 줄임말·이모티콘·MZ 신조어 매일 생성. "unfriend", "retweet", "bromance".
2. **암묵적(implicit) 지식** — 회사·업무 내에서 모두가 알기에 적지 않는 지식이 train data에 없음.
3. **Named Entity의 동적 갱신** — 매일 새 책·영화·음료 출시. 위키피디아도 못 따라옴. "Bug's Life", "Let It Be" 같은 제목 그라운딩.
4. **Bias / 차별** — 헝가리어처럼 성별 없는 대명사를 영어로 번역할 때 확률 통계가 편향(예쁜 건 여자, 똑똑한 건 남자)을 그대로 반영. 자기소개서 평가에 쓰일 때 큰 문제.

## 3. Corpus(코퍼스) — 언어 모델의 재료

### 3.1 정의와 예
- **Corpus** = 어떤 언어/도메인에 속하는 텍스트의 모음. 복수형 = **Corpora**.
- 예: Shakespeare 코퍼스, 반도체 코퍼스, 삼성전자 사내 문서 코퍼스.

### 3.2 Web Corpus와 Google
- 검색 서비스의 본질: 크롤러가 웹페이지를 따라가며 모두 다운로드 → 자기 디스크에 인덱싱.
- 그게 곧 거대한 web corpus. 검색 결과를 0.1초 안에 주려면 로컬 인덱스 필수.
- 이렇게 모인 web corpus가 LLM 시대의 발판이 됨.

### 3.3 코퍼스의 종류
| 유형 | 예 |
|------|----|
| 언어별 | 한국어, 영어 (수천 개 언어 코퍼스 존재) |
| 변종 | African American English, Twitter 영어 |
| **Code-switching** | 한국어+영어 섞임, 인도 힌디+영어 |
| 시대별 | 신문 시대별 |
| 분야별 | 픽션 / 논픽션 / 의학 / 법률 |
| 저자별 | 연령·성별·출신별 |
| 라이선스 만료 | **Project Gutenberg** (저작권 만료된 책) |

### 3.4 Code-switching 비유
- 강사·전청조·의사 = 영어 섞어 쓰는 대표 사례. 기술용어·의학용어가 자주 섞임.
- 한글 데이터가 부족할 때 영어 데이터의 일부를 한글로 섞어 **인공 code-switching corpus** 를 만들어 데이터 증강.

## 4. Token / Type / Vocabulary

### 4.1 정의
- **Token** : 처리하는 가장 작은 단위 (단어/서브워드).
- **Type** : 중복 제거된 토큰 = **Vocabulary**의 원소.
- 예: "the dog runs the dog" → tokens=5, types=3.

### 4.2 Heaps' Law
**🔥 통계적 관찰**
```
|V| ≈ k · N^β     (β < 1, 보통 0.5 ~ 0.7)
```
- N: 토큰 수 (코퍼스 크기)
- |V|: 보케뷸러리(타입) 크기
- 토큰 수가 늘어도 vocab은 √N 정도로 **천천히 증가** → 같은 단어가 자주 재활용된다는 의미.

### 4.3 토큰 / 타입 예시 (강의 표)
| 코퍼스 | 토큰 수 | 타입 수 |
|--------|---------|---------|
| 작은 대화 | 240만 | 2만 |
| 신문 | 4400만 | 3만 |
| Google | 1조+ | 빠르게 천천히 증가 |

## 5. Tokenization

### 5.1 영어 — Naive Tokenization
띄어쓰기로 split. Python:
```python
text.split()      # 공백 기준
```
하지만 단점:
- 구두점 결합 ("dog." vs "dog")
- 어퍼스트로피 ("I'd" → "I", "'d") — apostrophe-d가 하나의 토큰처럼 취급되는 오류
- 이렇게 만든 셰익스피어 토큰 빈도: the(2.3만), and, of, a, ...

### 5.2 ⭐ Tokenization의 어려움 — 언어별 차이
**🔥 시험 출제 가능**
- **중국어/일본어/태국어**: 띄어쓰기 없음. "아버지가방에들어가신다" 같은 모호성.
- 해결: 가능한 분절을 enumerate하고 가장 그럴듯한 것 선택. "아버지가 방에 / 가방에"
- 7000개 언어를 지원하려면 7000개 tokenizer가 필요? → BPE라는 **언어 무관 해법** 등장.

### 5.3 Regular Expression 기반 Tokenizer
```python
re.findall(r"[a-zA-Z0-9]+", text)        # 알파벳/숫자
re.findall(r"[a-zA-Z0-9']+", text)       # 컨트랙션 포함
```
한계: **언어학적 지식**이 필요. 고대 아즈텍어처럼 모르는 언어엔 적용 불가.

## 6. Normalization (정규화)

### 6.1 모티베이션
**🔥 핵심: Surface Form 통일**
같은 의미인데 한 끗 차이로 다르게 취급되는 문제:
- US / U.S. / USA / U.S.A. → 모두 "미국"
- am / is / are / be / was → 모두 be 동사
- cat / cats / cat's / cats' → 모두 cat 어근

### 6.2 Case Folding (대소문자 통일)
- 단순: 모두 소문자로.
- 문제: **고유명사 의미 손실**.
  - **US** (미국) ↔ us (우리)
  - **General Motors** (회사) ↔ general motors (일반적인 모터)
  - **SAIL** (서울대 AI Lab) ↔ sail (항해하다)
  - **Fed** (연방정부) ↔ fed (먹였다)
- 절충: 문장 첫 단어만 소문자로 (그래도 미흡).

### 6.3 ⭐ Lemmatization (정확한 원형 찾기)
**🔥 시험 출제 포인트 (Lemmatization vs Stemming 구분)**
- **사전(dictionary) 기반** — 언어학적 지식 필요.
- 단어를 **어근(root form, lemma)** 으로 환원.
- 영어: cats → cat, running → run (단, 품사 정보 필요), geese → goose, is/am → be
- 한국어: "잘 되다" → "되다" + 어미 "ㄴ"

#### Morphological Parsing
모든 언어는 **stem(어근)** + **affix(어미)** 로 구성.
- 영어: cat + s
- 스페인어: am + aren (미래시제 3인칭)
- 한국어: 한 글자 안에서 어근/어미 분리되기도.

#### POS-aware Lemmatization
NLTK 라이브러리 사용 시 품사 정보를 함께 주면 정확도 ↑:
- "running" + (POS=verb) → run
- "running" + (POS=noun) → running (그대로)

### 6.4 ⭐ Stemming (어간 추출 — 야매)
**🔥 시험 출제 포인트**
- **룰 베이스** — 사전 X, 언어학적 정확성 X.
- 목표: 정확한 어근이 아니라, **같은 어근의 다양한 형태를 하나로 묶어 카운팅**.
- 강사 비유: "단어를 도마 위에 올린 다음에 도끼로 찍어내는 것".

#### Porter Stemmer 룰 예시
```
SSES → SS    (caresses → caress)
IES  → I     (ponies   → poni)
SS   → SS    (caress   → caress, no change)
S    → ""    (cats     → cat)
ATIONAL → ATE  (relational → relate)
TIONAL  → TION (conditional → condition)
ING  → ""    (motoring → motor)
```

#### Stemming의 부작용 (강사 일화)
- **King → K** (잘못된 잘림)
- 게임 현지화 사례: "stun" → "기절"로 일괄 치환했더니 "**assistant**"가 "**기절턴트**"가 됨.
- 그래도 surface form 통일 효과는 있어 통계적으로 유의미.

### 6.5 Lemmatization vs Stemming 정리
**🔥 시험 출제 포인트 - 반드시 외우기**
| 항목 | Lemmatization | Stemming |
|------|---------------|----------|
| 기반 | **사전(dictionary)** | **규칙(rule)** |
| 결과 | 정확한 원형 (lemma) | 야매 어간 (정확한 어근 X) |
| 정확도 | 높음 | 낮음 |
| 비용 | 큼 (언어학적 지식 필요) | 쌈 |
| 언어 일반화 | 어려움 (사전 필요) | 어려움 (룰이 언어 특이적) |
| 예: running → | run (품사 알면) | run |
| 예: ran → | run | ran (룰로 못 잡음) |
| 예: studies → | study | studi |

## 7. Byte Pair Encoding (BPE) — 데이터 기반 Subword Tokenization

### 7.1 ⭐ 모티베이션
**🔥 매우 중요 — 현대 LLM의 표준**
- 룰베이스/사전 기반 모두 **언어학적 사전지식 필요** → 7000개 언어 X.
- 해결: **데이터 통계만 보고** 자주 함께 등장하는 글자 쌍을 묶어 토큰화.
- **언어 무관**, **subword 단위** (한 단어가 여러 토큰이 될 수 있음).

### 7.2 BPE 알고리즘 (압축 원리)
**🔥 알고리즘 시험 출제 가능**
1. 초기 vocabulary = 코퍼스의 **개별 문자들** (영어: a-z, 한글: ㄱ-ㅎ, ㅏ-ㅣ).
2. 코퍼스를 훑어 **가장 자주 함께 등장하는 두 심볼(쌍)** 을 찾음.
3. 그 쌍을 새 심볼로 vocab에 추가.
4. K번 반복 (K = 사전 크기 파라미터).

### 7.3 작동 예 (강의 슬라이드)
초기 코퍼스 (단어 별 빈도와 끝 표시 `_`):
```
low_     × 5
lower_   × 2
newest_  × 6
widest_  × 3
new_t    × 2
```
초기 vocab = {l, o, w, e, r, n, s, t, i, d, _}

머지 순서 (빈도 기준):
1. `e` + `r` (9회) → `er`
2. `er` + `_` → `er_`
3. `n` + `e` (8회) → `ne`
4. `ne` + `w` → `new`
5. ...
→ 어미와 어근이 **자동으로 분리**되는 효과.

### 7.4 BPE의 두 단계
**🔥 시험 포인트**
1. **Learn** (학습 단계): 코퍼스를 훑으며 vocab을 K번 확장. 한 번만 수행.
2. **Apply** (적용 단계): 새 텍스트를 학습된 vocab으로 가장 긴 매칭부터 토큰화.

### 7.5 BPE vs 다른 방법
| 항목 | Naive (공백) | Lemma/Stem | **BPE** |
|------|--------------|------------|---------|
| 언어 의존 | 높음 (공백 없는 언어 X) | 매우 높음 | **무관** |
| 사전지식 | 약간 | 사전/룰 필요 | **불필요** |
| 단위 | 단어 | 어근 | **subword** |
| 한 단어 = ? 토큰 | 1개 | 1개 | **여러 개 가능** |

### 7.6 압축과의 관계
**🔥 강사의 인사이트**
- BPE는 사실 **데이터 압축의 원리**.
- 자주 나오는 시퀀스(예: "삼성전자")를 짧은 코드로 매핑 → 비트 절약.
- 토크나이저 입장에선 압축은 안 하지만, "자주 함께 나오는 단위 식별"은 동일.

### 7.7 관련 알고리즘 (이름만 알기)
- **BPE (Byte Pair Encoding)**: 위 설명. GPT 시리즈가 사용.
- **WordPiece**: BERT가 사용. BPE와 비슷하나 likelihood 기반 머지.
- **SentencePiece**: Google 라이브러리. 공백도 토큰으로 처리.
- 디테일은 다르지만 본질은 모두 "데이터 통계로 subword 사전 학습 + 적용".

## 8. ⭐ Probabilistic Language Model (확률적 언어모델)

### 8.1 ⭐ 언어모델의 정의
**🔥 시험 핵심 정의**
**언어모델 = 임의의 텍스트에 확률을 할당하는 함수.**
```
LM(w₁, w₂, ..., w_N) = P(w₁, w₂, ..., w_N)
```
- 영어 LM이라면: "high winds tonight" → 0.9, "large winds tonight" → 0.8 (덜 자연스러움).

### 8.2 ⭐ 어디에 쓰나
**🔥 시험 출제 포인트 - 응용 사례**

#### (1) 번역 (Machine Translation)
"내일 바람이 세게 불겠습니다" → 영어 후보들 중 어느 게 자연스러운가:
- strong wind / high wind / large wind / big wind ...
- LM이 가장 큰 확률을 주는 후보 선택.

#### (2) 스펠체크 (Spell Correction)
**🔥 어려운 케이스**
"My office is about 15 **minuet** from here" — minuet도 사전에 있는 정상 단어(미뉴에트, 음악 용어).
- 사전만으로 못 잡음. **문맥상 어색**한 걸 LM의 확률로 판단.
- edit distance 가까운 후보(minute, minuet, ...) 중 LM이 가장 그럴싸하다고 평가하는 것 선택.

#### (3) 음성 인식 (Speech Recognition)
"I saw a van" vs "I saw a ban" — 음파는 거의 동일.
- LM이 "saw a van"이 "saw a ban"보다 그럴싸하다고 판단 → 전자 선택.

> **🔥 강사 강조**: "이 모든 NLP에 근본적으로 이 확률을 알고 있다는 거가 성패를 가르고 있습니다."

### 8.3 Joint vs Conditional 표현
같은 LM을 두 형태로 쓸 수 있음:

#### Joint Probability
```
P(I, love, you)
```

#### Chain Rule로 분해 → Conditional
```
P(w₁, w₂, ..., w_n) = P(w₁) · P(w₂|w₁) · P(w₃|w₁,w₂) · ... · P(w_n|w₁,...,w_{n-1})
```

조건부 형태가 가능하다는 건 **다음 단어 예측**도 가능하다는 뜻 → **생성 능력**.

### 8.4 ⭐ 데이터 부족 문제 (Sparsity)
**🔥 시험 출제 포인트**

```
P("its water is so transparent that the") = ?
```
이렇게 긴 컨텍스트는 코퍼스에 등장 안 했을 가능성이 매우 큼.
- 분모(컨텍스트 빈도) = 0 → 확률 정의 불가.
- 분자도 거의 항상 0.
- **컨텍스트가 길어질수록 추정 불가능**.

## 9. ⭐ Markov Assumption과 N-gram 모델

### 9.1 ⭐ Markov Assumption
**🔥 시험 출제 핵심**
긴 컨텍스트는 버리고, **직전 N-1개 단어만 본다**는 가정.

```
P(w_n | w_1, ..., w_{n-1}) ≈ P(w_n | w_{n-N+1}, ..., w_{n-1})
```

### 9.2 N-gram 종류
**🔥 반드시 외울 표**
| 모델 | 컨텍스트 길이 | 식 |
|------|--------------|----|
| **Unigram** | 0 (컨텍스트 무시) | P(wₙ) |
| **Bigram** | 1 (직전 1개) | P(wₙ \| wₙ₋₁) |
| **Trigram** | 2 (직전 2개) | P(wₙ \| wₙ₋₂, wₙ₋₁) |
| **4-gram (Quadrigram)** | 3 | P(wₙ \| wₙ₋₃, wₙ₋₂, wₙ₋₁) |

- N이 클수록 더 자연스러운 언어, 하지만 sparsity 증가 (코퍼스 더 커야 함).
- N이 작을수록 추정은 가능, 하지만 부자연스러움.

### 9.3 ⭐ N-gram으로 언어 생성 (랜덤 샘플링)
**🔥 강사 강조 포인트**
LM이 있으면 **주사위만 굴려서 그 언어로 문장 생성 가능**.

#### 알고리즘
1. 시작 단어 w₁ 정함 (예: "I").
2. P(w₂ | w₁) 분포에서 w₁ 다음 단어를 **빈도에 비례하도록 샘플링** (그냥 가장 큰 거 고르면 뻔한 결과).
3. 새 컨텍스트로 다음 단어 샘플링.
4. 반복.

#### 강의 예시 (셰익스피어 코퍼스로 학습한 LM이 생성한 문장)
| 모델 | 생성 결과 (대략) |
|------|------------------|
| **Unigram** | "Deep of future, the unincorporated..." (의미 X, 단어 무작위) |
| **Bigram** | "parking lot, agreement reached..." (숙어 등장) |
| **Trigram** | "...without permission from..." (꽤 유창) |
| **4-gram** | 셰익스피어가 쓴 것처럼 됨 |

→ N이 커질수록 진짜 그럴듯한 셰익스피어 문장 생성.

### 9.4 ⭐ N-gram의 한계 — Long-distance Dependency
**🔥 시험 출제 포인트**
```
"어제 내가 사랑하는 그녀가 사준 3층에 뒀던 그 컴퓨터가 [crashed]"
```
- "computer" 와 "crashed"의 의존성은 **여러 단어 떨어져 있음**.
- 직전 3-4 단어만 보는 N-gram으론 절대 못 잡음.
- 이 문제 해결이 **RNN → LSTM → Transformer** 발전의 동기.

## 10. 실습 정리 (코드 레벨)

### 10.1 Levenshtein Distance 구현
오전에 본 Edit Distance에서 substitution cost = 2 로만 변경. 모티베이션: "치환은 delete + insert니까 2여야 한다."

### 10.2 Spell Checker
- Dictionary와 misspelled 단어 사이의 edit distance 계산.
- **Top-K candidates**를 거리 기준으로 정렬해 추천.
- 이게 곧 "검색"의 본질 — 거리 메트릭만 cosine similarity로 바뀌면 임베딩 검색이 됨.

### 10.3 Backtrace
DP 채울 때 어떤 연산(D/I/S)을 선택했는지 기록 → 변환 경로 재구성 가능.

### 10.4 Tokenization 실습 (NLTK)
```python
import nltk
from nltk.stem import WordNetLemmatizer, PorterStemmer
from nltk import pos_tag

# Lemmatization
lemmatizer = WordNetLemmatizer()
lemmatizer.lemmatize("running", pos="v")   # → "run"
lemmatizer.lemmatize("geese")              # → "goose"

# Stemming
stemmer = PorterStemmer()
stemmer.stem("running")    # → "run"
stemmer.stem("relational") # → "relat"
```

### 10.5 BPE 구현 핵심
```python
# 1. 페어 빈도 카운팅
def get_stats(vocab):
    pairs = Counter()
    for word, freq in vocab.items():
        symbols = word.split()
        for i in range(len(symbols)-1):
            pairs[(symbols[i], symbols[i+1])] += freq
    return pairs

# 2. 가장 빈번한 페어를 머지
def merge_vocab(pair, vocab):
    new_vocab = {}
    bigram = ' '.join(pair)
    replacement = ''.join(pair)
    for word in vocab:
        new_word = word.replace(bigram, replacement)
        new_vocab[new_word] = vocab[word]
    return new_vocab

# 3. K번 반복
for _ in range(num_merges):
    pairs = get_stats(vocab)
    best = max(pairs, key=pairs.get)
    vocab = merge_vocab(best, vocab)
```

### 10.6 Unigram 언어모델 — 가장 단순한 LM
**🔥 강사 위트**
```python
from collections import Counter
tokens = re.findall(r"\w+", text.lower())
freq = Counter(tokens)
total = sum(freq.values())
prob = {w: c/total for w, c in freq.items()}
```

## 11. 한 페이지 요약 (오후)

1. **NLP 역사**: ELIZA(1966) → AI 겨울 → Watson(2011 QA) → ChatGPT(2022 만능). 한 모델이 모든 태스크를 처리하는 시대.
2. **NLP 태스크들**: Spam·Sentiment·POS·NER·Coreference·WSD·Parsing·Translation·Paraphrase·Summarization·QA·Dialogue.
3. **Grounding**: 명사가 세상의 실재에 연결되어야 진정한 이해. 물리 AI에서 결정적.
4. **Paraphrase**: Extractive(추출) vs Abstractive(생성). MT/Summarization도 광의의 paraphrase.
5. **Corpus**: 토큰 수 N, 타입 수 |V|. **Heaps' Law**: |V| ≈ kN^β (β<1, 천천히 증가).
6. **Tokenization 진화**:
   - Naive(공백) → 룰베이스(regex) → **Lemmatization(사전)** / **Stemming(룰)** → **BPE(데이터 기반)**
7. **Lemmatization vs Stemming**: 사전 vs 룰, 정확 vs 야매. 둘 다 surface form 통일 목적.
8. **BPE**: 언어 무관 + subword. 코퍼스에서 빈번한 쌍을 K번 머지해 vocab 구성. 현대 LLM 표준.
9. **Language Model 정의**: 임의 텍스트에 확률을 부여하는 함수. **Joint** ↔ **Conditional** 변환 가능 (chain rule).
10. **LM 응용**: 번역·스펠체크·음성인식 — 후보 중 가장 그럴싸한 것을 LM 확률로 선택.
11. **N-gram + Markov Assumption**: Unigram→Bigram→Trigram→4-gram. N 클수록 자연스럽지만 sparsity ↑.
12. **N-gram 한계**: **Long-distance dependency** 못 잡음. Transformer 등장의 동기.
13. **N-gram도 생성 가능**: 빈도에 비례하는 주사위로 단어 샘플링 → 문장 생성. N이 커질수록 셰익스피어 같음.

### 시험 대비 핵심 포인트 정리

#### ⭐ 반드시 외울 것
1. **Lemmatization vs Stemming**: 사전기반 vs 룰기반 / 정확 vs 야매.
2. **BPE 두 단계**: Learn(코퍼스 훑어 vocab 확장) + Apply(새 텍스트 토큰화).
3. **N-gram 가족**: Unigram·Bigram·Trigram·4-gram. 컨텍스트 길이 = N-1.
4. **Markov Assumption**: 긴 컨텍스트 → 직전 N-1개로 근사.
5. **언어모델 정의**: 텍스트에 확률 부여하는 함수. Joint = ∏ Conditional.
6. **LM 3대 응용**: Translation, Spell correction, Speech recognition.
7. **N-gram 한계**: Long-distance dependency.
8. **Heaps' Law**: |V| ≈ kN^β.
