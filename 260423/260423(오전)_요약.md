## 2. Regular Expression (정규표현식)

### 2.1 왜 필요한가 — Surface Form 문제
**🔥 핵심 개념: Surface Form**
"같은 단어를 다양한 겉모습(대소문자·단복수·축약형)으로 표현"하는 자연어의 본질적 어려움. NLP에서 가장 어려운 점 중 하나로 강사가 반복 강조.
- 예: 다람쥐를 woodchuck/woodchucks/Woodchuck/WOODCHUCKS 등으로 표현. 룰을 짤 때 모두 처리해야 함.

### 2.2 Kleene이 정의한 Regular Expression 기호
| 기호 | 의미 | 예시 |
|------|------|------|
| `.` | 임의 한 글자 (0 글자 X) | `.상은` → 한상은, 황상은 |
| `*` | 앞 문자 0회 이상 | `5*` → "", 5, 55, 555... |
| `+` | 앞 문자 1회 이상 (최소 1번) | `5+` → 5, 55, 555... (`5*`와 다른 점) |
| `?` | 앞 문자 0 또는 1회 (옵셔널) | `colou?r` → color, colour |
| `[ ]` | 대괄호 안 문자 중 하나 | `[Tt]he` → The, the |

`5*` ≡ `5+` 의 차이: 전자는 0번 등장도 매칭, 후자는 최소 1번.

### 2.3 ⭐ 룰의 함정 — False Positive vs False Negative
**🔥 시험 출제 가능 핵심**

규칙: "기계가 틀렸을 때(False) 그 다음 단어는 **기계의 답**".
| 용어 | 의미 |
|------|------|
| **False Positive (FP)** | 기계가 잘못 찾았다고 답함 (사실은 X인데 O라고 함) |
| **False Negative (FN)** | 기계가 잘못 못 찾았다고 답함 (사실은 O인데 X라고 함) |

예시: 영어 정관사 "the"를 찾으려는 룰 `[Tt]he`
- FP: "there", "other"의 부분 문자열까지 매칭 → 잘못 찾음
- FN: "THE" (대문자) 미매칭 → 못 찾음
- 해결: 단어 경계(`\b`)로 둘러싸서 정확히 매칭.

### 2.4 어떤 에러가 더 중요한가 — Case-by-case
강사 비유:
- **범죄자 검거**: positive=범죄자라 정의했을 때, FP는 무고한 사람을 잡는 것 → 사회는 보통 FP 회피 (무죄추정).
- **코로나 진단**: FN(아픈 사람 놓침)이 더 위험 → FP를 감수.

**⭐ 시험 포인트: "한쪽만 최적화하면 안 된다"**
- "리콜만 챙길게" → 모든 걸 다 가져옴 (학생이 책 무더기 쌓아놓는 비유).
- "프리시전만 챙길게" → 아무것도 안 가져옴 (학생이 시도조차 안 함).

## 3. Precision / Recall / F1

### 3.1 정의 (Confusion Matrix 기반)
```
                실제 Positive    실제 Negative
예측 Positive       TP                FP
예측 Negative       FN                TN
```

| 지표 | 공식 | 직관 |
|------|------|------|
| **Precision (정밀도)** | TP / (TP + FP) | "갖다 드린 것 중 얼마가 맞나" — FP 줄이기 |
| **Recall (재현율)** | TP / (TP + FN) | "전체 정답 중 얼마를 잡아왔나" — FN 줄이기 |

### 3.2 Precision vs Recall 비유
- **High Precision**: "바쁜 교수 비유" — 쓸데없는 거 가져오지 마, 딱 필요한 것만.
- **High Recall**: "박사과정 학생 비유" — 인용 빼먹으면 리젝트, 문헌조사 다 긁어오기.

### 3.3 ⭐ F1 Score (Harmonic Mean = 조화 평균)
**🔥 시험 출제 포인트**
산술평균(arithmetic)이 아니라 **조화평균(harmonic mean)** 을 쓴다.
```
F1 = 2 · (Precision · Recall) / (Precision + Recall)
```
**왜 조화평균?**
- 산술평균: P=100, R=0 → 평균 50 (한쪽만 최적화해도 점수 받음. 트리비얼)
- 조화평균: P=100, R=0 → 0에 가까워짐 (한쪽 펑크 나면 강하게 페널티)
- 50-50일 때 조화평균이 50 → 두 지표가 균형 잡혀야 점수 높음.

> "하나만 옵티마이즈하는 건 너무 트리비얼하고, 그 트리비얼한 건 아무도 원하지 않는 솔루션이다."

### 3.4 가중치 일반화 — Fβ
α나 β 파라미터로 P/R 중요도를 조절하는 일반형도 존재 (이 강의에선 F1 위주).

## 4. Edit Distance (편집거리)

### 4.1 모티베이션 — Boolean Match를 넘어
정규식은 매치/언매치의 **이진(boolean)** 만 알려줌. 하지만 "거의 매치됐어"를 표현하려면 **연속적 거리** 가 필요.
- 예: "graffe"라는 오타를 보면 사람은 "graph"를 치려다 빼먹은 것일 가능성을 추측.
- **응용**: 스펠체크, DNA 시퀀싱(시퀀스 일부가 빠지거나 더해지거나 뒤집힘).

### 4.2 Intention vs Execution
- **Intention** (의도) = 사용자가 치려던 단어
- **Execution** (실행) = 실제로 친 단어
- Edit Distance = "intention에서 몇 번 삐끗해야 execution이 되는가"

### 4.3 기본 연산 (Operations)
| 연산 | 의미 | 비용 |
|------|------|------|
| **Insertion** (삽입) | 글자 하나 더 침 | 1 |
| **Deletion** (삭제) | 글자 하나 빼먹음 | 1 |
| **Substitution** (치환) | a를 치려다 b를 침 | 1 (또는 2: Levenshtein) |

### 4.4 Levenshtein Distance — 변형
**Levenshtein 선생의 이의 제기**: "치환은 한 번 delete + 한 번 insert니까 비용 2여야 한다."
→ Substitution cost = 2 인 변형이 **Levenshtein Distance**.

> 강사 코멘트: "여러분이 아무거나 제안하고 이름을 붙이시면 됩니다. 그게 받아들여지면 [이름] Distance 가 됩니다."

### 4.5 ⭐ Minimum Edit Distance와 트리 탐색
**🔥 핵심**
같은 두 문자열 사이에 가능한 편집 경로는 **무수히 많다** (deletion으로 다 지웠다가 insertion으로 다시 쓰기 등). 그 중 **최단 경로(minimum)** 가 곧 우리가 말하는 Edit Distance.

```
intention → ... → execution
```
이 변환을 트리로 그리면, 각 노드의 자식이 a-z로 치환되거나 삭제되거나 삽입된 모든 경우의 수 → 지수적 폭발.

### 4.6 ⭐ Dynamic Programming으로 구현
**🔥 시험 출제 포인트**
- Naive 트리 펼치기 = O(3^n) (지수)
- DP 사용 = O(m × n) (Quadratic)

#### 점화식
두 문자열 S1, S2의 i번째까지 / j번째까지 부분문자열 간 거리 D[i][j]:

```
D[i][j] = min {
    D[i-1][j]   + 1,                              // deletion
    D[i][j-1]   + 1,                              // insertion
    D[i-1][j-1] + sub_cost(S1[i], S2[j])          // substitution
}

sub_cost(a, b) = 0  if a == b
               = 1  (Edit Distance) or 2 (Levenshtein)  if a != b
```

#### 베이스 케이스
```
D[i][0] = i    // S1[1..i] → "" : i번 deletion
D[0][j] = j    // "" → S2[1..j] : j번 insertion
```

#### 직관
- **Deletion**: S1[1..i-1]을 S2[1..j]로 변환해놓은 뒤 S1의 마지막 글자 하나 삭제.
- **Insertion**: S1[1..i]를 S2[1..j-1]로 변환해놓은 뒤 S2의 마지막 글자 하나 삽입.
- **Substitution**: 둘 다 마지막 글자를 빼고 변환한 뒤, 마지막 글자를 치환.

### 4.7 Application-specific 변형
| 응용 | 추가 연산 / 비용 조정 |
|------|----------------------|
| 스펠체크 | **Transposition** (AS↔SA, 손가락 꼬임) 추가 |
| 키보드 뻑뻑 | Deletion cost ↑ |
| DNA 시퀀싱 | 도메인별 에러 패턴에 맞게 cost 부여 |

### 4.8 키보드/발음 기반 비대칭 비용
- A → E/O/U 치환은 흔함 (발음 유사 — "I love you too" vs "two")
- A → S 치환도 흔함 (키보드 인접)
- 한글 ↔ 영어 모드 미스도 잦음

이런 패턴을 substitution cost에 반영하면 더 정교한 스펠체크가 됨.

## 6. 확률 복습 — 100% 고등학교 수학

### 6.1 Conditional Probability (조건부확률)
```
P(A | B) = P(A ∩ B) / P(B)
```
- B가 일어났다는 조건 하에 A가 일어날 확률.
- 강사 비유: 비/우산 2x2 매트릭스. 비 온 날만 골랐을 때 우산 확률이 변함.

### 6.2 Joint / Marginal Probability
- **Joint Probability** P(A ∩ B): A와 B가 함께 일어날 확률.
- **Marginal Probability** P(A): 다른 변수에 대해 모두 더해(marginalize) 얻는 단일 변수 확률.
```
P(rain) = P(rain, umbrella) + P(rain, no_umbrella)
```

> "조인트 프로버빌리티, 마지널라이즈 한다 막 이러면은 그냥 더하는 거랑 곱셈하는 건데 어 이게 뭐지 하고 그런 생각이 들 때가 있다." — 영어 용어가 어렵게 느껴질 수 있다는 강사 위로.

### 6.3 ⭐ Bayes' Theorem (베이즈 정리)
**🔥 시험 출제 핵심**
```
P(A | B) = P(B | A) · P(A) / P(B)
```

#### 유도
P(A ∩ B) = P(A | B) · P(B) = P(B | A) · P(A) → 양변을 P(B)로 나눔.

#### 용어 정리 (베이지안 분류기에 재등장)
| 용어 | 의미 |
|------|------|
| **Prior** P(A) | 사전 확률. 아무 관찰 전 믿음. |
| **Posterior** P(A \| B) | 사후 확률. 증거 B 관찰 후 갱신된 믿음. |
| **Likelihood** P(B \| A) | 우도. A가 일어났을 때 증거 B가 관찰될 확률. |
| **Evidence** P(B) | 증거의 주변 확률 (정규화 상수). |

### 6.4 Spam Classification 예제
- Train set에서 관찰: 스팸 비율 40%, 스팸에 "free" 등장 80%, 햄에 "free" 등장 10%.
- 새 메일에 "free"가 등장 → 스팸일 확률은?

```
P(free) = P(free|spam)·P(spam) + P(free|ham)·P(ham)
        = 0.8·0.4 + 0.1·0.6 = 0.32 + 0.06 = 0.38

P(spam | free) = P(free | spam) · P(spam) / P(free)
              = 0.8 · 0.4 / 0.38 ≈ 0.8421 (약 84.21%)
```

**핵심 통찰**: train set에서는 라벨이 있는 상태에서 likelihood를 셀 수 있고, Bayes 정리로 방향을 뒤집어 **라벨 없는 새 샘플에 대한 posterior(즉 라벨 추론)** 를 계산할 수 있다.

## 7. Log Probability — 왜 로그를 쓰는가

### 7.1 ⭐ 동기 — Underflow 방지
**🔥 자주 등장하는 트릭**
```
P(w₁, w₂, ..., w_N) ≈ ∏ P(wᵢ)   (독립 가정)
```
확률은 0~1. 곱할수록 작아진다.
- 10단어 시퀀스 → 10^-16 수준
- 50단어 시퀀스 → 10^-66
- **컴퓨터의 유한 비트(부동소수점)로는 표현 불가능 → underflow**

### 7.2 해결: log 공간에서 계산
```
log P(w₁, ..., w_N) = ∑ log P(wᵢ)
```
- 곱셈 → 덧셈으로 변환 → 수치적으로 안정.
- 작아지는 속도가 exponential → linear로 완화.
- "수식에서 갑자기 log가 나오면 어려운 미적분이 아니라 underflow 회피라고 이해" — 강사 위로.

### 8.1 Python `re` 라이브러리
```python
import re
re.findall(pattern, text)            # 매칭 결과 리스트
re.findall(r"...", text)             # raw string 권장
re.findall(pattern, text, re.IGNORECASE)  # 대소문자 무시 플래그
```

### 8.2 정관사 the 매칭 — Word Boundary `\b`
```
[Tt]he         → "there", "other" 부분 매칭 됨 (FP 다수)
\b[Tt]he\b     → 단어 경계 안에 있는 the만 매칭 → 정확
```

### 8.3 가장 단순한 룰 베이스 스팸 필터
```python
pattern = r"(viagra|cialis)"
re.findall(pattern, email_text, re.IGNORECASE)
```
세로 줄(`|`)이 OR을 의미.

### 8.4 룰베이스가 여전히 중요한 이유
- LLM은 비싸고 외부 서버 호출이 보안 이슈 (민감정보 반출 제한).
- 빠르게 처리해야 할 케이스, 명확한 키워드는 룰베이스가 적합.
- "LLM에게 이런 조건의 정규식 짜줘" 라고 시킬 수 있으니 직접 짜는 게 필수는 아니지만, 도구의 존재를 아는 것 자체가 100% 유용.

## 9. 실습: Precision / Recall / F1 평가

### 9.1 Confusion Matrix 채우기
간단한 스팸 필터를 데이터셋에 돌려 TP/FP/FN/TN을 직접 계산. 다음을 함수로 구현:
```
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
Accuracy  = (TP + TN) / (TP + TN + FP + FN)
F1        = 2·P·R / (P + R)
```

### 9.2 ⭐ 도메인별 트레이드오프
**🔥 자주 묻는 응용**
| 도메인 | 우선 지표 | 이유 |
|--------|----------|------|
| 주식 매수 시그널 | High Precision | 사라고 할 때만 사도 손해 X. 놓쳐도 무방. |
| 암 진단 | High Recall | "음성인데 사실 걸렸을 수도 있어요"는 안 됨. |
| 스팸 필터 | 균형 | 중요 메일 잃으면 곤란 (Recall), 광고 다 통과해도 곤란 (Precision). |

## 10. 실습: Edit Distance with DP

### 10.1 테이블 초기화
```python
dp = [[0]*(n+1) for _ in range(m+1)]
for i in range(m+1): dp[i][0] = i
for j in range(n+1): dp[0][j] = j
```
"CAT" → "CAR" 변환을 예시로:
- `dp[3][3]` 자리에 정답.
- 첫 행/열은 단순 deletion/insertion 비용.

### 10.2 점화식 채우기
```python
for i in range(1, m+1):
    for j in range(1, n+1):
        sub_cost = 0 if S1[i-1] == S2[j-1] else 1   # 또는 2 (Levenshtein)
        dp[i][j] = min(
            dp[i-1][j]   + 1,         # delete
            dp[i][j-1]   + 1,         # insert
            dp[i-1][j-1] + sub_cost   # substitute
        )
return dp[m][n]
```

### 10.3 적용
- "intention" → "execution" 같은 긴 문자열도 quadratic 시간에 풀림.
- 트리를 거의 펼치지 않고도 답.

## 11. 한 페이지 요약 (오전)

1. **NLP는 함수 학습 문제** — input(텍스트) → output(클래스/구조). 분류·추출·번역·요약·QA·대화가 핵심 태스크.
2. **Surface Form 다양성** 이 NLP의 본질적 어려움 — 같은 의미의 다양한 표기.
3. **Regular Expression**: Kleene의 `* + ? . [ ]` + word boundary `\b`. 룰베이스의 출발점.
4. **두 가지 에러 타입**: **False Positive**(잘못 잡음) / **False Negative**(놓침). 도메인이 무엇이 더 치명적인지 결정.
5. **Precision = TP/(TP+FP)**, **Recall = TP/(TP+FN)**. 한쪽만 최적화하면 안 됨 → 조화평균인 **F1** 사용.
6. **Edit Distance** = "intention → execution 삐끗 횟수". 연산: insert/delete/substitute. **Levenshtein**은 sub cost=2.
7. **Minimum Edit Distance**는 **DP로 O(m·n)**. 점화식: `D[i][j] = min(D[i-1][j]+1, D[i][j-1]+1, D[i-1][j-1]+sub)`.
8. **확률 기초**: Conditional → Joint → Marginal → Bayes. P(A|B) = P(B|A)P(A)/P(B).
9. **Bayes 용어 4종**: Prior · Posterior · Likelihood · Evidence. 스팸 분류기에 그대로 적용.
10. **Log Probability**: 곱셈 폭주를 underflow 없이 처리하기 위한 표준 트릭.
11. **LLM 시대의 활용**: API + JSON 강제 출력으로 함수 위탁. 단, 룰베이스/통계 NLP는 비용·보안·속도 면에서 여전히 중요.
