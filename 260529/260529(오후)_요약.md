# 1. TPU v1 상세 ⭐

## 1.1 폼팩터와 워크로드

- **2015년 디플로이**, PCIe 슬롯에 꽂는 가속기(GPU와 동일 폼팩터), 서버당 최대 4개. 옆에 DDR3 메모리(삼성 칩) 다수.
- 그 당시 중요 워크로드: **MLP 61% / LSTM 29% / CNN 5%** (RNN/LSTM은 현재 트랜스포머로 대체됨).
- TPU v4 발표 시점(2022) 워크로드: **트레이닝의 60%가 트랜스포머** → 지금은 80~90%로 추정.
- 목표(2013 기획): 음성 어시스턴트가 모든 유저에게 3분/일만 서비스해도 데이터센터 2~3배 capa 필요 → **CPU 대비 10배 이상 perf/TCO** 달성하기.

## 1.2 CISC 아키텍처와 5 메인 명령어 ⭐

명령어 한 개가 굉장히 굵은 덩어리(CISC).

| 명령어 | 역할 |
|---|---|
| Read Host Memory | 호스트 DRAM → 가속기 DRAM (weight) |
| Write Host Memory | 가속기 → 호스트 (결과 copy) |
| Load Weights | 가속기 DRAM → MXU에 weight preload |
| **Matrix Multiply / Convolution** | systolic array로 행렬곱/컨볼루션 실행 |
| Activation Function | ReLU/Sigmoid/Pooling/Normalization 등 element-wise |

- **호스트 CPU가 명령어 fetch/dispatch**, TPU 칩은 데이터패스+실행 유닛만 있는 깡통 칩(비용 절감).
- 평균 CPI 10 사이클 이상, 4-stage pipelining: 각 스테이지가 서로 다른 타입의 명령어를 실행해 overlap 가능.

## 1.3 MXU와 Floorplan ⭐

\[
\text{MXU} = 256 \times 256 \text{ FMA cells} = 65{,}536 \text{ MAC units}
\]

\[
\text{Peak TOPS} = 65{,}536 \times 2 \text{ (mult+add)} \times 700 \text{ MHz} \approx 92 \text{ TOPS}
\]

| 구성요소 | 면적 비중 |
|---|---|
| Matrix Multiply Unit (MXU) | 약 25% |
| 로컬 Activation Memory (Unified Buffer) | 약 30% (24MB) |
| Accumulator Memory + 기타 | 나머지 |

Weight-stationary 동작 복습: weight를 MXU에 preload → 한쪽에서 input vector가 한 사이클씩 흘러들어가며 곱셈+partial sum 누적 → n사이클 후 출력. **모든 FMA가 이웃과만 통신**(글로벌 와이어 없음) → area/energy 효율.

## 1.4 TPU vs CPU/GPU 성능 ⭐

| 항목 | CPU | GPU (K80) | TPU v1 |
|---|---|---|---|
| TDP | 145W | 150W | **75W** |
| 워크로드 throughput (relative) | 1.0 | ~2.5× | **~15.3×** |
| 전성비(perf/W) | 1.0 | ~30× | **~80×** |

- **Roofline 분석**: DDR3(34 GBps) 사용 시 MLP/LSTM은 메모리 바운드(arithmetic intensity 낮음). CNN만 compute bound.
- 시뮬레이션: GDDR5(180 GBps, 약 5×)로 업그레이드하면 reach point가 **더 낮은 arithmetic intensity 쪽으로 이동**해 메모리 바운드 워크로드들도 peak FLOPs에 근접 가능 → 100×/70× 추가 향상 예상 → 실제 다음 세대는 GDDR, 그 이후 HBM으로 진화.

# 2. TPU v2/v3: 인퍼런스+트레이닝 통합과 bf16 ⭐

## 2.1 트레이닝 vs 인퍼런스의 요구사항 차이 ⭐

| 요소 | Inference | Training |
|---|---|---|
| 연산량 | forward only | forward + backward + weight update, 약 3× |
| 메모리 | activation 즉시 버림 | **모든 intermediate activation 보존**(backprop용) → 메모리 大 |
| 정밀도 | INT8/INT4 가능 | gradient 정확도 중요 → **wider/고정밀** |
| 분산 | 단일 칩 가능 | **distributed training** 네트워크 스택 중요 |

## 2.2 bf16 (Brain Float) ⭐⭐

TPU v2의 핵심 코디자인: **새로운 16bit 부동소수점 포맷**.

| 포맷 | Sign | Exponent | Mantissa | Range |
|---|---|---|---|---|
| FP32 | 1 | 8 | 23 | \(10^{-38}\) ~ \(3 \times 10^{38}\) |
| 표준 FP16 | 1 | 5 | 10 | \(10^{-8}\) ~ \(10^{4}\) (좁음) |
| **bf16** | 1 | **8 (유지)** | **7 (대폭 축소)** | **FP32와 동일** |

장점:

- **FP32와 range 동일** — 트레이닝 시 gradient에서 range가 정밀도보다 중요(주장).
- **FP32와 호환성 우수**: bf16↔FP32 변환이 mantissa truncation/zero padding만으로 됨(복잡한 shift/scaling 불필요) → 기존 FP32 소프트웨어와 잘 맞음.
- mantissa 축소가 area/power 절감에 효과적.

## 2.3 MXU와 데이터 패스 변화 ⭐

| 사양 | TPU v1 | TPU v2 |
|---|---|---|
| MXU 크기 | 256×256 | **128×128** × 2개 (코어당 1개, 칩당 2 코어) |
| 이유 | - | 너무 크면 utilization↓(텐서 크기 채우기 어려움), 너무 작으면 오버헤드. 128×128이 sweet spot |
| 데이터 타입 | INT8 fixed-point | **bf16 input, FP32 internal accumulation** |

내부 데이터패스는 **32bit accumulation**(partial sum 누적에 wider range 필요). 입력/저장은 bf16.

## 2.4 What You Train Is What You Serve ⭐

(What you see is what you get을 패러디.)

- 트레이닝 아키텍처와 서빙 아키텍처가 동일하면 number format 차이로 인한 "깨짐"이 없다.
- 실제로 GPU에서 트레이닝한 모델을 다른 아키텍처(NPU)로 가져가면 가끔 인퍼런스가 깨지는 경우가 있음.
- 단, **최근 트렌드**: 토큰 가격을 낮추는 게 중요해지면서 **다시 인퍼런스/트레이닝 분리**가 일반적. 이 분리/통합은 세대마다 갈라졌다 합쳐졌다 반복.

## 2.5 Scalar/Vector/MXU/Transpose 4-Unit 구조 ⭐

코어당 4개의 유닛:

| 유닛 | 역할 |
|---|---|
| **Scalar Unit** | **VLIW (Very Long Instruction Word) 322bit** 명령어 — 하나에 scalar 2개 + vector 4개 + matrix 2개 + misc 1개 + intermediate constant 6개 패키징 |
| **Vector Unit** | element-wise 연산(activation function, softmax 등) |
| **Matrix Multiply Unit** | 128×128 systolic array (MXU) |
| **Transpose / Permute / Reduction Unit** | reshape/transpose/reduction(max pooling, sum 등) 전용 — 메모리 액세스 많이 발생하는 오버헤드 회피 |

메모리:

- 코어당 HBM 1 스택(300 GBps × 2 = 600 GBps), 칩당 64GB HBM.
- 두 코어는 주소공간 공유하지만 **cache coherence 없음** — 필요하면 소프트웨어 레벨에서 명시적 동기화.

## 2.6 Pod와 Network ⭐

| 네트워크 | 역할 |
|---|---|
| **ICI** (Inter-Chip Interconnect) | TPU 칩 간 (NVLink 같은 역할). 이더넷 포트지만 프로토콜을 스트립다운(checksum 등 제거)해 low latency. **별도 스위치** |
| **InfiniBand** | 서버 간 통신(호스트 CPU 통해) |
| **PCIe** | 호스트와 가속기 연결(일부 레인) |

- TPU v2: 4 칩 보드 → 64 TPU pod.
- TPU v3: v2와 거의 동일(frequency 30%↑, bandwidth 30%↑, capacity 2×, 노드 수 4×). 1024 칩 슈퍼컴퓨터.
- 분산 학습용 멀티칩 스케일링을 처음부터 염두에 두고 설계 — distributed training 워크로드 대응.

## 2.7 Multi-tenancy ⭐

큰 HBM이 필요한 이유 중 하나는 **여러 모델을 한 메모리에 들고 있다가 쿼리에 따라 선택 서빙**.

- **A/B testing**: 새 모델과 옛 모델을 일정 비율로 섞어 서빙해 metric 비교.
- **Language switching**: 멀티링구얼 모델이나 언어별 모델.
- **Load-based switching**: 시스템 부하 낮을 때 좋은 모델, 몰리면 가벼운 모델로 스위칭(GPT/Claude도 종종 모델이 바뀜).

**SLO (Service Level Objective)** — 쿼리당 응답 시간 보장이 데이터센터는 중요, **피지컬 AI**(VLA: Vision-Language-Action)에서는 더 중요(실시간 컨트롤, 사람을 때리지 않게).

# 3. TPU v4와 Sparse Core ⭐

A100 세대와 비슷한 시기 출시.

## 3.1 주요 변화

- 2 코어/칩 (각 코어에 매트릭스 멀티플라이 유닛 4개, 벡터/트랜스포즈 유닛).
- **bf16** 입력, **FP32 accumulation** 유지.
- Pod 규모: **4096 칩**(8 rack × 4 trays × 4 칩 × 64 chip = 4×4×4 cube × 64 pod).
- **TPU v4i** (인퍼런스용 별도 칩): v4의 절반 슬라이스 — 저가형으로 분리.

## 3.2 OCS와 3D Torus 네트워크 ⭐

| 구간 | 인터커넥트 |
|---|---|
| Rack 내 | Copper (전통 금속선) |
| Rack 간 | **OCS (Optical Circuit Switch)** 옵티컬 — 3D Torus 토폴로지 |

Liquid cooling(고속 동작으로 패시브 에어 쿨링 불가) — CDU(Cooling Distribution Unit)로 쿨런트 공급.

## 3.3 Sparse Core ⭐

추천 시스템(DLRM, Deep Learning-based Recommendation Model)을 위한 **임베딩 가속 전용 IP**.

배경: 추천 시스템(검색·광고·썸네일)이 가장 중요한 워크로드. 유저/프로덕트/히스토리를 **벡터 공간에 임베딩**해 두고, 쿼리 시 해당 임베딩들을 읽어와 aggregation.

특성:

- **랜덤 메모리 액세스 매우 많음** (임베딩 공간에서 랜덤한 ID들을 패치).
- **연산은 매우 가벼움** (단순 sum/평균 aggregation).

→ 메모리 인텐시브 + 가벼운 연산이라는 특수 패턴을 위한 전용 코어. CPU 임베딩 처리 대비 **최대 30× 향상**.

## 3.4 CMEM (Constant Memory)

스크래치패드 SRAM, 이전 세대 대비 2배 확대 → 트레이닝 시 활용.

# 4. 최신 NVIDIA GPU ⭐

## 4.1 세대 로드맵

| GPU | 세대 | 출시 | 비고 |
|---|---|---|---|
| H100 | Hopper | 2022 | FP8 최초 지원, Transformer Engine, 2:4 sparsity |
| B200 | Blackwell | 2024 | 2 reticle-limited dies, 200B+ transistors |
| **B300 / GB300** | Blackwell Ultra | 2024~25 | **현재 살 수 있는 가장 하이엔드**, FP4 15/20 PFLOPs |
| Vera Rubin | Vera + Rubin | 2025~26 (예정) | CPU(Vera) + GPU(Rubin) 통합 슈퍼칩 |

가격: B200 8장 DGX 서버 약 7~8억원, B300 DGX 약 11~12억원.

NVIDIA의 **Tick-Tock 모델** (인텔과 유사):

- 한 해는 메모리 시스템 업그레이드(HBM3→3E→4→4E)
- 한 해는 코어 파이프라인/GPU 로직 업그레이드
- 매년 새로운 칩 발표

## 4.2 NVLink와 NVLink C2C

| 인터커넥트 | 역할 |
|---|---|
| **NVLink** | GPU-to-GPU 통신 (1.8 TBps 칩 간) |
| **NVLink C2C** | Grace CPU↔Blackwell GPU 통합 슈퍼칩 내부 (LPDDR을 capacity memory tier로) |

## 4.3 Transformer Engine ⭐

하드웨어-소프트웨어 코디자인. **FP8의 속도 + FP16의 정확도**를 동시에 달성하는 기법.

핵심 아이디어:

- FP8은 dynamic range가 좁음 → 표현 가능 범위를 **scaling factor**로 동적 이동.
- 하드웨어가 **dynamic value profiling**: 트랜스포머 layer output activation의 분포(히스토그램)를 측정해 적절한 scaling factor를 자동 결정.
- 실제 연산은 **internally FP16**으로 수행, 저장/메모리 R/W는 FP8(scaling factor 적용).
- 결과: 값이 가장 dense하게 분포된 지점 중심으로 FP8 표현 → accuracy loss 없이 메모리/대역폭 절감.

# 5. AMD 칩렛 기반 아키텍처 ⭐

AMD는 전통적으로 칩렛(chiplet) 인테그레이션에 강함.

## 5.1 구조

| 다이 | 역할 |
|---|---|
| **IO Die** | HBM 인터페이스, CCD/XCD 연결 (옛 PC의 North Bridge 같은 역할). TSV로 위에 다른 다이 적층 |
| **CCD** (CPU Complex Die) | CPU 칩렛 (AMD CPU 디자인과 공유) |
| **XCD** (Accelerator Die) | GPU 칩렛 (discrete GPU에도 reuse 가능) |

300A 같은 디자인: 일부는 GPU, 일부는 CPU를 인테그레이션한 APU.

## 5.2 V-cache (3D Integration)

원래 CPU에서 시작 — Cache(SRAM)는 공정 미세화에도 잘 스케일링되지 않으므로, **베이스 다이는 5nm**, **위 SRAM 캐시는 7nm**로 TSV 적층해 비용 효율 향상. GPU에도 적용.

## 5.3 칩렛 Trade-off

| 장점 | 단점 |
|---|---|
| 수율 좋음(작은 칩) | Die-to-die 통신 오버헤드(bandwidth↓) |
| Design reuse | 에너지 코스트 |
| 옛 공정 mix(SRAM은 옛 공정) | 실제 프로파일링하면 die 건너갈 때마다 성능 떨어짐 — 이론적으로는 아름답지만 실제로는 쉽지 않음 |

# 6. 현재 트렌드: Scaling Law ⭐

## 6.1 Frontier Model 컴퓨트

- 매년 4~5배씩 트레이닝 compute 증가.
- GPT-4 학습에 약 \(10^{25}\) FLOPs 사용.
- Med-PaLM(2023): 6,144 TPU v4 × 56일 = 약 500B 모델.

**Frontier vs Open-weight**의 구도(HBM vs commodity DRAM과 유사):

| 진영 | 모델 | 특징 |
|---|---|---|
| **Proprietary Frontier** | GPT-4, Claude, Gemini | 미국 (OpenAI/Anthropic/Google), 최첨단 |
| **Open-weight 추격** | DeepSeek, Qwen, GLM | 중국, 가성비 — DeepSeek은 Anthropic/OpenAI의 약 **1/10 토큰 가격**, 성능은 비등 |

## 6.2 Scaling Law의 한계 후보 ⭐

언젠가 스케일링이 멈춘다면 무엇이 한계?

| 후보 | 설명 |
|---|---|
| **Power** | 데이터센터 전력 수요 폭증 → 미국 원전 재가동(TMI). **가장 먼저 도래할 한계** |
| **Supply Chain** | HBM, TSMC capa(공급망) |
| **Data** | 학습 데이터 고갈 |
| **Latency Wall** | Amdahl's law — 병렬화로 줄일 수 없는 시간 |

스터디 결론: 이런 한계가 모두 있음에도 **2024 대비 약 10000× 헤드룸**이 2030년까지 있다 — 즉 "스케일링은 당분간 계속". Sam Altman 등은 무조건 compute 확보가 가장 중요한 챌린지.

NVIDIA Jensen: 내년 토큰 사용량 **100배** 증가 예상.

# 7. DeepSeek V3와 트랜스포머 메모리 bottleneck ⭐

## 7.1 DeepSeek의 혁신

| 기법 | 효과 |
|---|---|
| **MLA (Multi-head Latent Attention)** | KV cache를 토큰 공간이 아닌 **latent 공간**에 표현 → KV cache 약 **90% 절감** |
| **MoE (Mixture of Experts)** | 큰 모델 표준 (DeepSeek, Qwen 등) |
| **FP8 Training** | low-precision 학습으로 비용 절감 |
| **Reinforcement Learning 효율화** | 트레이닝 비용 절감 |

## 7.2 Self-attention 메모리 bottleneck ⭐

- 시퀀스 길이 \(L\)이 길어질수록 self-attention이 도미넌트(\(O(L^2)\)).
- 이전 컨텍스트 토큰을 **한 번씩 다 액세스**해야 해 메모리 바운드.
- 대응:

| 기법 | 아이디어 |
|---|---|
| **Sparse Attention** | 모든 토큰을 보는 대신 라이트한 prediction으로 중요한 top-K 토큰만 |
| **State Space Model (SSM)** | \(L\log L\) 또는 constant KV size — 시퀀스 길이가 길어져도 KV가 늘지 않음 |
| **Hybrid (Kimi 등)** | SSM + Transformer 혼합. accuracy 최고지만 트레이닝 어려움, asymptotic하게는 결국 \(O(L^2)\) 지배 |
| **PIM/FIM for Attention** | 메모리 근처에서 attention 실행 (삼성/하이닉스/마이크론 제안) |

발표자 연구실의 self-attention 가속기(2018~2022 테이프아웃): GPU 대비 self-attention 20~150× 빠름. NVIDIA의 다음 세대 GPU에 트랜스포머 가속기 아키텍처가 들어갈 거란 루머도.

# 8. 마지막 Q&A 요약 ⭐

## 8.1 Heterogeneous Cluster

- 여러 세대 GPU 혼합, GPU+NPU, GPU+CPU+NPU 등.
- 동기: **토큰/달러 낮추기** + 레거시 장비 활용.
- 이미 인더스트리 표준 기법: **PD (Prefill/Decode) Disaggregation**

| Phase | 특성 | 적합 하드웨어 |
|---|---|---|
| **Prefill** | 여러 입력 토큰 한 번에 처리, arithmetic intensity 高 | compute-centric (최신 GPU) |
| **Decode** | 토큰 하나씩 auto-regressive, arithmetic intensity 低, 메모리 바운드 | memory-centric (레거시 GPU — **cheap memory bandwidth**) |

- 한 단계 더: **Attention/FFN Disaggregation**. decode phase에서 FFN은 배치에 비례해 compute-centric으로 변하나 attention은 항상 메모리 바운드 → 다른 아키텍처에서 돌림. 발표자 연구실: attention은 **PIM**에서, FFN은 GPU에서.

## 8.2 학교 vs 인더스트리

학교의 역할 두 가지:

1. **Post-Transformer 연구**: 트랜스포머 이후의 새 아키텍처 탐색(AlexNet이 GPU 2장으로 AI 겨울 끝낸 것처럼).
2. **창업/현업 밀착**: 1000분의 1 자원으로 비슷한 성능 모델 만들기 같은 기술 차별화.

# 9. 오후 핵심 정리 ⭐

- **TPU v1**은 INT8 인퍼런스 향, 256×256 MXU 92 TOPS, CISC 5 명령어로 호스트가 컨트롤하는 깡통 가속기 칩이다. CPU 대비 약 15× throughput, 80× 전성비.
- **bf16**은 FP32와 **exponent 비트 수(8)·range가 동일**하고 mantissa만 7비트로 축소한 포맷으로, FP32와 truncation/zero-padding만으로 변환되어 호환성이 높다 — TPU v2에서 인퍼런스+트레이닝 통합 시 도입되었다.
- **TPU v2의 4 유닛 구조**(VLIW Scalar 322bit, Vector, 128×128 MXU 2개, Transpose/Reduction)와 **ICI** 인터커넥트 + **"What you train is what you serve"** 철학.
- **TPU v4**는 A100 세대로 4096칩 pod와 **OCS 옵티컬 3D Torus** 네트워크를 도입했고, 추천 시스템 임베딩의 **랜덤 메모리 액세스 + 가벼운 연산** 패턴을 위한 **Sparse Core**를 추가했다.
- 최신 NVIDIA GPU는 **Hopper(FP8) → Blackwell(2 reticle die, FP4) → Vera Rubin(CPU+GPU 슈퍼칩)**으로 진화하며, **Transformer Engine**은 dynamic value profiling으로 scaling factor를 자동 결정해 FP8 속도와 FP16 정확도를 동시 달성한다.
- **AMD 칩렛**(IO Die + CCD/XCD + V-cache TSV)은 수율·design reuse·옛 공정 mix(SRAM은 5/7nm) 측면에서 유리하나 die-to-die 통신 오버헤드가 존재한다.
- **Scaling Law의 한계 후보**는 power(가장 먼저)/supply chain/data/latency wall이며, 2024 대비 10000× 헤드룸은 2030년까지 있을 것으로 추정된다.
- **DeepSeek V3**는 **MLA(KV cache latent 공간 표현, 90% 절감)** + MoE + FP8 training으로 1/10 비용에 비등한 성능을 달성했고, self-attention의 \(O(L^2)\) 메모리 bottleneck은 sparse attention/SSM hybrid(Kimi)/PIM으로 대응 중이다.
- **PD Disaggregation**(Prefill→최신 GPU, Decode→레거시 GPU)은 이미 인더스트리 표준이며, 더 나아가 attention/FFN을 분리해 attention은 PIM에서 돌리는 연구가 진행 중이다.
