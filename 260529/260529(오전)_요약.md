# 260529 오전 강의 요약 — SmoothQuant 실습 · ViT/MobileViT 구현 · Mixed Precision/Token Pruning · NAS(Search Space·Strategy·Estimation) · Compound Scaling · Once-for-All

> 오전 흐름: **SmoothQuant 실습(W+A INT8을 위한 activation outlier를 weight로 옮기는 등가 변환, alpha 값/레이어별 그리디 서치) → ViT 직접 구현(Patch Embedding을 conv 1번으로, Block: LN→Attention→residual→LN→MLP→residual, classification token) → MobileViT(Conv로 spatial 축소 후 Transformer, Local-Global-Fusion 구조, patch embedding 없음) → Mixed Precision(Attention/MLP/Head 레이어별 다른 bit width, MLP 비중 66%·둔감 / Attention 32%·민감 / Head 메모리 영향 없으나 민감) → Token/Patch Pruning(CLS token 기반) → NAS의 3 키 팩터(Search Space·Strategy·Performance Estimation) → Search Strategy(RL/Evolution/Differentiable/Bayesian) → NASnet→mNASnet→EfficientNet/MobileNetV3 → ProxylessNAS/Single-Path NAS → Compound Scaling vs Once-for-All → 트랜스포머 NAS(hyperparameter tuning + Once-for-All)**
> 강의 주제: **LLM 양자화의 핵심 기법인 SmoothQuant를 직접 구현해 activation outlier 문제와 등가 변환의 효과를 체득하고, ViT/MobileViT를 직접 구현하며 mixed precision/pruning으로 경량화 trade-off를 실험하며, NAS의 3대 구성요소(Search Space·Strategy·Estimation)와 Compound Scaling/Once-for-All 같은 다중 하드웨어 대응 기법을 이해하는 것**.

# 1. 실습: SmoothQuant ⭐

## 1.1 LLM Quantization 분류

| 종류 | 표기 예 | 주 목적 |
|---|---|---|
| **Weight-only Quantization** | W4A16 | 메모리 사용량/대역폭 절약 (activation은 FP16/32) |
| **Weight + Activation Quantization** | W8A8 | 추론 속도 추가 향상 — SmoothQuant의 타깃 |

**SmoothQuant**는 W8A8을 잘 수행하기 위한 사전 작업: activation outlier 때문에 activation을 INT8로 양자화하기 어려우므로 **난이도를 weight로 옮겨** W8A8을 가능하게 한다.

## 1.2 핵심 원리 ⭐

LLM activation에는 **특정 채널에만 큰 값이 몰리는 outlier**가 자주 나타난다. 큰 outlier에 스케일이 맞춰지면 작은 값들이 INT8 표현 범위를 충분히 쓰지 못해 정보 손실이 일어난다.

등가 변환:

\[
y = (x \cdot s^{-1}) \cdot (s \cdot W)
\]

- activation \(x\)는 \(s\)로 **나누고**, weight \(W\)는 \(s\)로 **곱하면** 최종 출력은 동일.
- \(s\)는 채널마다 결정. activation이 많이 튀는 채널일수록 더 많이 줄이고 그만큼 weight가 부담.
- 왜 가능한가: **weight는 미리 알 수 있어**(offline) 대처하기 쉬움.

알파 값:

\[
s_j = \frac{\max(|x_j|)^{\alpha}}{\max(|W_j|)^{1-\alpha}}
\]

- \(\alpha\)가 크면 activation에서 더 많이 옮김, 작으면 덜 옮김. 보통 디폴트 0.5.

## 1.3 실습 결과 ⭐

OPT-125M 모델 + WikiText-2 데이터셋, **perplexity**(낮을수록 좋음)로 평가.

| 모델 | Perplexity 순서 |
|---|---|
| FP32 baseline | 가장 낮음(좋음) |
| **SmoothQuant INT8** | 중간 |
| Naive INT8 | 가장 높음(나쁨) |

- Activation/weight max 비율(레이어별): activation outlier가 더 도드라짐(예: 6.6 vs 4.1). SmoothQuant 적용 후 activation 비율 6.6→3.7로 감소.
- **레이어별로 분포가 다름** → uniform alpha보다 레이어별 그리디 서치가 perplexity 더 개선. 단, search chunk가 작으면 alpha 변동성이 커 신뢰도 낮음 → 실제로는 큰 calibration set 필요.
- Perplexity 평가에서 calibration은 사전에 한 번 하느냐(SmoothQuant), 매 forward마다 max로 잡느냐(dynamic) — 둘 다 PTQ.

# 2. 실습: Vision Transformer (ViT) 직접 구현 ⭐

## 2.1 Patch Embedding ⭐

ViT의 출발점. 트랜스포머는 토큰 벡터 시퀀스를 입력으로 받으므로, 이미지를 벡터 시퀀스로 변환해야 한다.

두 가지 기능을 **Conv 한 번**으로 수행:

1. **자르기**: 224×224×3 → 16×16 패치로 나누면 14×14 = **196개 패치**, 패치당 픽셀 \(3 \cdot 16 \cdot 16 = 768\).
2. **Projection**: 각 패치를 임베딩 차원으로 변환.

Conv 인자:

| 인자 | 값 |
|---|---|
| in_channels | 3 |
| out_channels | 192 (embedding dim) |
| kernel_size | 16 (patch_size) |
| **stride** | **16** (패치가 겹치지 않게) |

## 2.2 Attention과 CLS Token ⭐

- Attention은 첫 패치~끝 패치까지 **글로벌하게** 한 번에 정보 교환(CNN의 옆 픽셀과 대조).
- **Classification 토큰** 추가: 이미지에서 자른 게 아닌, 처음부터 학습되는 빈 토큰. 여러 레이어를 거치며 이미지 전체를 요약하는 대표 토큰이 됨. 마지막에 이것만 뽑아 분류 → **토큰 수 196 → 197**.

## 2.3 ViT Block ⭐

\[
\begin{aligned}
z' &= x + \text{Attention}(\text{LN}(x)) \\
z  &= z' + \text{MLP}(\text{LN}(z'))
\end{aligned}
\]

CIFAR-10 비슷한 ImageNet-ette 데이터셋(10 클래스, 224×224, 약 13,000장), epoch 10, accuracy 약 0.55~0.60.

# 3. 실습: MobileViT ⭐

## 3.1 핵심 아이디어

기본 ViT는 첫 레이어부터 196개 토큰 전부 attention → **\(O(n^2)\)이 heavy**.

**MobileViT**: **Conv로 먼저 feature map을 충분히 줄인 뒤** Transformer 돌림(224→28→14→7로 점진 축소). 토큰이 작아지니 attention 연산량 급감.

## 3.2 블록 구조

\[
\text{입력} \to \text{Local Repr} \to \text{Unfold} \to \text{Transformer} \to \text{Fold} \to \text{Fusion} \to \text{출력}
\]

| 단계 | 구성 | 역할 |
|---|---|---|
| **Local Representation** | n×n conv → 1×1 conv | Conv가 잘하는 로컬 정보를 라이트하게 섞음 |
| **Unfold→Transformer→Fold** | feature map↔토큰 변환 | 글로벌하게 멀리 떨어진 영역끼리 정보 교환 |
| **Fusion** | 1×1 conv → 입력 residual → n×n conv | 채널 맞춰 원본과 합치고 conv로 융합 |

⭐ MobileViT는 **patch embedding이 없다**: stride 16의 한 번 conv 대신 stride 2로 conv를 **여러 번**해 천천히 줄임. 가벼운 conv로 미리 줄인 뒤 토큰화하는 것이 경량화 핵심.

# 4. 실습: Mixed Precision ⭐

ViT 블록을 **레이어 종류별로 다른 bit width**로 양자화(PTQ, dynamic — 매 forward에서 activation max로 스케일).

레이어 분류와 특성:

| 레이어 종류 | 전체 메모리 비중 | 정밀도 민감도 | 메모리 절감 효과 |
|---|---|---|---|
| **Head** (분류기) | 거의 없음 | **민감**(3bit 이하 비권장) | 없음 — 4 또는 8 권장 |
| **MLP** (FFN) | **66%** | **둔감** (2bit에서 0.55→0.416) | 가장 큼 — 많이 낮추기 권장 |
| **Attention** | 32% | **매우 민감** (2bit에서 0.1 붕괴) | 중간 — 4 또는 6 |

핵심 메시지: **레이어별 민감도와 메모리 비중이 다르므로** 단순 일괄 양자화보다 mix가 효과적이다.

# 5. 실습: Token / Patch Pruning ⭐

ViT 196개 토큰 중 배경 등은 분류에 거의 기여하지 않음 → CLS 토큰이 별로 주목하지 않는 패치부터 버리는 로직.

- `keep_ratio` 파라미터(1.0 → 0.25)로 토큰을 줄이며 accuracy 낙폭 관찰.
- **토큰을 절반 이하로 줄여도 낙폭이 작다면 패치에 redundancy가 많다는 증거**.
- 실습에서는 pruning 후 fine-tuning 생략(실무는 prune → fine-tune 반복).

# 6. NAS (Neural Architecture Search) 개요 ⭐

알고리즘 자체를 자동으로 찾자는 영역(AutoML의 일부). 임베디드에서는 **Hardware-aware** 가 매우 중요(메모리/compute 제약 내에서 accuracy 최대화).

## 6.1 NAS의 3 키 팩터 ⭐⭐ (시험 출제 슬라이드)

| 팩터 | 의미 |
|---|---|
| **Search Space** | 어떤 범위의 네트워크 구조에서 찾을 것인가 |
| **Search Strategy** | 그 공간을 어떻게 탐색할 것인가 |
| **Performance Estimation** | 두 후보 아키텍처의 성능을 어떻게 비교/예측할 것인가 |

- **무엇이 가장 중요한가?** 직관적으로 **Search Space** — 정답이 search space 안에 없으면 아무리 잘 찾아도 sub-optimal밖에 못 찾음. 단 search space가 좋은지 알기 어려움.
- Performance Estimation: 풀 트레이닝은 불가능 → CIFAR-10 등 작은 데이터셋·소수 epoch로 빠르게 평가하며 accuracy loss 감수.

## 6.2 Search Strategy 종류 ⭐

| Strategy | 특징 | 시간 |
|---|---|---|
| **Reinforcement Learning** (NASnet, mNASnet) | 컨트롤러 RNN이 hyperparameter를 바꾸며 reward 받음 | 매우 김 (7만 GPU hour, 학교 불가) |
| **Evolutionary Search** (AmoebaNet) | Genetic Algorithm — parent에서 crossover/mutation으로 next generation | 매우 김, 단 RL보다 가속 가능 |
| **Differentiable Search** (DARTS, ProxylessNAS, Single-Path) | 아키텍처 선택을 **확률(softmax)로** 만들어 gradient descent로 학습. 학교발 빠른 기법 | 빠름 (1 GPU·1일) |
| **Bayesian Optimization** | Exploitation(좋은 근처) vs Exploration(멀리) trade-off | 중간 |
| **Random Search** | 그냥 랜덤하게 샘플링 | 빠름 |

⭐ Single-Path NAS의 흥미로운 발견: 잘 정한 search space에서 **random search조차 정교한 strategy와 1% 차이**밖에 안 남 → **Search Space가 더 중요**하다는 강력한 증거. 또한 1% 올리는 게 매우 힘들다는 의미이기도.

## 6.3 주요 NAS 결과들 ⭐

| 네트워크 | 출처 | Strategy | Search Space 특징 |
|---|---|---|---|
| **NASnet** | Google (NAS pioneer) | RL | **Cell 기반**: Normal Cell / Reduction Cell 반복. 7단계 셀, 13개 operation candidate(3×3/1×1/separable/avg pooling/max pooling 등) |
| **mNASnet** | Google (모바일 향) | RL | **Block 기반**(고정 7 블록), 각 블록 안 layer 수/conv 종류/SE block 포함 여부 가변 |
| **EfficientNet/MobileNetV3** | Google | RL | mNASnet과 **같은 search space** 위에서 만듦 |
| **AmoebaNet** | Google | Evolutionary | NASnet의 search space 재사용 |
| **ProxylessNAS** | Song Han (MIT) | Differentiable | 하드웨어별 최적 구조가 다름 — GPU/CPU/모바일 GPU별로 다른 구조. **ImageNet에서 직접** NAS |
| **ChamNet** | FB+Berkeley | - | MobileNetV2 기반 hyperparameter tuning |
| **DARTS** | - | Differentiable | 아키텍처 자체를 trainable. softmax로 op 선택 |
| **Single-Path NAS** | CMU | Differentiable | **하나의 필터에서** 3×3/5×5/7×7을 동심원으로 뽑아냄(weight 공유). 30 GPU hour |
| **S3NAS** | 발표자 연구실 | - | EfficientNet보다 더 좋은 스케일링 |

핵심 패턴: **구글이 search space를 잘 정의해 발표하면**(NASnet, mNASnet) 다른 연구들은 그 search space를 그대로 차용하고 **search strategy만** 바꾼다.

# 7. 다중 하드웨어 대응: Compound Scaling vs Once-for-All ⭐

문제: 하드웨어 플랫폼이 너무 많고(갤럭시·Jetson도 1년마다 새 모델), 매번 NAS 다시 하기 어렵다.

| 기법 | 출처 | 방향 |
|---|---|---|
| **Compound Scaling** (EfficientNet B0~B7) | Google | **작은 모델을 NAS로 찾고 scale up** |
| **Once-for-All (OFA)** | Song Han | **큰 모델을 트레이닝하고 작은 하드웨어로 scale down** |

## 7.1 Compound Scaling

depth(layer 수), width(필터 수), resolution(입력 이미지 크기) — 하나만 올리면 saturation, **세 가지를 균형 있게 올려야** accuracy가 계속 증가.

\[
d = \alpha^\phi, \quad w = \beta^\phi, \quad r = \gamma^\phi
\]

EfficientNet B0이 baseline, B7까지 scale up. CNN 이미지 분류 SOTA.

## 7.2 Once-for-All

- 큰 모델을 트레이닝하되, **작을 때도 잘 동작하도록** progressive shrinking + knowledge distillation으로 학습.
- 하드웨어 스펙이 작으면 그 부분만 잘라(slice) 바로 씀(NAS 재실행 불필요). Single-Path NAS의 weight 공유 아이디어와 유사.

## 7.3 트랜스포머 NAS

- 트랜스포머는 토폴로지를 거의 바꾸지 않음 → **hyperparameter tuning**(head/dimension/hidden size/depth) 위주.
- Differentiable Search가 트랜스포머에서 잘 안 됨(stochastic gradient로 아키텍처 변경 어려움).
- **Evolutionary Search**가 가장 많이 쓰임 + **Once-for-All**을 많이 사용(progressive shrinking으로 학습 → slice).

# 8. 오전 핵심 정리 ⭐

- **SmoothQuant**는 W+A INT8 양자화에서 activation outlier 문제를 해결하기 위해 \(x \to x/s\), \(W \to s W\)의 **등가 변환**으로 난이도를 weight 쪽으로 옮긴다. weight는 offline에서 미리 알 수 있어 대처가 쉽다.
- **Perplexity 순서**: FP32 < SmoothQuant INT8 < Naive INT8. 레이어별로 alpha를 다르게(그리디 서치) 적용하면 추가 개선되나 큰 calibration set 필요.
- **ViT의 patch embedding**은 conv 1번(stride=patch_size)으로 자르기+projection을 동시 수행한다. CLS token이 추가되어 197 토큰이 되고, 이것이 분류의 대표 토큰이 된다.
- **MobileViT**는 conv로 spatial을 먼저 줄인 뒤 Transformer를 돌려 \(O(n^2)\) 부담을 줄이며, Local Representation → Transformer → Fusion 구조에 patch embedding이 없다(stride 2 conv 여러 번).
- **Mixed Precision**에서 레이어별 메모리 비중은 MLP 66% / Attention 32% / Head ≈0이고, 민감도는 Head·Attention 高 / MLP 低 → **MLP를 가장 많이 낮추는** 전략이 효율적.
- **NAS의 3 키 팩터**는 **Search Space·Search Strategy·Performance Estimation**이며, 시험에 출제될 가능성이 높다. Single-Path NAS의 random 비교에서 보듯 **Search Space가 가장 중요**하다.
- Search Strategy는 **RL → Evolution → Differentiable**로 발전했고, NASnet/mNASnet의 search space를 후속 연구들이 차용했다. EfficientNet/MobileNetV3는 mNASnet과 같은 search space에서 나왔다.
- **Compound Scaling**(작은 모델→scale up)과 **Once-for-All**(큰 모델→scale down)은 매번 NAS를 다시 하지 않으면서 다양한 하드웨어를 지원하기 위한 두 패러다임이며, 트랜스포머 NAS는 hyperparameter tuning + Once-for-All 기반이다.
