## 1.4 확정문제별 심층 분석

### Q1 — Self-supervised pretraining + fine-tuning

**시험이 검증하는 것**: Weakly SL **전략 ② Pre-training** — 왜 라벨 부족 상황에서 먹히는가.

**(1) Label 없이 대규모 학습이 가능한 이유**
- **Pretext task**(self-supervised)로 데이터 **자체 구조**에서 supervision 신호 생성
- Unlabeled data는 **값싸고 대량** 확보 가능 → 대규모 pretraining

**(2) Downstream에서의 역할**
- 학습된 **general representation**이 좋은 **initial point(warm-up)**
- 소량 labeled data로 **fine-tuning만** 해도 좋은 성능 → **비싼 labeled data 요구량 감소**

**강의 연결**: Loss = Likelihood + Regularizer(Prior = Inductive Bias). Pre-training은 데이터 많을 때 주류.

### Q2 — In-context learning

**핵심**: Parameter는 **고정**이지만, prompt에 **demonstration**을 넣으면 **출력(행동)이 변한다**.

→ Context에 의해 모델 행동이 달라지므로, gradient/update 없이도 **“learning”** 이라 부른다.

**강의 연결**: Meta-learning / MAML 정신 — 빠른 적응. Instruction FT의 zero-shot 성능 향상과 같은 계열.

### Q3 — Instruction FT 한계 + RLHF

**(a) 근본적 한계 두 가지**

| 한계 | 내용 |
|------|------|
| **1. Creative / open-ended** | 정답(right answer)이 없는 생성 — 정답 맞추기 학습 부적합 |
| **2. Token-level 동등 penalty** | LM objective는 모든 토큰 오류에 **동일 페널티** — 실제로는 오류 **심각성**이 다름 → LM 목적과 **human preference** 사이 **mismatch** |

**(b) RLHF의 해결**
- **Human preference**를 직접 최적화 대상으로
- 출력 $s$에 **reward $R(s)$** 정의 → 기대 reward **최대화** (강화학습)
- 정답 없는 task도 학습 가능, 오류 심각성을 **reward에 반영**

**강의 보강 — RL을 쓰는 이유**: 각 스텝이 좋은지 애매하고 **시퀀스 끝**에야 reliable feedback → 챗봇·바둑·로봇 보행과 동일한 RL 프레임.

# 2부. 확정문제 (공개 PDF 원문)

---

## 문제 1 (확정)

Label이 부족한 상황에서, 대규모 unlabeled data로 self-supervised pretraining을 한 뒤 downstream task에 fine-tuning하는 것은 효과적인 weakly supervised learning 방법이다. 그 이유를 **(1) label 없이도 대규모 학습이 가능한 이유**와 **(2) pretraining된 모델이 downstream task 학습에서 하는 역할**을 포함하여 설명하시오.

### 답안

**(1)** Self-supervised pretraining은 **pretext task**를 통해 데이터 자체의 구조로부터 supervision 신호를 만들어내므로, label 없이 값싸게 대량 확보 가능한 **unlabeled data**로 대규모 학습이 가능하다.

**(2)** 이렇게 학습된 **general한 representation**은 downstream task 학습의 좋은 **initial point(warm-up)** 역할을 하므로, 소량의 labeled data로 fine-tuning만 해도 좋은 성능에 도달할 수 있다. (결과적으로 비싼 labeled data 요구량이 크게 줄어든다.)

---

## 문제 2 (확정)

In-context learning은 prompt에 task의 demonstration을 넣어주는 방식으로, gradient 계산도 없고 model parameter의 update도 일어나지 않는다. 그럼에도 불구하고 이것을 **'learning'**이라고 부르는 이유를 한두 문장으로 쓰시오.

### 답안

입력 sequence가 진행됨에 따라(demonstration이 주어짐에 따라) **모델의 출력이 달라지기** 때문이다. 즉, parameter는 고정되어 있지만 **context에 의해 모델의 행동(출력)이 변화**하므로 learning이라 부른다.

---

## 문제 3 (확정)

Instruction finetuning은 LM을 다양한 task에 빠르게 적응하게 만드는 핵심 기법이지만, ground-truth 데이터 수집 비용 외에도 근본적인 한계들이 존재한다.

**(a)** Instruction finetuning의 근본적인 한계 **두 가지**를 설명하시오.

**(b)** RLHF(Reinforcement Learning from Human Feedback)는 이 한계들을 **어떤 방식으로** 해결하고자 하는지 설명하시오.

### 답안

**(a)**

- **한계 1**: 창의적 생성(open-ended creative generation)과 같은 task는 **정답(right answer)이 존재하지 않아**, 정답을 맞추도록 학습하는 방식이 적합하지 않다.
- **한계 2**: language modeling objective는 모든 **token-level 오류에 동일한 페널티**를 부여하지만, 실제로는 오류마다 **심각성이 다르다**. 즉, LM objective와 “인간의 선호를 만족시킨다”는 실제 목적 사이에 **mismatch**가 존재한다.

**(b)** RLHF는 정답을 맞추는 대신 **인간의 선호(preference)를 직접 최적화 대상**으로 삼는다. LM의 출력 샘플 $s$에 대해 인간의 선호를 반영하는 **reward $R(s)$**를 정의하고(높을수록 좋음), 이 reward의 기대값을 최대화하도록 **강화학습**으로 LM을 학습한다. 이를 통해 정답이 없는 task에서도 학습이 가능하고, 오류의 심각성 차이도 reward에 반영할 수 있다.

## 예상문제 1 — Continual Learning (★ 최우선)

Continual Learning(또는 Lifelong Learning)에서 **Stability–Plasticity dilemma**가 무엇인지 설명하고, **Catastrophic Forgetting**이 왜 발생하는지 서술하시오. Class incremental learning 연구가 초기에 catastrophic forgetting에 집중했던 이유와, **Large language model** 환경에서 forgetting보다 더 중요해지는 이슈가 무엇인지 함께 설명하시오.

### 모범 답안

**Stability–Plasticity dilemma**

- **Stability(안정성)**: 이미 학습한 지식을 **유지**하고 잊지 않으려는 성질 → Catastrophic Forgetting 방지
- **Plasticity(가소성)**: **새로운 지식**을 수용·적응하려는 성질 → Adaptability
- 둘은 **상충**한다. 너무 안정적이면 새 것을 못 배우고, 너무 가소적이면 옛것을 까먹는다. Continual Learning의 목표는 **둘의 밸런스**이다.

**Catastrophic Forgetting 발생 이유**

- **파라미터 수 < 저장해야 할 지식의 양**
- 새 데이터로 순차 학습(특히 **1 epoch** online) 시, gradient가 새 task에 맞춰지면서 **이전 task에 맞던 파라미터가 덮어씌워짐**
- Multi-epoch 복습 없이 지나가면 forgetting에 더 취약

**Class incremental이 forgetting에 집중한 이유**

- 작은 모델·순차 클래스 추가 셋업에서 **forgetting이 가장 눈에 띄는 실패 모드**였기 때문
- “새 클래스를 배우면 옛 클래스 accuracy가 0에 가깝게 떨어진다”는 현상이 벤치마크의 핵심 과제가 됨

**Large model에서의 변화**

- 파라미터 용량이 커지면 forgetting은 **상대적으로 작아지는** 경향
- 대신 **Adaptability**(새 task·새 도메인에 얼마나 잘 적응하는가)가 **dominant issue**
- Large model continual learning에서는 **forgetting 완화만**이 아니라 **효율적 adaptation**(예: layer freezing, LoRA)이 더 중요

## 예상문제 2 — Weakly SL 두 전략 (Inductive Bias vs Pre-training)

Label이 부족한 상황(Weakly Supervised Learning)에서 취할 수 있는 **두 가지 전략**을 설명하고, 각 전략이 무엇을 전제로 하며 어떤 비용·지식이 필요한지 비교하시오.

### 모범 답안

Weakly SL에서 라벨이 부족할 때 해법은 크게 **두 가지**이다.

**1. Inductive Bias(귀납적 편향)를 잘 설계한다**

- **도메인 지식**과 **모델·loss 설계**로 prior를 넣는다.
- 예: CNN의 **패치 단위 인코딩**(CV 40년 결론), L1/L2 **regularizer**, 트리 깊이 제한, 아키텍처 선택
- Bayesian 관점: **Likelihood = 데이터**, **Prior = Inductive Bias**
- **장점**: labeled data가 **적을 때** 특히 유리 (Bias–Variance에서 bias 쪽 설계)
- **비용**: **Hand-crafted** 도메인 지식·설계 노력 필요

**2. Pre-training을 잘 한다**

- **Unlabeled data**를 대량 활용해 representation을 먼저 학습(self-supervised 등) → 소량 라벨로 **fine-tuning**
- **장점**: 요즘 **주류**; 대규모 데이터·연산만 있으면 general representation 확보
- **비용**: Pre-training **인프라·연산**; downstream은 상대적으로 적은 라벨

**비교 한 줄**

- **옛 주류**: Inductive Bias (도메인 지식으로 데이터 효율 ↑)
- **현 주류**: Pre-training (언라벨 대량 → warm-up → 소량 SFT)
- 둘 다 “이미 있는 지식/표현을 transfer”한다는 Weakly SL 철학과 맞닿아 있다.

## 예상문제 3 — RLHF에서 Reinforcement Learning을 쓰는 이유

RLHF에서 language model을 학습할 때 **supervised learning만으로는 부족**하고 **reinforcement learning**을 사용하는 이유를, 챗봇·로봇 보행·바둑 등 다른 도메인의 사례와 연결하여 설명하시오.

### 모범 답안

**공통 패턴**

- **각 스텝(토큰·모터 각도·한 수)** 이 “지금 잘했는가”를 **즉시 판단하기 어렵다**
- **시퀀스 전체가 끝난 뒤**에야 좋은지 나쁜지 **reliable한 피드백**을 줄 수 있다

| 도메인 | RL이 맞는 이유 |
|--------|----------------|
| **챗봇** | 매 턴마다 정답 라벨 없음 → 대화 끝 **good/bad** human feedback |
| **로봇 보행** | 매 순간 각도 프로그래밍 불가 → 끝에 “넘어지지 않음” reward |
| **바둑(AlphaGo)** | 한 수의 승률 불명확 → **수십~수백 수 뒤 승패**로 역산 |

**RLHF에 적용**

- Instruction FT는 **token-level supervised** objective → 모든 오류 동등 페널티, creative task 부적합
- RLHF는 출력 **시퀀스 $s$** 에 **reward $R(s)$** (human preference)를 부여
- **기대 reward 최대화**로 학습 → 시퀀스 **전체 품질**을 최적화하는 프레임
- 중간 토큰별 GT 없이도, **시퀀스 끝의 선호**로 policy(LM)를 업데이트할 수 있어 **RL이 자연스럽다**

# 5부. 시험 직전 체크리스트

### 확정 3문제 (반드시 서술 연습)

- [ ] Pretraining: **pretext task** → unlabeled 대량 / representation = **warm-up**
- [ ] In-context: **parameter 고정**, **context**로 출력 변화 → learning
- [ ] Instruct FT 한계: **no right answer**(creative) + **token 동등 penalty**
- [ ] RLHF: **preference → reward $R(s)$** → RL로 기대 reward 최대화

### 예상 4번째 (우선 대비)

- [ ] **Stability vs Plasticity** 트레이드오프
- [ ] **Catastrophic forgetting** = 파라미터 < 지식량, 순차 학습 덮어쓰기
- [ ] Large model: **forgetting ↓**, **adaptability**가 핵심
- [ ] Weakly SL **두 전략**: Inductive Bias vs Pre-training
- [ ] RL 이유: **시퀀스 끝**에만 reliable reward

### 보조 대비

- [ ] Bias–Variance: 데이터 적을수록 **inductive bias** 중요
- [ ] CNN = **패치 인코딩** = CV 40년 결론
- [ ] CIL 3법: **Replay / Regularization / Parameter isolation**
- [ ] **GDUM**: 버퍼 + from scratch, 단순한데 SOTA
- [ ] GPT-3 → InstructGPT → **RLHF** → ChatGPT
- [ ] Zero-shot vs Unsupervised — **직접 비교 문제는 안 냄**
