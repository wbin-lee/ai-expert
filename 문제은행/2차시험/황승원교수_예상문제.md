랩업 세션에서 다뤄졌지만 11문제가 cover하지 못한 주제들. ⭐ 개수는 강사가 부여한 중요도.

### A. ⭐⭐⭐ **Knowledge Graph Embedding + Relation Extraction** (랩업 §20)

- **Relation Extraction 방법론 3가지**:
  - **Rule-based**: Hearst Pattern + Regular Expression. (e.g., `"X such as Y"` → Y is a hyponym of X)
  - **Supervised**: Logistic Regression Classifier로 학습.
  - **Distant Supervision**: 사전(외부 KB)을 이용. 사전에 있는 트리플이 텍스트에 나타나면 그 문장을 positive로 라벨링.
- **Snowballing**: 사전 → 릴레이션 추출 → 트리플 → 사전 키움 → 더 많은 릴레이션. **부트스트래핑** 방식.
- **Graph 만들기 파이프라인**: Tokenization → POS Tagging → NER → Entity → Relation Extraction → Triple → Graph.
- **GraphRAG**: Vector RAG로는 답하기 어려운 **multi-hop / 집계** 질문에 강함.
- **KG Embedding (TransE 류)**:
  - **Head + Relation = Tail** 이 성립하도록 vector를 학습.
  - **Margin Loss**: positive 트리플의 score는 ↓, negative 트리플의 score는 margin γ 이상 ↑.
  - $$\mathcal{L} = \sum_{(h,r,t)\in S}\sum_{(h',r,t')\in S'}\left[\gamma + d(h+r, t) - d(h'+r, t')\right]_{+}$$
  - 여기서 $d(\cdot, \cdot)$는 거리 함수 (L1 또는 L2), $[x]_+ = \max(0, x)$.

### B. ⭐⭐⭐ **MCTS (Monte Carlo Tree Search) + Alpha-SQL** (랩업 §9~11)

- **MCTS 기본**: 알파고 성공 비결. 트리에 expand하며 여러 상태 탐색 → leaf node에서 reward 확인 → 위로 전파.
- **바둑 비유**: State = 바둑판, Action = 돌 놓기. 가능 경우의 수 ≈ $10^{30+}$ (우주 원자 수보다 많음). 알파고 제로는 **1600번 깊이**에서 끊고 뉴럴 모델로 reward 백업.
- **Alpha-SQL 컨셉**:
  - **Node = Partial Reasoning Step** (부분적으로 완성된 SQL 상태).
  - **Edge = Action** (SQL 변형 오퍼레이션 / SELECT, FROM, WHERE 절 등).
  - **Self-Supervised Reward = Consistency**: 정답을 알 수 없으므로 대용으로 사용. 완성 시점에 여러 번 샘플링 (temperature ≠ 0) → 일관된 결과 비율이 reward. (e.g., 5번 중 3번 일치 → 0.6)
- **MCTS 4단계**:
  | 단계 | 의미 |
  |------|------|
  | **Select** | 트리에서 다음 어떤 자식 노드로 갈지 UCB1로 선택 |
  | **Expand** | 리프 노드에서 다음 state 생성 → 트리 확장 |
  | **Simulate** | 완결될 때까지 SQL 만들어 reward 관찰 |
  | **Backprop** | reward 위로 전파 (모든 부모 노드까지 누적) |
- **UCB1 식**:
  $$UCB1(n) = \bar{R}_n + c \cdot \sqrt{\frac{\ln N_{\text{parent}}}{N_n}}$$
  - 앞 항 = **Exploitation** (지금까지의 평균 reward).
  - 뒤 항 = **Exploration** (덜 방문된 노드일수록 가중).
  - `c`는 exploration constant (하이퍼파라미터).
  - **log + sqrt**: $N_{\text{parent}}$의 영향력을 regularize.
- **Backpropagation**: 뉴럴넷의 backprop과 컨셉은 유사하지만 수식 무관. 그냥 **카운트**:
  - `visit_count += 1`
  - `value_sum += reward`
  - 평균값 = `value_sum / visit_count`.

### C. ⭐⭐⭐ **Chase-SQL 파이프라인** (랩업 §4~8)

- **큰 그림**: "Multi-Path Reasoning + Preference Optimized + Candidate Selection". 한 꺼풀 벗기면 LLM 딸깍이지만, 워크플로우 디자인이 좋은 참고가 됨. (ICLR 2025)
- **전체 파이프라인**:
  1. **Value Retriever** — DB terminology 검색. (e.g., 병원 DB의 "PO" = Per Os = 경구투여, 컬럼 `unit_price`)
  2. **3 Generators**:
     - **DC-CoT (Divide and Conquer CoT)**: 복잡 질문 → sub-question 분해 → 각 sub-question에 작은 SQL → 합쳐 최종 SQL. (GraphRAG와 유사한 패턴)
     - **QP-CoT (Query Execution Plan CoT)**: Reasoning 단계를 명확히 3가지로 분리 — ① 어떤 테이블에서 가져올지 → ② 어떤 오퍼레이션 (JOIN, GROUP BY) → ③ 결과 어떻게 정리. 논문에서는 "Inductive Bias" 주는 디자인이라 부름.
     - **OS (Online Synthetic example)**: Few-shot이 더 잘하므로 **예시를 자기가 직접 생성**. Schema 주고 (Question, SQL) 페어를 LLM이 생성 → few-shot으로 삽입.
  3. **Query Fixer** — 에러 트레이스 보고 수정.
  4. **Pairwise Selection** — Tournament 방식. 둘씩 비교하여 더 나은 쿼리 선택. 결과 같으면 carry-over, 다르면 LLM 호출로 점수.
- **Pairwise인 이유**: 단순 design choice. 3개씩 비교하면 "왜 3개죠?" 리뷰어 질문이 생김. 페어와이즈가 가장 미니멀.
- **머저리티 보트 한계**: SQL은 surface form이 다양해서 string-level majority voting은 불가능. 그래서 **실행 결과 기반** majority voting이 베이스라인이지만, 모델이 잘못된 bias면 잘못된 답이 우세할 수 있어 발전된 방법 필요.

### D. ⭐⭐⭐ **Execution Accuracy + SQL Surface Form** (랩업 §3)

- **Surface Form 문제**: 같은 의미인데 여러 표현. 예: "United States" vs "USA".
- **SQL의 Surface Form**:
  - 논리적으로 같지만 쿼리 문자열은 다를 수 있음.
  - **예시 (MAX 구하기)**: ① Ascending 정렬 → 맨 뒤, ② Descending 정렬 → 맨 앞 — 둘 다 논리적 equivalent.
- **Execution Accuracy 정의**:
  - String 1:1 매치가 아니라 **직접 돌려본 결과 같으면 OK**.
  - 가장 실용적인 메트릭.
  - 한계: 결과가 binary이면 로직이 엉뚱해도 우연히 맞을 수 있음.
- **출제 포인트**:
  - 이미 공개된 11문제 중 SQL evaluation 자체를 직접 묻는 문제가 없으므로, 남은 3문제 후보로 충분히 가능하다.
  - Chase-SQL 문제의 소문항으로 들어가도 좋고, 짧은 독립 설명 문제로도 출제하기 좋다.

### E. ⭐⭐ **Cross-lingual NLP (Orthogonal Procrustes + Isomorphism)** (랩업 §18)

- **Orthogonal Procrustes**: 단어 벡터 분포에 **Linear Transformation** (회전 / 대칭)을 적용해, 언어 간 같은 의미 단어가 임베딩 공간에서 가까이 매핑되도록 학습.
- **Isomorphism**: 의미가 비슷한 단어 벡터들은 **다른 벡터들과 이루는 관계도 비슷하다**는 가정. (e.g., 언어 X든 Y든 "왕"은 "여왕"과 가깝고 "책"보다는 멀다.)
- **다국어 LLM**: 여러 언어 한꺼번에 학습 시 이 정렬 과정이 **암묵적으로** 수행됨.

### F. ⭐⭐ **Lemmatization vs Stemming + Normalization** (랩업 §14.1, 14.2)

- 강사 멘트: **"중간고사 단골 출제 포인트"**. 학교 코스에서 자주 나옴.
- **Normalization 필요성**: 다양한 surface form 존재 → 통일 필요.
- **방식**: Case folding, Lemmatization, Stemming.

| 항목 | Lemmatization | Stemming |
|------|---------------|----------|
| 방식 | **사전 + 언어학적 지식 / 형태소 분석** | **도끼로 자르듯** 룰 베이스 |
| 정밀도 | 높음 | 단순 |
| 예시 | "running" → "run" (사전 기반) | 앞뒤 어미 떼서 치환 |
| 비용 | 비쌈 | 가벼움 |

### G. ⭐⭐ **Add-One (Laplace) Smoothing** (랩업 §15.2)

- **문제**: 관찰 안 된 단어가 하나만 있어도 카운트 곱셈에서 **전체 확률 0** 발생.
- **처방**: **분자 +1, 분모 +|V|** (Vocabulary 크기).
  $$P_{\text{Laplace}}(w_i \mid w_{i-1}) = \frac{C(w_{i-1}, w_i) + 1}{C(w_{i-1}) + |V|}$$
- 모든 단어가 한 번씩 관측된다고 간주하는 효과.

### H. ⭐⭐ **Pass@K + Code Metric** (랩업 §21)

- **Code Metric**:
  - 일반 QA: 토큰 겹치면 좋음.
  - **코드는 그렇지 않음** — Syntax / Semantics 고려 필요.
  - **CodeBLEU**: syntax tree / data-flow까지 고려한 BLEU의 확장.
- **Pass@K**:
  - **Execution 기반**: 코드가 실제로 잘 돌아가는지를 본다.
  - "K번 샘플링 중 하나라도 정답이면 통과"의 확률.
  - **Unbiased Estimator**:
    $$\text{pass@}k = \mathbb{E}_{\text{problems}}\left[1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}\right]$$
    - `n`: 문제당 생성한 총 샘플 수
    - `c`: 그 중 정답인 샘플 수
    - `k`: pass@k의 k
  - 이 식은 분산이 작아서 안정적 추정 가능.

### I. ⭐⭐ **RL Action 단위 + Reward Shaping + Continuation 평가** (랩업 §1)

- **RL의 액션 단위는 토큰만이 아님**: 토큰 / 툴 콜 / 트래젝토리 전체 모두 가능. MDP는 sequential decision making이므로 단위는 자유.
- **Reward Shaping의 부정적 효과**:
  - Outcome reward + 잘 만든 shaped reward 조합이 좋아 보이지만,
  - **너무 인위적인 shaped reward는 방해**가 됨.
  - 예: Prefix BLEU shaping → marginal gain만. 이유: 초반에 다른 토큰 만드는 게 크게 페널티 → 모델 위축 / 다양성 억제.
- **대안 - Continuation 기반 평가**:
  - Prefix에서 **continuation을 여러 번 만들어 봄**.
  - 괜찮은 prefix는 이어서 좋은 결과가 더 자주 나옴.
  - 비용 절감: **Beam Search 트리에서 이미 공짜로 만든 continuation 활용**.
  - 결론: 이 아이디어가 **MCTS / Inference-time search**와 연결됨.
- **출제 포인트**:
  - 공개 11번 GRPO가 RLHF 식을 이미 물었지만, 랩업 첫 부분의 **액션 단위 / reward shaping 부작용 / continuation 평가**는 아직 직접 커버되지 않았다.
  - 단독 긴 문제보다는 MCTS/Alpha-SQL 문제의 앞부분 소문항으로 묶여 나올 가능성이 높다.

## 예상 1. Knowledge Graph Embedding + Relation Extraction

Answer the following questions regarding Knowledge Graph construction and embedding.

**(1)** State the three main approaches to Relation Extraction discussed in class.

> 수업에서 다룬 Relation Extraction의 세 가지 주요 접근법을 서술하라.

**(2)** Briefly explain **Snowballing** in Relation Extraction.

> Relation Extraction에서 Snowballing이 무엇인지 간단히 설명하라.

**(3)** List the full NLP pipeline that takes raw text and produces a Knowledge Graph.

> 원본 텍스트에서 시작해 Knowledge Graph를 만드는 전체 NLP 파이프라인을 나열하라.

**(4)** In TransE-style Knowledge Graph Embedding, what is the core constraint imposed on `(head, relation, tail)` vectors? Write down the **margin loss** used for training.

> TransE 류의 Knowledge Graph Embedding에서 (head, relation, tail) 벡터에 부과되는 핵심 제약은 무엇인가? 학습에 사용되는 **margin loss**를 식으로 적어라.

**(5)** State one type of question that **GraphRAG** can answer but a plain Vector RAG struggles with.

> GraphRAG가 답할 수 있지만 단순 Vector RAG로는 어려운 질문 유형을 한 가지 들어라.

### Answer

**(1)** Three approaches:
- **Rule-based**: Hearst Pattern + Regular Expression. (e.g., `"X such as Y, Z"` 패턴으로 hyponymy 관계 추출)
- **Supervised**: Logistic Regression Classifier 등의 분류기를 학습.
- **Distant Supervision**: 외부 사전(Knowledge Base)에 있는 트리플이 텍스트에 등장하면 그 문장을 positive 예시로 자동 라벨링.

**(2)** Snowballing은 **부트스트래핑**의 일종. 초기 사전 → 그 사전을 이용해 텍스트에서 새로운 릴레이션 추출 → 추출한 트리플로 사전을 키움 → 키운 사전으로 더 많은 릴레이션 추출 — 이 과정을 반복하여 점진적으로 KB를 확장한다.

**(3)** Tokenization → POS Tagging → NER (Named Entity Recognition) → Entity Linking → Relation Extraction → Triple `(head, relation, tail)` → Graph 구성.

**(4)** 핵심 제약: **`head + relation = tail`** (벡터 덧셈으로 관계 표현).

Margin loss (TransE 표기):

$$\mathcal{L} = \sum_{(h,r,t)\in S} \sum_{(h',r,t') \in S'} \left[\gamma + d(h+r,\ t) - d(h'+r,\ t')\right]_{+}$$

- `S`: positive 트리플 집합, `S'`: corrupted (negative) 트리플 집합.
- `d(·, ·)`: 거리 함수 (L1 또는 L2 norm).
- `γ`: margin 하이퍼파라미터.
- `[x]_+ = max(0, x)`: hinge loss.

즉 positive 트리플에 대해서는 `d(h+r, t)`가 작아지고, negative 트리플은 적어도 margin γ 이상 멀어지도록 학습.

**(5)** **Multi-hop reasoning 질문** 또는 **집계(aggregation) 질문**.
예: "A의 친구의 친구 중 의사인 사람은?" — 여러 hop의 릴레이션을 따라가야 하므로 단순 vector similarity로는 풀 수 없지만, 그래프 위에서는 자연스럽게 답할 수 있다.

## 예상 2. MCTS for SQL Generation (Alpha-SQL)

Consider Monte Carlo Tree Search (MCTS) applied to SQL generation in **Alpha-SQL** (ICML 2025).

**(1)** In Alpha-SQL, what is defined as a **node** and what is defined as an **edge** of the search tree?

**(2)** Why can we **not** directly use the correctness of a generated SQL query as the reward in MCTS? What is used **instead** as the **Self-Supervised Reward**, and how is it computed?

> 생성된 SQL 쿼리의 정답 여부를 MCTS의 reward로 **직접** 사용할 수 **없는** 이유는 무엇인가? 그 대신 **Self-Supervised Reward**로 무엇을 사용하며, 어떻게 계산하는가?

**(3)** Name the **four stages** of MCTS and briefly describe each.

> MCTS의 **네 단계**를 나열하고 각각을 간단히 설명하라.

**(4)** Write down the **UCB1** formula used during the Select phase. Explain the role of the first term and the second term.

**(5)** During Backpropagation, MCTS uses the **same name** as the backpropagation in neural networks. State whether the underlying computation is the same or different, and briefly justify.

> MCTS의 Backpropagation은 뉴럴넷의 backpropagation과 **이름이 같다**. 두 연산이 실제로 같은지 다른지 답하고, 간단히 이유를 설명하라.

**(6)** Before introducing MCTS, the lecturer discussed reward shaping and continuation-based evaluation. Why can prefix-level reward shaping such as Prefix BLEU hurt generation, and how does continuation-based evaluation address this?

> MCTS를 소개하기 전에 reward shaping과 continuation 기반 평가가 언급되었다. Prefix BLEU 같은 prefix-level reward shaping이 왜 생성에 악영향을 줄 수 있으며, continuation 기반 평가는 이를 어떻게 보완하는가?

### Answer

**(1)**
- **Node** = **Partial Reasoning Step** = 부분적으로 완성된 SQL 상태.
- **Edge** = **Action** = SQL 변형 오퍼레이션. (e.g., 어떤 column을 SELECT 할지, 어떤 테이블에서 FROM 할지 등 — SELECT/FROM/WHERE 같은 절 단위가 액션이 될 수 있음)
- Leaf node = 완성된 SQL 또는 곧 샘플링 가능한 상태.

**(2)** 정답을 안다면 그냥 그걸 제출하면 되므로, **테스트 타임에는 정답을 알 수가 없다**. 따라서 정답 여부를 reward로 직접 쓸 수 없다.

대용으로 사용하는 것은 **Consistency (자기 일관성)**:
- 완성된 partial SQL 상태에서 **temperature ≠ 0** 으로 여러 번 샘플링.
- 샘플된 SQL들을 실행한 결과가 얼마나 일관되는지를 측정.
- **예**: 5번 샘플링 중 3번이 같은 결과 → reward = 3/5 = **0.6**.

(직관: 모델이 confidence가 높을수록 일관된 답을 내놓는다는 prior에 기반.)

**(3) MCTS 4단계**:

| 단계 | 설명 |
|------|------|
| **Select** | 트리의 현재 위치에서 어떤 자식 노드로 갈지 **UCB1**로 선택. 리프 노드에 도달할 때까지 반복. |
| **Expand** | 리프 노드에서 LLM에 다음 액션 후보(K개)를 제안받아 자식 노드들로 트리 확장. |
| **Simulate** | 확장된 노드에서 SQL이 완결될 때까지 따라가 reward (Self-supervised consistency)를 관찰. |
| **Backprop** | 관찰된 reward를 경로 위 모든 부모 노드까지 카운트 누적으로 전파. |

**(4) UCB1**:

$$UCB1(n) = \bar{R}_n + c \cdot \sqrt{\frac{\ln N_{\text{parent}}}{N_n}}$$

- `R̄_n`: 노드 n까지의 평균 reward (= value_sum / visit_count).
- `N_n`: 노드 n의 방문 횟수.
- `N_parent`: 부모 노드의 방문 횟수.
- `c`: exploration constant.

**역할**:
- **첫 번째 항 `R̄_n`** = **Exploitation**. 지금까지 reward가 높았던 노드를 우선 선택.
- **두 번째 항** = **Exploration**. 부모 대비 덜 방문된 노드일수록 가중치를 더해, 새로운 경로 탐색을 장려.

**(5) 다르다.** 이름은 같지만 **컨셉만 유사**할 뿐 **수식적으로는 거의 무관**하다.
- 뉴럴넷의 backprop = chain rule을 통한 gradient 계산.
- MCTS의 backprop = 단순히 경로 따라 올라가며 **카운트 누적**:
  - `node.visit_count += 1`
  - `node.value_sum += reward`
- 두 연산 모두 "끝 정보를 위로 전파"한다는 점만 비슷하다.

**(6)** Prefix BLEU 같은 shaped reward는 정답 prefix와 조금만 달라도 초반 토큰에 큰 페널티를 줄 수 있다. 그러면 모델이 다양한 경로를 탐색하지 못하고 보수적으로 위축되어, 실제 최종 품질 개선은 marginal gain에 그칠 수 있다.

Continuation 기반 평가는 prefix 자체를 즉시 점수화하지 않고, 그 prefix에서 여러 continuation을 생성해 본다. 좋은 prefix라면 뒤이어 좋은 완성 결과가 더 자주 나오므로, continuation의 품질/일관성을 통해 prefix를 더 안정적으로 평가할 수 있다. Beam Search 과정에서 이미 생성된 prefix tree와 continuation을 활용하면 추가 비용도 줄일 수 있고, 이 아이디어가 MCTS 및 inference-time search와 연결된다.

## 예상 3. Chase-SQL Pipeline (Multi-Path Reasoning)

**Chase-SQL** (ICLR 2025) is a workflow for text-to-SQL generation. Answer the following.

**(1)** The Chase-SQL pipeline has four major stages. Name them in order.

> Chase-SQL 파이프라인의 네 가지 주요 단계를 순서대로 나열하라.

**(2)** Chase-SQL uses three different **generators** to produce SQL candidates. Briefly explain the core idea of each:
- DC-CoT
- QP-CoT
- OS

> Chase-SQL은 SQL 후보를 만들기 위해 세 가지 **generator**를 사용한다. 각각의 핵심 아이디어를 간단히 설명하라:
> - DC-CoT
> - QP-CoT
> - OS

**(3)** The final stage uses **Pairwise Selection** (tournament-style) instead of, e.g., comparing all three candidates at once. State the **rationale** that the lecturer gave for choosing pairwise.

> 마지막 단계에서 세 후보를 한 번에 비교하지 않고 **Pairwise Selection** (토너먼트 방식)을 사용하는 이유를 강사가 제시한 대로 답하라.

**(4)** A simple baseline for selecting the final SQL among candidates is **majority voting on execution results**. Explain why string-level majority voting is **not** suitable for SQL, and state **one limitation** of execution-based majority voting.

> 후보 SQL 중 최종 답을 고르는 단순한 베이스라인은 **실행 결과에 대한 majority voting**이다. SQL에 대해 string-level majority voting이 **부적합한** 이유를 설명하고, 실행 결과 기반 majority voting의 **한계 하나**를 답하라.

**(5)** Define **Execution Accuracy** in text-to-SQL evaluation. Why is it more appropriate than exact string match, and what is one limitation?

> Text-to-SQL 평가에서 **Execution Accuracy**를 정의하라. 이것이 exact string match보다 적절한 이유와 한계 하나를 설명하라.

### Answer

**(1) 4 stages**:
1. **Value Retriever** — DB terminology (column 이름, 약어 등)을 자연어 질문에 매핑.
2. **3 Generators** — DC-CoT, QP-CoT, OS 세 방식으로 SQL 후보들을 생성.
3. **Query Fixer** — 에러 트레이스를 보고 안 돌아가는 SQL을 수정.
4. **Pairwise Selection** — 후보들을 토너먼트로 비교하여 최선의 쿼리 선택.

**(2)**
- **DC-CoT (Divide-and-Conquer CoT)**: 복잡한 질문을 **여러 작은 sub-question으로 분해** → 각 sub-question에 대한 작은 SQL 생성 → 합쳐서 최종 SQL 구성. (GraphRAG의 community 분해와 유사한 패턴.)
- **QP-CoT (Query Execution Plan CoT)**: 질문을 나누는 게 아니라 **reasoning 단계를 명확히 3가지로 분리**:
  ① 어떤 테이블에서 가져올지 → ② 어떤 오퍼레이션 (JOIN, GROUP BY 등) 적용할지 → ③ 결과를 어떻게 정리할지.
  논문에서는 이를 "Inductive Bias를 준다"고 표현.
- **OS (Online Synthetic example)**: Few-shot prompting이 zero-shot보다 잘하므로 **few-shot 예시 자체를 LLM이 즉석에서 생성**한다. 학습 데이터 없이 schema만 주고 (Question, SQL) 페어를 만들게 한 뒤, 그것을 few-shot 예시로 본 쿼리의 프롬프트에 삽입.

**(3) Pairwise를 쓰는 이유**: 단순한 **design choice**. 세 개씩(또는 그 이상) 동시에 비교하면 리뷰어가 "왜 3개죠?"라는 의문을 가지게 되어 justification이 어렵다. **Pairwise가 가장 미니멀한 비교**이기 때문에 default로 채택한 것. (정확도 향상이라는 본질적 이유가 따로 있는 것은 아님.)

**(4)**
- **String-level majority voting이 부적합한 이유**: SQL은 surface form이 매우 다양함. 논리적으로 동일한 쿼리도 ascending 정렬 + LIMIT 1로 MAX를 구하느냐, descending 정렬 + LIMIT 1로 구하느냐 등 string은 다를 수 있어, string 비교로는 동치성을 판단할 수 없다.
- **실행 결과 기반 majority voting의 한계**: 모델이 **잘못된 bias**를 가지고 있다면, 잘못된 답이 우세하게 뽑힐 수 있다. (다수결이 항상 정답을 보장하지 않음.) 그래서 Chase-SQL은 더 발전된 Pairwise Selection을 도입.

**(5)** Execution Accuracy는 생성된 SQL 문자열이 gold SQL과 글자 단위로 같은지를 보는 것이 아니라, **실제로 DB에서 실행했을 때 결과가 gold query의 실행 결과와 같은지**를 보는 평가 지표다.

Exact string match보다 적절한 이유는 SQL에는 같은 의미의 여러 surface form이 있기 때문이다. 예를 들어 최대값을 구할 때 ascending 정렬 후 마지막 값을 고르거나 descending 정렬 후 첫 값을 고르는 쿼리는 문자열은 다르지만 실행 결과는 같다.

한계는 실행 결과가 우연히 같을 수 있다는 점이다. 특히 결과가 binary이거나 작은 도메인에서는 로직이 틀렸는데도 정답과 같은 결과가 나올 수 있다.

## 예상 4. Cross-lingual NLP — Orthogonal Procrustes & Isomorphism

**(1)** Define the **Isomorphism assumption** that underlies cross-lingual word embedding alignment.

> Cross-lingual word embedding 정렬의 기반이 되는 **Isomorphism 가정**을 정의하라.

**(2)** What is **Orthogonal Procrustes** and what kind of transformations does it apply to the embedding space?

> **Orthogonal Procrustes**는 무엇이며, 임베딩 공간에 어떤 종류의 변환을 적용하는가?

**(3)** Modern multilingual LLMs (e.g., trained on text from many languages simultaneously) do not explicitly perform Procrustes alignment. Explain briefly why their multilingual capability can still be understood **through the same lens**.

> 최신 다국어 LLM은 명시적으로 Procrustes 정렬을 수행하지 않는다. 그럼에도 그들의 다국어 능력이 **같은 관점**에서 이해될 수 있는 이유를 간단히 설명하라.

### Answer

**(1) Isomorphism 가정**: 의미가 비슷한 단어들의 벡터는 **다른 단어 벡터들과 이루는 관계 구조 또한 유사하다**. 즉, 단어들의 상대적 위치 관계(거리/방향)가 언어 X와 언어 Y 사이에서 비슷한 패턴을 형성한다.

예시: "왕(king)"을 나타내는 벡터는, 어떤 언어로 임베딩해도 **"여왕(queen)"과 가깝고 "책(book)"보다는 멀다**. 이런 상대적 관계가 언어 간 보존되므로, 두 임베딩 공간을 적절히 정렬하면 의미상 대응되는 단어들이 가깝게 매핑된다.

**(2) Orthogonal Procrustes**: 한 임베딩 공간을 다른 임베딩 공간에 가장 잘 정렬되도록 **직교 행렬(orthogonal matrix)** `W` (즉 `WᵀW = I`)를 찾는 방법. 직교 변환은 **회전(rotation)** 과 **반사/대칭(reflection)** 만 허용 — 벡터의 norm과 내적이 보존된다.

소스 언어 벡터들 `X`와 타겟 언어 벡터들 `Y`가 주어졌을 때:

$$W^* = \arg\min_{W^\top W = I} \|XW - Y\|_F$$

이를 통해 두 언어의 단어 벡터 분포를 정렬한다.

**(3)** 다국어 LLM은 여러 언어의 코퍼스를 **하나의 모델 파라미터 공간**에서 함께 학습하므로, 모델이 **암묵적으로(implicitly)** 의미가 비슷한 단어들을 유사한 representation으로 매핑하게 된다. Procrustes 같은 **명시적 선형 변환**은 안 하지만, 학습 과정 자체가 **동일한 정렬 효과**를 내부적으로 수행하는 것과 마찬가지다. 결국 isomorphism 가정이 모델 내부에서 자연스럽게 실현된 형태로 볼 수 있다.

## 예상 5. Normalization, Lemmatization vs Stemming, Add-One Smoothing

**(1)** Explain why **normalization** is needed in text preprocessing, and list two common normalization techniques besides case folding.

> 텍스트 전처리에서 **normalization**이 필요한 이유를 설명하고, case folding 외에 흔히 쓰이는 normalization 기법 두 가지를 나열하라.

**(2)** Explain the key differences between **Lemmatization** and **Stemming** in terms of methodology, output quality, and computational cost.

> **Lemmatization**과 **Stemming**의 차이를 방법론, 출력 품질, 계산 비용 측면에서 설명하라.

**(3)** Consider a bigram language model trained on a small corpus. Some bigrams are unobserved and have count 0, which causes the entire sentence probability to become 0. Write the **Add-One (Laplace) Smoothing** formula for a bigram and briefly explain its intuition.

> 작은 corpus에서 학습된 bigram language model을 생각하자. 일부 bigram이 관측되지 않아 count가 0이면 문장 전체 확률이 0이 된다. 이를 해결하는 **Add-One (Laplace) Smoothing** 식을 bigram에 대해 쓰고, 그 직관을 간단히 설명하라.

**(4) Numerical exercise.** Suppose the corpus consists of these three sentences:

```
<s> the cat sat </s>
<s> the dog ran </s>
<s> the cat ran </s>
```

The vocabulary is `V = {the, cat, dog, sat, ran, <s>, </s>}` (so `|V| = 7`).

Compute `P_Laplace(ran | dog)` using Add-One Smoothing.

> 위 코퍼스와 어휘를 가정하고, Add-One Smoothing을 적용해 `P_Laplace(ran | dog)`를 계산하라.

### Answer

**(1)** 같은 의미가 **여러 surface form**으로 나타날 수 있어 (e.g., "USA" vs "United States", "running" vs "run") 모델이 이들을 별개 단어로 취급하면 학습 효율과 정확도가 떨어진다. Normalization은 이런 변형들을 표준 형태로 통일한다.

흔한 기법: **Lemmatization**, **Stemming**, (그 외 punctuation removal, accent stripping 등).

**(2)**

| 항목 | Lemmatization | Stemming |
|------|---------------|----------|
| 방법론 | **사전(dictionary) + 언어학적 지식(형태소 분석)** 기반 | 단순 **규칙(rule)** 기반. 어미를 "도끼로 자르듯" 제거 |
| 출력 품질 | 정확한 표제어(lemma) 반환. e.g., "ran" → "run" | 어형 변화의 흔적은 지우지만 **실제 단어가 아닐 수도** 있음. e.g., "running" → "runn", "studies" → "studi" |
| 비용 | 비교적 비쌈 (사전 lookup, POS 정보 필요) | 가벼움 (단순 정규식/룰) |

**(3) Add-One Smoothing (bigram)**:

$$P_{\text{Laplace}}(w_i \mid w_{i-1}) = \frac{C(w_{i-1}, w_i) + 1}{C(w_{i-1}) + |V|}$$

**직관**: 모든 단어 쌍이 **최소 한 번씩은 관측된다고 가정**. 분자에 `+1`을 더해 unseen bigram이 0이 되지 않게 하고, 확률의 합을 1로 유지하기 위해 분모에 `+|V|`을 더한다. (즉, `w_{i-1}` 뒤에 올 수 있는 단어가 |V|가지 있고, 각각이 한 번씩은 등장했다고 간주.)

**(4) 계산**:

- `C(dog, ran)` = 1 (`<s> the dog ran </s>`에서 한 번 등장)
- `C(dog)` = 1 (전체 코퍼스에서 "dog"가 한 번 등장)
- `|V|` = 7

$$P_{\text{Laplace}}(\text{ran} \mid \text{dog}) = \frac{C(\text{dog, ran}) + 1}{C(\text{dog}) + |V|} = \frac{1 + 1}{1 + 7} = \frac{2}{8} = \frac{1}{4} = \mathbf{0.25}$$

(참고: smoothing 없이 MLE로 계산하면 `1/1 = 1.0`이 되어 과대평가됨. Laplace는 평탄화 효과로 0.25까지 낮춤.)

## 예상 6. Pass@K Unbiased Estimator + Code Metrics

**(1)** Explain why **token-overlap-based metrics** (such as BLEU) are not directly suitable for evaluating code generation, and name one metric specifically designed for code that addresses this.

> **토큰 중첩 기반 메트릭** (BLEU 등)이 코드 생성 평가에 부적합한 이유를 설명하고, 이를 보완하기 위해 코드 전용으로 제안된 메트릭 하나를 답하라.

**(2)** State the conceptual definition of **Pass@K** for code generation.

> 코드 생성에서 **Pass@K**의 개념적 정의를 서술하라.

**(3)** A naive way to estimate Pass@K is: "generate K samples, check if at least one passes". Why is this estimator **high-variance** when k is small relative to n? Write the **unbiased estimator** of Pass@K used in practice.

> Pass@K를 단순하게 "K개 샘플링해서 그 중 하나라도 맞으면 통과"로 추정하면 k가 n에 비해 작을 때 **분산이 큰** 이유는 무엇인가? 실무에서 사용되는 **unbiased estimator** 식을 쓰라.

**(4) Numerical exercise.** For a single problem, you generate `n = 10` samples, and `c = 4` of them pass the test cases. Compute **pass@1**, **pass@5**, and **pass@10** using the unbiased estimator.

> 한 문제에 대해 `n = 10` 샘플을 생성했고 그 중 `c = 4`개가 테스트를 통과했다. Unbiased estimator를 사용해 **pass@1**, **pass@5**, **pass@10**을 계산하라.

### Answer

**(1)** 코드는 **syntax**(구문)과 **semantics**(의미)가 핵심이며, 변수명을 바꾸거나 동치 표현으로 리팩토링해도 동작은 동일하다. 반대로 토큰이 1개만 달라도 (e.g., `==` → `=`) 의미가 완전히 바뀐다. 따라서 **token-overlap만 보는 BLEU**류는 코드 정확성을 측정하지 못한다.

**대안 메트릭**: **CodeBLEU** (BLEU에 AST/data-flow 비교를 더해 syntax/semantics를 고려). 또는 **Execution 기반 Pass@K**.

**(2) Pass@K**: 한 프로그래밍 문제에 대해 모델이 **K개의 샘플을 생성**했을 때, 그 K개 중 **하나라도** 테스트 케이스를 모두 통과하는 정답인 확률.

여러 문제로 평균 내어 모델의 종합 성능을 나타낸다. Execution 기반이므로 **코드가 실제로 정확히 동작하는지**를 측정한다.

**(3)** 단순 추정 (`K개 뽑아 → 통과 여부 확인`)을 한 번만 하면, 운에 따라 결과가 크게 달라져 분산이 매우 크다. 특히 정답률이 낮을수록 단순 K개 샘플로는 정답을 못 만날 가능성이 높다.

**Unbiased Estimator** (n번 샘플 중 c번 정답일 때):

$$\text{pass@}k = 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}$$

- `n`: 한 문제당 총 샘플 수.
- `c`: 그 중 정답인 샘플 수.
- `k`: 평가하려는 K.
- 의미: "k개를 뽑았을 때 그 안에 정답이 **하나도 없을** 확률"의 여집합. 즉 `1 - P(모두 오답)`.

이 식은 같은 n개의 샘플로 여러 k에 대해 계산할 수 있고, 분산도 작아 안정적이다.

**(4) 계산** (n = 10, c = 4):

$$\binom{n}{k} = \binom{10}{k},\qquad \binom{n-c}{k} = \binom{6}{k}$$

- **pass@1** = $1 - \frac{\binom{6}{1}}{\binom{10}{1}} = 1 - \frac{6}{10} = \mathbf{0.4}$

- **pass@5** = $1 - \frac{\binom{6}{5}}{\binom{10}{5}} = 1 - \frac{6}{252} = 1 - \frac{1}{42} \approx \mathbf{0.976}$

- **pass@10** = $1 - \frac{\binom{6}{10}}{\binom{10}{10}}$. 그런데 $\binom{6}{10} = 0$ (k > n-c). 따라서

  pass@10 = $1 - 0 = \mathbf{1.0}$

  (직관: 10개 모두 뽑으면 정답 4개를 반드시 포함하므로 통과 확률 1.)

## 부록 2. 남은 3문제 최종 베팅

| 우선순위 | 남은 3문제 후보 | 반드시 외울 것 |
|---|---|---|
| 1 | **Lemma/Stemming + Laplace** | Lemma vs Stem 차이, Add-One 식 `(count+1)/(context count+|V|)`, 간단 계산 |
| 2 | **KG/Relation Extraction** | RE 3방법, Snowballing, KG 파이프라인, `h + r = t`, margin loss |

- 공개 11문제와 가장 덜 겹치면서 랩업 별표가 강한 것은 **KG/RE**, **MCTS/Alpha-SQL**, **Lemma/Stemming + Laplace**다.
