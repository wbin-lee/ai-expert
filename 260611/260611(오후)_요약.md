# 1. AI에 대한 관점 — 모든 문제는 모델링이다 ⭐

## 1.1 모델 = 함수

- 과학/공학의 거의 모든 문제는 **모델링 이슈**로 요약된다: 입력이 주어지면 출력을 주는 **함수**를 만드는 일.
- 조지 박스(George Box): "모든 모델은 틀렸다, 그러나 일부는 쓸모 있다" — 얼마나 **쓸모 있는** 모델을 만드느냐의 문제.
- 물리의 대표 모델: \(F = ma\) — (질량, 가속도) 입력 → 힘 출력.
- 컴퓨터 과학의 모델 = **알고리즘**. 대표 예: 네비게이션 시스템 — (출발지, 도착지) 입력 → 경로 출력. Dijkstra의 최단경로 알고리즘.

## 1.2 진짜 어려운 건 "피처 발견" ⭐

- \(F = ma\)에서 m과 a를 **곱하는 관계**를 알아내는 것보다, 힘을 정의하려면 **M과 A라는 두 피처가 필요하다는 것** 자체를 알아내는 게 더 어렵다.
- 모든 문제에서 핵심: **"내가 뭘 보고 판단해야 하는가"라는 피처를 찾는 일** — 기존엔 사람이 머리 써서 찾았다.

## 1.3 이미지 분류는 왜 안 풀렸나 ⭐

- 고양이 사진 분류: 사람에게는 너무 쉽지만, 컴퓨터 비전에서는 "평생 해도 못 푼다"던 문제 (강사 박사과정, 20년 전 일화).
- 이유: 이미지 = 엄청나게 많은 숫자(픽셀)의 집합인데, 고양이의 포즈/생김새/occlusion 등 **variation이 무한대**.
  - 바둑의 경우의 수가 "우주의 원자 수보다 많다"지만, 고양이를 만드는 픽셀 조합은 **무한대**.
- 룰/특징을 명시적으로 기술하는 알고리즘으로는 고양이 하나도 설명 불가 → 1990년대 머신러닝(데이터 활용) 등장 → **딥러닝에서 빵 터짐**.

## 1.4 딥러닝의 본질 = Representation Learning ⭐⭐

- 딥러닝의 철학: "규칙은 모르겠고, **입력과 출력(데이터와 레이블)을 엄청나게 많이 보여줄 테니 네가 알아서 풀어봐**."
- 딥 뉴럴 네트워크의 파워: 여러 레이어를 통해 **피처를 스스로 학습** — 문제 해결에 near-optimal한 피처를 자동으로 찾는 것이 AI 시대의 핵심.
- 트랜스포머도 마찬가지: attention 레이어를 쌓으며 representation을 계속 변환 → 마지막에 가장 좋은 feature를 만들어냄.

## 1.5 사례: 야구 스트라이크/볼 판정 CNN ⭐

- 전통적 컴퓨터 비전 접근: 야구공 detection → tracking → 포수 미트 도달 순간의 위치로 스트라이크/볼 판정 (생각보다 잘 안 됨).
- 강사 연구실 접근: 지난 7년치 한국 프로야구 중계 영상 다운로드, 스트라이크 영상 ~1만 개 + 볼 영상 ~1만 개를 직접 레이블링 → CNN에 학습.
  - 정확도 **약 97%**, 나머지 3% 에러는 대부분 **심판의 오심** (AI가 맞고 심판이 틀린 경우).
- 시각화 실험: 네트워크가 어디를 보고 판정하는지 보니 — 공도, 포구 순간도 아닌 **전광판의 스트라이크/볼 표시등**에 attention이 집중.
  - 모델이 데이터에 의존해 **치팅**을 한 것: "딴 거 볼 필요 없이 전광판만 보면 거의 100% 맞출 수 있다"는 최적 피처를 스스로 발견.
- 교훈: 딥러닝은 규칙을 하나도 설명해주지 않아도 입력/출력만 많이 보여주면 (블랙박스이긴 하지만) **최적의 피처와 representation을 찾아낸다**.

## 1.6 아키텍처보다 데이터 ⭐

- 아키텍처 흐름: MLP → Deep NN → CNN → Transformer. 현재는 트랜스포머의 시대.
  - Mamba 같은 State Space Model이 떠올랐으나 단독으로는 부족 → Jamba/Samba 류 **하이브리드 모델** 등장. 그래도 attention이 완벽한 대세.
- 본질은 여전히 **데이터**: 데이터를 많이 보여주면 피처를 기가 막히게 찾고, 그 피처는 파운데이션 모델처럼 다양한 태스크에 갖다 붙일 수 있음.
- 메타(Meta) 방문 일화: 라지 스케일 모델 개발 시 **아키텍처는 거의 안 건드리고**, 비디오 퍼셉션 팀이 **일주일에 10억 원 이상을 데이터 레이블링에** 투입.
- 비유: 모델 capacity ↑ = 뇌세포 많은 모델, 데이터 ↑ = 문제집 많이 푸는 것 → 강력해질 수밖에 없음.
- Claude 사용 경험: 엑셀 양식 채우기 같은 작업까지 알아서 해내는 것은 트랜스포머 아키텍처만으로는 설명 불가 → Anthropic/OpenAI에는 **상상 이상의 데이터**가 들어갔을 것.
- 산업 현장의 보틀넥: raw 입력 데이터는 많은데 **출력(레이블) 데이터가 없음** → "무식하게 레이블을 다 다는 게 정답일 수도 있다."

# 2. 딥러닝 기반 Auto White Balance ⭐⭐

## 2.1 문제 정의

- **White Balance(Color Constancy)**: 조명의 색을 맞춰 **하얀 물체가 하얀색으로 보이도록** 하는 것.
- 애매한 케이스: 무대 조명(노란 조명 아래 흰 옷) — 노란빛을 제거하면 공간의 분위기도 사라짐 → 설계 정의 자체가 복잡.
- 블루/골드 드레스 논란도 조명 인지와 관련 — **물리적인 색과 사람이 인지하는 색은 다르다**.
- 파이프라인상 위치: 센서 RGB → XYZ 변환 전에 **White Balance가 선행**되어야 함.
- 보정 자체는 쉬움: 조명색을 알면 **조명색으로 나눠주기만** 하면 됨. 어려운 것은 **이미지 한 장에서 조명색을 추정**하는 것.
  - 고전 알고리즘: White Patch, Gray World 등.

## 2.2 데이터셋과 출력 형태

- 데이터셋: 이미지 + 조명색 ground truth. 이미지 안의 **컬러 차트(패치)**에서 값을 읽어 레이블로 사용.
- GT는 RGB 3개가 아니라 **2개의 숫자**: 녹색 채널을 1로 고정하고 나누기 때문 (조명의 밝기가 아니라 순수 **컬러만의 문제** → 2채널 게임).

## 2.3 CNN Regression의 실패 → Classification으로 전환 ⭐⭐

- 단순 접근: 입력 이미지 → CNN → 2개의 실수 출력 (regression). **학습해보니 성능이 생각보다 별로**.
- 핵심 레슨: **CNN으로 실수를 바로 출력하는 regression은 잘 안 된다 → classification으로 먼저 만들고 거기서 regression 하라.** (이후 다른 태스크에서도 유사 연구 다수)

## 2.4 방법: K-means 클러스터링 + 확률 가중합 ⭐⭐

```
1) 데이터셋의 GT 조명값들을 K-means로 클러스터링 (예: K=20)
   → "세상에 존재하는 조명은 K개다"로 이산화(discretize)
2) CNN은 K개 조명 중 하나를 고르는 classification으로 학습
   → softmax로 클래스별 확률 출력
3) 최종 조명값 = Σ (클래스 확률 × 클러스터 센터 조명값)  ← weighted sum
   → 확률 가중합이 곧 regression이 됨
```

- K개로 제한하면 정확한 조명값을 못 맞출 것 같지만, softmax 확률로 클러스터 센터들을 **weighted sum** 하면 연속값 회귀가 가능 → 정확도가 훨씬 좋아짐.
- 언어 모델도 본질은 비슷: regression스러워 보여도 마지막은 **softmax 기반 classification**.

## 2.5 학습 디테일 — 전형적인 딥러닝 파이프라인

| 단계 | 내용 |
|---|---|
| Data Augmentation | 레이블(조명색)이 변하지 않는 변환만 적용: rotation, translation, scale 등 (**컬러를 바꾸지 않는** 변환) |
| Transfer Learning | ImageNet pre-trained 모델을 이 데이터에 fine-tuning |
| 백본 | 당시 **AlexNet** (지금은 아무도 안 쓰지만, 태스크 규모상 굳이 ViT까지 갈 필요도 없음) |

## 2.6 결과와 분석

- 정량: Gray World/White Patch 등 고전 기법, 기존 CNN 직접 regression 대비 **뚜렷한 성능 향상** (mean/median 에러 기준).
- 정성: prediction이 ground truth와 거의 일치.
- 필터 시각화: ImageNet 학습 필터 대비 fine-tuning 후 1번째/5번째 레이어 필터에 **컬러 성분이 더 많이 등장** → "컬러에 대해 학습하고 있다"는 주장.
- t-SNE 임베딩: 비슷한 조명의 이미지들이 2D 공간에서 모여 있음 → representation을 잘 학습.

## 2.7 "조명이 아니라 씬을 분류한 건 아닐까?" 검증 ⭐

- 우려: 조명과 씬은 커플링됨 (야외 = daylight, 실내 = 형광등) → 조명색이 아니라 **장면 종류**를 보고 분류했을 가능성.
- 검증 실험: **씬은 동일하게 두고 인위적으로(synthetic) 조명색만 입혀** 테스트 → 통과 → 조명색 자체를 학습하고 있음을 확인.
- 여기서도 아키텍처는 거의 고정 — **데이터와 문제 변환(classification 후 regression) 아이디어**가 기여의 핵심.

## 2.8 Failure Cases와 한계

- 사진/그림을 다시 찍은 경우 — 촬영물 자체가 입력이라 조명 추정이 어려움.
- **Multi-illuminant** (실내 + 야외 조명 혼합): 대부분의 WB 알고리즘은 "조명색이 하나"라는 가정 위에 있는데, 이 가정이 깨지면 작동 불가.
- 후속 연구: multi-illuminant 데이터셋 구축 및 공개, 이를 푸는 논문도 발표 (타 기관과 콜라보).

# 3. Q&A ⭐

## 3.1 Out-of-Distribution 조명

- 학습 분포에서 멀리 떨어진 생뚱맞은 조명이 들어오면 맞힐 가능성 희박.
- AI가 정말 잘하는 것 = 많은 데이터 분포 안에서의 **interpolation**. **extrapolation이 가능한지는 아직 명확한 답이 없다**.
- 신약 개발/신소재 등은 본 적 없는 영역으로 가야 하는 문제 — AI가 그걸 할 수 있는지는 미해결. "새로운 discovery를 가능케 하는 방법을 푸는 사람은 대박."

## 3.2 Multi-Illuminant의 출력 형태

- 싱글: 모델 출력이 RGB(조명색) 하나.
- 멀티: **픽셀별로 조명값**이 필요 — 두 조명이 픽셀마다 어떻게 섞이는지가 핵심.
  - 강사 연구의 출력: 조명 2개의 색 + **픽셀별 혼합 weight 맵**.
- 레이블링 난제: 컬러 차트 하나로는 픽셀별 GT를 만들 수 없음 → 수학적으로 픽셀별 레이블을 계산하는 방법 자체가 논문 기여. (최대 3개 조명 가정으로 데이터셋 수집 중)
- 연산이 복잡해져 핸드폰 등에는 아직 미탑재로 추정되나 중요한 이슈.

## 3.3 컬러 vs 밝기(쉐이딩)

- 색에서 **brightness와 color를 분리**해서 생각해야 함: xyY 컬러 스페이스에서 x, y는 밝기와 무관한 순수 컬러, Y가 brightness.
- WB/Color Constancy는 순수 **컬러만**의 문제 (2채널).
- 밝기/쉐이딩은 **지오메트리** 문제: 조명 방향과 surface normal의 dot product(각도)에 따라 brightness가 변함 → **Intrinsic Imaging** 등 별개 분야, 훨씬 복잡.

## 3.4 조명색의 직접 측정

- 조명의 RGB를 직접 재려면 특수 장비 필요 — **하이퍼스펙트럴 카메라**(약 300채널, 가시광선 전 파장 측정)로 조명의 스펙트럼을 측정 가능.
- 그러나 매우 복잡하고 고가 → 배보다 배꼽이 더 큼.

## 4.1 Module 1 — 이미지 센서와 Raw Data 이해

핵심 개념:

- **Optical Black(OB)**: 센서는 전기 신호라 노이즈가 필연 → 실제 이미지가 되는 Aperture Pixel Part 외에, 발열/기종별 노이즈 속에서 **"진짜 검은색" 값을 계산하기 위한 영역**(First/Second OB Pixel Part).
- 모노크롬 센서: 픽셀마다 luminance만 측정, interpolation 없이 그대로 반영.
- 컬러 센서: **RGGB 베이어 패턴** — 각 픽셀은 색상 1개만 획득, 여러 픽셀을 interpolation해 이미지 생성.

실습 내용 (캐논 5D Mark III raw + JPEG 사용):

```python
import rawpy
raw = rawpy.imread(path)   # raw_pattern, sizes(top_margin, left_margin, height, width) 등 활용
```

- 센서 전체 크기 vs OB 제외 **활성 영역** 확인 (top/left margin으로 계산). 이 카메라는 OB가 **좌상단**에 쏠려 있음.
- 활성 영역 min/max: JPEG는 8비트(max 256)인데 raw는 **14비트** → 훨씬 넓은 표현 범위. 사진작가가 raw로 작업하는 이유 = JPEG에서 날아간 정보를 복원 가능.
- raw는 RGB 채널 없는 **1채널 2D 이미지** → 그대로 시각화하면 회색 (2^14로 normalize).
- RGGB 패턴을 적용해 "카메라가 본 색"을 시각화.
- Naive 색 복원(R, B 그대로 + G 2개 평균)을 하면 **초록색이 지배적인 이미지** — G 값이 크게 측정되기 때문 → demosaicing 외에도 WB, 컬러 스페이스 변환, photo finishing 등 **ISP 파이프라인이 필요한 이유**를 눈으로 확인.

## 4.2 Module 2 — Black Level Subtraction ⭐

목적: **하드웨어 노이즈를 고려해 어떤 값이 검정색인지 파악**.

- OB 영역에서 Black Level 직접 계산: 테두리 전체 평균보다는(비네팅 영향 가능) 가장 두꺼운 **left OB 영역에 threshold를 주고 mean** — 정답은 아니고 robust하게 하기 위한 한 방식.
- 효과 비교: 단순 14비트 normalize → 회색빛 / **black level을 빼고 (white−black)으로 normalize** → 검은색이 제대로 표현됨.
- rawpy 제공값 활용:
  - `black_level` = (2047, 2047, 2047, 2047) — **RGGB 패턴별** black level (이론적으로 컬러 필터마다 다를 수 있음).
  - `white_level` ≈ 15000: 14비트 지원이라도 흰색이 2^14까지 못 가는 경우가 있음.
- **다이나믹 레인지 인사이트** ⭐: black 2047 ~ white 15000이면 실효 레인지가 2^14보다 좁음 → "이 카메라는 다이나믹 레인지가 안 좋다"는 평가가 스펙이 아닌 이런 값에서 나옴.
- 정규화: \( \text{norm} = (raw - black) / (white - black) \).

## 4.3 Module 3 — Demosaicing (Bilinear Interpolation) ⭐

목적: RGGB에서 각 픽셀의 빠진 색 채널을 **주변 픽셀 interpolation**으로 채우기.

- 알고리즘: 예) R 위치에서 G는 상하좌우 4개 G의 평균, B는 대각 4개 B의 평균.
- 구현 1: naive **3중 for문** (x, y, color) — 쉽지만 비효율.
- 구현 2: **`scipy.signal.convolve2d`로 병렬화** — 채널별 필터를 설계 (중앙이 해당 색이면 그대로, 아니면 이웃 가중 평균이 되도록 4, 1 등의 계수 + 보정값 구성).
- 성능 비교: 128×128 기준 for문 **124.5초** vs convolve2d **5.2초** — 같은 결과, 효율 대폭 향상.
- Bilinear의 한계 ⭐: low frequency 영역에서는 충분히 괜찮으나, **텍스처/엣지 등 high frequency 영역에서 blurry**해지고 거슬리는 선(artifact)이 생김 → 현업에서 더 정교한 demosaicing 연구/모델을 쓰는 이유.

## 4.4 Module 4 — Auto White Balance 구현 ⭐

목적: 화이트 밸런스를 적용해 흰색을 흰색으로 맞추기.

- Demosaicing 후 R/G/B 채널 분포 확인 — **초록색 값이 지배적**임을 재확인.
- 직접 구현 (고민해서 짜볼 핵심 두 가지):
  - **Gray World**: 회색 = R, G, B가 같아야 한다 → 채널 평균을 일치시키는 gain 적용.
  - **White Patch**: 각 채널의 **max 값을 흰색**이라 가정하고 맞춤.
- 카메라 메타데이터 WB: raw의 `camera_whitebalance` 속성으로 촬영 시 저장된 WB 값을 읽음 (RGBG 구조 → 마지막 G 중복 제거) → gain으로 정규화.
- 결과 비교 (시각화): 초록 끼던 이미지가 Gray World에서 크게 개선, White Patch는 그보다 다소 아쉬움, **카메라 WB는 Gray World와 상당히 유사**.
- **Color Temperature 조작**: WB gain을 임의로 조절해 톤 연출 가능.
  - 웜톤: R × 1.2, B × 0.8 (R↑, B↓)
  - 쿨톤: B↑, R↓
  → WB 값 조절만으로 표준/따뜻한 톤/쿨한 톤을 만들 수 있음.

## 4.5 실습 정리 및 예고

오늘 구현한 파이프라인:

```
Raw 읽기 (rawpy) → Black Level Subtraction + White Level Normalization
→ RGB Demosaicing (bilinear) → [Noise Reduction은 생략] → White Balance
```

- 내일(다음 실습): Color Transform(컬러 스페이스 매핑), Color Manipulation, JPEG Compression, AI 기반 ISP 실습. 이론은 ISP의 다른 파트에 AI가 쓰이는 사례와 Super Resolution 등.

# 5. 오후 핵심 정리 ⭐

- 과학/공학의 거의 모든 문제는 **입력→출력 함수의 모델링**이며, 관계식보다 어려운 건 **어떤 피처를 봐야 하는지 발견하는 것**이다.
- 이미지 분류가 오래 안 풀린 이유는 픽셀 조합의 **무한한 variation** 때문이고, 딥러닝은 데이터와 레이블만으로 **near-optimal한 피처/representation을 자동 학습**해 이를 돌파했다.
- 야구 스트라이크/볼 CNN은 **전광판 표시등을 보는 치팅**을 스스로 학습했다 — 딥러닝이 데이터에서 최적 피처를 찾아내는 힘과, 데이터 편향의 위험을 동시에 보여주는 사례.
- 현재 AI의 본질은 아키텍처보다 **데이터** — 빅테크는 아키텍처를 거의 안 건드리고 레이블링에 막대한 비용을 투입하며, 산업 현장의 보틀넥도 레이블 데이터다.
- **White Balance를 CNN으로 직접 regression하면 잘 안 된다** → GT 조명을 **K-means로 K개 클래스로 이산화** → classification 후 **softmax 확률로 클러스터 센터를 weighted sum**하면 regression이 되고 성능이 크게 향상된다 — regression 태스크 일반에 적용 가능한 레슨.
- 모델이 조명이 아닌 **씬을 분류할 가능성**은 synthetic 조명 테스트로 검증해야 하며, multi-illuminant(조명 혼합)는 단일 조명 가정이 깨져 여전히 어려운 문제(픽셀별 조명 레이블 필요).
- AI는 분포 내 **interpolation**은 매우 잘하지만 **extrapolation(OOD)**이 가능한지는 미해결 — 신약/신소재 발견에의 적용이 열린 질문.
- 실습에서 Classical ISP를 직접 구현: **raw(14bit, RGGB) 읽기 → Optical Black 기반 Black Level Subtraction과 White Level Normalization(실효 다이나믹 레인지 확인) → bilinear demosaicing(for문 vs convolve2d 병렬화, 124.5s→5.2s) → Gray World/White Patch/카메라 메타데이터 WB 비교 및 color temperature 조작**.
