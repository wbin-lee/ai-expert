# 1. Computational Photography란? ⭐

## 1.1 정의 — 그래픽스와 비전의 중간

컴퓨터 그래픽스의 서브필드로도, 컴퓨터 비전의 서브필드로도 볼 수 있는, 그 중간 어딘가의 연구 분야 (~20년 된 분야).

| 관점 | 정의 |
|---|---|
| 그래픽스 쪽 | 그래픽스 콘텐츠 제작에 **사진을 활용** (예전 명칭: Image-Based Rendering — 리얼리즘을 위해 사진으로 텍스처 매핑) |
| 비전 쪽 | **전통적인 사진의 한계를 계산/알고리즘으로 개선** (사실상 이미지 프로세싱에 가까움 — 더 팬시한 이름) |

두 매체의 장단점:

- **그래픽스**: 새로운 것을 만들고 조작(카메라 위치, 물체 이동)하기 쉬움 ↔ 리얼리스틱하게 렌더링하기 어려움.
- **사진**: 찍자마자 리얼리스틱하고 캡처가 쉬움 ↔ 조작(물체 위치 변경 등)이 어려움.

→ 둘의 장점을 섞어 **realism + 조작 용이 + 캡처 용이**를 지향하는 분야.

## 1.2 전통 사진의 한계 (다루는 문제들)

- **Blur / Camera shake**: 카메라 흔들림 또는 물체 움직임에 의한 블러 → 디블러링.
- **Noise**: 이미지 노이즈 → 디노이징. 손상된 옛 사진 복원 등 → Image Restoration.
- **Dynamic Range**: 카메라가 담을 수 있는 밝기 범위가 실제 세상보다 좁음(역광 문제). → **HDR Imaging** — 실제로는 여러 장을 찍어 합성 (폰의 HDR 기능). 지금도 활발한 연구 분야.
- **Single Viewpoint / 2D**: 2D 사진을 3D스럽게 — AI 글래스 시대에 더 중요해질 수 있음.
- **Focus**: 오토포커스의 한계. **Light Field 카메라**는 촬영 후 재포커싱이 가능했으나 해상도가 너무 작아 사그라듦(폰에 들어가면 대박이지만 어려움).

## 1.3 Image Restoration = Ill-posed Problem ⭐

디노이징, 디블러링 등 대다수 복원 문제는 **수학적으로 풀 수 없는 ill-posed problem** — **답이 무수히 많아서**, 이런저런 가정을 넣어 풀어야 함.

→ 너무 어려우니 나온 접근들:

1. **Computational Camera**: 옵틱스/하드웨어를 건드려 추가 정보를 캡처하고 알고리즘과 결합 (예: 플래시 사진 + 노플래시 사진 혼합, 조명을 색깔별로 껐다 켰다 하며 촬영). 연구는 많았으나 현실 제품으로는 잘 못 들어옴.
2. **Interactive Vision**: 사람이 힌트를 주는 방식 (segmentation에 대략적 힌트, 흑백사진 컬러화에 포인트별 색 힌트 — 흑백→컬러 완벽 재현은 원래 불가능한 문제).
3. **최근**: 이 모든 문제가 사실상 **Diffusion 모델 하나로 수렴**하는 추세 — 생성 모델이므로 "더 좋은 사진 만들기"에 자연스럽게 들어옴.

## 1.4 왜 ISP인가 — 컴퓨터 비전의 수익성 문제

컴퓨터 비전은 기술적으로는 잘 되지만 **직접 돈을 벌기 어려움** (classification을 아무리 잘해도 수익화가 어려움; 로봇/자율주행 제외). 삼성 같은 회사 입장에서 비전/이미지 프로세싱이 가장 임팩트 있는 곳은 여전히 **카메라(In-camera rendering pipeline = ISP)** — 핸드폰의 가장 중요한 피처가 카메라이기 때문.

# 2. 카메라는 Light Measuring Device인가? ⭐⭐

## 2.1 컴퓨터 비전의 암묵적 가정

- 비전 알고리즘에게 사진은 "이쁜 그림"이 아니라 **0~255 숫자들의 집합**일 뿐.
- 보는 과정: 조명이 빛을 뿌림 → 물체에서 반사 → 이미징 시스템(센서)이 빛의 양을 측정.
- 거의 모든 비전 연구자가 깔고 가는 가정: **픽셀의 RGB 값 = 물리적인 값** (카메라 = light measuring device).

## 2.2 현실: 카메라는 light measuring device가 아니다 ⭐

- 소비자는 "있는 그대로"를 원하지 않음 — 피부톤은 더 뽀얗게, 더 예쁘게. **카메라 회사의 목표는 물리값 재현이 아니라 더 예쁜 사진**.
- 증거: 같은 씬을 캐논/니콘/소니로 찍으면 색감이 다름. 아이폰과 갤럭시도 다름 (옛 아이폰 모델의 "그때 톤"을 선호해 구형을 사기도 함). → **색감 = 각 회사의 시크릿 레시피**.
- 정확한 측정 기기였다면 어떤 카메라로 찍어도 색이 같아야 함.

## 2.3 두 가지 시사점

1. **카메라 회사 입장**: 센서 값을 어떻게 소비자에게 어필하는 컬러로 만들 것인가.
2. **비전 연구자 입장**: detection/classification에는 절대적 영향은 없지만, **빛의 물리량을 쓰는 알고리즘**(피부병 진단 등 과학적 응용)에 sRGB 이미지를 쓰면 값이 이미 왜곡되어 있어 **작동 불가**.

→ 오늘의 초점: 센서 값이 최종 RGB가 되는 과정(ISP). 내일: 그 과정의 AI 대체.

# 3. 컬러의 본질 — 컬러는 물리량이 아니다 ⭐⭐

## 3.1 컬러 = 인지 현상

- 컬러는 **주관적·심리적인 것**이지 물리적 절대값이 아님 ("true blue가 어느 패치냐"는 사람마다 갈림).
- 물리적으로 존재하는 것은 **전자기파(에너지)**뿐. **가시광선 영역 ≈ 400~700nm** (380~720). 백색광을 프리즘에 통과시키면 분리되는 무지개색이 이 대역.

## 3.2 눈의 센서 — 콘셀(Cone)과 로드셀(Rod)

- 망막의 **콘셀(cone cell)** 3종류: **L(Long, 빨강 영역) / M(Medium, 초록) / S(Short, 파랑)** — 각각 다른 spectral sensitivity를 가진 필터.
- 콘셀은 망막 중앙에 밀집, 주변부에는 **로드셀(rod)** — 컬러가 아니라 **low light / night vision** 담당 (어두운 데서 색감이 떨어지는 이유).
- 색약(강사 본인이 적록색약): 정상인도 L과 M의 sensitivity가 많이 겹치는데, 색약은 **L·M이 더 심하게 overlap** → 둘을 분리하는 능력 저하.
- 콘셀에서 감지된 신호가 **뇌로 가서 해석**되는 것이 컬러 → 컬러는 인지적 현상. (LMS sensitivity를 정확히 알아낸 것은 2000년대 — 비교적 최근)

## 3.3 Spectral Power Distribution과 Metamer ⭐

- 물체의 컬러 특성은 **Spectral Power Distribution(SPD)** — 연속적인 파장별 에너지 분포 (버터: 그린+레드에 에너지, 토마토: 레드에만).
- 눈/카메라는 이 연속 분포를 LMS 필터로 **3개의 값으로 압축(샘플링/필터링)**.
- **Metamer**: 물리적으로는 **다른 SPD**(예: 실제 상추 vs 그린 잉크)인데 3개 값으로 압축하면 **값이 똑같아져 같은 색으로 인식**되는 현상.

## 3.4 Trichromatic Color Theory와 Grassmann's Law

- LMS를 알기 전부터 과학자들은 "색은 3개로 표현 가능"을 알고 있었음 — **Tri-stimulus Color Theory**.
- **Grassmann's Law**: 색 혼합은 **linear system**. (R1,G1,B1)로 만든 색 + (R2,G2,B2)로 만든 색 = 가중치를 선형으로 더한 색. → RGB primary 3개가 **basis**가 됨.

# 4. 컬러의 수치화 — 유저 스터디의 역사 ⭐⭐

컬러를 수치화한 것은 **약 100년** 전. 컬러는 주관적이므로 수치화 방법은 결국 **유저 스터디** (페인트 판매 등에 수치가 필요했음).

> 비유: 생성모델 이미지 평가나 갤럭시 vs 아이폰 화질 비교(IQA, Image Quality Assessment)도 결국 주관적이라 블라인드 테스트(펩시 vs 코카콜라식)를 할 수밖에 없는 것과 같은 이슈.

## 4.1 실험 1 — Flicker Experiment → Luminosity Function ⭐

**밝기**에 대한 반응 측정:

- 오른쪽: 기준 밝기의 빛. 왼쪽: 400~700nm 파장을 바꿔가며 빛을 쏨.
- 유저는 왼쪽 빛의 **세기(볼륨)**를 조절. 양쪽이 교대로 깜빡이는데 **밝기가 일치하면 깜빡임(flicker)이 사라짐** → 그 시점의 세기를 기록.

결과 해석:

- 400/700 양 끝에서는 볼륨을 많이 올려야 했음 = 눈이 둔감. 가운데(초록)는 조금만 올려도 됨 = 민감.
- → **Luminosity Function**: 사람 눈은 **green에 가장 민감**, 파랑/보라/빨강 쪽은 덜 민감 (sensitivity는 측정 곡선의 역함수).

파생되는 실용적 사실 ⭐:

1. 카메라 센서 패턴이 **R-G-G-B (그린 센서 2배)**인 이유.
2. 비전 알고리즘에서 채널 하나만 쓸 때 **그린 채널을 쓰는 근거** ("사람 눈이 그린에 민감하므로").
3. SPD가 주어지면 밝기 \(Y\) = SPD와 luminosity function의 적분(연속) 또는 **dot product/inner product**(이산) → 밝기의 정량화 가능.

## 4.2 실험 2 — Color Matching Experiment → CIE RGB ⭐

**3개의 색 수치**를 얻기 위한 실험:

- 왼쪽: 테스트 파장의 빛(400, 405, 410nm... 순차적으로).
- 오른쪽: **R(700nm), G(546nm), B(435nm)** primary 조명 3개.
- 유저는 RGB의 혼합비를 조절해 왼쪽 테스트 컬러와 일치시킴.
- **음수 문제**: 어떤 파장은 빼야 매칭됨 — 물리적으로 빛을 뺄 수 없으므로 **왼쪽(테스트 쪽)에 그 primary를 더하는 방식**으로 빼기 효과를 구현.

결과: \(\bar{r}, \bar{g}, \bar{b}\) 세 개의 **basis function** (CIE RGB; CIE는 "컬러계의 IEEE"격 기관).

- 해석: 600nm 테스트 컬러를 만들려면 R 이만큼, G 이만큼, B 이만큼 필요.
- 길이를 잴 때 유클리디안 x,y,z축이 있어야 하듯, 컬러를 잴 **축(basis)**을 (논리니어하지만) 휴먼 스터디로 구한 것.

## 4.3 1931 CIE XYZ ⭐⭐

문제: CIE RGB의 **음수 구간** — 이걸로 기계를 만들 수 없음 (물리적으로 구현 불가).

→ **1931년** 사람들이 모여 **합의된 3×3 매트릭스**를 곱해 음수가 없는 **XYZ color space**를 정의.

- 의미: **컬러를 quantify할 수 있게 됨**. SPD가 주어지면 \(x, y, z\) basis function과의 적분(또는 inner product)으로 XYZ 3개 값을 얻음.
- **약 95년이 지난 지금도 모든 카메라/TV 등 색 표현의 근간은 CIE XYZ** — 모든 컬러 스페이스 변환의 기준(홈베이스).
- 단, XYZ basis는 요상하게 생겨서(non-orthogonal) 직관적으로 해석하기 어려운 공간.

## 4.4 xyY와 Chromaticity Diagram

- 컬러감과 밝기를 분리해서 보면 유용 → **xyY space**: 소문자 \(x, y\) = 컬러(chromaticity), 대문자 \(Y\) = 밝기(luminance).
- \(x\)-\(y\) 평면의 **말굽(horseshoe) 모양 chromaticity diagram**: 사람이 볼 수 있는 모든 색이 이 안에 들어감. 테두리에 파장이 표시되고, 가운데 **E point** = X=Y=Z인 white point.

## 4.5 17명의 영국 남자 ⭐ (재미있는 사실)

100년간 쓰이는 이 색공간의 유저 스터디 참가자는 **단 17명** — 그것도 영국 남성만. (연구실에서 "야 너 잠깐 와봐" 식으로 모은 수준의 실험을 인류가 100년째 사용 중)

# 5. 조명과 컬러 — Color Constancy & White Balance ⭐⭐

## 5.1 조명이 항상 곱해진다

지금까지 빠진 요소 = **조명**. 눈/카메라에 들어오는 빛의 색 = **조명 SPD × 물체의 반사 특성**.

- 같은 빨간 토마토라도 조명색에 따라 물리적으로는 다른 빛이 들어옴.
- 흰 종이에 파랑/노랑/빨강 조명을 쏘면 물리적으로는 각각 파랑/노랑/빨간 빛이 들어옴.

## 5.2 Color Constancy ⭐

그런데도 사람은 모두 "빨간 토마토", "흰 종이"로 인식 → **뇌가 조명색을 보정**하는 기능 = **Color Constancy** (정확한 메커니즘은 미상). 단, 배경색에 따라 같은 색이 다르게 보이는 착시처럼 100% 정확하지는 않음.

카메라의 대응 기능 = **White Balance**.

## 5.3 수학적 모델 — von Kries Transform

입력 색(조명이 입혀진 색)을 보정하려면 **조명의 색을 알아야** 하고, 알면 **채널별로 나눠주면** 됨:

\[
\begin{pmatrix} R' \\ G' \\ B' \end{pmatrix} = \begin{pmatrix} 1/\ell_R & & \\ & 1/\ell_G & \\ & & 1/\ell_B \end{pmatrix} \begin{pmatrix} R \\ G \\ B \end{pmatrix}
\]

→ **대각(diagonal) 3×3 매트릭스** 하나로 보정 = **von Kries Transform**. 지금까지도 사용.

> 참고로 인용된 George Box(통계학의 아버지)의 말: **"All models are wrong; the question is how wrong do they have to be to not be useful."** — 모든 수식은 모델일 뿐이며 유용성이 기준 (AI/뉴럴넷 시대와 연결, 내일 부연).

## 5.4 Color Temperature ⭐

조명색을 표현하는 **1차원 수치**:

- 이론적인 **Blackbody Radiator**를 특정 온도(Kelvin)로 가열했을 때 내뿜는 색 = 그 조명의 색.
- 온도에 따라 변하는 조명색의 궤적을 XYZ 상에 그리면 → **Planckian Locus** 곡선.
- 낮은 K(예: 2,000K) = 붉은색, 높을수록 파랗게/하늘색 쪽. **6,500K ≈ 데일라이트(거의 흰색)**.
- 자연광(sky, sunlight)의 SPD는 대체로 **플랫**(흰색에 가까움), 인조 조명(형광등)은 특정 파장이 **튀는** 특성.
- CIE 표준 조명: Standard A, B, C... **D가 붙는 것들이 Daylight 계열**.
- 실생활: 카메라 화이트밸런스 설정 = color temperature 조절. 전구를 살 때도 "5000K daylight", "3000K"로 표기됨.

## 5.5 White Point도 조명에 종속

흰색 자체도 조명색이 입혀지므로 white point는 **Planckian locus를 따라 변함**. → 색을 표현/변환할 때는 **항상 조명을 고려해야 한다**는 결론.

# 6. 컬러 스페이스 — sRGB의 탄생과 그 외 ⭐

## 6.1 Device Independent vs Device Dependent

- **XYZ**: 어떤 기기에도 종속되지 않는 **device independent** 공간.
- 카메라/디스플레이/프린터는 각자의 RGB primary(잉크, 필터)를 가짐 → **device dependent**. 삼성 프린터와 HP 프린터의 같은 RGB 값이 **다른 색**이 됨. chromaticity diagram에서 각 기기가 커버하는 삼각형도 다름. → 변환이 필요.

## 6.2 sRGB — 1996년 Microsoft + HP의 합의 ⭐

- RGB 값의 비통일이 산업 문제가 되자 당시 최대 IT 기업 **MS와 HP가 1996년 합의**로 RGB primary를 정함 — **과학적 근거 없이** 당시 기기들이 재현 가능한 색 기준의 약속.
- **White point = D65 (6,500K daylight)** ⭐: sRGB 이미지를 본다는 것에는 "**뷰잉 컨디션이 6,500K 데일라이트(대낮)**"라는 가정이 깔려 있음.
- 한계: 세상의 모든 컬러를 표현하지 못함 (말굽 안의 일부 삼각형). 그러나 표준은 한 번 정해지면 모든 시스템이 맞춰져 있어 바꾸기 매우 어려움 — **지금도 가장 많이 쓰는 표준**.

## 6.3 XYZ ↔ sRGB 변환과 Gamma ⭐

- XYZ ↔ linear sRGB는 정해진 **3×3 매트릭스**로 언제든 왕복 가능.
- 단, 실제 사진에서 읽는 sRGB 값은 linear sRGB에 **gamma curve를 한 번 더 씌운 non-linear 값** (더 보기 좋은 이미지를 만들기 위함).

```
XYZ --(3x3 matrix)--> linear sRGB --(gamma curve)--> sRGB (우리가 읽는 값)
```

## 6.4 기타 컬러 스페이스

- **Lab space**: XYZ에서는 "같은 색으로 인지되는 영역(타원)"의 크기가 위치마다 달라 **perceptually uniform하지 않음**. Lab은 **perceptually uniform** — 어느 영역에서든 같은 거리 = 같은 인지적 색 차이. → 컬러 차이를 잴 때는 Lab에서 거리를 재는 것이 인지적으로 합당.
- **Gamut**(표현 가능한 색 범위): ProPhoto RGB(Kodak) > Adobe RGB > **sRGB(가장 좁음)** — 그래도 표준은 sRGB.

# 7. In-Camera ISP 파이프라인 ⭐⭐

모바일 칩(엑시노스/스냅드래곤)에서 **ISP가 차지하는 면적·전력·프로세싱 비중이 상당히 큼** — 그만큼 폰에서 카메라가 중요. 전체 흐름(회사마다 다르고 실제는 훨씬 복잡하지만 대략적인 단계):

```
렌즈 → CMOS 센서(Bayer) → [RAW] → Gain(ISO)/전처리(black light subtraction, dead pixel, lens shading)
  → Demosaicing → Noise Reduction                       ← 여기까지 카메라 고유의 RAW color space
  → White Balance + Color Space Transform → [CIE XYZ]   ← 표준 공간으로 통일
  → Photo Finishing (3D LUT, Tone Curve, selective color…)  ← 시크릿 레시피
  → Resizing/Super-resolution → sRGB 변환 → JPEG 압축 → 저장
```

## 7.1 센서단 ⭐

- 요즘은 모두 **CMOS 센서** (예전엔 CCD). 센서가 클수록 좋은 사진 — DSLR > 스마트폰 (전문가들이 DSLR을 쓰는 이유). 화소수 경쟁은 한때 핵심 스펙이었으나 지금은 글쎄.
- 센서 위 구조 (위→아래):
  - **Microlens**: 픽셀마다 하나씩, 빛을 더 모아주는 역할.
  - **Color filter (RGB)**: 눈의 콘셀에 해당. **픽셀 하나당 R/G/B 중 하나만** 감지 가능.
  - **NIR(near-infrared) filter**: 레드 센서에 적외선이 섞여 들어오는 것을 차단 (사람은 가시광선만 보므로).
- **Bayer Pattern** ⭐: Kodak의 **Bryce Bayer**가 고안한 R-G-G-B 배열 (그린 2배 — luminosity function과 연결). 과거 사진 기술 대부분이 코닥 연구소 출신.
- 회사/카메라마다 컬러 필터의 **spectral sensitivity가 다름** → 센서단부터 색 차이 발생. 외부인은 특수 장비 없이는 알 수 없음.

## 7.2 RAW 파일 ⭐

- **RAW = 순수한 물리적 값** (radiance/irradiance). "카메라를 light measuring device로 쓸 수 있나?" → **RAW를 쓰면 된다**.
- 전문 사진가들은 RAW를 받아 Lightroom/Photoshop으로 직접 "현상"함. 일반인은 카메라가 주는 sRGB(JPEG)를 사용.
- 단점: **용량이 매우 큼** (수백 MB), **12~16bit** (8bit가 아님).
- 주의 ⭐: 서로 다른 카메라의 RAW는 컬러 필터 sensitivity가 달라 **서로 다른 색 공간(축)에 사는 값** — RGB 숫자가 같아도 같은 색이 아님.
- RAW를 sRGB 가정의 디스플레이에 그대로 렌더링하면 어둡고 초록빛 도는 이미지가 나옴.

## 7.3 Gain (ISO)과 센서단 전처리

- **ISO = gain**: 이론적으로는 센서 민감도 조절이지만 실제로는 대부분 **RAW 값에 숫자를 곱하는 것**. 평상시 100~200, 어두운 환경에서 높임 (플래시 없이 분위기를 살리는 방법).
  - 치명적 문제 ⭐: ISO를 높이면 **노이즈도 같이 곱해져** 노이즈가 심해짐.
- 픽셀 인텐시티는 ISO(gain)와 **노출**(조리개를 얼마나, 몇 초 열었나)로 결정 — 단위가 없는 **상대적인 값**.
- **Black Light Subtraction**: 완전 암흑에서도 0이 아닌 픽셀이 있음(노이즈/하드웨어 문제) → 렌즈 캡을 닫고 찍어 0이 아닌 값을 빼줌 (공장 단계 처리).
- **Defective Pixel Mask/Detection**: 공정상 죽은 픽셀을 사전에 탐지해 주변 픽셀 값으로 interpolation.
- **Lens Shading (Vignetting) 보정**: 옵틱스 때문에 가장자리가 어두워지는 현상을 보정.

## 7.4 Demosaicing ⭐

- 센서는 픽셀당 한 컬러만 갖고 있으므로, **갖고 있지 않은 나머지 두 채널 값을 만들어내는 과정**.
- 가장 단순: 이웃 픽셀 값 평균 **interpolation**. 단점: 엣지에서 선명도 문제 → **edge-aware interpolation** 등.
- 최근 카메라는 Bayer를 넘어 **Tetra, Nona** 등 더 복잡한 패턴, 심지어 RGB가 아닌 색(CMY 계열, 소니의 시도)도 사용 → 패턴에 맞는 demosaicing 품질이 **좋은 컬러의 시작**.
- 지금도 회사 현장에서 활발히 연구되는 영역 (이 과정의 개별 프로젝트에서도 다루는 주제).

## 7.5 Noise Reduction ⭐

- 모든 센서는 본질적으로 노이즈를 가짐. **스마트폰은 센서/픽셀이 작아 노이즈에 더 취약** → 더 공격적으로 디노이징.
- 센서 노이즈의 수학적 모델이 있지만, George Box의 말처럼 **모델일 뿐** — 실제 노이즈 형태를 정확히 아는 것은 어려움 (측정 자체가 어려움).
- 단순한 방법(Gaussian blurring)은 노이즈와 함께 **high frequency/엣지 디테일이 뭉개지는** 문제.
- **Ground Truth 문제** ⭐: 디노이징 연구가 PSNR로 우월성을 주장하지만 **노이즈의 GT를 아는 것이 거의 불가능**.
  - 실용적 방법: 무거운 삼각대에 카메라를 고정하고 천장을 여러 장 찍어 **평균을 noise-free GT로 가정**.
  - 그마저도 셔터 누를 때의 미세 흔들림으로 3~4픽셀씩 shift 발생 → 리모트/무선 셔터로 찍어도 어려움.
  - → 딥러닝 디노이징도 **데이터 캡처가 가장 어려운** 분야.

## 7.6 White Balance + Color Space Transform (RAW → XYZ) ⭐⭐

여기서 카메라 고유의 RAW color space에서 **표준 XYZ space로 통일**됨. 핵심 포인트:

- XYZ ↔ sRGB 매트릭스는 공개되어 있고, 카메라 회사는 자기 센서의 sensitivity를 아니까 RAW → XYZ 3×3 매트릭스를 구할 수 있음.
- 그러나 **sensitivity만으로는 변환 불가** ⭐: XYZ space는 **white point(흰색 조명) 가정** 위에 만들어졌으므로, **촬영 순간의 조명색을 알아야** 변환 가능.
- → 순서: **조명색 추정 → White Balance(대각 매트릭스) → 3×3 full matrix로 XYZ 변환**.

\[
\text{RAW} \xrightarrow{\text{WB (diagonal matrix)}} \text{조명 보정된 RAW} \xrightarrow{\text{CST (full 3×3)}} \text{XYZ}
\]

- WB는 좋은 이미지를 위해서뿐 아니라 **컬러 스페이스 변환 자체에 필수**.
- 카메라 회사들은 조명 종류별 대각 매트릭스를 미리 정해둠. JPEG **메타데이터**에 적용된 WB 매트릭스가 기록되어 있는 경우도 많음 (GPS, exposure 정보와 함께).

### Auto White Balance 알고리즘 ⭐

픽셀 값 = 조명 × 물체 본연의 색 → 100 = 50×2일 수도, 무수히 많은 해 → **ill-posed, 가정 필요**:

| 알고리즘 | 가정 | 방법 |
|---|---|---|
| **Gray World** | 세상은 회색이다 | 이미지 전체 픽셀 RGB **평균**을 구해, 그 평균이 그레이가 되도록 나눠줌 (평균 = 조명색) |
| **White Patch** | 가장 밝은 부분이 흰색이다 | \(R_{max}, G_{max}, B_{max}\)를 white point(=조명색)로 보고 보정 |

- Gray World가 의외로 잘 되는 이유: **실제 세상이 꽤 그레이/무채색** (강의실 옷, 서울 시내 건물 대부분 회색/흰색; 색을 찍으려면 꽃가게·과일가게에 가야 할 정도).
- WB 연구용 GT 확보: 씬에 **Macbeth Color Chart**를 놓고 촬영, 차트의 그레이/화이트 패치에 찍힌 색 = 조명색으로 가정. (컬러 연구하는 랩에는 하나씩 있음)
- 최근에도 딥러닝 기반 WB 논문이 계속 나오는 활발한 분야.

## 7.7 Photo Finishing (Color Manipulation / Color Rendering) ⭐⭐

XYZ 이후 단계 — **카메라 회사들의 진짜 시크릿 레시피** (앞 단계가 재료 손질이라면 여기가 맛을 내는 단계).

- **지역별로 다를 수 있음**: 미국/한국/유럽 판매 폰의 색감이 선호도에 따라 다르게 튜닝되기도.
- DSLR의 **Picture Style** (Standard/Portrait/Landscape 등): 2단계(XYZ)까지는 동일하고 이 단계에서 모드별 색감을 입힘.
- 핵심 구성 ⭐: RGB→RGB **3D 함수**가 필요한데, 빠른 계산을 위해 **3D Look-Up Table(LUT)** + **1D Tone Curve**(감마커브처럼 contrast 개선)로 구현. 이 둘의 설계가 엔지니어들의 어마어마한 튜닝 결과 (Photoshop의 Curves 기능이 이 1D tone curve 조정에 해당).
- **리버스 엔지니어링 사례** (강사가 포닥 때 Michael Brown 랩에서 수행): 캐논 vs 니콘이 이 단계에서 RGB를 어떻게 비트는지 분석 → **니콘은 비교적 그대로 두고, 캐논은 훨씬 많이 손댐** — 사진가들의 직관("니콘은 그대로, 캐논은 터치한다")과 일치.
- **Tone Mapping**: 이미지 전체에 적용하는 **global** 방식과 특정 영역별로 다르게 적용하는 **local** 방식.
- **Selective Color Manipulation** ⭐: 얼굴 디텍션 후 **피부톤만 별도 보정** — 모든 카메라 회사가 가장 중요하게 생각하는 부분.

## 7.8 Resizing / Super-Resolution

- 뷰파인더용 축소, 줌을 위한 확대(예: 갤럭시 "문샷" — 제너러티브로 알려짐).
- Super-resolution은 computational photography의 주요 연구 주제 (내일 별도로 다룸).

## 7.9 sRGB 변환과 압축

- 지금까지 XYZ에서 작업했으므로 최종적으로 **sRGB로 변환** 후 압축.
- 표준은 **JPEG**: 이미지를 **8×8 패치**로 나눠 **DCT(Discrete Cosine Transform)** 기반 압축. Fine = 저압축, 강하게 압축하면 **JPEG artifact** 발생 (이를 줄이는 연구도 존재). RAW+JPEG 동시 저장 옵션도 있음.
- 새 표준 움직임: 애플의 **P3** 컬러 스페이스(더 넓은 gamut), **HEIC** 코덱(JPEG보다 우수, 아이폰이 사용 시작). 다만 빅플레이어들(삼성, 화웨이 등)의 이해관계가 달라 **합의가 쉽지 않음**.

## 7.10 Zero Shutter Lag — Circular Buffer

- 셔터를 누른 순간과 실제 캡처 사이에는 딜레이가 존재.
- 대응: 카메라를 켜둔 동안 **circular buffer에 RAW가 비디오처럼 계속 저장**되고, 셔터를 누르면 **그 전에 저장된 프레임**을 프로세싱해 보여줌 (= zero shutter lag).

## 7.11 정리와 현업 이야기

- 파이프라인 앞단(Bayer processing front-end) = 기본 재료 손질, 뒷단 = 복잡한 photo finishing. 스마트폰에서는 훨씬 더 복잡한 일이 일어남.
- **머신 비전 카메라** (예: Point Grey): 중간 단계 터치를 최소화하고 빛 본연의 값을 찍음 → 로봇/자율주행 등 **light measuring device가 필요한 곳에 적합** (자율주행에 피부 뽀얗게 만들 이유가 없음).
- 삼성 내 ISP 조직: **LSI**(AP에 들어가는 알고리즘), **MX**(갤럭시 특화 ISP), **SR**(연구소) — 각각 팀이 있고 끊임없이 튜닝 중. 애플도 ISP 엔지니어를 상시 채용.
- 학교에서는 비디오 세그멘테이션 같은 분야가 더 팬시해 보이지만, **산업 수요는 ISP 쪽이 분명함** (제자 사례: ISP 연구를 유지해 삼성 토론토랩 진출).
- 오후: White Balancing 심화 + 실습(파이프라인 단계별 구현).

# 8. 오전 핵심 정리 ⭐

- **Computational Photography** = 그래픽스(조작 용이)와 사진(리얼리즘·캡처 용이)의 장점을 결합하려는 분야. 전통 사진의 한계(blur, noise, dynamic range, focus 등)를 계산으로 극복하며, 대부분의 복원 문제는 **ill-posed**(해가 무수히 많음). 최근에는 diffusion 모델로 수렴하는 추세.
- **카메라는 light measuring device가 아니다** — 카메라 회사의 목표는 물리값 재현이 아니라 예쁜 사진이며, 색감은 회사별 시크릿 레시피. 물리량이 필요한 작업에는 **RAW**를 써야 한다.
- **컬러는 물리량이 아닌 인지 현상**. 물리적으로는 SPD(연속적 파장 분포)이고, 눈의 **LMS 콘셀** 3개 필터로 압축되며, 다른 SPD가 같은 색으로 보이는 **metamer**가 존재한다.
- 컬러 수치화는 **유저 스터디**로 이루어짐: **Flicker 실험 → Luminosity Function**(눈은 green에 가장 민감 — RGGB Bayer 패턴, 그린 채널 사용의 근거), **Color Matching 실험 → CIE RGB → (음수 제거 합의) → 1931 CIE XYZ**. 17명의 영국 남성 실험을 100년째 쓰고 있으며, **XYZ는 지금도 모든 색 표현의 홈베이스**.
- 들어오는 색 = **조명 × 물체 반사 특성**. 뇌의 **Color Constancy** = 카메라의 **White Balance**이며, 조명색을 알면 **대각 매트릭스(von Kries transform)**로 보정. 조명색은 **Color Temperature**(blackbody, Planckian locus, 6500K ≈ daylight)로 표현.
- **sRGB**는 1996년 MS+HP의 합의(white point D65)로 만들어진, 과학적 근거 없는 표준이며, linear sRGB에 **gamma curve**를 씌운 값이 우리가 읽는 RGB다. 인지적 색 차이는 **Lab space**에서 재는 것이 합당하다.
- **ISP 파이프라인**: Bayer 센서 RAW → ISO gain/전처리(black light subtraction, dead pixel, vignetting) → **Demosaicing**(픽셀당 1채널 → 3채널) → **Noise Reduction**(GT 확보가 거의 불가능한 문제) → **White Balance + CST**(RAW→XYZ; 조명색을 모르면 변환 불가 ⭐) → **Photo Finishing**(3D LUT + tone curve + 피부톤 selective 보정 = 시크릿 레시피) → sRGB + JPEG 압축.
- **Auto White Balance**의 고전 알고리즘: **Gray World**(이미지 평균을 그레이로 — 세상이 의외로 무채색이라 잘 됨), **White Patch**(채널별 최댓값을 흰색으로 가정). GT는 **Macbeth color chart**로 확보.
- ISP는 모바일 칩에서 면적·전력 비중이 크고, 학계보다 **산업에서 수요가 분명한** 분야다.
