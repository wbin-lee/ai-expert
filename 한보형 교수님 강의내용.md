## 1.1 컴퓨터 비전의 짧은 회고 (개인 일화)

- 그 당시 비전 알고리즘은 **거의 아무것도 되는 게 없었다**. *"지금 생각하면 정말 상전벽해"*.
- 당시 비전 알고리즘은 대부분 **군사용**.
- **첫 커머셜라이즈는 디지털 카메라의 페이스 디텍션 → 오토포커스** (2006~2007년경). 페이스 디텍션 알고리즘 자체는 **2000년 Viola-Jones** 가 처음, 약 6년 후 커머셜화.

## 1.2 Object Detection의 정의

> **Object Detection**: 이미지가 주어졌을 때 다양한 클래스 오브젝트들의 **클래스 + 위치(bounding box)** 를 동시에 맞추는 태스크.

- 한 이미지에 **여러 클래스가 동시에** 등장 가능, 각 클래스당 **여러 인스턴스** 가능.
- 기본적으로 **수퍼바이즈드 러닝** — 미리 정해진 클래스만 디텍션.

**Open-set Detection**: 최근 CLIP 같은 비전 파운데이션 모델을 활용해 학습되지 않은 클래스도 디텍션. 교수님 평: *"오픈셋으로 넘어가기 시작하면 굉장히 훨씬 더 지저분한 문제가 돼가지고 개인적으로 좋아하지는 않는데."*

**Bounding Box 컨벤션**:
- 컨케이브한 오브젝트(예: 배에 돛 달린 경우)는 어떻게 그리느냐?
- **오브젝트 전체를 다 포함하는 bounding box**가 컨벤션. 백그라운드가 많이 들어가는 단점은 있지만.

## 1.3 관련 태스크 비교

| 태스크 | 설명 |
|---|---|
| **Image Classification** | 한 클래스, 한 인스턴스 분류만 |
| **Classification + Localization** | + bounding box. **하나의 오브젝트가 어디 있는지** |
| **Object Detection** | 여러 클래스, 여러 인스턴스 각각의 bounding box |
| **Semantic Segmentation** | 픽셀-와이즈 클래시피케이션. 인스턴스 구분 X |
| **Instance Segmentation** | Object Detection과 거의 동일. 최종 결과가 bbox가 아니라 **픽셀 레벨 바운더리** |

교수님 강조: *"Object Detection과 Instance Segmentation은 굉장히 유사하다."*

## 1.4 Evaluation — 매우 중요한 섹션

> 🔥**시험포인트**🔥
> 교수님이 명시적으로 *"이게 좀 트리키한 면이 많이 있어서 정확히 이해하고 넘어가는 게 낫다"* 라고 강조한 섹션.

### 1.4.1 IoU (Intersection over Union)

$$ IoU = \frac{\text{Intersection}}{\text{Union}} $$

- 모델 prediction bbox와 ground truth가 얼마나 겹치는지.
- **0.5 이상**이면 디텍션됐다고 본다.
- 교수님 직접 코멘트: *"제가 재보진 않았는데 실제로 이 정도 되면 제 생각에는 0.7 조금 넘는 정도일 것 같아요. 생각보다 0.5가 굉장히 오벌랩이 큽니다."*

### 1.4.2 Precision, Recall, TP/FP/FN/TN

> 🔥**시험포인트**🔥 — 디텍션에서 가장 헷갈리는 부분이라며 풀어주심.

- 약어: **T/F = True or False**, **P/N = Positive or Negative**. 따라서 TP=True Positive, FP=False Positive, FN=False Negative.
- **Precision** = TP / (TP + FP): 모델이 뽑은 bbox 중 진짜 object인 비율.
- **Recall** = TP / (TP + FN): 실제 씬에 있는 오브젝트 중 모델이 찾아낸 비율.

**Trade-off**:
- Threshold ↓ → bbox 많아짐 → False Alarm 증가 → **Precision ↓, Recall ↑**
- Threshold ↑ → 정확하지만 못 찾는 게 많아짐 → **Precision ↑, Recall ↓**
- 그래서 **둘 다 동시에 봐야 한다**.

### 1.4.3 Detection의 PR Curve가 지그재그인 이유

> 🔥**시험포인트**🔥
> 일반 ML과 다른 디텍션 특유의 PR curve 모양.
>
> 일반: monotonic하게 감소.
> **디텍션**: 지그재그 형태.
>
> 왜? bbox들을 score로 정렬해 위에서부터 보면 맞고-맞고-틀리고(예외)-맞고-맞고-틀리고 식으로 진행됨. 틀린 게 나올 때마다 precision이 확 떨어졌다가 다음에 맞으면 다시 올라감.
>
> *"이게 티피컬한 프리시전 리콜 커브예요, 디텍션에서."*

### 1.4.4 AP와 mAP

> 🔥**시험포인트**🔥
>
> - **AP (Average Precision)**: PR curve 아래 면적 = 다양한 threshold에서 precision의 평균(적분).
> - **mAP (mean AP)**: 각 클래스의 AP를 평균낸 값.
> - Threshold가 두 종류 있음: **Classification score threshold** 와 **IoU threshold**.
> - 보통 IoU를 **0.5 ~ 0.95까지 0.05 단위로** 잘라 평균. (mAP@0.5 vs mAP)
> - 교수님 정리: *"민 AP는 여러 클래스를 가지고 평균을 낸 거다 라고 생각하시면 될 것 같습니다."*

## 1.5 옛날 방식 — Exhaustive Search

이미지 클래시피케이션 네트워크는 그대로 디텍션에 쓸 수 없음(여러 오브젝트가 여러 위치). 그래서 옛날엔 **위치/스케일/aspect ratio/rotation 다 search**.

**옛날 디텍터의 한계**:
- 각 디텍터가 **특정 클래스에만** 학습됨 (페이스/페데스트리언/자동차 따로).
- 페이스 디텍터: **정면 페이스만**, 보통 1:1 정사각형만.
- 자동차 디텍터: **자동차 뒷면만** (ADAS에서 뒷면 보니까).
- **Pedestrian detection**: 사람이 아니라 *"페데스트리언"* 이라 부른 이유 — **서 있는 사람만**, 1:2 비율 고정. 누워있거나 앉아있으면 못 찾음.

교수님 부연:
- *"딥러닝 되기 전에는 자동차, 페데스티안, 페이스 그런 디텍션들은 그래도 상당히 잘 됐고요. 나머지는 거의 디텍션을 못하고 있었다고 봐야죠."*
- 옛날 디텍터는 빠르지만 CNN은 그게 안 됨 → 새 접근(Object Proposal) 필요.

## 1.6 교수님의 솔직한 여담 — 아키텍처의 가치

> *"이런 알고리즘이나 아키텍처, 특히 아키텍처, 알고리즘은 그래도 조금 밸류가 있는데 **아키텍처의 밸류가 하나도 없다고 생각**하거든요. 그냥 데이터가 많고 그냥 GPU 받으면은 그냥 그 아키텍처가 상관없이 다 잘됩니다. 성능도 똑같다고 봐야 돼요. (...) **파라미터가 충분히 많이만 있으면 똑같은 개수의 파라미터를 갖고 있는 아키텍처는 다 똑같은 퍼포먼스를 내야 맞다고 생각을 해요**. (...) **데이터가 많으면 쉬운 문제고, 데이터가 없으면 어려운 문제고**."*

## 1.7 Object Proposal

> 🔥**시험포인트**🔥 — 딥러닝 기반 디텍션의 핵심 아이디어. 교수님이 *"기억하시면 되겠습니다"* 라고 명시적으로 강조.

### 1.7.1 정의
- **Object일 것 같은 bbox**들의 집합. **Class-agnostic**.
- 모든 위치/스케일을 보는 대신 **약 2000개의 proposal만** 뽑아 CNN에 넣음.

### 1.7.2 핵심 Properties
1. **Recall이 가장 중요**: proposal에 뽑히지 않은 오브젝트는 영영 못 찾음.
2. **빠르게 동작해야 함**: pre-processing이라 디텍션보다 빨라야 의미.
3. **여러 스케일, 여러 aspect ratio 고려** (rotation은 보통 X).

### 1.7.3 두 가지 알고리즘

**(1) Selective Search**
- 기반: **Superpixel Segmentation** — 비슷한 색·위치 픽셀들 묶기.
- **계층적(hierarchical)** 으로 — low-level에서 시작해 비슷한 superpixel끼리 또 묶고...
- 다양한 color space, feature, similarity metric(Euclidean, L-infinity, Pearson, cosine) → 다양한 proposal set.

**(2) Edgebox**
- **Edge map** 만들고, 가정: *오브젝트 안에는 edge가 많고, 오브젝트를 가로지르는 edge는 적다.*
- Score = (box 안 edge pixel 수) - (box를 가로지르는 edge pixel 수).

### 1.7.4 성능 비교 (구체 숫자)
> 🔥**시험포인트**🔥
> - **Edgebox**: 800 박스로 recall 75%.
> - **Selective Search**: 75% recall에 1400개 필요.
> - **Edgebox가 40배 더 빠름**.
> - 그런데 논문에서는 **Selective Search를 더 많이 씀** — Edgebox는 edge detection 모듈도 필요하고 구현 복잡. 실제 동작 시스템 만드는 사람들은 Edgebox 선호.

## 1.9 R-CNN (2014)

### 1.9.1 파이프라인
1. Selective Search/Edgebox로 **proposal 2000개**.
2. 각 proposal을 **CNN(AlexNet, VGG)에 넣어** 마지막 feature 추출.
3. Feature로 **SVM** 학습 → classification.
4. **Bounding Box Regression**으로 위치 refine.

### 1.9.2 Bounding Box Regression — R-CNN의 진짜 Contribution

> 🔥**시험포인트 — 교수님이 명시적으로 강조**🔥
> *"R-CNN 논문의 가장 큰 컨트리뷰션은 제가 보기에는 바운딩박스 리그레션이고, 이 바운딩박스 리그레션은 유니버셜하게, 지금도 디텍션 알고리즘들도 거의 이런 식으로 디텍션 바운딩박스의 얼라인먼트를 더 좋게 만드는 것을 합니다."*

- Proposal: $(P_x, P_y, P_w, P_h)$, Ground truth: $(G_x, G_y, G_w, G_h)$
- Transformation $d(P)$: 4개 파라미터 (translation 2 + width/height scaling 2)
- $d(P) = w^T \cdot \phi_5(P)$ — Pool5 feature에 linear vector $w$ 내적
- 학습: **simple ridge regression** (L2 regularization).

교수님: *"왜 단순 내적으로 충분한가? 그냥 그런 이유 없어요. 그냥 이렇게 해서 그냥 우리가 피팅을 하겠다 그런 얘기죠."* — 단순 linear로도 충분.

### 1.9.3 R-CNN의 Bottleneck
- Proposal 2000개 → CNN 2000번 → **컴퓨테이셔널 코스트가 엄청남**.

## 1.10 Fast R-CNN

**핵심 차이**: CNN을 한 번만 돌린다.

### 1.10.1 파이프라인
1. 이미지 **전체**를 CNN 한 번에 통과 → **feature map**.
2. Proposal을 feature map의 **같은 상대적 위치에 적용** → feature 추출.
3. **RoI Pooling**으로 dimension 맞춤.
4. FC layer → **Softmax classification + Bounding box regression**.

### 1.10.2 왜 RoI Pooling이 필요한가
- CNN은 보통 224×224 고정 입력 (FC layer 때문).
- Fast R-CNN은 feature map에서 proposal 영역을 찍는데, proposal 크기가 다 다르니 다음 FC layer dimension 불일치 → **RoI Pooling이 해결**.

### 1.10.3 RoI Pooling 동작
> 🔥**시험포인트 — 교수님이 한 번 더 읽어보라고 강조**🔥
> *"region proposal의 코디네이트를 integer value로 라운딩을 하고, 그 다음에 렉탱글을 7×7로 더 나누고, 그 다음에 거기서 각각의 7×7 셀에서 max pooling을 한다."*
>
> 1. Proposal을 feature map에 매핑 시 좌표 그리드와 정확히 안 맞음 → **rounding**.
> 2. Rectangle을 **7×7로 분할**.
> 3. 서브셀 크기는 제각각.
> 4. 각 서브셀에서 **Max Pooling** → 최대값 하나씩.
> 5. 결과: 어떤 proposal이든 같은 차원의 vector.

### 1.10.4 성능
- **Training 9배, Testing 200배 빨라짐**.
- 모든 걸 CNN 안에서 (Softmax classification + Bbox regression).

### 1.10.5 Multi-task Loss
> 🔥**시험포인트**🔥
>
> $$ L = L_{cls} + \lambda \cdot \mathbb{1}[u \geq 1] \cdot L_{loc} $$
>
> - 0번 클래스 = **백그라운드 클래스** (K+1 클래스).
> - **Classification loss**: 정답 클래스의 negative log probability.
> - **Localization loss**: Indicator $\mathbb{1}[u \geq 1]$ — u가 백그라운드(0)이면 무시. 오브젝트일 때만.
> - $L_{loc}$: **Smoothed L1 loss** — L1은 0에서 뾰족해 미분 안 됨 → 그 근처에 quadratic 넣어 미분 가능. -1~1은 quadratic, 그 밖은 linear.

## 1.11 Faster R-CNN

Fast R-CNN의 가장 느린 부분은 **Selective Search/Edgebox 부분** → **proposal까지 네트워크 안에서**.

### 1.11.1 Region Proposal Network (RPN)
- 각 feature map 셀마다 **K개의 anchor box** (사이즈/aspect ratio 변주).
- 두 branch:
  - **Classification (binary)**: 오브젝트일 것 같은가? → **2K**
  - **Bounding Box Regression**: 4 파라미터 → **4K**

### 1.11.2 동작 순서
1. RPN이 anchor가 object일지 binary 판별.
2. Object면 어떻게 transform할지 결정.
3. Transformed anchor로 feature 다시 떠서 진짜 object class classification.
4. 최종: **위치는 RPN에서, class는 classification head에서**.

> 🔥**시험포인트 가능**🔥
> *"여기서 일단 바이너리로 오브젝트일 것 같은지 아닌지를 판별을 했지만, 제가 정확히 기억이 나는데 여기서 다시 약간 백그라운드인지 아닌지를 **다시 한 번 판별하는 그런 스텝이 있을 겁니다**."*

### 1.11.3 성능
- RPN이 약 **10ms** — 기존 proposal 모듈의 bottleneck 해소.
- VGG + Selective Search 2000개 vs VGG + RPN 300개 → **RPN이 300개만 써도 성능 더 좋음**.

### 1.11.4 mAP@0.5 vs mAP
- **mAP@0.5**: IoU 0.5에서 mAP.
- **mAP** (general): IoU 0.5~0.95 0.05 단위로 평균. *"mAP인 거죠 어떻게 보면."* — 한 단계 더 추상화.

## 1.12 SSD (Single Shot Multibox Detector) — Google, 2016

### 1.12.1 개요
- **Single Shot** = 한 번에 (two-stage 아님).
- **Fully Connected 없이 모든 걸 Convolution**.
- R-CNN과의 큰 차이: **여러 스케일 feature map을 동시에 사용**해 다양한 스케일 오브젝트 디텍션.

### 1.12.2 네트워크 구조

VGG backbone:
- Conv4 feature map: **38×38**
- 이후 **19×19**, 더 작은...

> 🔥**시험포인트 — 멀티스케일 의미**🔥
> - **얕은 layer (큰 feature map)**: 작은 오브젝트 좋지만 semantic level 낮음.
> - **깊은 layer (작은 feature map)**: semantic 높지만 작은 오브젝트 못 찾음.
> - 작은 오브젝트는 왼쪽(얕은 곳), 큰 오브젝트는 오른쪽(깊은 곳).

### 1.12.3 Default Box
- Faster R-CNN의 anchor와 비슷. SSD에서는 **default box**.
- 각 셀마다 최대 **6개**: aspect ratio 1, 2, 3, 1/2, 1/3 + (ar=1에서) 인접 스케일 하나 더.
- 어떤 feature map에선 6개, 어떤 데선 4개.
- **총 8,732개의 default box**.

### 1.12.4 Prediction (모두 Convolution)
- 각 default box: (C 클래스 + 백그라운드) classification + 4 bbox regression.
- 셀마다 (C + 4) × K → 3×3 conv로 처리.

### 1.12.5 Loss Function
- 본질적으로 Faster R-CNN과 같음. **Smoothed L1**.
- 교수님: *"복잡하게 썼지만 사실 Loss function 자체는 Faster R-CNN과 똑같다."*

### 1.12.6 Hard Negative Mining

> 🔥**시험포인트 — 교수님이 "가장 중요"라고 명시**🔥
> *"디텍션 할 때 사실은 가장 중요한 건 뭐냐면은 요 **하드 네거티브 마이닝**입니다."*
>
> - 그냥 아무데서나 negative 뽑으면 너무 쉬워서 학습 효과 낮음.
> - **현재까지 학습된 디텍터가 positive로 잘못 판단한 sample**들을 negative로 계속 넣어줌.
> - → 어려운 negative를 정확히 negative로 분류하게 됨.

### 1.12.7 Dilated Convolution (Atrous Convolution)

> 🔥**시험포인트**🔥
>
> SSD의 FC6, FC7에서 사용.
> - 일반 3×3 conv: 연속 3×3에서 input.
> - **Rate 2 dilated**: 한 칸씩 띄어서.
> - **필터 파라미터 수는 같지만 receptive field가 더 넓다.**
>
> **Receptive Field 설명**:
> - 3×3 conv 한 번: receptive 3×3
> - 두 번 중첩: 더 커짐
> - **Pooling 쓰면 receptive 극적으로 커지나 feature map 사이즈 반으로 줄어듦.**
> - **Dilated convolution은 receptive만 키우고 feature map 사이즈는 유지** → 매우 효과적.
>
> **출처**: Google의 **DeepLab** (semantic segmentation)에서 처음 사용. 현재도 많이 씀.

### 1.12.8 성능 비교
- 객관적: **Faster R-CNN 성능 최고, 가장 느림**.
- SSD: 빠르지만 mAP 낮음.
- **명확한 trade-off**.

## 1.13 YOLO (You Only Look Once) — AI2

### 1.13.1 핵심 특징
> 🔥**시험포인트**🔥
> - **Object detection을 regression problem으로**.
> - **Single stage detection**.
> - 가장 인상적: **속도** — Fast 버전은 **155 FPS**.

교수님: *"성능은 초창기에 YOLO가 나왔을 때는 그렇게 좋지 않은데 되게 빠른 디텍터였다."*

YOLO가 지금까지 가장 유명한 이유:
- 다른 알고리즘보다 빠름(본질적이라기보다 사람들이 많이 써서 optimization 잘 됨).
- 새 버전이 계속 나옴 (강의 시점 v12/v13).

### 1.13.2 YOLO V1 구조

> 🔥**시험포인트 — 학습 방식**🔥
>
> 1. 이미지를 **S×S 그리드** (보통 **S=7, 7×7**)로 나눔.
> 2. 각 오브젝트의 **bbox 중심이 어느 셀에 떨어지는지** 보고, 그 셀이 담당.
> 3. 각 셀은 **B=2개의 bbox** prediction 가능.
> 4. 각 셀 prediction:
>    - 각 box: (x, y, w, h, confidence) = 5개
>    - 클래스 확률 (Pascal VOC면 20개)
> 5. 총 output: 7×7×30 = **1470개** (5×2 + 20 = 30)

**Confidence Score** = P(Object) × IoU
- P(Object): 0 또는 1
- IoU: 오브젝트 있을 때 얼만큼 오버랩

### 1.13.3 총 박스 수 비교 (단골)

> 🔥**시험포인트**🔥
> - R-CNN: 2000개 proposal
> - Faster R-CNN: grid × K = 수백 개
> - SSD: 8,732개
> - **YOLO: 7×7×2 = 98개** — 극단적으로 적음

교수님: *"누구에게나 그럴듯한 계획이지만 사실은 98개가 너무 적기 때문에 **성능은 굉장히 낮았습니다**."*

### 1.13.4 YOLO V1의 한계 — "한 놈만 패는" 문제

> 🔥**시험포인트**🔥
> 각 셀이 2개 bbox만 → 여러 오브젝트의 센터가 같은 셀에 떨어지면?
>
> *"이 논문은 어떻게 하냐? 논문은 그냥 포기합니다. 하나만 찾아요. 하나만. **한 놈만 패는 거죠**, 어떻게 보면."*

### 1.13.5 성능 비교
- Fast YOLO: 155 FPS, mAP 52.7
- YOLO: 63.4 mAP
- Faster R-CNN: 73.2 mAP — *"상대도 안 되게 좋은"*, 매우 느림.
- YOLO with VGG16: 66.4 mAP

### 1.13.6 분석
- 저자들 분석: **가장 큰 문제는 localization error**.
- 백그라운드에 false positive 많음.

## 1.14 YOLO V2 — Better, Faster, Stronger

교수님 평:
- **Stronger** = 관념적
- **Better** = 여러 트윅으로 성능 향상
- **Faster** = 속도 개선

성능: 67 FPS에서 76.8 mAP, 40 FPS에서 78.6 mAP (Pascal VOC 2007).

### 1.14.1 Stronger — 9000 클래스 디텍션
- ImageNet은 detection label 없고 classification label만.
- WordNet hierarchy 활용해 detection label 없는 클래스도 prediction.
- 총 9000 클래스. 교수님 평: *"이건 우리가 할 수 있었다는 그런 걸 보여주는 것에 가까웠고요. 그냥 해봤다 그런 차원입니다."*

### 1.14.2 Better — 성능 개선 테크닉

**(1) Batch Normalization** (2015): batch마다 activation normalize. 학습 안정화.

**(2) High Resolution Classifier**: 마지막 pooling 제거 → feature map 7×7 → **13×13**. Input 416×416. Localization 좋아짐.

**(3) Anchor Box**
> 🔥**시험포인트**🔥
> - YOLO V1은 98개 → 너무 적음.
> - V2는 **Faster R-CNN/SSD 같은 anchor box**.
> - 한 셀에 여러 오브젝트도 handling.
> - 1000개 넘는 박스 → mAP 0.3%p 감소 but **Recall 7%p 증가**.

**(4) Dimension Priors — "치팅" 느낌**
> 🔥**시험포인트 — 교수님이 비판적으로 언급**🔥
> - Faster R-CNN, SSD는 anchor box를 사전 정보 없이 정함.
> - YOLO V2는 **training data의 GT bbox들을 모아 clustering** → cluster된 box를 anchor로.
> - *"프라이어 날리지를 좀 과도하게 쓴 경향이 있다... **약간 좀 애매하지만 좀 치팅의 느낌이 좀 있다**."*

**(5) Direct Location Prediction (Sigmoid)**
> 🔥**시험포인트**🔥
> - 기존 bbox regression: transformation 파라미터 그대로 regression → **잘못되면 산으로 갈 수 있음**.
> - V2는 **sigmoid**로 0~1 사이 bounded → 학습 stable.

**(6) Pass-through (Fine-Grained Features)**
> 🔥**시험포인트**🔥
> - 여러 layer feature map을 합쳐 쓰는 경우. 사이즈 다름 (26×26 vs 13×13).
> - 해결: 큰 26×26 feature map을 13×13으로 줄이되 pooling/conv가 아니라 **2×2 패턴을 4개 채널로 펼침** → 채널 수 4배.
> - 그 다음 13×13과 채널 방향 concat.

**(7) Multi-Scale Training**: 학습 때도 여러 스케일 → 성능 향상.

**(8) New Backbone — DarkNet**: VGG/GoogLeNet보다 FLOPs 적음 → 더 빠름.

## 2.2 RetinaNet (2017) — "사실 이거 소개하려고 준비한 건 Focal Loss다"

교수님이 본격적으로: *"RetinaNet 자체에 새로운 아키텍처가 있는 건 아니지만, **Focal Loss를 소개하려고** 오늘 준비했다."* 논문 제목도 *"Focal Loss for Dense Object Detection"*.

### 2.2.1 Cross Entropy의 한계 — 시험 비유

교수님의 인상적 비유:
> *"시험 볼 때 70-80점 맞는 건 쉽지만 100점 가까이 맞는 건 어려운 거랑 비슷한 거죠."*
>
> - 모델 아웃풋은 어차피 argmax. 정답 클래스 스코어를 1까지 올릴 필요 없음.
> - 바이너리는 0.5만 넘으면 OK, 1000 클래스면 0.23만 돼도 1등 가능.
> - **스코어 0.5는 쉽지만 0.9~1로 만드는 건 어려움** → 어려운 거 하느라 **애매한 애들·틀리는 애들의 성능 향상에 소홀**.

### 2.2.2 Focal Loss 수식

> 🔥**시험포인트**🔥
>
> $$ \text{Cross Entropy: } -\log(p) $$
> $$ \text{Focal Loss: } -(1-p)^\gamma \log(p) $$
>
> - $p$: 정답 클래스 확률
> - $\gamma$: 하이퍼파라미터 (보통 2)
> - $p$가 1에 가까워질수록 $(1-p)^\gamma$ → 0 → 로스 **익스포넨셜 감소**

그래프 해석:
- 파란색 = Cross Entropy (γ=0).
- γ=5(가장 극단): 정답 클래스 스코어가 낮을 때 조금만 높여도 로스 많이 감소.
- 처음에 모델 잘 안 됐을 때 낮은 스코어 애들에 튜닝 잘 됨.
- 반대로 스코어 0.5 정도면 더 높여도 베네피트 없음.

> 🔥**시험포인트**🔥
> *"레티나넷에서 아마 포컬로스가 처음 나온 것 같고, 최근에 나온 논문들 보면 포컬로스를 썼다는 게 되게 많을 거예요."*
>
> 마지막 정리 슬라이드에서도 강조:
> *"오늘 **레티나넷, 포칼로스 같은 건 되게 중요하니까 기억하시면 좋을 것 같고**."*

### 2.2.3 Dilated Convolution 재정리

지난 시간에 빨리 넘어간 부분 다시:
- Receptive field 극적으로 키우는 건 **풀링** — 단점 feature map 사이즈 ↓.
- Detection/Segmentation 같은 dense prediction에서는 작은 오브젝트 다루기에 패널티 큼.
- **Dilated convolution**: rate로 인접하지 않은 element를 띄엄띄엄 봄. **receptive 키우면서 feature map 사이즈 유지**.

교수님의 솔직한 정정:
> *"사실은 그 얘기도 솔직히 말씀드리면 약간 좀 과장된 게 있는데... 2×2에서 맥스 풀링을 하고 스트라이드를 1로 하면 피쳐맵 사이즈는 작아지지 않습니다. **풀링을 해서 작아지는 게 아니고 사실은 스트라이드를 2로 하기 때문에 피쳐맵이 작아지는 거예요**. 약간 그거는 좀 말장난인 부분도 있는데."*

> 🔥**시험포인트**🔥
> *"최근에는 일반적인 컨볼루션보다 다이얼리티드 컨볼루션을 굉장히 많이 써요. 그리고 또 한 가지가 크로스 엔트로피로스에 비해서 최근에는 **포컬로스를 굉장히 많이 쓴다**."*

### 2.2.4 Feature Pyramid Network (FPN)

- CNN을 태우면 여러 레이어 feature map. 밑은 크고, 위는 시멘틱하지만 작다.
- 작은 오브젝트는 큰 feature map이, 큰 오브젝트는 작은 feature map이 담당.
- 문제: 시멘틱 레벨이 다름. 위 레이어가 성능 좋은데 사이즈가 작아 불리.
- 해결: **feature map들을 합친다.** 작은 걸 업샘플링해서 합침. 더 나아가 디코딩 레이어를 더 두어 사이즈를 키움.

### 2.2.5 Focal Loss 임팩트 분석
누적분포 그래프:
- Positive example: Cross Entropy는 스코어 낮은 애들 비율 많아 손해. Focal Loss는 줄어듦.
- Negative example: 더 극명. Focal Loss는 스코어 높은 애들 거의 없음.

## 2.3 FCOS (Fully Convolutional One-Stage, 2019)

### 2.3.1 Two-stage vs One-stage — 교수님 견해

> 🔥**시험포인트**🔥
> *"얘네들이 주장하는 2 stage 1 stage는 제가 보기에는 좀 좀 좀 이상한 것 같고요 결국은 뭐 큰 차이는 없는 것 같습니다. 근데 요 **FCOS는 조금 달라요. FCOS는 기존 object detection과 다른 게 좀 더 원스테이지라는 말이 어울리는 것 같아요**."*
>
> *"원스테이지 디텍션 알고리즘이라고 부르는 것 중에 이게 정말로 원스테이지스럽고, 다른 거는 원스테이지라고 하지만 사실상 투 스테이지랑 별로 상관이 없어 보이는데, **얘들이 훨씬 더 원스테이지라는 말에 어울리는 방법**인 것 같다."*

### 2.3.2 FCOS의 핵심 — 세그멘테이션처럼
- **앵커박스나 프로포절 없다.**
- Feature map의 **모든 location마다 prediction** (semantic segmentation처럼).
- 단점: 쓸데없는 디텍션이 많이 나오고, 같은 오브젝트에 여러 픽셀 디텍션 가능. (NMS로 보통 해결하는데 FCOS는 NMS가 없다고 교수님 기억)

### 2.3.3 Loss 구성
- 모든 location마다 **classification loss (Focal Loss)**.
- **Regression loss**: IoU 기반.
- Indicator: GT 오브젝트 안 location에만 regression loss.

### 2.3.4 Centerness Loss — FCOS의 시그니처

> 🔥**시험포인트 — 핵심 정의**🔥
>
> 현재 픽셀이 bbox 안에 있을 때, bbox의 left/right/top/bottom까지 거리 L, R, T, B.
>
> $$ \text{Centerness} = \sqrt{\frac{\min(L,R)}{\max(L,R)} \times \frac{\min(T,B)}{\max(T,B)}} $$
>
> - 센터: min=max라서 centerness=1
> - 한쪽 치우치면 0에 가까움
> - 0~1 사이 값 → binary cross entropy loss로 학습 → 센터에서 멀면 페널티

### 2.3.5 인퍼런스 — 최종 스코어

> 🔥**시험포인트**🔥
> *"그 스코어가 뭐냐면 **classification 스코어 곱하기 centerness 스코어**에요. 얼마나 그 클래스에 대한 확신이 있느냐 라는 거에다가 가급적이면 더 센터에 가까운 애가 픽스하는 게 더 맞다고 보는 거죠."*

### 2.3.6 FCOS의 장점
> *"FCOS 같은 거는 (다른 디텍션이 신경 쓸 게 많은데) 세그멘테이션 파이프라인이랑 되게 비슷해서 그런 게 별로 없어요. 그래서 훨씬 더 **깔끔하다**는 장점이 있습니다."*

### 2.3.7 Limitation
가장 크리티컬: **오버랩된 오브젝트**. A와 B가 highly overlap되어 있을 때 픽셀마다 그냥 regression시키면 자기가 찾기 쉬운 것만 찾을 수 있음(예: 큰 사람만). 멀티레벨 feature map, regression 파라미터 바운드로 해결 시도.

## 2.5 DETR (DEtection TRansformer)

### 2.5.1 DETR 소개 — 컨셉적으로 새로움

> 🔥**시험포인트 — DETR의 핵심 차별점**🔥
>
> - **트랜스포머 기반**
> - **Bipartite matching** 수행
> - **Handicraft 앵커가 없다**
> - **Non-maximum suppression이 없다** ← 결정적
> - = **NMS-free + Set Prediction + 1-to-1 매칭**

교수님:
> *"디터 그린 그림을 보면 그냥 입력 넣고 아웃풋이 나오는 중간 단계가 없는 듯한 굉장히 썰렁한 그림이 나온다. 이게 디터의 장점이다."*

### 2.5.2 기존 디텍터 NMS의 문제
1. **NMS threshold에 매우 민감** — 결과가 크게 달라짐.
2. **다른 오브젝트끼리 겹칠 때 두 개로 인식해야 하는데 하나가 지워지는 경우** 발생.

### 2.5.3 Object Query
- 앵커 없는 대신 **query token**을 트랜스포머 입력으로.
- 입력에 뭐 들어가야 할지 모르니 **학습**.
- 학습 시 더미 토큰 + 같이 업데이트, 인퍼런스 시 학습된 토큰 사용 (입력 이미지와 무관).
- 더미 토큰 의미: 교수님 추측 — *"YOLO v2에서 GT bbox들을 클러스터링해서 앵커를 만든 거랑 비슷한 일을 하는 게 아닌가."* 즉 frequently observed 위치 임베딩.
- 토큰 수는 몇백 개 (트랜스포머에 너무 많이 넣으면 안 좋으니까).

### 2.5.4 Bipartite Matching + Hungarian Algorithm

> 🔥**시험포인트**🔥
> - Object query set과 GT bbox set 간 **명시적 1대1 매칭**.
> - **Hungarian algorithm**: 두 set 사이 가장 어울리는 매칭을 글로벌하게 찾아주는 알고리즘.
> - 매칭 결과 σ. σ_i는 i번째 query와 매칭되는 GT 인덱스.

### 2.5.5 Matching Cost
- **Classification term**: i번째 bbox가 실제 오브젝트와 매칭됐을 때의 classification 확률
- **Box regression term**: L1 차이 + (1 - GIoU)

### 2.5.6 GIoU (Generalized IoU) — 매우 중요

> 🔥**시험포인트 — 단골**🔥
>
> - A와 B 두 bbox, C는 두 박스를 둘러싸는 가장 타이트한 영역.
> - **GIoU = IoU - (C \ (A∪B)) / C**
>
> **IoU의 한계**: A와 B가 오버랩 없으면 IoU = 0 → 어떤 위치든 똑같이 0 → **그래디언트 없음**.
>
> **GIoU**: 멀리 있을수록 enclosing box C 영역이 커져 페널티 → 그래디언트 흐름 생김.
>
> 교수님: *"Object query가 GT를 찾아가야 되는데, 재수없게 GT bbox랑 학습된 query로 얻은 bbox 로케이션들이 오버랩이 하나도 안 되어 있는 애가 있을 수 있잖아요. 그냥 IoU를 쓰면 얘랑 얘랑 구별이 안 되니까... GIoU를 쓰면 이 영역이 패널티가 되니까 얘를 더 고를 수 있을 거란 얘기죠."*

### 2.5.7 Detection Loss
거의 매칭 코스트와 같은 꼴인데:
- 매칭할 땐 그냥 확률, detection loss는 log probability.
- Classification loss는 **모든 bbox에 적용** (background class 스코어 포함).
- Box loss는 **실제 오브젝트가 있는 애들에 대해서만**.

### 2.5.8 매칭과 학습의 분리

> 🔥**시험포인트 — 개념적 정리**🔥
> *"이 부분은 학습을 하는 부분, 로스가 백프롭 되는 건 아니고요. **현재 학습된 모델을 기반으로 이런 것들을 계산해가지고 매칭을 하는 거예요**."*
>
> 흐름: **매칭(Hungarian, 옵티마이저) → σ 결과 → detection loss 계산 → 백프롭**.

### 2.5.9 DETR Architecture
- **CNN backbone** (ResNet, 32x downscale, 2048 채널)
- **2D positional encoding** (이미지가 2D이므로)
- **Transformer encoder**: feature map + positional encoding
- **Transformer decoder**: object query를 **parallel**하게 디코딩 (autoregressive 아님)
- **Prediction FFN** (3-layer perceptron, ReLU): classification + bbox regression
- 오브젝트 없으면 **null class**

### 2.5.10 Positional Encoding 디테일
> 🔥**시험포인트 — DETR PE의 특징**🔥
> 1. **2D Positional Encoding** (Y축으로 하나 더 생긴 형태)
> 2. **Query/Key에만 더하고 Value에는 안 더함**
> 3. **매 레이어마다 추가** (입력에 한 번만이 아님)
>
> 또한 입력 이미지 사이즈가 다 다를 수 있으니 **Binary mask** (실제 이미지 부분 1, 패딩 0) 사용. Cumulative sum으로 각 픽셀의 x/y 좌표 만들어냄. 그 다음은 일반 sin/cos PE와 비슷.

교수님 평: *"쓸데없이 복잡하게 썼다."*

### 2.5.11 DETR 결과
> *"결과는 디터가 사실 Faster R-CNN에 비해서도 특별히 엄청 좋다 그런 느낌은 아닌데, 여기서는 좀 좋네요."*
>
> *"이 당시에 어떻게 하면 트랜스포머를 비전 테스크에 잘 쓸 수 있을까 그런 게 굉장히 중요했던 시기이기 때문에 그런 크레딧이 가장 큰 것 같습니다."*

Self-attention map 시각화로 각 오브젝트의 attention 볼 수 있음.

## 2.6 Deformable DETR

### 2.6.1 DETR의 약점

> 🔥**시험포인트**🔥
> 1. **느리다.** 학습 컨버전스 느림 — **DETR 500 epoch vs Faster R-CNN 50 epoch**.
> 2. **작은 오브젝트 디텍션 어려움.** 레졸루션 커지면 토큰 수 증가 → 컴퓨테이션 급증.
> 3. **인퍼런스 느림.**

### 2.6.2 Deformable Attention 핵심 아이디어

> 🔥**시험포인트**🔥
> - Attention: query와 모든 key의 outer product → softmax → value 곱. Quadratic 복잡도.
> - **Key 위치를 샘플링**해서 query가 보는 key 개수를 제한.

교수님이 설명한 **Deformable Convolution** 비유:
> *"디포머블 컨볼루션이 뭐냐면, 컨볼루셔널 필터는 3×3면 윈도우에서 컨볼루션 하는 거잖아요... 디포머블 컨볼루션은 그렇게 하지 않고 **필터의 각각의 파라미터가 자기가 뭐랑 곱해져야 되는지를 찾아가는 거예요**. 필터 사이즈가 더 이상 스퀘어가 아니고 학습된 옵셋 파라미터에 의해서 필터의 쉐입이 다이나믹하게 결정되는 거죠."*

같은 맥락으로 deformable attention은 각 query마다 **sampling offset Δp**를 학습해 소수 key 위치만 본다.

### 2.6.3 복잡도 비교
- 원래: Nq × Nk × C²
- Deformable: Nk가 샘플된 작은 개수 K로 감소
- Encoder complexity: spatial resolution에 linear
- Decoder complexity: spatial size와 무관, Nk와만 상관

### 2.6.4 Multi-Scale Deformable Attention

> 🔥**시험포인트**🔥
> - 작은 오브젝트 단점 보완: 여러 스케일 동시 attention.
> - K개 위치 샘플링 × L개 레이어 = **LK개 키**.
> - **별도의 FPN 안 써도** attention 자체가 멀티스케일 정보 통합.

### 2.6.5 두 가지 추가 트릭

**(1) Iterative Bounding Box Refinement**
- Decoder layer마다 bbox refine
- 이전 레이어 결과를 받아 다음 레이어에서 업데이트

**(2) Two-Stage Deformable DETR**
- 인코더에서 **region proposal** 생성
- 4개 멀티스케일 피처에 디텍션 헤드 → prediction → 스코어 높은 애들만 디코더로 → **object query로 사용**
- 원래 DETR object query는 학습된 고정값 → 이미지마다 다른 게 더 좋음
- 학습된 query 대신 인코더 출력에서 뽑은 proposal 사용

### 2.6.6 결과
- Training epoch이 한 10배 정도 줄어듦
- 작은 오브젝트 AP 좋아짐
- Two-stage variant가 추가 성능 향상

## 2.7 RT-DETR (Real-Time DETR)

### 2.7.1 모티베이션
Deformable DETR도 여전히 느리니까 진짜 real-time을 노림. 주로 다룬 건 **NMS 모듈 비판** (YOLO와 비교).

### 2.7.2 YOLO NMS의 단점

> 🔥**시험포인트**🔥
> - NMS에 **confidence threshold, IoU threshold** 같은 하이퍼파라미터 필요
> - NMS 시간이 **입력에 따라 다름** (중복 prediction 많으면 더 오래)
> - 하이퍼파라미터에 sensitive
> - YOLO inference 시간 측정 시 NMS 시간 빼고 보고 → **unfair**
> - Anchor-based detector가 anchor-free보다 **3배 더 많은 bbox** 출력 → NMS 시간 더 많음
> - YOLOv5(anchor-based) vs YOLOv8(anchor-free) 박스 개수 비교

교수님 농담: *"디턴은 옛날부터 NMS가 없었는데 NMS 얘기를 요로 해서 하면서, 뭐 하여튼 그건 좀 이상한 것 같고."*

### 2.7.3 DETR의 문제 (RT-DETR이 해결)
- NMS 없지만 **자체로 느림**
- 멀티스케일 피처 → 토큰 개수 많아짐
- 인코더 아예 없애면 GFLOPs 49% 줄고 accuracy 11% drop — 11%면 작은 숫자 아님
- 피처가 classification에는 유용하지만 **localization에는 별로**

### 2.7.4 RT-DETR Architecture
- **Backbone**: ResNet, 마지막 3 stage (S3, S4, S5)
- **Efficient Hybrid Encoder**: AIFI + CCFF
- **Uncertainty-Minimal Query Selection**
- **Decoder** (iterative refinement)

### 2.7.5 Efficient Hybrid Encoder

> 🔥**시험포인트 — 발견**🔥
> **로우 레벨 피처에서의 intra-scale interaction은 별로 필요 없다** (시멘틱스 적으니까).

두 모듈:
- **AIFI (Attention-based Intra-scale Feature Interaction)**: **S5에만** single-scale transformer로 intra-scale attention. 시멘틱 리치한 피처에만.
- **CCFF (CNN-based Cross-scale Feature Fusion)**: S3, S4, F5(=S5 attention 결과)로 cross-scale fusion. **PAN-Net style** 양방향. 1×1 conv + N개 RepConv block + element-wise add.

### 2.7.6 Uncertainty-Minimal Query Selection

> 🔥**시험포인트**🔥
> - Vanilla query selection은 classification score만 사용 → localization 나쁨.
> - **Uncertainty = classification과 localization distribution의 차이**.
> - Classification 잘 되는 애가 localization도 잘 되어야 함 (correlation).
> - Minimize 로스 추가 → 클래스피케이션·로컬라이제이션 둘 다 높은 피처가 **120% 더 나옴**, AP 향상.

교수님: *"꼭 그래야 되는지는 애매하긴 하지만 말이 아주 안 되는 건 아닌 것 같아요."*

### 2.7.7 Flexible Decoder
- Decoder layer 6개 → 5개 써도 AP 0.1%만 떨어지고 레이턴시 줄어듦
- 따로 학습 안 하고 auxiliary detection head로 처리
- 스피드 sensitive한 상황에서 layer 생략 가능

### 2.7.8 결과
- YOLO 대비 FPS와 AP 측면에서 빠름
- DINO backbone + Deformable DETR보다 빠르고 좋음
- 참고 (실습에서 본 수치): **DETR 28 FPS vs RT-DETR 108 FPS (논문 3.86배)**

# 부록 B. 시험 전 빠른 복습 — 한 페이지 정리

## 6월 2일 오전 (Object Detection 기초 & 1세대 디텍터)

### 평가 (Evaluation)
- **IoU**: Intersection / Union. **0.5 이상**이 디텍션. 0.5는 의외로 큰 오버랩.
- **Precision / Recall / TP, FP, FN, TN**.
- **Detection PR Curve의 지그재그 형태** — 일반 ML과 다름.
- **AP = PR curve 아래 면적, mAP = 클래스별 AP 평균, mAP@0.5 vs general mAP (IoU 0.5~0.95)**.

### Object Proposal
- 핵심 properties: **Recall이 가장 중요, 빠르게, 다양한 스케일/aspect ratio**.
- **Selective Search**: superpixel hierarchical / **Edgebox**: edge map 기반.
- **숫자**: Edgebox 800박스로 recall 75%, Selective Search 1400개 필요. Edgebox 40배 빠름.

### R-CNN family
- **R-CNN의 진짜 contribution = Bounding Box Regression** (지금도 유니버셜).
- **Fast R-CNN**: CNN 한 번 + **RoI Pooling** (rounding → 7×7 분할 → max pool). Training 9배, Testing 200배 빠름.
- **Multi-task Loss**: cls + λ × 𝟙[u≥1] × loc. Loc은 **Smoothed L1** (-1~1은 quadratic).
- **Faster R-CNN**: **RPN** 으로 proposal까지 네트워크 내. 약 10ms. 300개로 SS 2000개보다 좋음.

### SSD (Google)
- Single shot, all convolution, **multi-scale feature maps** (작은 오브젝트는 얕은 곳, 큰 건 깊은 곳).
- **Default box 8,732개**. aspect ratio 1, 2, 3, 1/2, 1/3.
- **Hard Negative Mining** = 디텍션에서 "가장 중요한 것" (교수님 명시).
- **Dilated Convolution** (DeepLab 출처): receptive field 키우되 feature map 사이즈 유지.

### YOLO (AI2)
- Regression problem으로, **Single stage**, **Fast YOLO 155 FPS**.
- **V1**: 7×7 grid, 셀당 2 박스 = **총 98개 박스** (극단적으로 적음).
- **V1 한계**: 같은 셀에 여러 오브젝트 센터 → "한 놈만 패는" 문제.
- **V2 (Better)**: BN, High Resolution Classifier, **Anchor box**, **Dimension Priors (clustering, "치팅" 느낌)**, **Sigmoid (학습 안정)**, **Pass-through (2×2 → 4채널)**, Multi-Scale Training, DarkNet.
- **박스 수 비교**: R-CNN 2000 / Faster R-CNN 수백 / SSD 8732 / YOLO 98.

## 6월 4일 오전 (RetinaNet, FCOS, YOLOv7, DETR family)

### RetinaNet (Focal Loss + FPN)
- **Focal Loss**: $-(1-p)^\gamma \log p$. γ=2가 보통. **"100점 만점" 비유** — 어려운 거에 집중하느라 애매한 거 소홀해지는 문제 해결.
- **RetinaNet에서 Focal Loss 처음 등장**. 최근 매우 많이 쓰임.
- **Dilated Convolution** 재정리: receptive field 키우면서 feature map 사이즈 유지.
- **FPN**: 멀티스케일 feature map 합치기 (업샘플링 + 디코딩 레이어).

### FCOS (진정한 One-Stage)
- 교수님 명시: *"이게 정말로 원스테이지스럽다."*
- **앵커 없음, proposal 없음**. **모든 location마다 prediction** (세그멘테이션처럼).
- Loss: classification (**Focal Loss**) + regression (IoU 기반).
- **Centerness Loss**: $\sqrt{(\min(L,R)/\max(L,R)) \times (\min(T,B)/\max(T,B))}$ — 0~1, BCE 학습.
- **최종 score = classification × centerness**.
- Limitation: 오버랩된 오브젝트.

### YOLO v7 (교수님은 회의적)
- "Bag of freebies". 트레이닝 코스트는 늘어도 인퍼런스 영향 X.
- **RepConv**: 학습 시 3×3 + 1×1 + identity 병렬 (activation 없음) → 인퍼런스 시 하나의 conv로 collapse. ResNet에는 identity 빼야.
- ELAN/Extended ELAN, Compound Scaling, Coarse-to-Fine Label Assignment.
- 교수님 평: *"이게 좋아야 되는 이유가 하나도 없다. 그냥 GPU가 좋아져서."*

### DETR
- **트랜스포머 기반, NMS-free, no anchors, set prediction**.
- **Object Query** (학습된 토큰). 토큰 수 수백 개.
- **Bipartite Matching + Hungarian Algorithm**: 1대1 매칭.
- **Matching Cost**: classification + L1 + (1-GIoU).
- **GIoU = IoU - (C \ (A∪B)) / C** — IoU의 그래디언트 0 문제 해결.
- **매칭은 백프롭 안 됨**, 매칭 후 detection loss 계산해서 백프롭.
- Architecture: CNN backbone + **2D PE** + Transformer Encoder + Decoder + FFN.
- **DETR PE 특징**: 2D / Q,K에만 (V엔 X) / 매 레이어마다.
- 약점: **500 epoch (vs Faster R-CNN 50)**, 작은 오브젝트, 인퍼런스 느림.

### Deformable DETR
- **Deformable Attention**: query마다 sampling offset Δp 학습 → **K개 key만** attention.
- 복잡도: Encoder는 spatial resolution에 linear, Decoder는 spatial size 무관.
- **Multi-Scale Deformable**: K × L개 키 → 별도 FPN 불필요.
- **Iterative Bbox Refinement** + **Two-Stage Deformable DETR** (인코더가 region proposal → 디코더 object query로).
- Training epoch 약 10배 감소.

### RT-DETR
- **YOLO NMS 비판**: 하이퍼파라미터, 입력에 따라 시간 다름, anchor-based가 박스 3배.
- **Efficient Hybrid Encoder**:
  - **AIFI** (S5에만 intra-scale attention)
  - **CCFF** (PAN-Net style cross-scale)
- **Uncertainty-Minimal Query Selection**: classification ↔ localization alignment를 uncertainty로 모델링.
- Flexible Decoder: layer 5개로도 OK.
- DETR 28 FPS → RT-DETR 108 FPS (논문 3.86배).
