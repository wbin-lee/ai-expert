## 2.2 성능 평가가 왜 사이언스인가

- 조직이 망하는 길 중 하나는 **숫자로 장난치는 것**.
- 인텔 옛 사례: 특정 컴파일러 플래그를 숨겨놓고 그 플래그가 있으면 미리 핸드튜닝된 어셈블리 바이너리가 나오게 만들어 벤치마크 부풀린 일.
- 삼성도 발열 관련 게임 벤치마크 플레이로 한 차례 철퇴를 맞은 적 있음.
- 그러므로 우리는 **공평한 측정과 의미 있는 매트릭**을 알아야 한다.

## 2.3 성능의 정의

> ⭐ "성능"은 평가하는 사람이 무엇을 보느냐에 따라 다르다.

비행기 예시:

| 매트릭 | 의미 |
|---|---|
| Passenger Capacity | 사람 많이 태우기 |
| Range | 한 번 주유로 멀리 가기 |
| Cruising Speed | 최고 속도 |
| Composite (capacity × speed × range) | 종합 KPI |

곤란한 질문엔 "정의에 따라 다르다"고 빠져나갈 수 있다 (강사 농담).

# 3. 펀더멘탈 매트릭 — Time과 Rate

## 3.1 두 펀더멘탈

| 종류 | 매트릭 | 예시 |
|---|---|---|
| **Time** | 주어진 일을 끝낼 때까지 걸리는 시간 | Response time, Latency |
| **Rate** | 단위 시간당 처리량 | Throughput, QPS, MIPS, FLOPS |

추가로 **Energy per Operation** 이 또 하나의 핵심 매트릭.

## 3.2 Wall-clock Time

- 시작 ~ 종료까지 측정한 가장 정직한 시간.
- 모든 것을 포함: CPU 계산, 디스크 I/O, 네트워크, 렌더링 등.
- 예: 웹브라우저 벤치마크는 `samsung.com` 친 다음 화면이 다 그려질 때까지의 시간.

## 3.3 CPU Time

- 좁게 본 시간: **CPU가 실제로 busy하게 일한 시간**만.
- I/O 대기 중에는 CPU가 놀기 때문에 wall-clock과 다르다.

| 구분 | 의미 |
|---|---|
| **User Time** | 어플리케이션 코드가 도는 시간 |
| **System Time** | OS (네트워크/스토리지 등)가 도는 시간 |

`time` 명령어 예시: 전체 159초 중 90.7초가 user, 12.9초가 system → CPU 활용도 약 65%.

# 4. CPU Performance Equation

## 4.1 Iron Law of CPU Performance

> ⭐ ⭐ ⭐ ⭐ 수업의 핵심 공식.

\[
\text{CPU Time}
=
\text{Instruction Count (IC)}
\times
\text{CPI}
\times
\text{Cycle Time}
\]

| 항 | 의미 | 결정 요인 |
|---|---|---|
| **IC** | 명령어 개수 | 알고리즘 / 컴파일러 / ISA |
| **CPI (Cycles per Instruction)** | 명령어당 평균 사이클 | ISA / Microarchitecture (조직 구조) |
| **Cycle Time** | 한 사이클의 시간 | 공정 (2nm / 3nm) / 파이프라인 구조 |

예시: 1 GHz CPU → 1 cycle = 1 ns. 4 GHz → 0.25 ns.

## 4.2 CPI는 프로그램에 의존한다

- ALU 명령어: CPI ≈ 1
- Load / Store: CPI ≈ 2 (메모리 접근 느림)
- Branch: CPI ≈ 2 (조건문 분기)

따라서 명령어 mix에 따라 평균 CPI가 다르다.

## 4.3 두 CPU 비교

| 명령어 종류 | S사 CPU | Q사 CPU |
|---|---:|---:|
| ALU | 1 | 1 |
| Load/Store | 더 느림 | — |
| Branch | — | 더 느림 |
| 평균 CPI | 2.1 | 1.8 |

⭐ 핵심 메시지: **CPI만 보고 성능을 단정할 수 없다.** Cycle Time까지 함께 봐야 한다.

- Intel은 전통적으로 **Cycle Time을 최소화** (클럭 ↑).
- AMD는 **CPI를 줄이는** ILP 강화에 더 집중.

## 4.4 사용자가 컴퓨터 살 때 빠지는 함정

소비자는 보통 클럭 스피드만 보고 결정한다. 하지만 클럭만으로는 성능이 결정되지 않는다 — IC × CPI도 같이 봐야 함.

# 5. RISC vs CISC

## 5.1 두 아키텍처 개념

| 분류 | 명령어 개수 | 명령어 복잡도 | 하드웨어 |
|---|---:|---|---|
| **RISC** (Reduced ISC) | 수십~100개 | 단순 | 가볍고 클럭 ↑ 가능 |
| **CISC** (Complex ISC) | 1,500개 이상 (x86) | 명령어 하나가 매우 강력 | 복잡, 클럭 한계 |

## 5.2 동일 프로그램 비교 (MIPS vs VAX)

| 매트릭 | RISC | CISC |
|---|---|---|
| Instruction Count | ↑ (약 2배) | ↓ |
| CPI | ↓ | ↑ (약 4배) |
| Cycle Time | ↓ | ↑ |

\(\text{IC} \times \text{CPI}\) 만 보면 어느 쪽이 좋다 단정할 수 없다. 세 항을 모두 곱해야 의미가 있다.

> ⭐ **단일 매트릭만 보고 결론 내리면 안 된다.**

# 6. Rate Metric의 함정

## 6.1 MIPS / FLOPS / TOPS

- **MIPS** = Million Instructions Per Second
- **FLOPS** = Floating-point Operations Per Second
- **TOPS** = Tera Operations Per Second (요즘 NPU 광고에 흔함)
- **QPS** = Queries Per Second (서버, 검색엔진)

## 6.2 MIPS의 한계

- 명령어 복잡도를 반영하지 않는다.
- RISC가 같은 프로그램에서 항상 MIPS가 높지만, 그것이 곧 성능 우위는 아님.
- 같은 컴퓨터에서도 프로그램마다 MIPS가 매우 달라진다.

## 6.3 Ratio (Speedup)

같은 벤치마크 wall-clock 시간 기준:

\[
\text{Speedup}_{X \text{ over } Y}
=
\frac{T_Y}{T_X}
\]

- 1.25 = X가 Y보다 25% 더 빠르다.
- 10배 = 1 order of magnitude.
- 100배 = 2 orders of magnitude.

# 7. 여러 벤치마크 요약 — 평균을 어떻게 낼 것인가

## 7.1 핵심 원칙

> ⭐ ⭐ ⭐ ⭐ 시험에 잘 나오는 국룰.

| 평균 | 사용 대상 |
|---|---|
| **Arithmetic Mean** | 시간 (Execution Time) |
| **Harmonic Mean** | 처리율 (Rate, MIPS, FLOPS, QPS) |
| **Geometric Mean** | 비율 (Speedup, Ratio) |

## 7.2 Arithmetic Mean 함정

10개 벤치마크 중 9개가 1~2초, 1개만 100초인 경우, 단순 평균은 그 100초에 거의 다 끌려간다. 이를 보정하려면 **Weighted Arithmetic Mean**을 쓰고, 보통 wall-clock의 역수를 weight로 둔다.

## 7.3 Harmonic Mean

\[
H = \frac{n}{\sum_i \frac{1}{x_i}}
\]

예: 10, 100, 50, 20 MFLOPS → \(H \approx 22.2\) MFLOPS.

## 7.4 예제 문제

벤치마크 X, Y, Z를 두 머신 A, B에서 wall-clock 기준으로 측정한 결과, 어리스매틱 평균과 MIPS 하모닉 평균이 모두 **A가 B보다 2.66배 빠르다**로 동일하게 나오는 경우가 있다.

⭐ 평가 포인트: **첫 번째는 Arithmetic Mean, 두 번째는 Harmonic Mean을 써야 한다**는 사실을 정확히 안다는 것.

# 8. Roofline Model

## 8.1 동기 — 병목 분석

시스템 성능을 개선하려면 먼저 **병목 분석**을 해야 한다. Roofline은 모든 분석을 세 결론 중 하나로 압축한다.

1. **Compute Bound** — 연산이 병목.
2. **Memory Bound** — 메모리 대역폭이 병목.
3. **Neither** — 하드웨어가 병목이 아니라 **알고리즘 자체** 또는 **소프트웨어 구현**이 suboptimal.

## 8.2 가속기의 두 자원

- **Computation Units**: 수만~수십만 개의 MAC, FMA 유닛.
- **Memory**: On-chip SRAM (Scratchpad), HBM 같은 high-bandwidth DRAM.
- 예시: Google TPU v2 — 128×128 systolic MAC array, BF16 input + FP32 accumulate.

## 8.3 가속기 성능을 결정하는 세 요소

| 요소 | 의미 |
|---|---|
| Computation | 연산 자원의 양 |
| Communication | 메모리 R/W 대역폭 |
| Locality | 한 번 가져온 데이터를 단물 빠질 때까지 재사용하기 |

⭐ 같은 alg를 돌려도 **하드웨어와 알고리즘의 궁합**이 맞아야 진짜 피크 성능에 도달한다.

# 9. Arithmetic Intensity (AI)

## 9.1 정의

> ⭐ ⭐ ⭐ ⭐ 핵심 정의.

\[
\text{Arithmetic Intensity}
=
\frac{\text{Total FLOPs}}{\text{Total DRAM Bytes}}
\]

- 한 바이트를 메모리에서 가져왔을 때 평균 몇 번의 부동소수점 연산을 하는가.
- 단위는 **FLOPs / Byte**.

## 9.2 예시

- 100만 FLOPs, 1,000 bytes 읽음 → AI = 1,000.
- 한 바이트당 1,000번 연산 → 데이터 재사용이 매우 많다.

## 9.3 AI를 측정하는 방법

1. **프로파일러**: Intel VTune 등에서 FP instruction count와 DRAM bytes를 자동 측정.
2. **Paper & Pencil**: 단순한 커널은 직접 손으로 카운트.

## 9.4 AI는 알고리즘만의 함수가 아니다

같은 알고리즘이라도 GPU, NPU, TPU 어디서 돌리느냐에 따라 분모의 DRAM bytes가 달라져 AI가 달라진다.

\(\text{(분모는 cache로 filtering 된 후 DRAM까지 도달한 bytes만을 의미한다는 점에 주의)}\)

# 10. Roofline Plot

## 10.1 두 축

- **X축**: Arithmetic Intensity (FLOPs / Byte) — **log scale**.
- **Y축**: Attainable Performance (FLOPs / Second) — **log scale**.

## 10.2 두 선

| 선 | 의미 | 결정 요인 |
|---|---|---|
| 수평선 | **Peak FLOPS** | 하드웨어가 매 사이클 연산기를 비지하게 한다고 가정한 이론적 한계 |
| 사선 (기울기 1) | **Peak Memory Bandwidth × AI** | 메모리 대역폭이 병목일 때 도달 가능한 최대 FLOPS |

식으로:

\[
\text{Attainable Perf}
=
\min\!\left(
\text{Peak FLOPS},\;
\text{Peak BW} \times AI
\right)
\]

## 10.3 Ridge Point

두 선이 만나는 점.

\[
AI_{\text{ridge}}
=
\frac{\text{Peak FLOPS}}{\text{Peak BW}}
\]

- **AI > AI_ridge**: Compute Bound 영역.
- **AI < AI_ridge**: Memory Bound 영역.

## 10.4 Ideal Region

> ⭐ Peak FLOPS를 찍을 수 있으면서 가장 낮은 AI 지점이 **이상적**이다.

AI를 높이는 것은 SIMD, vectorization 등 큰 엔지니어링 노력을 요구하므로, **가장 적은 노력으로 Peak를 찍는 지점**이 ridge point다.

## 10.5 Performance Optimization Workflow

1. 내 어플리케이션의 AI와 실제 FLOPS를 측정해 점으로 찍는다.
2. 그 점이 Roofline에서 어디 있는지 본다.
   - **수평선 바로 아래** → Compute bound. 더 이상 헤드룸 없음.
   - **사선 바로 아래** → Memory bandwidth bound. 하드웨어 limit.
   - **어중간한 지점** → 헤드룸이 있다. 소프트웨어 최적화 여지.
3. 헤드룸이 있으면 SIMD, cache blocking, loop unrolling 등 적용.

## 10.6 Roofline에 Ceiling을 추가하기

실제 환경에서는 이론적 Peak에 도달하지 못하게 막는 여러 요인이 있다. 이를 **Ceiling** 으로 그림에 추가.

### 10.6.1 Compute Ceilings (위쪽)

| 요인 | 설명 |
|---|---|
| **FMA Imbalance** | 알고리즘에 ADD만 있고 Multiply 없으면 곱셈기 idle → Peak의 절반밖에 못 찍음 |
| **SIMD 미사용** | 컴파일러나 코드가 SIMD를 못 쓰면 추가로 Peak가 1/4 등으로 떨어짐 |
| **ILP 부족** | 명령어 간 의존성 강하면 동시 실행 불가 |

### 10.6.2 Memory Ceilings (오른쪽 아래)

| 요인 | 설명 |
|---|---|
| **No Unit Stride** | 순차 접근이 아니라 strided/random access면 BW 하락 |
| **No Prefetching** | latency 숨기지 못해 BW 하락 |
| **No NUMA Optimization** | Dual-socket 서버에서 데이터가 멀리 있으면 BW 하락 |

### 10.6.3 정렬 컨벤션

- 위쪽 / 오른쪽 끝에 **가장 펀더멘탈하고 극복하기 어려운** ceiling.
- 아래쪽 / 왼쪽으로 갈수록 소프트웨어로 극복 가능한 가벼운 제한.

## 10.7 Roofline 그리는 절차 (Log-Log)

- y절편의 위치: \(\log_{10}\text{Peak BW}\).
- 사선의 기울기: 항상 **1** (log-log 좌표에서).
- AI가 10배 늘면 y축도 10배 늘어남.
- Peak BW가 2배 늘면 사선 전체가 위로 \(\log 2\) 만큼 평행이동.

## 10.8 TPU v1 사례

구글이 TPU v1 페이퍼에서 Roofline 분석을 한 결과:

- 절반 이상 워크로드(MLP, LSTM 등)가 **사선에 딱 붙음 → Memory Bandwidth Bound**.
- CNN만 peak FLOPS 찍었음 (compute bound).
- 원인: TPU v1이 DDR3 메모리를 썼음.
- ⭐ 그래서 TPU v2부터 **HBM**으로 교체 → 사선을 위로 push → MLP/LSTM도 함께 성능 상승.

## 10.9 Cache의 Non-blocking 최적화

- Cache는 SoC 위 작은 scratchpad.
- 데이터가 한 번만 쓰이면 cache에 올렸다가 버리는 것이 낭비.
- 그래서 **bypassing**: 한 번만 쓸 데이터는 cache를 거치지 않고 그냥 흘려보내고, 재사용이 많은 데이터에만 cache 공간을 할당해 **locality를 극대화**.

# 11. 예제 — NPU v2의 PE 개수 결정

## 11.1 문제

S사의 NPU v1 스펙:

| 항목 | NPU v1 |
|---|---|
| SoC Clock | 500 MHz |
| PE (Processing Element, FMA 단위) | 2,000개 |
| DRAM Bandwidth | 10 GB/s |

NPU v2에서는 DRAM bandwidth를 10배 늘려 **100 GB/s**로 설정. 메모리 접근 패턴은 동일하다고 가정.

> v2의 DRAM 스펙에서 **최대 성능**을 끌어내려면 PE가 몇 개 있어야 하는가? (PE 하나는 매 사이클당 1 FP op 가능, 1 GHz로 동작 가정)

## 11.2 풀이 흐름

1. AI는 변하지 않는다 (메모리 패턴 동일).
2. Peak BW가 10배 늘었으니 Roofline 사선 전체가 \(\log 10 = 1\) 만큼 평행 이동.
3. 같은 AI에서 도달 가능한 최대 FLOPS는 사선이 위로 올라간 만큼 **10배**가 된다.
4. 이때 그 새로운 사선이 Peak FLOPS 수평선과 만나는 지점을 새 ridge point로 보면, 필요한 Peak FLOPS는 \(10^5\) GFLOPS 수준.
5. PE 한 개가 1 GHz에 1 FP op → \(N\) PE → \(N\) GFLOPS.
6. 따라서 \(N \ge 10^5\) → **약 \(10^5\) 개의 PE**가 필요.

> ⭐ 이런 형태의 문제가 **거의 매년 출제되는 단골 패턴**이다.

# 13. 오후 핵심 정리

## 13.1 큰 그림

- 성능 평가는 사이언스다. 숫자로 장난치면 회사 망한다.
- 모든 성능 매트릭은 **Time**과 **Rate** 두 펀더멘탈로 환원된다.
- 단일 매트릭(IPC, MIPS, 클럭)만으로 결론 내리지 말 것.

## 13.2 Iron Law of CPU Performance

\[
\text{CPU Time} = IC \times CPI \times \text{Cycle Time}
\]

세 항이 어떤 요인에 영향을 받는지 외워두기.

## 13.3 평균 내기 국룰

| 평균 | 대상 |
|---|---|
| Arithmetic | Time |
| Harmonic | Rate |
| Geometric | Ratio |

## 13.4 Roofline 분석

핵심 정의:

\[
AI = \frac{\text{FLOPs}}{\text{DRAM Bytes}}, \quad
\text{Perf} = \min(\text{Peak FLOPS},\; \text{Peak BW} \times AI)
\]

- AI가 Ridge Point 우측 → Compute Bound.
- AI가 Ridge Point 좌측 → Memory Bound.
- Roofline을 그리고 내 어플리케이션 위치를 점으로 찍어 헤드룸 분석.
- Ceiling은 FMA imbalance, SIMD, ILP, prefetch, NUMA 등으로 더 낮아짐.
- ⭐ 시험 단골: NPU v1 → v2처럼 DRAM bandwidth가 \(k\)배 늘면 사선이 \(\log k\) 만큼 위로 평행이동, 그에 맞는 PE 개수 계산.

## 13.5 MLPerf

- 산업 표준 ML 벤치마크.
- Inference 6.0 / Training 5.1 (2026년 4월 기준).
- 시나리오: Offline / Server / Single Stream / Multi Stream.
- Closed / Open Division.

## 13.6 강사 메시지

- API 위에서만 작업하는 엔지니어와 시스템 밑단을 이해하는 엔지니어는 경쟁력이 다르다.
- 학습이든 추론이든 **속도가 답답하면 Roofline을 그려라** — 헤드룸과 병목이 한눈에 보인다.
- 시험은 강의 자료 중심이지만, Roofline 계산처럼 변별력 있는 응용 문제 한 두 개가 끼어들 수 있다.
