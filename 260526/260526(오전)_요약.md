## 1.1 강의 시리즈 개요

- Day 2~3: 트랜스포머 등 후속 토픽
- Day 4: 실습

## 1.2 Embedded Computer란

- **General-purpose computer**: 서버 / PC 등 사용자가 자유롭게 프로그래밍하는 기계.
- **Embedded computer**: 그 외의 모든 컴퓨팅 시스템. 사용자는 보통 미리 만들어진 응용을 소비한다.
- 예시: 가전, IoT, 스마트팩토리, 로봇, 자율주행차, 드론, 스마트글래스, 그리고 일부 특성을 공유하는 스마트폰.
- 최근에는 **Physical AI** (물리 환경에서 사람과 상호작용하는 AI)가 각광받으며 인베디드 시스템이 다시 주목된다.

## 1.3 Embedded System의 주요 제약

⭐ 시험적 관점에서 **Embedded ML 분석의 3대 축**.

| 축 | 의미 |
|---|---|
| **Latency** | 첫 입력이 들어오고 출력이 나오기까지의 시간. 자율주행처럼 반응이 늦으면 사고로 직결. |
| **Throughput** | 단위 시간당 처리량. 입력이 끊임없이 들어올 때 막힘 없이 출력을 내는 능력. |
| **Memory / Energy** | 메모리 용량 안에서 모델이 돌아야 하고, 배터리 기반이면 에너지 효율이 중요. |

추가 특성:

- **이질적 병렬 시스템**: CPU + GPU + NPU + (장기적으로) PIM이 공존. GPU는 파워가 너무 커서 모든 인베디드 시스템에 적합하진 않다.
- **Correctness 중요**: 자율주행·드론·로봇 등은 죽으면 곤란하므로 디자인 정확성과 신뢰성이 본질.
- **Cost-Performance Ratio, 가성비, 휴대성, Dependability, Time-to-Market**이 모두 중요.

## 1.4 Embedded ML의 통합 설계 (Co-Design)

강사가 30년 강조해 온 메시지: **알고리즘 · 하드웨어 아키텍처 · 소프트웨어**를 따로 만들면 안 된다.

세 가지 방향의 최적화:

| 방향 | 내용 |
|---|---|
| **하드웨어** | AI 가속기 설계 (별도 강의자 담당) |
| **소프트웨어 최적화** | 알고리즘·하드웨어 고정, 그 위에서 코드를 튜닝 — **수배~10배** 성능 개선 가능 |
| **NAS (Neural Architecture Search)** | 하드웨어 고정, 알고리즘 자체를 그 하드웨어에 맞게 재설계 |

> ⭐ 강사 경험담: 삼성 NPU에 GPU용 알고리즘을 그대로 던지면 NPU가 비효율적으로 동작한다. 알고리즘은 **NPU-aware**하게 만들어야 한다.

## 1.5 Embedded ML 응용 분야

- **컴퓨터 비전**: Image Classification, Object Detection, Segmentation — 초창기 CNN, 현재 Transformer 기반으로 이동 중. 단 매우 작은 시스템에서는 여전히 CNN이 유효.
- **오디오·언어**: Edge LLM 등 트랜스포머 기반.
- **IoT 센서 AI**.

자율주행차 한 대에는 동시에 돌아가야 하는 ML 모델이 **수십 개**라서 모두 트랜스포머로 채울 수 없다. 따라서 CNN을 잘 이해하고 다룰 줄 알아야 한다.

# 2. CNN 알고리즘의 복잡도 분석

## 2.1 Convolution의 차원과 MAC 개수

Input feature map 크기 \(F_H \times F_W \times C\), 필터 크기 \(K \times K\), 필터 개수 \(N\)일 때 padding으로 spatial size를 유지한다고 가정하면 출력은 \(F_H \times F_W \times N\)이 된다.

**한 출력 픽셀당 곱셈 수**: \(K \times K \times C\).

**전체 MAC 수**:

\[
\#\text{MAC} = F_H \cdot F_W \cdot N \cdot K \cdot K \cdot C
\]

**필터 메모리**: \(K \cdot K \cdot C \cdot N\) (필터 전체).

## 2.2 Arithmetic Intensity (연산 강도)

> ⭐ 시험 출제 포인트: 같은 데이터를 DRAM에서 한 번 가져왔을 때, 그 데이터로 몇 번 연산을 하느냐.

- DRAM 접근 시간은 CPU 한 사이클 대비 **수백~수천 배** 길다.
- 따라서 데이터를 자주 재사용할수록 메모리 접근 비용이 상각된다.
- CNN의 경우 필터 하나를 한 번 가져오면 \(F_H \cdot F_W\)번 곱셈에 재사용 가능 → CNN은 **가속하기 좋은** 알고리즘.
- 트랜스포머는 데이터 재사용 빈도가 낮아 **가속이 어렵다**.

## 2.3 TOPS와 모델 복잡도의 연결

**TOPS (Tera Operations Per Second)** = 1초당 가능한 연산 수.

\[
\text{(1초당 CNN 추론 횟수)} = \frac{\text{TOPS}}{\#\text{MAC per inference}}
\]

이런 식으로 하드웨어 파워와 모델 복잡도를 같이 따져 본다.

# 3. 메모리 레이아웃 — NCHW vs NHWC

## 3.1 코드 한 줄로 다 보이는 차이

알고리즘을 아는 사람은 nested loop 코드만 보고 "별 문제 없다"고 한다. 그러나 임베디드 ML 관점에서 보면 **루프 순서가 메모리 레이아웃을 암시**한다.

대표적 CNN 6-nested loop:

```
for n in N: for c in C: for h in H: for w in W: for ky in K: for kx in K:
    out[n,h,w,...] += in[n,c,h+ky,w+kx] * filter[n,c,ky,kx]
```

가장 안쪽에서 `kx, ky`가 돌면 메모리에서 K 영역을 먼저 읽는다. 이때 채널 방향 데이터를 만나려면 멀리 점프해야 한다.

## 3.2 NCHW vs NHWC

| 레이아웃 | 메모리에 저장되는 가장 안쪽 축 | 효과 |
|---|---|---|
| **NCHW** | width → height → channel → batch | 채널 데이터가 멀리 떨어져 있어 캐시 활용이 나쁨 |
| **NHWC** | channel → width → height → batch | 채널 방향이 연속이라 **3×3×C** 필터에서 캐시 라인을 그대로 활용 |

> ⭐ Conv에서 채널 C는 32 → 64 → 128로 갈수록 커지므로 채널 방향이 연속이어야 한다. **NHWC가 더 좋다.**

## 3.3 강사 메시지

- 메모리 레이아웃은 소프트웨어 엔지니어가 결정.
- 그러나 하드웨어 팀이 레이아웃을 먼저 못 박고 소프트웨어 팀에 넘기면 컴파일러 개발자들이 고생한다.
- 좋은 NPU 디자인은 **소프트웨어 친화적인 메모리 레이아웃**부터 정해야 한다.

# 4. CNN의 다른 layer들

## 4.1 Activation Function

- ReLU, Leaky ReLU, Sigmoid, Tanh, Swish, H-Swish 등.
- FP32 floating point에선 그대로 계산하면 되지만, INT8 등 quantize된 데이터에선 **Table Lookup으로 근사**한다.
- ⭐ **ML 모델 자체가 근사 연산이므로 activation도 정확히 계산할 필요 없다.**

## 4.2 Pooling

- Max Pooling / Average Pooling.
- Feature map의 spatial size를 줄이기 위해 사용.
- 일반 conv에서 stride 2로 sub-sampling을 대체할 수 있지만 Inception v3처럼 별도 Pooling layer를 둔 모델도 존재.
- ⭐ **Pooling은 연산량은 적지만 메모리 접근 오버헤드는 그대로**이므로 Arithmetic Intensity가 나쁘다 — NPU에서 비효율적인 layer.

## 4.3 Fully Connected Layer

- Softmax 직전의 classification head.
- Weight 수가 매우 많아 **메모리 중심**의 연산.
- Conv 대비 데이터 재사용 빈도가 낮아 메모리와 연산 비용이 거의 1:1 수준.

## 4.4 Training과 Backward Convolution

- 강의 시간은 inference 중심이지만, training의 backward도 **convolution + weight gradient computation = 모두 convolution**으로 표현된다.
- Inference는 INT8, training은 FP16 등 데이터 타입만 다를 뿐 본질은 동일.

# 5. Convolution 알고리즘 4종

## 5.1 Direct Convolution

- 수식 그대로 계산.
- 임베디드 NPU에서 주로 사용.

## 5.2 GEMM (im2col)

- Convolution을 **행렬 곱셈으로 재구성**.
- input feature map의 patch를 `im2col`로 펼쳐 메모리 크기는 \(K^2\)배 늘리지만, 메모리 접근 횟수를 줄이고 cache line 단위로 한 번에 읽는다.
- NVIDIA cuBLAS, cuDNN의 표준 컨볼루션 구현.
- ⭐ ⭐ GPU/CPU에서는 GEMM 기반이 Direct보다 **훨씬 빠르다**. 메모리는 크게 쓰지만 메모리 접근이 적기 때문.
- 단, NPU에선 메모리 capacity가 부족해 GEMM을 못 쓰고 Direct를 사용.

## 5.3 Winograd Convolution

- ⭐ 수학자들이 만든 **Minimum Filtering Algorithm**의 응용.
- \(m \times r\) 곱셈을 \(m + r\) 곱셈으로 줄임 (덧셈은 늘어남).
- 2D Winograd를 적용하면 3×3 conv의 곱셈 36개를 16개로 줄일 수 있다.
- **3×3 필터까지**는 Winograd가 좋은 것으로 알려져 있다.
- ⭐ NVIDIA cuDNN: 3×3 → Winograd / 나머지 → GEMM. 자동 선택.
- NPU에선 멀티플라이와 덧셈의 비용이 거의 같아 Winograd가 큰 이점이 없어 **잘 안 쓴다**.

## 5.4 FFT-based Convolution

- 이론상 큰 필터일수록 효과적.
- 하지만 최근 CNN의 필터는 3×3, 1×1, 1×3 등 작아지는 추세 → ⭐ **현재는 거의 사용 안 함**.

## 5.5 정리 — 플랫폼별 추천

| 플랫폼 | 선호 알고리즘 |
|---|---|
| **GPU / CPU** | 3×3 → Winograd, 그 외 → GEMM |
| **NPU / Systolic Array** | 대부분 Direct Convolution (메모리 capacity 제한) |

# 6. 소프트웨어 최적화의 두 축

ML 모델은 모두 **근사 계산**이라서 정확하게 계산할 필요가 없다. 이 사실이 소프트웨어 최적화의 출발점.

> ⭐ ⭐ ⭐ 소프트웨어 최적화의 두 축:
> 1. **Approximation (근사)**
> 2. **Parallelization (병렬화)**

Approximation은 다시 세 가지 기법으로 나뉜다.

| 기법 | 줄이는 대상 |
|---|---|
| **Quantization** | 데이터 (operand 크기) |
| **Pruning** | 연산 자체 (storage + computation) |
| **Low-Rank Approximation** | 행렬 분해를 통한 연산 + 저장 동시 감축 |

(이 외에 Approximate Arithmetic은 하드웨어 접근이라 본 강의 scope는 아니다.)

# 7. Quantization

## 7.1 동기

DRAM 접근이 가장 비싼 연산이다.

| 연산 종류 | 상대 에너지 |
|---|---|
| 8-bit add | 매우 작음 |
| 32-bit FP add/mul | 그보다 크지만 비슷한 수준 |
| **DRAM access** | **압도적으로 큼** |

따라서 데이터 크기를 줄여 메모리 접근 비용을 낮추는 것이 핵심이다.

## 7.2 Floating Point Format

| 포맷 | Exponent | Mantissa | 특징 |
|---|---:|---:|---|
| **FP32 (IEEE)** | 8 | 23 | 과학 계산용, 정밀도 우선 |
| **FP16 (IEEE)** | 5 | 10 | GPU 학습용 |
| **BF16 (BFloat16)** | 8 | 7 | ⭐ AI 가속기 표준, FP32와 같은 **dynamic range** 유지하면서 mantissa만 줄임 |
| **FP8 (IBM 등)** | 5 | 2 (1 sign) | 회사별로 비트 배분 다름 |

> ⭐ ⭐ ML은 근사 연산이라 정밀도 (mantissa)는 줄여도 무방하지만, **dynamic range (exponent)** 는 유지해야 한다.

## 7.3 Fixed-Point Quantization

- 보통 **INT8**이 대세, 점차 **INT4**, 그 이하로.
- Radix point(소수점) 위치는 하드웨어가 모름 → 사람이 알아두어야 한다.
- 비트 수가 더 작을 때 radix point를 어디 두느냐가 정확도를 좌우.

## 7.4 Codebook Quantization

- 2-bit, 3-bit 자체로 숫자를 표현하지 않고 **주소**로 사용.
- Codebook 안에는 8-bit 숫자가 저장됨 → 메모리 크기는 줄이고, 표현력은 8-bit 수준 유지.

## 7.5 Linear Quantization

### 7.5.1 Symmetric vs Asymmetric

| 종류 | 특징 | 채택 회사 |
|---|---|---|
| **Symmetric** | max값만 정함 (양/음 대칭) | NVIDIA TensorRT |
| **Asymmetric** | α (양수 최대) + β (음수 최대) 따로, 0의 위치도 별도로 계산 | Google TPU |

Asymmetric은 곱셈 시 두 번의 뺄셈과 한 번의 덧셈이 추가되어 비싸지만 정확도가 더 보존된다.

### 7.5.2 Max값 결정 방법

- **단순 max**: 분포의 가장 큰 값.
- **NVIDIA TensorRT**: **KL Divergence** 최소화. FP 분포와 INT 분포의 distribution 차이를 가장 작게 만드는 threshold \(T\)를 찾는다.

## 7.6 Non-linear Quantization

| 방식 | 동기 | 단점 |
|---|---|---|
| **Log Quantization** | 0 근처에 분포가 몰려있어 자연스러워 보임. 곱셈을 shift+add로 대체 가능. | ⭐ **큰 값들이 결과에 더 중요한데 로그는 큰 값을 듬성듬성 표현** → 정확도 손실 |
| **Weighted (Codebook-based)** | 값과 값의 간격에 규칙 없이 클러스터링으로 대표값 결정 | sub/no bit 정도에서 유리 |
| **Outlier Quantization** | 95% 정도의 분포만 quantize, 나머지 5% outlier는 FP로 그대로 | 하드웨어가 두 경로를 지원해야 함 |
| **Learnable Quantization** | 학습 중 분포 범위를 함께 결정 | QAT 형태 |

## 7.7 QAT vs PTQ

| 방식 | 학습 필요 | 정확도 | 비용 |
|---|---|---|---|
| **QAT (Quantization-Aware Training)** | 학습하면서 quantize와 backward를 반복 | 높음 | 시간이 오래 걸림 |
| **PTQ (Post-Training Quantization)** | 학습 후 calibration data 몇 개만으로 quantize 결정 | 8-bit는 충분히 좋음, 4-bit는 어려움 | 빠름 |

> ⭐ PTQ Pipeline: Cross-Layer Equalization (레이어 간 weight 분포를 비슷하게 곱해주기) → Bias Correction → AdaRound (라운딩을 항상 nearest로 하지 않고 학습) → 등.

## 7.8 Fake Quantization vs Integer-Only

| 방식 | 학습 시점 | 정확도 보장 |
|---|---|---|
| **Simulated / Fake Quantization** | 입출력만 quantize, 중간 계산은 FP32 | NPU 실제 동작과 차이가 있어 최적 아님 |
| **Integer-Only Quantization** | 중간 accumulation까지 INT로 | NPU 실제 동작과 일치 |

## 7.9 Quantization Granularity

| 단위 | 정확도 | NPU 적합성 |
|---|---|---|
| **Layer-wise** | 가장 거침 | NPU 친화적 (단순) |
| **Channel-wise** | 정확도 높음 | NPU에서 구현 어려움 |
| **Group-wise** | 중간 | NPU에서 절충안으로 채택 가능 |

## 7.10 Mixed Quantization

- 레이어마다 다른 bit 사용 (초기·말단은 8-bit, 중간은 4-bit 등).
- 레이어가 100개 넘어가면 mix 결정이 난제 → 연구 영역.

## 7.11 강사의 현장 메시지

- ⭐ 가속기 연구실의 가장 큰 고민은 **weight quantization** — 사전 학습된 weight들이 모두 FP로 되어 있는데 NPU는 INT로 돌려야 하기 때문.
- 다행히 PTQ pipeline은 툴화되어 있어 따라가면 된다.

# 8. Pruning

## 8.1 동기

CNN에는 **Redundancy**가 많다.

- Sparsity가 증가할수록 일정 구간까지 accuracy가 오히려 **증가**한다 (Occam's Hill).
- 이유: 너무 많은 파라미터는 training set에 과적합 → generalization 저하. 줄이면 일반화 효과.

## 8.2 Pruning의 대상

| 단위 | 줄이는 대상 |
|---|---|
| **Fine-grain** | 개별 weight (값이 작은 weight를 0으로) |
| **Vector level** | weight vector |
| **Kernel level** | 채널 |
| **Filter level** | 필터 통째로 |

## 8.3 Fine-Grain의 함정

- 단순히 weight 0이 많아진다고 GPU/CPU/NPU에서 계산이 빨라지지는 **않는다**.
- 메모리에서 데이터는 일괄로 읽혀서 0 여부와 무관하게 로드되기 때문.
- Fine-grain은 주로 **메모리 압축**에 효과적이다.

## 8.4 Filter Pruning (가장 효과적)

> ⭐ ⭐ ⭐ CNN에서 가장 효과적인 pruning은 **filter pruning**.

- 필터를 하나 제거하면 다음 레이어의 채널도 하나 줄어든다 → \(N\)도 줄고 \(C\)도 줄어 **전체 연산량 감소**.
- 50% 필터 제거 시 약 2배 가속.

## 8.5 어떤 필터를 제거할까

| 기준 | 의미 |
|---|---|
| **L1 norm** ⭐ | weight 절댓값 합 — 평균적으로 가장 좋다 |
| **L2 norm** | weight 제곱합 |
| **Statistical Sensitivity** | 필터 제거 후 validation accuracy 변화량 |
| **First-Order / Second-Order Taylor** | 손실 함수의 1차·2차 미분 정보 |

> ⭐ 결국 평균적으로 가장 좋고 **가장 단순한 L1 norm**이 표준으로 쓰인다.

## 8.6 Layer-wise Sensitivity

- 초반 conv layer와 마지막 conv layer는 pruning에 **민감** → 많이 자르면 정확도 급락.
- 중간 layer는 **둔감** → 더 많이 자를 수 있다.

## 8.7 Pruning Schedule

| 방식 | 설명 |
|---|---|
| Train → Prune → Retrain | 가장 보편적인 방식 |
| Iterative Prune + Retrain | QAT와 유사하게 학습 도중 반복 |
| ⭐ **Sparse Training / Pruning at Initialization** | 학습 전부터 sparse하게 시작 — 현재 활발한 연구 |

## 8.8 Lottery Ticket Hypothesis (LTH)

> ⭐ ⭐ "큰 네트워크 안에는 같은 정확도를 달성하는 작은 sub-network (winning ticket)가 존재한다."

- **SNIP** (Connection Sensitivity 기반)처럼 학습 없이 winning ticket을 찾는 시도들이 제안됨.
- 사람들이 가장 원하는 것은 **PTQ처럼 trained 없이 ticket을 찾는 방법**.

## 8.9 Hardware-Aware Pruning

- 삼성 NPU 사례: Sparsity-aware NPU를 만들었지만 zero weight 판별 오버헤드와 ReLU 의존이 약해진 최근 activation 추세로 인해 효과가 제한적.
- 대신 **압축 (Run-Length Encoding 등)** 으로 메모리 크기를 줄이는 데 효과적 — 1.5배 정도 메모리 절감.

## 8.10 Deep Compression (Song Han)

- ⭐ Pruning + Quantization + Huffman Encoding을 결합 → 메모리 약 **13배** 압축, 속도 **3~4배** 향상.
- MIT 교수 Song Han이 박사과정 시절 발표한 논문으로 임베디드 ML의 기념비적 연구.

# 9. Low-Rank Approximation

## 9.1 핵심 원리

> ⭐ ⭐ ⭐ "필터가 보려고 하는 **receptive field**는 그대로 유지하되, weight 수는 줄여서 비슷한 효과를 낸다."

5×5 필터의 효과를 얻는 두 가지 분해:

| 분해 | weight 수 | receptive field |
|---|---:|---|
| 5×5 conv | 25 | 25 pixel |
| 3×3 conv × 2 | 18 | 25 pixel ✓ |
| 5×1 + 1×5 | 10 | 25 pixel ✓ |

곧 큰 필터 하나를 작은 필터 여러 개로 **matrix factorization** 하는 것.

## 9.2 Tucker Decomposition

- 입력 채널 \(C_I\)에서 출력 \(C_O\)로 가는 3×3 conv를 다음 세 단계로 분해:
  1. **1×1 conv**로 채널을 \(C_I\)에서 \(C_I'\) (감소)으로 매핑.
  2. **3×3 conv**로 spatial filtering ( \(C_I'\)에서 \(C_O'\)로).
  3. **1×1 conv**로 채널을 \(C_O'\)에서 \(C_O\) (증가)로 확장.
- 채널을 반으로 줄이면 약 **40% 연산 감소**.

## 9.3 Bottleneck Architecture (ResNet)

- ResNet에서 사용하는 구조가 곧 Tucker decomposition의 사례.
- 1×1로 채널 축소 → 3×3 spatial conv → 1×1로 채널 확장.

## 9.4 Depthwise Separable Convolution (MobileNet)

> ⭐ ⭐ ⭐ 가장 극단적인 형태의 low-rank approximation.

- **Depthwise Conv**: 각 채널을 독립적으로 spatial 3×3 conv (채널 간 섞임 없음).
- **Pointwise (1×1) Conv**: 채널 간 정보를 합침.
- 원래 3×3 conv 한 번에 보던 spatial + channel correlation을 **두 단계로 분리**.

## 9.5 MobileNet 연산 감소 비율

원래 3×3 conv:

\[
F_H \cdot F_W \cdot C \cdot N \cdot 3 \cdot 3
\]

Depthwise separable:

\[
F_H \cdot F_W \cdot C \cdot 3 \cdot 3 \;+\; F_H \cdot F_W \cdot C \cdot N
\]

비율:

\[
\frac{1}{N} + \frac{1}{K^2}
\]

\(N\)은 보통 100~1000 정도로 크기 때문에 사실상 \(\frac{1}{K^2}\)에 가까운 감소 효과 → **약 9분의 1**.

> ⭐ ⭐ ⭐ ⭐ "Low-Rank Approximation은 시험 예상 문제 중 하나"라고 강사가 명시적으로 언급한 항목.
> 핵심: **convolution 하나가 필요로 하는 MAC 수를 계산할 수 있어야 하고**, **분해된 후의 MAC 수가 어떻게 줄어드는지 비교**할 수 있어야 한다.

# 10. 오전 핵심 정리

## 10.1 큰 그림

- 인베디드 시스템은 시간·메모리·에너지의 강한 제약 위에서 ML을 돌려야 한다.
- 알고리즘 / 아키텍처 / 소프트웨어를 **통합 설계 (Co-Design)** 해야 진짜 성능이 나온다.
- CNN의 본질은 6-nested loop이고, 메모리 레이아웃이 성능을 좌우한다. **NHWC 표준**.
- Convolution은 Direct / GEMM / Winograd / FFT 네 가지가 있고 **플랫폼별로 최적 선택**이 다르다.

## 10.2 ML 모델은 근사 연산이라는 사실에서 출발

소프트웨어 최적화 = **Approximation + Parallelization**.

- Quantization: 데이터 크기 ↓ → 메모리 비용 ↓.
- Pruning: 연산 ↓.
- Low-Rank Approximation: 둘 다 동시에 ↓.

## 10.3 Quantization 요약

| 분류 | 옵션 |
|---|---|
| 분포 | Linear (Uniform) vs Log/Weighted/Outlier (Non-uniform) |
| 대칭성 | Symmetric (NVIDIA) vs Asymmetric (Google) |
| 비트수 | INT8 → INT4 → … |
| 단위 | Layer-wise / Channel-wise / Group-wise |
| 학습 방식 | QAT vs PTQ (Cross-Layer Equalization, Bias Correction, AdaRound) |
| 시뮬레이션 | Fake / Simulated vs Integer-Only |

## 10.4 Pruning 요약

- Fine-grain은 메모리에 좋고, **Filter pruning이 연산에 가장 효과적**.
- 기준은 **L1 norm**이 가장 보편적이고 평균적으로 최선.
- Layer-wise sensitivity 고려, 초기·말단은 보수적으로.
- Lottery Ticket Hypothesis와 SNIP은 학습 없이 winning ticket을 찾으려는 시도.
- Pruning + Quantization + Huffman 결합한 **Deep Compression** 으로 약 13배 메모리, 3~4배 속도 가능.

## 10.5 Low-Rank Approximation 요약

- 큰 필터 → 작은 필터들의 연쇄로 분해.
- **Tucker** = 채널 축소 → spatial → 채널 확장 = ResNet의 **bottleneck**.
- MobileNet의 **Depthwise Separable Conv** = spatial과 channel을 완전히 분리, 약 \(K^2 \approx 9\)배 연산 감소.
- ⭐ 시험 예상 문제: MAC 개수 비교.

## 10.6 강사 메시지

- 디테일을 다 외울 필요 없다.
- "큰 그림"과 강사가 ⭐로 강조한 부분만 정확히 잡으면 충분하다.
- 임베디드 ML은 **하드웨어를 모르면 소프트웨어 최적화를 할 수 없다**.
