# 260430 오후 강의 요약 — LLM 정렬 파이프라인 (SFT → RLHF → DPO → ORPO → GRPO) · LoRA · 실습

> 오후 흐름: **LLM 발전 개관 → Pre-training의 한계 → ICL → SFT(Instruction Tuning) → MMLU → Self-Instruct/Alpaca → RLHF의 등장(HH-RLHF) → Pairwise Comparison(Bradley-Terry) → Reward Model → PPO + KL Penalty → InstructGPT 파이프라인 → DPO → ORPO → GRPO(DeepSeek-R1) → LoRA → 실습 (Pre-training → SFT → DPO → ORPO → GRPO)**
> 강의 주제: 오전에 다룬 RLHF 개념을 실용적 알고리즘 계보로 전개하고, 마지막에 LoRA + Unsloth로 직접 실습.

## 1. ⭐ LLM 발전 개관과 정렬의 필요성

### 1.1 모델 크기 ↑ ≠ 의도 정렬
- 초기에는 LM, GPT-1, BERT처럼 수억 파라미터 모델 → 이후 GPT-2, GPT-3, GPT-4로 수십~수천억 파라미터.
- 단순히 "커졌다"가 아니라 **하나의 모델이 다양한 task를 prompt만으로 수행**하는 시대로 전환.
- ⭐ 핵심 질문: "큰 모델이 정말 좋은 모델인가?"
  - 지식 ↑, 문장 능력 ↑.
  - 그러나 **사용자 의도에 맞게 행동**한다는 보장은 없음.
  - 사용자 지시 무시 / 틀린 답 / 안전하지 않은 답변 가능.
- → **큰 모델을 어떻게 사람의 의도와 선호에 맞게 정렬할 것인가** = 오후 강의의 핵심 질문.

### 1.2 Pre-training의 본질
- ⭐ Auto-regressive LM = **다음 토큰 예측** (`I roll goes to make tasty tea` 같은 시퀀스에서 다음 단어 예측).
- 학습 목표는 "어떤 단어가 다음에 올 가능성이 높은가" — **명령을 따르도록 학습되지 않음**.
- Pre-training 단계는 언어의 패턴 / 문법 / 지식 / 상식 학습.
- ⭐ **다음 토큰을 잘 예측 ≠ 사용자가 원하는 응답** — 이 차이가 오후 강의의 출발점.

### 1.3 ⭐ Pre-training만으로 부족한 이유 — "달 착륙을 6살에게 설명해줘"
- GPT-3에 직접 프롬프트로 주면 **답변 대신 비슷한 패턴의 문장**만 반복 생성.
  - 예: "중력이론을 6살 아이에게 설명해줘", "상대성이론을 6살 아이에게 설명해줘" 등 패턴 복제.
- 사람이 직접 쓴 정답은 6살이 이해할 수 있는 자연스러운 흐름.
- ⭐ 핵심 한계: 모델은 **질문에 답변**한 게 아니라 **웹에서 봤을 법한 텍스트 패턴을 따라**한 것.
  - 언어 패턴은 잘 배웠으나 **"사용자 작업 수행"이라는 행동은 미학습**.

### 1.4 ICL (In-Context Learning) — 부분 해결책
- GPT-3 (2020.6) 이후 **프롬프트에 예시 몇 개 → 그 패턴을 따라 답변** 능력 발현.
  - 영→불 번역 예시 후 모델이 "번역 task구나" 추론.
- Zero-shot / One-shot / Few-shot의 차이.
- ⭐ ICL의 한계:
  - 파라미터 업데이트 X — 임시적인 작업 방식 파악만.
  - 프롬프트 잘못 넣으면 성능 흔들림.
  - 복잡한 지시는 안정적으로 못 따름.
  - 프롬프트에 넣을 데이터 수에 한계 (context window).

## 2. ⭐ Instruction Tuning / SFT

### 2.1 Fine-Tuning의 기본
- Pre-training 후 **특정 입력-정답 쌍**으로 학습 → 특정 입력에 특정 출력을 내도록 조정.
- 단계:
  - **Step 1**: Pre-training (대량 일반 텍스트).
  - **Step 2**: Instruction + Output 쌍으로 Fine-Tuning.
- 예시 instruction:
  - `다음 질문에 답하라`
  - `이 문장을 요약하라`
  - `이 문제를 단계적으로 풀어라`
- 추론 과정 무시하고 정답값(`-320.4` 같은) 직접 학습.

### 2.2 ⭐ Instruction Tuning 데이터 규모
- **약 1600개 task / 300만+ instruction** (Super Natural Instructions 수준).
- ⭐ 핵심 포인트: **단일 task가 아니라 매우 다양한 task 동시 학습** → 범용 instruction following 능력.
- 이게 **ChatGPT의 기본 핵심 원리**.

### 2.3 ⭐ MMLU — Massive Multi-Task Language Understanding
- 우주 / 화학 / 수학 / 역사 / 의학 / 법률학 등 **다양한 영역의 종합 객관식**.
- 단순 패턴 매칭으론 못 푸는 사고형 문제.
- "좋은 모델"의 기준이 **다양한 task를 두루 잘하는 모델**로 변화 → MMLU가 LLM 성능 평가의 핵심 벤치마크로 자리잡음.

### 2.4 ⭐ Instruction Tuning의 효과 — Before vs After
- **Before**: 질문(Q) + 옵션(A/B/C) 중 골라야 하는 문제에 대해 **거의 비슷한 패턴 문장**을 반복 생성, 엉뚱한 답.
- **After**: 추론 과정도 따라 나오고 `정답은 C` 식으로 **선택지 중 적절한 답**을 선택해 생성.
- ⭐ 핵심: 단순히 문장을 예쁘게 쓰게 된 게 아니라, **질문 형식 이해 + 선택지에서 고르라는 형식 이해 + 그 형식에 맞춰 답하는 행동**을 학습.

### 2.5 ⭐ Instruction Tuning 데이터는 어떻게 만드는가 — Self-Instruct / Alpaca
- 핵심 아이디어: **좋은 모델을 써서 좋은 데이터를 만든다**.
- 절차 (Alpaca):
  1. **175개 Seed Task**만 사람이 직접 작성.
  2. **Text-Davinci-003** 같은 강력한 모델로 Instruction-Output을 자동 생성.
  3. 약 **5만+ instruction following 데이터** 생성.
  4. **LLaMA**(작은 모델)에 SFT 적용 → Alpaca 완성.
- ⭐ 두 가지 의미:
  1. **데이터 생성 비용 절감** — 175개만 사람 손 거침.
  2. **작은 모델 + Instruction Tuning ≈ GPT-3.5 수준 성능** — 비용 효율.
- → "좋은 모델로 데이터 생성 / 작은 모델로 학습"이라는 새로운 패러다임의 시작점.

## 3. ⭐ HH-RLHF의 등장 — Helpfulness & Harmlessness

### 3.1 Instruction Tuning의 잔존 문제
- 명령은 잘 따르지만 **여전히 사회적으로 부적절한 답변** 가능.
- 예: "옷가게를 좀 털고 싶다" → 두 답변 모두 부정적 답변이지만 한쪽이 더 친절/안전.
- ⭐ HH = **Helpful + Harmless** — 두 H의 의미.
- 두 응답 중 더 선호되는 것에 **chosen** 라벨, 덜 선호되는 것에 **rejected** 라벨.

### 3.2 ⭐ Instruction Tuning의 두 가지 한계 (시험 출제 가능)
1. **Open-ended task에 정답이 없음**:
   - "강아지와 메뚜기가 나오는 이야기를 써줘" → 정답이 단 하나가 아님.
2. **Token-level loss의 한계**:
   - 언어모델은 각 토큰의 예측 오류를 **비슷하게** 다룸 (cross-entropy 평균).
   - 그러나 인간 직관에서는 **사소한 오류와 중대한 오류의 무게가 다름**.
   - 예: 문장 끝의 어색한 표현 vs 의료 조언에서 다른 진단명을 말하는 것 → **수준이 완전히 다른 오류**인데 동일 페널티.
- → SFT만으로는 **인간의 선호를 충분히 반영 불가** → Reward 기반 학습 필요.

### 3.3 Reward의 직관
- 요약 task에서 두 응답 S1, S2.
  - S1: 핵심을 잘 담음 → reward 8.0.
  - S2: 불필요/틀린 내용 포함 → reward 1.2.
- 목표: **모델이 생성한 답변 중에서 기대 reward가 높은 답변을 자주 생성**하도록.

## 4. ⭐⭐ RLHF Pipeline 3단계 (시험 출제 핵심)

### Step 1 — SFT (Supervised Fine-Tuning)
- 사람이 작성한 **양질의 정답 예시**로 instruction following 모델 만들기.
- 기본기를 만드는 단계.

### Step 2 — Reward Model 학습
- 같은 prompt에 대해 LM이 여러 답변 생성 → 사람이 비교.
- ⭐ 비교 데이터(chosen/rejected)로 **사람의 선호를 흉내내는 채점기** 학습.
- 사람이 매번 평가하기엔 비싸고 시간적 부하 ↑ → 자동 채점기 필요.

### Step 3 — Reward Model로 RL
- LM이 응답 샘플링 → Reward Model이 점수 → 점수 최대화 방향으로 LM 업데이트.
- ⭐ 핵심: **SFT는 기본기, RLHF는 사람의 선호에 맞게 조정**.

## 5. ⭐ Pairwise Comparison (Bradley-Terry) — 절대 점수의 함정

### 5.1 ⭐ Calibration 문제 (시험 출제 가능)
- "이 답변에 10점 만점에 몇 점?" → A는 6.6, B는 3.2.
- 사람마다 점수 기준이 다름 → **calibration이 안 됨** → 절대 점수는 불안정.

### 5.2 ⭐ Pairwise Comparison
- 두 답변 사이 **어느 쪽이 더 좋은지**만 결정하는 방식.
- 부등호 형태 (`Y1 > Y2`)로 라벨링.
- N개 답변이면 nC2 페어로 데이터셋 구성.
- 선택된 답변 = **winning sample / chosen**.
- 선택되지 않은 답변 = **losing sample / rejected**.

### 5.3 왜 페어인가 (ranking 대신)
- 사람에게 5~6개 중 하나 고르기 < 둘 중 하나 고르기 (인지적으로 더 안정).
- 학습에서도 페어가 더 안정적.
- Listwise / Ranking feedback도 사용 가능하긴 하지만 페어가 표준.

### 5.4 RLHF의 한계 — Bias
- ⭐ **사람 피드백 → 편향 발생 불가피**.
- **아첨 편향 (Sycophancy) / 과개입 편향 / Reward Hacking** 등 별도 한계 존재.
  - 강의 후반부에 "아첨 편향이라 했던 것은 정확히는 **Reward Hacking**"이라고 정정됨.
- 의료 / 법률 / 금융 같은 **고가치 도메인에 RLHF만으로는 신뢰 어려움** — 사용자 경향을 학습해 더 잘못된 방향으로 강화될 위험.

## 6. ⭐ Reward Model 학습과 PPO

### 6.1 Reward Model 입출력
- 입력: prompt + response.
- 출력: scalar reward.
- 학습 목표: **chosen에는 높은 점수, rejected에는 낮은 점수**.

### 6.2 ⭐ PPO + KL Penalty (시험 출제 핵심)
- 단순히 reward 최대화하면 모델이 **reward model의 허점을 이용** → reward hacking.
- ⭐ PPO 기반 RLHF의 objective:
  ```
  E[r(y_hat)] − β · KL(π_RL ∥ π_pretrained)
  ```
  - 앞 항: 답변에 대한 보상 최대화.
  - 뒤 항: **현재 정책이 사전학습된 정책에서 얼마나 벗어났는지**를 측정하는 페널티.
  - β: 벗어남 제어 강도.
    - β 크면 → 기존 모델에서 멀어지는 것을 강하게 벌점.
    - β 작으면 → reward를 적극 따라감.
- ⭐ **KL-Divergence**: 현재 policy와 reference policy의 분포 차이.
- 직관: **"reward는 높이되, 원래 모델에서 너무 멀어지지 마라"** = 안정성 유지를 위한 보조 장치.

### 6.3 Cross-Entropy와 KL의 관계 (참고)
- Cross-Entropy: 정답 분포 P와 예측 분포 Q 사이 거리 측정. SFT에서 답변 확률을 높이는 데 사용.
- KL Divergence = Cross-Entropy − H(P).
- SFT는 정답 확률을 높이고, PPO는 reward를 높이되 KL로 안정성 유지.

### 6.4 ⭐ RLHF 효과 (그래프)
- **Pre-trained Only < Supervised Fine-Tuning < RLHF (with Human Feedback)**.
- 사람이 선호하는 답변 생성 능력에서 RLHF가 가장 우수.

## 7. ⭐ InstructGPT Pipeline (실제 OpenAI 적용)

### 7.1 단계별 디테일
1. **사람 작성 Demonstration → SFT**:
   - Plain prompt / Few-shot prompt / API 사용 사례 prompt 등 **다양한 프롬프트 수집** (13K).
   - 사용자가 던질 수 있는 다양한 요청에 대응.
2. **Comparison Data → Reward Model**:
   - SFT 모델이 여러 답변 생성 → 사람 평가자가 chosen/rejected 라벨링.
   - 두 답변 중 어느 쪽이 나은지만 비교.
3. **Reward Model로 PPO**:
   - prompt + response → scalar score.
   - 사람의 선호를 흉내내는 채점기.

### 7.2 ⭐ GPT-3 vs InstructGPT 비교
- **달 착륙 6살 설명**:
  - GPT-3: 비슷한 패턴 문장 반복.
  - InstructGPT: 6살에게 맞는 실제 답변 생성.
- **Open-ended (창작) prompt**:
  - GPT-3: 비슷한 문장 반복.
  - InstructGPT: 프롬프트를 **수행해야 할 지시**로 받아들여 짧은 시 생성.
- ⭐ 차이의 본질: **지식의 양 차이 X, 행동 방식 차이 O**.

## 8. ⭐ DPO — Direct Preference Optimization

### 8.1 PPO의 부담
- Reward Model **별도 학습 필요** → 추가 단계, 추가 비용.
- 학습 시 **3개 모델 동시 메모리 로드** (Reference / Policy / Reward) → 메모리 부담.
- Online sampling 필요 + 하이퍼파라미터 민감.
- 현업 실용주의 관점에서 **부담스러운 파이프라인**.

### 8.2 ⭐ DPO 핵심 아이디어 (시험 출제 가능)
- **Reward Model 없이** chosen/rejected preference pair만으로 policy 직접 학습.
- 외부에서 들어오는 정보는 결국 **"이 답변이 저 답변보다 더 좋다"**는 preference label뿐.
- → 그 정보를 곧바로 policy 학습에 사용.

### 8.3 DPO 수식 직관
- Reward의 차이를 **policy의 log probability 차이**로 변환.
- ⭐ Preference Gap:
  ```
  log π(chosen) − log π(rejected)
  ```
  - chosen 답변의 확률 ↑.
  - rejected 답변의 확률 ↓.
- 두 답변의 log probability 갭이 클수록 더 강한 선호 학습.

### 8.4 DPO의 효과
- Reward Model 없이도 안정적 학습 가능.
- 다양한 task에서 **PPO와 비슷하거나 약간 더 안정적인 성능**.
- 평균 성능에서 DPO가 PPO를 우위.

### 8.5 DPO의 한계
- ⭐ **OOD (Out of Distribution) 데이터에 취약**:
  - 학습 시 보지 못한 유형의 데이터/상황에서 응답 생성에 한계.

## 9. ⭐ ORPO — Odds-Ratio Preference Optimization

### 9.1 SFT의 잔여 한계
- ⭐ SFT는 **정답 토큰 기반의 cross-entropy** → rejected 응답을 직접 낮추는 메커니즘 없음.
- 결과: SFT를 하면 **chosen뿐 아니라 rejected의 확률도 같이 올라감** (정답 토큰만 supervisied하기 때문).
- → 나쁜 출력에 **불이익**을 주는 게 필요.

### 9.2 ⭐ ORPO의 핵심
- **Reference Model 없이** chosen ↑ + rejected ↓를 **단일 단계**로 통합.
- 구성:
  - **SFT 파트**: chosen에 대한 토큰 확률 학습 (기존 SFT loss).
  - **Preference 파트**: σ(log π(chosen) − log π(rejected))로 chosen이 rejected보다 높아지도록.

### 9.3 단계 비교
```
RLHF:   SFT → Reward Model 학습 → RLHF (Reference 고려)
DPO:    SFT → DPO 학습 (Reward Model 제거)
ORPO:   SFT + RLHF 학습이 단일 단계로 통합
```

### 9.4 ORPO의 학습 추이
- Loss는 빠르게 감소.
- Preference Gap은 초반 ↓ → 학습 진행 시 완만하게 ↑.
- 초반 ↓는 초기 random bias에서 진짜 preference gap을 찾아가는 과정.

## 10. ⭐ GRPO — Group Relative Policy Optimization (DeepSeek)

### 10.1 GRPO 등장 배경
- DeepSeek가 2024년 초 **DeepSeek Math 논문**에서 처음 소개.
- DeepSeek-R1을 통해 효율성 입증.

### 10.2 ⭐ DeepSeek-R1-Zero의 Rule-Based Reward
- **Accuracy Reward**: 수학/코딩처럼 정답이 명확한 경우 → 정답 일치 여부로 reward.
- **Format Reward**: CoT(Chain-of-Thought) 응답이 지정된 형식을 지켰는지로 reward.
- ⭐ 핵심: **사람의 주관적 판단 생략** → 더 효율적이고 객관적.

### 10.3 ⭐ DeepSeek-R1 학습 흐름
1. 긴 CoT 예시 + 사람 annotation → **Cold-start data**.
2. 수학/코딩/논리 reasoning task에서 **rule-based RL**.
3. **Rejection Sampling** → 좋은 reasoning 응답 선별 → SFT 데이터로 재사용.
4. Reasoning뿐 아니라 **HH(Helpful, Harmless)** 시나리오까지 포함하여 RL 확장.

### 10.4 ⭐ PPO vs GRPO 차이 (시험 출제 가능)
| | PPO | GRPO |
|---|---|---|
| Value Model | **있음** | **없음 (제거됨)** |
| Reward 측정 | Reference + Reward + Value 동시 작동, GAE로 advantage 계산 | 그룹 내 후보 답변끼리 상대 비교, 그룹 평균 대비 advantage 계산 |
| 메모리 부담 | Value Model이 policy와 같은 크기 → 큼 | Value Model 제거로 ↓ |
| 비유 | 판사(Value Model)가 평가 | 학생들끼리 그룹 시험 보고 평균보다 잘한 학생에게 상 |

### 10.5 GRPO Advantage 계산
```
advantage_i = (reward_i − mean_reward) / std_reward
```
- 평균보다 좋은 답변 → positive advantage.
- 평균보다 나쁜 답변 → negative advantage.
- ⭐ 이 advantage가 **gradient의 크기와 방향을 결정하는 weight**로 작동:
  - 양수 → log probability ↑.
  - 음수 → log probability ↓.

### 10.6 ⭐ DeepSeek-R1 성능
- AIME, Codeforces, GPQA, MMLU, SWE Bench 등 다양한 reasoning 벤치마크.
- 수학/코딩 중심 reasoning task에서 **OpenAI o1과 거의 동등**한 성능.
- 모든 벤치마크에서 SOTA는 아니지만 RL 기반 학습이 **reasoning 능력 발현**에 효과적임을 입증.
- → **오픈소스 거대 모델**의 가능성을 연 사례.

## 11. ⭐ LoRA — Low-Rank Adaptation

### 11.1 왜 필요한가
- 기존 Fine-tuning은 **모델의 모든 파라미터** 업데이트 → 계산량 ↑↑, 메모리 ↑↑.
- 수백억 파라미터 모델에서는 비현실적.

### 11.2 ⭐ LoRA의 핵심
- **기존 weight W는 고정**, **작은 업데이트만 학습**.
- Low-rank decomposition:
  ```
  W → W + (A · B)
  ```
  - A, B는 작은 rank의 행렬.
- Attention 같은 중요 layer에 적용.
- 추론 지연 시간 추가 X (학습 후 W에 합쳐 사용 가능).
- 성능 유지 + 학습 비용 대폭 ↓.

## 12. 실습 — Pre-training → SFT → DPO → ORPO → GRPO

### 12.1 환경
- **Unsloth**: 학습/추론 속도를 2배 이상 가속하는 오픈소스 라이브러리.
- 베이스 모델: **LLaMA 3.2 1B Instruct (BNB 4-bit)** + LoRA 어댑터.
- T4 GPU.

### 12.2 Part A — Pre-training
- 토이 코퍼스 10문장을 100번 반복 → 1000개 데이터.
  - 같은 문장을 100번 학습 → 패턴 학습 / 오버피팅 발생.
- 외부 코퍼스 3000문장 옵션도 제공.
- Epoch=3에서 loss가 급격히 감소 (epoch=5 추천했으나 3이면 충분).
- 모델은 **TinyGPT-2** 사용 (실습 시간 제약).

### 12.3 Part B — SFT
- 추적 질문: `Define DPO in one sentence`.
- 토이 Instruction Dataset 작성:
  - User role → instruction.
  - Assistant role → answer.
- 학습 전 결과:
  - "Disability Protection Officer" / "Delegated Personal Officer" 등 **완전히 잘못된 의미**.
  - "DPO는 Dynamic Programming"으로 잘못 답변.
- 학습 후 결과:
  - "Direct Preference Optimization"이라는 정답을 정확히 생성.
  - Loss 0.1 → 0.0 (Epoch 3, 약 4분).
- ⭐ SFT의 효과 입증.

### 12.4 Part C — DPO
- Preference dataset 직접 작성:
  - chosen: `Direct Preference Optimization`.
  - rejected: `Direct Policy Optimization` (Policy로 잘못된 표현).
- ⭐ Preference Gap = log π(chosen) − log π(rejected).
- DPO 학습 전 평균 gap: **52.83**.
- DPO 학습 후:
  - log π(chosen) ≈ −8.7.
  - log π(rejected) ≈ −140.
  - **Gap ≈ 131.5** (학습 중 최대 142.1까지 도달).
- Loss는 0으로 수렴.
- ⭐ 단, DPO step 너무 많이 돌리면 답변 품질 저하 → 30 step 권장.

### 12.5 Part D — ORPO
- ORPO loss = SFT loss + Preference loss (시그모이드 기반).
- 학습 전: SFT loss = 8.79, Preference part ≈ 0.
- 50 step 후:
  - SFT loss: 0.023 → 0.002.
  - Gap: 67 → 73 (점진적 ↑).
- ⭐ 초반 gap은 randomness로 ↓ → 점차 진짜 preference gap을 찾아 ↑.

### 12.6 Part E — GRPO
- 토이 Reward 직접 부여 (사람이 매긴 점수).
  - 좋은 답변 = 1.0, 살짝 부정확 = 0.8, 완전 잘못 = −1.0 등.
- Advantage = (reward − mean) / std.
- ⭐ GRPO loss에 advantage가 포함:
  - Advantage 양수 → log probability ↑.
  - Advantage 음수 → log probability ↓.
- 30 step 학습 → loss 점진적 ↓.

## 13. ⭐ 한 페이지 요약

### 13.1 시험 핵심 포인트 (오후)

| 주제 | 핵심 |
|------|------|
| **Pre-training의 한계** | 다음 토큰 예측 잘함 ≠ 사용자 의도 따르기 |
| **ICL 한계** | 예시에 의존, 파라미터 업데이트 X, 안정성 ↓ |
| **SFT** | Instruction-Output 쌍으로 명령 following 능력 학습 |
| **MMLU** | 다영역 종합 객관식, LLM의 핵심 벤치마크 |
| **Self-Instruct / Alpaca** | 큰 모델로 데이터 생성 + 작은 모델 학습 패러다임 |
| **HH-RLHF** | Helpful + Harmless, chosen/rejected 라벨링 |
| **Instruction Tuning 한계** | open-ended 정답 없음 + token-level loss 부적절 |
| **Pairwise Comparison** | 절대 점수 calibration 어려움 → 페어 비교 |
| **Bradley-Terry** | 점수의 절대값이 아닌 페어 차이 학습 |
| **PPO** | Reward 최대화 + β·KL(π‖π_ref) 페널티 |
| **KL의 역할** | "원래 모델에서 너무 멀어지지 마라" 안정성 보조 |
| **InstructGPT 3단계** | SFT → Reward Model → PPO |
| **GPT-3 vs InstructGPT** | 지식 차이 X, 행동 방식 차이 O |
| **DPO** | Reward Model 없이 chosen/rejected 페어로 직접 policy 학습 |
| **DPO Preference Gap** | log π(chosen) − log π(rejected) |
| **DPO 한계** | OOD 데이터에 취약 |
| **ORPO** | Reference Model 없이 SFT + Preference를 단일 단계 통합, rejected 직접 ↓ |
| **GRPO** | Value Model 제거, 그룹 평균 대비 advantage로 학습 |
| **DeepSeek-R1-Zero** | Rule-based reward (Accuracy + Format) + RL → reasoning 발현 |
| **LoRA** | W 고정 + 저차원 A·B만 학습, 비용 ↓ 성능 유지 |
| **Reward Hacking** | reward model 허점을 모델이 이용 → 과적합/편향 |

### 13.2 정렬 알고리즘 발전 흐름
```
SFT (정답 따라하기)
  ↓ open-ended 한계 + token-level loss 한계
RLHF (PPO)
  Reward Model 학습 → KL penalty + reward 최대화
  ↓ 비용/메모리/하이퍼파라미터 부담
DPO
  Reward Model 제거, 페어로 직접 policy 학습
  ↓ OOD 한계 + SFT가 rejected 못 낮춤
ORPO
  SFT + Preference를 단일 단계로 통합, rejected 직접 ↓
  ↓ rule-based reasoning을 위한 RL
GRPO
  Value Model 제거, 그룹 평균 대비 상대적 advantage
```

### 13.3 RLHF 3개 알고리즘 비교
| | PPO | DPO | ORPO | GRPO |
|---|---|---|---|---|
| Reward Model | 필요 | 불필요 | 불필요 | 필요 (rule-based 가능) |
| Reference Model | 필요 (KL) | 필요 (DPO 내) | **불필요** | 필요 (KL) |
| Value Model | 필요 | 불필요 | 불필요 | **불필요** |
| 학습 단계 | SFT + Reward + RL | SFT + DPO | **SFT+Preference 통합** | SFT + GRPO |
| 메모리 부담 | ↑↑↑ | ↑ | ↑ | ↑↑ (Value 제거로 ↓) |
| 특징 | 정통 RLHF | 페어 직접 학습 | rejected 직접 ↓ | 그룹 비교 |

### 13.4 비유 / 직관 모음
| 개념 | 비유 / 직관 |
|------|----------|
| **Pre-training의 한계** | "달 착륙 6살에게 설명해줘"에 비슷한 패턴 문장만 반복 |
| **SFT 효과** | "질문 형식 + 선택지 형식까지 이해해서 답하는 행동" 학습 |

### 13.5 강사/조교 어록
- "다음 토큰을 잘 예측하는 것 ≠ 사용자가 원하는 응답을 하는 것."
- "큰 모델이 똑똑해도 사용자의 지시를 무시하거나 안전하지 않은 답을 할 수 있다."
- "좋은 모델로 데이터를 생성하고 더 작은 모델로 학습한다."
- "절대 점수는 calibration이 안 되어 평가로 부적합 — 페어 비교를 쓴다."
- "PPO는 reward를 높이되 원래 모델에서 너무 멀어지지 않도록 한다."
- "DPO는 reward model 없이도 안정적으로 학습할 수 있다."
- "GRPO는 판사 역할의 value model 대신 학생들끼리 그룹 시험을 보게 한다."
- "리워드 해킹 때문에 RLHF 이후에도 연구는 계속되고 있다."
- "RLHF만으로 의료/법률/금융 같은 고가치 분야에 적용은 어렵다."
