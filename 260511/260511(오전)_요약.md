## 3. ⭐⭐ Temperature — Softmax 직관

### 3.1 ⭐ 동작
- LLM의 last layer = logits (raw output).
- Softmax 씌워서 ⭐ **카테고리컬 분포** 만듦.
- ⭐ ⭐ Temperature는 ⭐ **exp 씌우기 전 logit을 T로 나눔**.

### 3.2 ⭐⭐ 직관 — 시험 출제 포인트
```
logit = [1, 2]
T=1: exp(1) vs exp(2)        → 2배 차이
T=0.5: 2배 ÷ 0.5 = 4배 차이 → exp 후엔 더 sharp
```
- ⭐ **T를 작게 할수록 가장 큰 logit이 더 강조됨 = sharp 분포**.
- ⭐ **T → 0**: ⭐ argmax (greedy decoding과 같음).
- ⭐ **T → ∞**: uniform 분포.
- ⭐ ⭐ **시뮬레이티드 어닐링의 temperature와 같은 개념** → 이름 유래.

### 3.3 ⭐ 실무 기본값
- ⭐ 보통 ⭐ **0.6 ~ 0.8**.
- 1은 너무 다양/랜덤 — 잘 안 씀.
- ⭐ Deterministic 필요 시 0 (= greedy).

## 5. ⭐⭐ Sparse Reward 두 가지 의미 — 시험 출제 포인트

### 5.1 ⭐ 의미 1 — 시간적 sparse
- ⭐ 토큰 하나하나 = action (RL 용어).
- ⭐ ⭐ 대부분 토큰엔 reward 없고 ⭐ **마지막 종료 시점에만 reward** 발생.
- ⭐ ⭐ 이전 sparse 검색 매트릭스의 sparse와 같은 뉘앙스.

### 5.2 ⭐ 의미 2 — 그라뉴럴리티 sparse
- 마지막 한 번에 reward를 주는데:
  - ⭐ Binary (0 / 1).
  - ⭐ Dense (0.1, 0.2, ..., 1.0) — 더 그라뉴럴.
- ⭐ ⭐ 현대 용어 = **Outcome Reward** vs **Process Reward**.

### 5.3 ⭐⭐ 결론 — 시험 출제 포인트
- ⭐ ⭐ ⭐ "10개 중 9개 맞은 게 8개 맞은 것보다 진짜 나은가?" — 테스트 케이스를 사람이 ⭐ **퀄리티 align하기 힘듦**.
- ⭐ ⭐ **최근 연구 결론: 0.8, 0.9로 그라뉴럴 만들기 ⭐ 별로 도움 안 됨**.
- ⭐ ⭐ 대안: Process Reward / Progress Reward — ⭐ "이 정도 만들었을 때 잘 가고 있다 / 별로다" 미리 주는 것.
- ⭐ 단점: ⭐ **훨씬 비쌈**.

## 6. ⭐⭐ Pass@K — 직접 구현

### 6.1 ⭐ 동기
- ⭐ 옛날 모델은 너무 못해서 한 번에 못 맞춤.
- ⭐ Temperature 영향으로 ⭐ **같은 모델도 매번 결과 달라짐**.
- ⭐ ⭐ **K번 찬스를 주면 하나라도 맞을 확률** = Pass@K.

### 6.2 ⭐⭐ Naive vs Smart 추정
| 방식 | 설명 | 문제 |
|------|------|------|
| **Naive** | K번 생성 묶음 m번 반복 → 그 중 hit 비율 | ⭐ 너무 비쌈 (mK개 샘플) |
| **Smart** | ⭐ **총 N개 생성 → C(N, K)로 모든 K개 부분집합 평균** | ⭐ 샘플 재활용, 같은 비용으로 variance 작음 |

### 6.3 ⭐⭐ 콤비네이션 식
- ⭐ ⭐ ⭐ **Pass@K = 1 − C(n − c, K) / C(n, K)**.
- 여기서 n = 총 생성 수, c = 맞은 개수, K = 평가 사이즈.
- 의미: ⭐ ⭐ **K개를 다 틀린 거에서만 골랐을 확률을 빼면** 하나라도 맞을 확률.

### 6.4 ⭐ Numerical Stability
- ⭐ ⭐ 그냥 곱하고 나누면 ⭐ **integer overflow** 위험.
- 원본 OpenAI 코드는 ⭐ **하나씩 나누는 형태로 재배치**:
```python
# 분모 시작 n부터 1씩 감소, 분자 시작 n-c부터 1씩 감소, 총 K번
pass_k = 1 - np.prod(1 - K / np.arange(n - c + 1, n + 1))
```
- ⭐ floating-point 오차는 감수.
- ⭐ np.prod = sum의 곱 버전 / arange ≈ range지만 numpy array 반환.

### 6.5 ⭐ Sanity Check
- 10개 만들었는데 0개 맞음 → ⭐ K 뭐든 0.
- 모두 맞음 → 항상 1.
- ⭐ Pass@1 = ⭐ **맞은 개수 / 전체 수** (반복 없음, variance 큰 추정).
- Pass@K(K=n) = 하나라도 맞으면 1, 없으면 0.

### 6.6 ⭐ Unbiased Estimation 보너스
- 모평균 / 모표준편차는 모집단 알아야 진실값.
- 표본평균은 모평균의 ⭐ **unbiased estimate**.
- ⭐ 표본표준편차는 ⭐ **n−1로 나눠야 unbiased** (n으로 나누면 bias).
- ⭐ unbiased = 무한 반복하면 expectation이 실제 값과 같음.

## 7. ⭐ 문제별 Pass@K → 모델 Pass@K

### 7.1 ⭐ 계산
- ⭐ **각 문제마다 Pass@K 계산** → 모든 문제에 대해 ⭐ **평균** = 모델의 Pass@K.

### 7.2 ⭐ 결과 관찰
- ⭐ ⭐ K 늘리면 Pass@K 당연히 ↑.
- ⭐ ⭐ ⭐ Greedy 가까운 디코딩 (T 낮음) → **Pass@1 높음** / K 클수록 saturate 빨리.
- ⭐ ⭐ ⭐ Diverse 디코딩 (T 높음) → Pass@1 낮지만 ⭐ **K 클수록 우상향**.
- 직관: ⭐ "한 번만 확실히 맞춰야 하면 T 낮게 / 여러 찬스가 있으면 다양성 ↑".
- 실험: 20문제 중 T 낮은 게 7개, T 높은 게 9개 맞춤 (K=10).

## 8. ⭐⭐ Verifier — 코드 분야의 강점

### 8.1 ⭐ Verifier의 의미
- ⭐ ⭐ ⭐ **수학**: proof-assistant tools (Lean / Coq) = ⭐ **진짜 verifier**.
- ⭐ ⭐ **코드**: 테스트 케이스로 검증 = ⭐ **조금 약한 verifier**.
- ⭐ 일반 QA / RAG: "정말 문서에 충실한 답인가" = ⭐ **포멀한 verification 아님**.

### 8.2 ⭐⭐ Code Generation = Sampling + Verifier
- ⭐ ⭐ ⭐ 좋은 테스트 케이스만 있으면 ⭐ **다양한 샘플을 많이 만들고 verifier로 거르기**.
- ⭐ 누가 좋은지 verifier가 다 알려주니 ⭐ **고민 X**.
- 코드 생성 = ⭐ **테스트 케이스로 환원**.

### 8.3 ⭐⭐ 프로덕션의 진짜 문제
- ⭐ ⭐ 실무에서는 ⭐ **테스트 케이스가 미리 없음** → 두 가지 접근:
  1. ⭐ **없는 테스트 케이스도 LLM이 만들기** — 단, 잘못된 테케면 맞는 코드 틀렸다고 채점할 수 있음.
  2. ⭐ **확률 / Confidence 같은 단서로 리랭킹**.

## 9. ⭐⭐ CodeT — 코드 ↔ 테스트 그루핑 (Anna Karenina 비유)

### 9.1 ⭐ 컨셉
- ⭐ ⭐ 코드 N개 + 테스트 케이스 M개 모두 ⭐ **cross 실행 → N×M 결과 매트릭스**.
- 결과 = ⭐ 단순 pass/fail이 아니라 ⭐ **실행 출력값** (e.g., return value)으로 완화.

### 9.2 ⭐⭐⭐ Anna Karenina 비유 — 시험 출제 포인트
- ⭐ ⭐ ⭐ "행복한 가정은 비슷비슷하고 ⭐ **불행한 가정은 다 이유가 다르다**".
- ⭐ ⭐ 맞는 코드와 맞는 테스트 케이스는 ⭐ **큰 한 그룹**.
- ⭐ ⭐ 틀린 애들은 ⭐ **작은 여러 그룹** (각자 다른 이유로 틀림).
- → ⭐ **그루핑 결과만 봐서 큰 그룹에 속한 코드를 선택**하면 ⭐ 외부 시그널 없이 정답 확률 ↑.

### 9.3 ⭐ Property-based Testing
- ⭐ ⭐ "add(짝수, 홀수) = 홀수" 같은 ⭐ **property 정의**.
- 자동으로 수십~수백 개 테스트 케이스 생성 가능.
- "input 하나 늘리면 output 1 늘어야" 같은 ⭐ **변화량 테스트**.
- ⭐ 사람이 golden 테케 안 줘도 LLM이 불완전한 거 보충 가능.

## 10. ⭐⭐ BLEU의 한계 (코드 평가용)

### 10.1 ⭐ 컨셉
- ⭐ n-gram precision 합산.
- 어쨌든 ⭐ **lexical match 기반** → 코드 평가에 부적합.

### 10.2 ⭐ 코드 특수성
- ⭐ ⭐ **변수명 swap만 해도 BLEU 떨어짐** (같은 코드인데도).
- ⭐ Python에서 ⭐ **tab vs space** indentation도 차이 만듦.
- ⭐ 맞은 코드인데도 ⭐ **canonical solution과 BLEU 매우 낮음**.
- ⭐ ⭐ ⭐ **틀린 코드와 맞은 코드의 BLEU 분포가 크게 오버랩** → 변별 불가.
- 한 줄만 빠져도 ⭐ 컴파일 X인데 BLEU는 거의 같음 → ⭐ 신뢰성 X.

### 10.3 ⭐ 다양한 정답
- 리스트 합계 = built-in `sum` / for loop / 재귀 — 모두 가능.
- ⭐ canonical solution은 보통 ⭐ 하나만 주어짐 → BLEU에 불리.

### 10.4 ⭐ CodeBLEU
- ⭐ identifier에 덜 sensitive하게.
- ⭐ ⭐ **data flow 정보** 더함.
- ⭐ 실행해보는 거랑 100%는 아니지만 ⭐ **correlation 향상**.

## 11. ⭐⭐ Iterative Refinement — Blind Retry보다 효율적

### 11.1 ⭐ 컨셉
- ⭐ K번 처음부터 다시 = ⭐ **blind retry**.
- 대신 ⭐ ⭐ ⭐ "한 번 시도하고 ⭐ **에러 메시지 보고 부분 수정**" → 효율적.
- ⭐ 직관: 같은 실수 반복 안 함.

### 11.2 ⭐ 미니멀 버전
```
이전 시도: <코드>
실행 결과: <에러>
→ 뭐가 틀렸는지 생각해보고 다시 만들어봐
```

### 11.3 ⭐⭐ 한계 / 문제점
- ⭐ ⭐ ⭐ **Context length 누적** — 입력이 점점 길어짐.
- ⭐ ⭐ ⭐ A 문제 고침 → B, C 고치다가 ⭐ **A 다시 introduce** (도돌이).
- ⭐ ⭐ 처방: ⭐ **branch** / 고치는 LLM과 분석하는 LLM 분리 / 협동.
- ⭐ 보통 ⭐ **3~4번 돌면 saturate** — 더 돈다고 monotonic 개선 X.

### 11.4 ⭐⭐ Communication-based — 초보자 vs 전문가
- ⭐ ⭐ ⭐ ChatGPT가 코드 짠 뒤 사용자 피드백:
  - **초보자**: "이게 안 됐어 / 이렇게 출력됨" 그대로 copy-paste.
  - **전문가**: "여기 if 조건이 문제야 / 이렇게 고쳐줘" — 자세한 피드백.
- ⭐ ⭐ 강사 연구실 워크: ⭐ **모델 A는 자세하지 않은 피드백에 강함 / 모델 B는 자세할수록 잘 받아먹음**.
- ⭐ ⭐ ⭐ 요즘은 사람 대신 ⭐ **LLM이 simulated user** 역할 — 정답과 비교해 어디 고쳐야 한다는 피드백 생성.
- 실패 모드: ⭐ **피드백 LLM이 정답을 그대로 다 까버림** (1000개 hint).

## 12. ⭐⭐ HumanEval+의 약점 + 더 큰 벤치마크

### 12.1 ⭐ HumanEval 약점
- ⭐ ⭐ **테스트 케이스가 약함** → false positive 多.
- ⭐ 너무 비효율적인 코드도 timeout 널널해서 통과.
- 학습 누출 의심도 큼.
- ⭐ **HumanEval+** — 사람이 테케 강화.

## 18. ⭐⭐ NL → SQL — 실습 + 직관

### 18.1 ⭐ 설정
- ⭐ Python + ⭐ **SQLite3** (기본 내장).
- ⭐ Dialect 많음 (PostgreSQL / MySQL / ...) — 우리는 가장 보편 SQLite.

### 18.2 ⭐ 기본 SELECT
```sql
SELECT url FROM keyword WHERE word = 'Bogota'
-- 결과: 3행
```
- COUNT(*), SUM(freq) 등 그대로 활용.

### 18.3 ⭐⭐ JOIN — Multi-hop 추론
```sql
SELECT hyperlink.url FROM hyperlink, keyword
WHERE keyword.word = 'Bogota' AND hyperlink.link = keyword.url
```
- ⭐ "Bogota 단어가 있는 페이지로 ⭐ **링크하는** 페이지의 url".
- ⭐ ⭐ 충돌 방지 → ⭐ **테이블명.컬럼명** 명시.
- ⭐ ⭐ **순서 다를 수 있어 set으로 비교**.

### 18.4 ⭐⭐ 어떤 테이블을 봐야 하는가 — 시험 출제 포인트
- ⭐ ⭐ ⭐ **자연어 질문 → 어떤 테이블 / 컬럼 봐야 하는지 결정도 nontrivial**.
- ⭐ 회사 데이터는 테이블 100개, 1000개 → ⭐ **선정만으로도 큰 작업**.
- ⭐ ⭐ **Semantic Layer** 제공하면 LLM이 lookup 쉽게 가능.

### 18.5 ⭐ Semantic Parsing
- ⭐ 자연어 → DSL (Domain-Specific Language) / SQL / SPARQL.
- 핵심 = ⭐ ⭐ **자연어 entity를 어떤 컬럼 / 테이블에 매핑할지**.
- ⭐ DB 업체들이 그런 annotation 제공.

### 18.6 ⭐ 실무 팁
- ⭐ ⭐ SELECT 결과가 ⭐ **너무 길어서 context 다 잡아먹는 문제** 자주.
- ⭐ ⭐ **Truncate** 적극 활용.
- 코드와 공통: ⭐ **퀴리 실행해보고 결과로 평가 가능**.

### 19.4 ⭐ 강사 어록
- "트랜스포머는 next token prediction으로 학습되어 prefix → continuation에 체중화 — 만들다 만 함수를 자연스럽게 끝맺는다."
- "프롬프트가 사람마다 달라서 reproducibility 깨졌다 — 그래서 bigcode-evaluation-harness 같은 게 등장."
- "Temperature를 0으로 보내면 argmax — 어떻게 디코딩하든 greedy와 같다."
- "Python은 sandbox가 어렵다 — 학부 수업 환경 만들 때도 진짜 골치 아프다."
- "Sparse reward는 두 가지 의미가 섞여 쓰인다 — 시간적 sparse와 그라뉴럴 sparse."
- "0.8, 0.9로 더 그라뉴럴하게 줘봤더니 별 도움 안 되더라 — 테스트 케이스 자체가 progress와 100% align 안 되니까."
