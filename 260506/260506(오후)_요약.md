## 1. ⭐ BM25 시각화 — TF Saturation 곡선

### 1.1 TF-IDF의 log TF vs BM25의 TF 항
- log TF: TF가 커질수록 **무한히** 증가 (느리지만 끝없음).
- BM25 TF 항: `(K1+1)·tf / (K1 + tf)` — TF가 커지면 ⭐ **(K1+1)에 bound**.
- K1=1이면 → 약 2에 saturate. 그 이상 늘어나도 절대 2 이상 안 됨.
- ⭐ K1, B는 **사용자가 그냥 잘 골라서 쓰는 하이퍼파라미터** (대개 정해진 범위).

### 1.2 ⭐ Length Normalization
- BLEU score와 비슷한 **brevity penalty / length penalty** 아이디어.
- 짧은 다큐먼트에서 한 번 등장 vs 긴 다큐먼트에서 한 번 등장 → ⭐ **짧은 데서 등장이 더 강한 신호**.
- `(1-B + B × |d|/avg|d|)` — 평균 길이 대비 보정.

### 1.3 BM25 IDF — Laplace Smoothing 형태
- TF-IDF의 단순 IDF와 미묘하게 다른 형태.
- 모든 보캡에 c만큼 더하고 분모를 cV만큼 키우는 형태와 유사.

### 1.4 ⭐ BM25의 위상
- "**검색에서 빠지지 않는 강력한 베이스라인**" — 단순한 카운트 기반인데도 검증된 좋은 baseline.
- 어떤 검색 방법 논문이든 BM25를 항상 비교 대상으로.

## 2. ⭐ Inverted Index — Sparse의 효율성 핵심

### 2.1 책의 색인(Index)과의 비유
- 책 뒤 "찾아보기": 단어 → 페이지 번호 매핑.
- ⭐ **Inverted Index도 똑같음**: term → 등장한 document ID 리스트.

### 2.2 ⭐ Sparse Matrix의 본질
- Term-Document Matrix는 **거의 0** — 대부분의 단어는 일부 문서에만 등장.
- 단순 2D Array는 메모리 낭비 → **non-zero entry 위치만 저장**.
- 구조: `term → [(doc_id, count), (doc_id, count), ...]`.

### 2.3 ⭐ PageRank와의 유사성
- Adjacency Matrix (10억 × 10억) — 웹페이지 간 하이퍼링크.
- ⭐ 대부분 0 → Inverted Index 같은 sparse 표현으로 저장.
- "검색의 sparse 자료구조 = 그래프의 adjacency sparse 표현".

### 2.4 ⭐ 효율성 — Query 처리
- BM25 식 분해: 일부 항은 **다큐먼트에만 의존** → 미리 오프라인 계산 가능.
  - Length normalization, doc length, average length.
- ⭐ Query가 들어오면 **Inverted Index에서 query term의 후보 문서 union**만 보면 됨.
- 전체 문서 처음부터 보는 게 아니라 **후보군을 좁힘** → 수백만 문서도 빠르게 검색.

### 2.5 실습 — Inverted Index 구현
- Raw text → tokenize → Counter로 단어별 횟수 → `term → [(doc_id, count), ...]` 딕셔너리 구축.
- Search: query term들의 등장 doc ID들을 set에 union → 후보군만 점수 계산.

## 3. ⭐ Sparse vs Dense — 질적 차이

### 3.1 키워드 매치의 한계
- Sparse: **Murder vs Assassination** — 같은 의미인데 다른 단어 → 매치 X.
- ⭐ "Query에 Murder만 있는데 Assassination이 등장한 다큐먼트를 찾아라"는 Sparse는 못 함.

### 3.2 ⭐ Dense의 장점
- 임베딩 공간에서 Murder ≈ Assassination (의미적으로 가까움) → **소프트 매치 가능**.
- 다국어 임베딩 정렬 시 **Cross-lingual** 검색까지.

### 3.3 ⭐ 의료/법률 도메인의 Sparse 우월성
- 정확한 키워드 매칭이 더 중요한 도메인.
- 전문 용어는 IDF가 매우 높아 매치 신호가 강력.

## 4. ⭐⭐ Bi-Encoder vs Cross-Encoder

### 4.1 Bi-Encoder (Two-Tower / Dual Encoder)
- ⭐ Query와 Document를 **각각 따로** 인코더에 통과시켜 벡터화.
- 보통은 같은 모델 공유 (Siamese).
- 점수 = `cosine(q_vec, d_vec)`.
- ⭐ **장점**: Document 벡터를 **테스트 타임 전에 미리 오프라인 계산** 가능.
- 검색 시 query만 임베딩 + 시밀러리티 서치.

### 4.2 ⭐ Cross-Encoder
- Query와 Document를 **concat해서 하나의 입력**으로 인코더에 동시 통과.
- 두 텍스트가 **서로 attention**할 수 있어 풍부한 인터랙션 가능.
- ⭐ **장점**: 성능 ↑ (Bi-Encoder보다 일관되게 좋음).
- ⭐ **단점**: Query마다 **모든 document와 concat해서 forward pass** 필요 → 매우 비쌈.

### 4.3 ⭐ Trade-off — 성능 vs 효율
- Bi-Encoder: 빠름, 미리 계산 가능, 압축된 단일 벡터의 표현력 제한.
- Cross-Encoder: 느림, 벡터 미리 못 만듦, 성능 ↑.

### 4.4 ⭐ Multi-Phase Retrieval — 표준 패턴
- ⭐ **Sparse (BM25) → Top 1000 후보** → **Cross-Encoder Re-ranking → Top 10**.
- 또는 **Bi-Encoder → Top 100 → Cross-Encoder → Top 10**.
- 산업 표준 — 가성비 최적.

## 5. ⭐ ColBERT — Late Interaction의 균형

### 5.1 컨셉
- Bi-Encoder처럼 Query와 Doc을 따로 임베딩하되, **단일 벡터가 아닌 토큰별 벡터**.
- ⭐ Query 각 토큰 vs Doc 각 토큰의 **pairwise similarity** 계산.
- Document 차원으로 **max** 취해서 — 각 query 토큰별 best match 찾기.

### 5.2 ⭐ Soft Match
- Sparse는 hard match: "Murder" query → Doc에 "Murder" 정확히 있어야.
- ColBERT는 soft match: "Murder" query 토큰 ≈ "Assassination" doc 토큰 임베딩 가까우면 매치.
- ⭐ Sparse의 효율 + Dense의 의미 매치 모두 챙김.

### 5.3 효율성
- Document 임베딩은 여전히 **오프라인 계산 가능**.
- Cross-Encoder보다 훨씬 저렴, Bi-Encoder보다는 비쌈 — 중간 위치.
- Stanford 박사가 만들고 MIT 교수로 임용 → 거의 모든 주요 검색엔진의 표준.

## 6. ⭐ FAISS — Facebook AI Similarity Search

### 6.1 컨셉
- Meta(구 페이스북) 라이브러리 — 대규모 Dense 검색 인덱싱.
- 임베딩 미리 계산 → 효율적 시밀러리티 서치 인프라 제공.
- GPU/CPU 분산 처리 + Approximation 옵션.

### 6.2 Index 구축
- Document 벡터들을 미리 등록 (`index.add`).
- 검색 시 query 벡터로 `index.search(top_k)` 호출.
- 정확 검색 (Exact NN) vs 근사 검색 (ANN) 선택 가능.

### 6.3 ⭐ Approximation 기법들
- **비트 수 줄이기** (Quantization).
- **차원 축소** (PCA / SVD 또는 학습 시 앞쪽 차원에 정보 몰아넣기 — Matryoshka).
- ⭐ **Clustering-based**: 미리 문서를 클러스터링 → query는 가까운 클러스터만 보기.
- **HNSW (Hierarchical Navigable Small World)**: 다큐먼트 간 그래프 → 비슷한 애들 따라 hopping.

### 6.4 ⭐ SVD / 차원 축소 직관
- 큰 singular value들만 남기고 작은 건 버림 → 정보 손실 최소화한 압축.
- ⭐ **학습 시점에 인코더가 앞쪽 차원에 중요 정보 몰아넣게** 학습 가능 → 그냥 슬라이싱으로 차원 축소.
- 비용 줄이고 싶을 때 짧은 벡터 쓰면 자연스럽게 효율 ↑.

## 7. ⭐⭐ RAG (Retrieval Augmented Generation)

### 7.1 동기
- ⭐ LLM의 한계: KV cache는 **휘발성**, 파라미터는 **pre-training 시점에 고정**.
- 반도체 기밀 / 회사 내부 정보 / 최신 뉴스는 LLM이 못 알 수밖에 없음.
- ⭐ 해결: **외부 지식을 long-term memory로 저장 + LLM이 검색해서 사용**.

### 7.2 기본 RAG
1. Query 들어옴.
2. Retriever(Sparse/Dense)로 관련 문서 Top-K 검색.
3. 검색된 문서를 prompt context에 넣어서 LLM 생성.
4. ⭐ **출처 인용 (Citation)** 가능 → 검증성 ↑.

### 7.3 ⭐ RAG의 장점
- **Hallucination 감소** (대부분의 경우).
- ⭐ **Verifiability**: 사람이 출처를 따라가 검증 가능 — 마치 논문 사이테이션처럼.
- ⭐ "헐루시네이션이 없다고 단언할 수 없으니, 사람들이 검증할 수 있는 근거를 주는 게 RAG의 본질".

### 7.4 ⭐ RAG의 함정
- Retriever가 잘 안 되거나, 답이 retrieval에 있어도 generator가 못 쓰는 경우.
- ⭐ 검색 엔진이 **사람용으로 옵티마이즈** → LLM에게 최적은 아님 (뒤에서 다룸).

## 8. ⭐ DPR — Dense Passage Retriever

### 8.1 컨셉
- BERT 기반 Bi-Encoder를 검색에 맞게 ⭐ **Fine-tune**.
- Query encoder + Passage encoder (보통 같은 모델 공유).

### 8.2 ⭐ Contrastive Loss
- Positive pair (query, gold passage) → 가깝게.
- Negative pair → 멀게.
- ⭐ **In-Batch Negatives**: 다른 query의 gold가 내 query에는 negative.
  - 배치 안 다른 샘플들을 그냥 negative로 재활용 → 매우 효율적.
  - 하드 네거티브는 아니지만 충분히 효과적.

### 8.3 ⭐ Hard Negatives
- Positive와 거의 비슷해 보이지만 답이 안 되는 negative.
- 사람이 일부러 만들거나 변별력 있게 큐레이션.
- 미세한 차이 변별 학습 가능.

### 8.4 ⭐ DPR의 위상
- ⭐ Dense Retrieval의 **빠지지 않는 베이스라인** (Sparse의 BM25에 대응).
- 논문에서 자주 보이는 조합: **BM25 (Top-100) + DPR (Re-ranking) → Top-10**.

## 9. ⭐ HyDE — Hypothetical Document Embedding

### 9.1 동기
- ⭐ Query는 짧고 Document는 김 — **분포 / 길이가 다름** → 임베딩 매칭 어려움.
- 코사인이 이걸 보정하지만 한계 있음.

### 9.2 ⭐ HyDE의 천재적 아이디어
- Query를 **가상의 답변(Document)으로 변환** — LLM에게 일단 hallucinate 시킴.
- "사랑니 뽑는데 얼마 걸려?" → LLM "30분 걸려" (이게 정확할 필요 X).
- ⭐ 이 가상 답변이 **새로운 query** 역할 → 진짜 document와 분포가 비슷 → 매칭 ↑.
- 시밀러리티: query ↔ document 대신 **hypothetical document ↔ document**.

### 9.3 반대 방향
- 또는 **document에 미리 LLM으로 "어떤 query가 이 doc을 찾을까"** 해시태그 붙이기.
- query와 hashtag 비교 (둘 다 query 분포).

## 10. ⭐⭐ AI vs Human Optimization — 검색 엔진의 갈림길

### 10.1 ⭐ 핵심 질문
- 검색 엔진은 **사람의 클릭 데이터**로 매일 fine-tune됨 (Learning to Rank).
- ⭐ "그러면 사람용으로 최적화된 구글이 LLM에게도 최적인가?"
- ⭐ **아니다** — A 리트리버 vs B 리트리버 결과를 LLM에 주면, 사람이 좋아하는 구글 결과가 항상 좋은 건 아님.

### 10.2 사람용 vs AI용 도구 분리
- "네이버도 LLM용 네이버를 따로 만들어야 하는가?" 연구 활발.
- 강사 견해: 향후 **논문도 사람용 / AI용 두 버전이 필요할 수 있다**.
  - 지금 논문은 사람 리뷰어가 읽기 좋게 작성.
  - 미래엔 LLM 에이전트가 읽으니 LLM 친화적 포맷이 따로 필요.

### 10.3 ⭐ AI의 검색 결과 활용 Bias
- ⭐ **Lost in the Middle**: 중간에 있는 정보를 **있어도 못 본다**.
- ⭐ **Recency Bias**: 생성 직전(prompt 끝)에 있는 정보 선호.
- 1등 + 마지막 정보는 잘 활용, 중간은 무시.
- ⭐ **Polar Bear Bias**: "백곰을 생각하지 마"라고 negative로 언급해도 → **백곰을 찍어내는 경향**.

### 10.4 사람의 검색 vs AI의 검색
- 사람: NDCG 평가 — 1등 위치에 가장 중요한 정보.
- AI: 1등 + 마지막 위치 선호 + 중간 무시.
- ⭐ 단순 랭킹 외에 **요약(Summarize)이 더 나을 수도 있다** — Top-10을 종합해서 한 덩어리로.

### 10.5 ⭐ RL for Retrieval
- Generator의 피드백으로 Retriever fine-tune.
- 답이 더 잘 맞도록 retriever 발전 → 발전된 retriever로 generator 더 잘 → 반복.
- ⭐ **이런 연구는 구글 정도만 가능** — retriever와 generator를 동시에 fine-tune할 자본 필요.

## 11. ⭐ Multi-turn RAG / Deep Research Agent — 미래

### 11.1 현재 RAG의 한계
- ⭐ "I'm Feeling Lucky" 모드 — top-K를 한 번 뽑고 그대로 사용.
- Retriever를 무조건 신뢰.

### 11.2 ⭐ 추세 — Session-based RAG
- 사람이 검색할 때 결과 보고 query 바꿔가며 세션 진행.
- ⭐ LLM도 multi-turn으로:
  - 검색 → 결과 보고 → 새 query 생성 → 다시 검색 → 종합 → 답변.
  - **하이드처럼 검색 결과가 새 query가 되어 친구 다큐먼트 추가 검색**.

### 11.3 ⭐ Deep Research Agent — 새 벤치마크
- "사랑니 뽑는데 얼마?" 같은 단순 query가 아니라:
  - **"반도체 주가가 어디까지 오를지 최근 연구 다 조사해줘"**.
  - **"HBM 컴프레션 때문에 메모리 주가 하락이 맞는지 분석해줘"**.
- → **문서 여러 개를 모아 보고서 형태로 작성**.
- ⭐ 강화학습 기반으로 **여러 번 검색 + 결과 조직 + 생성**.

## 12. ⭐ 한 페이지 요약

### 12.1 오후 핵심 포인트

| 주제 | 핵심 |
|------|------|
| **BM25 시각화** | TF saturation (~K1+1에 bound) + Length normalization |
| **Inverted Index** | term → [(doc_id, count)] sparse 자료구조 |
| **PageRank Adjacency** | Inverted Index와 동일한 sparse 발상 |
| **후보군 Union** | query term 등장 doc만 점수 계산 — 효율성 핵심 |
| **Sparse vs Dense 한계** | Murder ↔ Assassination 매치는 Dense만 가능 |
| **Bi-Encoder** | Q/D 따로 임베딩 / 미리 오프라인 계산 가능 |
| **Cross-Encoder** | Q+D concat / 인터랙션 ↑ / 비쌈 |
| **ColBERT** | 토큰별 Late Interaction / Soft Match |
| **FAISS** | 대규모 Dense 인덱싱 / Approximation (Quantization, Clustering, HNSW) |
| **Multi-phase Retrieval** | Sparse (1000개) → Dense → Cross-Encoder Re-rank |
| **RAG** | LLM의 long-term memory — Hallucination ↓, Citation 가능 |
| **DPR** | Bi-Encoder + Contrastive + In-Batch Negatives — Dense 베이스라인 |
| **HyDE** | Query를 hallucinate한 가상 doc으로 변환 → 진짜 doc와 매칭 |
| **AI vs Human Opt.** | 사람용 검색엔진이 LLM에 최적은 아님 |
| **Lost in the Middle** | 중간 정보를 있어도 못 봄 |
| **Recency Bias** | 마지막 정보 선호 |
| **Polar Bear Bias** | "백곰 생각하지 마"라고 해도 백곰 떠올림 |
| **Multi-turn RAG** | Session 기반 / Deep Research Agent — 강화학습으로 |

### 12.2 전체 구조 도식
```
[BM25 심화]
TF Saturation (K1) + Length Normalization (B) + Laplace IDF

[Sparse 효율성의 본질]
Inverted Index — term → [(doc_id, count)]
   ↓
   후보군 Union으로 좁힌 뒤 점수 계산

[Sparse → Dense 진화]
Murder vs Assassination — Sparse 매치 X
   ↓
Bi-Encoder (오프라인 인덱싱) ← DPR 학습 (Contrastive + In-Batch Negatives)
Cross-Encoder (Q+D concat, 비쌈)
ColBERT (토큰별 Late Interaction, Soft Match)
   ↓
FAISS (Quantization / Clustering / HNSW로 근사)

[Multi-phase 산업 표준]
Sparse (BM25) → Top 1000 → Cross-Encoder/ColBERT → Top 10

[RAG]
LLM + Long-term Memory 컨셉
   ↓
HyDE (Hypothetical Document Embedding)
   ↓
   AI vs Human 검색 — 다른 최적해
   ↓
   Lost in the Middle / Recency / Polar Bear Bias
   ↓
RL for Retrieval — Generator 피드백으로 Retriever 학습
   ↓
Multi-turn / Session-based RAG → Deep Research Agent
```

### 12.3 비유 / 직관 모음

| 개념 | 비유 / 직관 |
|------|-----------|
| **BM25 Saturation** | "K1=1이면 TF가 무한히 커도 약 2까지만" |
| **Inverted Index** | "책 뒷부분 색인 — 단어에서 페이지 거꾸로 찾기" |
| **PageRank ↔ Inverted Index** | "10억x10억 0이 대부분 / 1도 sparse 표현으로" |
| **Sparse 후보 좁히기** | "수백만 문서 다 볼 필요 없이 query term 등장 doc만" |
| **Bi-Encoder** | "Two-Tower — 다큐먼트 임베딩은 미리 다 만들어 둔다" |
| **Cross-Encoder** | "Q+D를 한 입력으로 — 인터랙션 ↑ / 매번 처음부터" |
| **ColBERT의 균형** | "Soft match면서도 doc 임베딩은 미리 계산" |
| **FAISS Clustering** | "비슷한 문서끼리 묶어 두고 가까운 클러스터만 본다" |
| **차원 축소 학습** | "앞쪽 차원에 중요 정보 몰아넣게 학습 → 슬라이싱으로 압축" |
| **RAG = Long-term Memory** | "LLM의 KV cache는 휘발성 — 외부에 저장하고 가져다 쓰자" |
| **RAG Citation** | "사람이 출처를 검증할 수 있다 — 가짜 사이테이션은 걸린다" |
| **DPR In-Batch Negatives** | "다른 query의 gold가 내겐 negative — 무료 negative" |
| **Hard Negatives** | "Positive와 거의 비슷해 보이는 교묘한 가짜" |
| **HyDE** | "사랑니 뽑기 → LLM이 30분 답 생성 → 그게 새 query" |
| **AI ≠ Human 최적해** | "구글이 사람을 위해 옵티마이즈 → LLM에 최적은 아님" |
| **Lost in the Middle** | "10개 줘도 중간 5개는 있어도 못 본다" |
| **Polar Bear Bias** | "백곰을 생각하지 마라 → 백곰을 떠올리는 인간성" |
| **Deep Research** | "사랑니 30분이 아니라 반도체 주가 보고서를 만들어라" |

### 12.4 강사 어록
- "Sparse Matrix가 대표적으로 큰 데이터의 효율 표현 — PageRank의 Adjacency도 동일."
- "Bi-Encoder의 가장 큰 장점은 다큐먼트를 미리 다 만들어둔다는 것."
- "Cross-Encoder는 거부할 수 없이 성능이 높아진다 — 하지만 사용자는 못 기다린다."
- "ColBERT는 Stanford 박사 시절 작품 — MIT 교수로 바로 갔다."
- "BM25 + DPR — Sparse와 Dense 베이스라인의 표준 조합."
- "구글 비밀번호 변경에 페이스북 비밀번호 결과를 주면 안 된다 — Sparse가 살아남는 이유."
- "HyDE는 Hallucinate해도 된다 — 그게 새 query가 된다."
- "LLM은 중간에 있는 정보를 있어도 못 본다 — Lost in the Middle."
- "백곰 떠올리지 말라고 해도 떠올리는 게 LLM의 Polar Bear Bias."
- "사람용 도구가 LLM에 최적이 아니라 — LLM용 도구가 따로 필요할 수도 있다."
- "RL로 retriever와 generator 동시 fine-tune — 구글 정도만 가능한 연구."
- "RAG가 단순 한 번이 아니라 — Multi-turn Session으로 가야 한다."
- "벤치마크 자체가 'Deep Research Agent'로 바뀌고 있다."

### 12.5 오전 + 오후 통합 — IR 전체 구조

| 단계 | 핵심 |
|------|------|
| **표현 — Sparse** | Term-Document Matrix / 원핫 / 차원 = 단어 수 |
| **표현 — Dense** | Word2Vec / BERT / 차원 = 수백 |
| **스코어링 — Sparse** | TF, IDF, TF-IDF, BM25 |
| **스코어링 — Dense** | Cosine Similarity (Bi-Encoder) / Joint Score (Cross-Encoder) / Late Interaction (ColBERT) |
| **인덱싱 — Sparse** | Inverted Index (term → doc IDs) |
| **인덱싱 — Dense** | FAISS (Quantization / Clustering / HNSW) |
| **학습 — Sparse** | 학습 X (통계 기반) |
| **학습 — Dense** | DPR (Contrastive + In-Batch Negatives + Hard Negatives) |
| **결합** | Multi-phase: Sparse → Dense → Cross-Encoder |
| **활용 — RAG** | LLM 외부 메모리 / Citation / HyDE / Multi-turn Session |
| **AI 특이성** | Lost in Middle / Recency / Polar Bear Bias 고려 |
