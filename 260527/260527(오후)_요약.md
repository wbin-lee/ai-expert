> 오후 흐름: **임베디드 ML의 세 축(하드웨어 가속/소프트웨어 최적화/알고리즘) 복습 → LPIRC 대회 소개와 우승 스토리 → 하드웨어 플랫폼(Jetson TX2)·알고리즘(Object Detection) 선정 → MAP/IoU/Precision-Recall → YOLO vs SSD vs Two-stage(Faster R-CNN) → Tiny-YOLO 선정 → 소프트웨어 최적화 절차(파이프라이닝→Tucker→OpenMP→FP16) 2.5배 → DVFS로 에너지 최적화 → 2회차 대회(네트워크 선정에 최적화 영향 고려, YOLOv2) → ResNet 경량화 실습(Bottleneck/Depthwise-Separable/Quantization-Calibration/Pruning) → 병렬화(파이프라이닝+멀티 컨텍스트+DLA/GPU)로 8배 → JEDI 프레임워크와 멀티 애플리케이션 스케줄링(Pareto/Genetic Algorithm)**
> 강의 주제: **연구실의 LPIRC 대회 우승 사례를 통해, 잘 알려진 경량화 기법들을 "어떤 순서로 어떻게 적용하느냐"라는 방법론(methodology)이 어떻게 큰 성능 향상을 만드는지 이해하고, ResNet 경량화를 직접 실습하며 병렬화로 하드웨어 유틸리제이션을 극대화하는 원리를 파악하는 것**.

# 1. 복습: 임베디드 ML의 구조

임베디드 머신러닝(Embedded ML)은 세 축으로 구성된다.

- **하드웨어 가속** 파트
- **소프트웨어 최적화** 파트
- **알고리즘**(모델을 잘 만드는) 파트

소프트웨어 최적화는 다시 두 큰 틀로 나뉜다.

| 틀 | 내용 | 다룬 시점 |
|---|---|---|
| **Approximation** | Quantization, Pruning, Low-rank Approximation | 전날(어제) |
| **Parallelization** | 병렬화 | 오늘 |

Low-rank Approximation 세부:

- **Tucker Decomposition** → ResNet의 **Bottleneck 아키텍처**
- **Depthwise Separable Convolution** → MobileNet 계열

# 2. LPIRC 대회와 우승 방법론 ⭐

## 2.1 대회 소개

**LPIRC (Low Power Image Recognition Challenge)**

- 2만 장의 이미지를 10분 안에 object detection.
- 스코어 = Accuracy / (Power, Energy). 즉 **실시간성 + 에너지 제약**.
- ImageNet object detection 데이터셋 사용. 2만 장을 다 못 하면 MAP가 0.
- 하드웨어 플랫폼·알고리즘·소프트웨어 최적화 **모두 자유 선택(open)**.

연구실은 임베디드 ML 경험이 전혀 없는 상태에서 3개월 준비 → **2회 연속 우승**. 이후 임베디드 SW 최적화 전문 연구실로 알려짐.

## 2.2 핵심 메시지 ⭐

- **새로운 기술을 개발한 게 아니라, 잘 알려진 기술을 어떻게 적용하느냐(methodology)가 핵심.**
- 삼성이 박사급 엔지니어 수십 명으로 수년 걸려 만든 NPU보다, 연구실 학생 2명이 만든 것이 성능이 더 좋았던 이유는 **하드웨어-소프트웨어를 함께 고려해(co-design) 설계했기 때문**. 구조가 단순해 학회에서는 "novelty 없음"으로 리젝당했지만, novelty만 추구한 다른 논문들은 소프트웨어 고려가 없어 실제로는 안 돌아가는 경우가 많다.
- **비효율을 파악하고 개선하는 것은 모르는 데서 나오지 않는다. 알고 있는 것을 어떻게 잘 활용할 것인가가 핵심.**

# 3. 하드웨어/알고리즘 선정 ⭐

## 3.1 하드웨어 플랫폼

스마트폰, Jetson TX2, Arduino, Exynos, Snapdragon 검토 → 성능이 가장 좋은 **Jetson TX2** 선정.

- 10분에 2만 장 = 약 33.3 FPS(실제 약 3 FPS 필요) 기준을 만족할 만한 성능 필요.

## 3.2 Object Detection 알고리즘

Object Detection = bounding box 찾기 + box 안 물체 분류(classification).

| 종류 | 예시 | 특징 |
|---|---|---|
| **Two-stage** | R-CNN → Fast → Faster R-CNN | RPN으로 region 먼저, 그 다음 classification. MAP 높으나 너무 느림 |
| **One-stage** | YOLO, SSD | fully CNN으로 box+classification 동시. 빠르나 MAP 낮음 |

평가 지표 ⭐:

- **IoU (Intersection over Union)**: bounding box 정확도 = (겹치는 면적) / (전체 면적).
- **Precision / Recall**:
  - Recall = (찾은 true) / (전체 true).
  - Precision = (예측한 true 중 실제 true 비율).
  - 둘은 trade-off(recall을 높이려 아무거나 맞다고 하면 precision 하락).
- **MAP (Mean Average Precision)**: recall에 대한 average precision의 평균(precision-recall 곡선 아래 면적).

## 3.3 YOLO vs SSD

- **YOLO** (You Look Only Once): CNN으로 feature를 뽑으며, 마지막 단에서 각 cell마다 bounding box + class probability + confidence score를 예측. 작은 물체를 못 찾음(cell이 너무 큼).
- **SSD** (Single Shot Detector): YOLO와 달리 앞단(작은 receptive field)에서 작은 물체, 뒷단에서 큰 물체를 찾는 **multi-resolution** 방식 → MAP 향상.
- CNN의 receptive field는 3×3 필터 한 층마다 2씩 커진다(3×3 → 5×5 → 7×7...).

선정: Faster R-CNN은 너무 느림 → SSD, YOLOv2, Tiny-YOLO 비교 → 스코어 기준 **Tiny-YOLO** 선정(conv layer 9개, 416×416 시작 → 13×13 종료).

# 4. 소프트웨어 최적화 절차 (1회차) ⭐

연구실의 최적화 **methodology**:

1. Throughput이 중요하고, CPU/GPU 두 리소스를 **둘 다 busy하게(balanced utilization)** 만든다.
2. **Per-layer optimization**(레이어별로 적용).
3. **에너지는 마지막에** 고려.

핵심 원칙: **항상 bottleneck을 찾아 그것을 줄인다.** Bottleneck이 아닌 것을 줄이면 도움이 안 된다.

| 단계 | 기법 | 효과 |
|---|---|---|
| 1 | **Pipelining** (출력 레이어 NMS는 CPU, CNN은 GPU → 중첩) | 1.7배. Tiny-YOLO는 거의 33.3 FPS 만족 |
| 2 | **Tucker Decomposition** (Low-rank, 큰 필터를 1×1→3×3→1×1로) | 채널 긴 뒷부분은 줄고 앞부분은 오히려 늘어 → per-layer 적용. 적용 후 CPU NMS가 bottleneck이 됨 |
| 3 | **OpenMP** (CPU의 NMS를 멀티코어 병렬화) | CPU 시간 급감 → 다시 GPU가 bottleneck |
| 4 | **Quantization (FP16, half precision)** | 메모리 반감, 대부분 빨라짐. 단 앞부분은 FP16이 더 느려(NVIDIA가 FP32에 최적화한 탓 추정) → **앞부분 FP32, 뒷부분 FP16** 혼용 |

**총 2.5배(250%) 성능 향상** → 실시간성 만족.

이후 **DVFS (Dynamic Voltage Frequency Scaling)**: 너무 빨라서 에너지를 줄이기 위해 bottleneck 아닌 CPU 클럭을 반으로 낮춤(2GHz→), GPU는 약간만 낮춤 → 에너지 최적화. → 2등과 압도적 차이로 우승.

# 5. 2회차 대회: 네트워크 선정에 최적화 영향 고려 ⭐

핵심 발견: **소프트웨어 최적화의 효과는 네트워크마다 다르다. 따라서 최적화 결과를 가지고 네트워크를 선정해야 한다.**

- 1회차에는 raw 속도 때문에 Tiny-YOLO 선택. 2회차에는 "최적화하면 속도는 맞출 수 있다"는 것을 알게 됨.
- YOLOv2, SSD, RetinaNet, YOLOv3 등을 각각 소프트 최적화 범위와 함께 비교.
- **우리가 원하는 33.3 FPS 지점에서 가장 좋은 것**을 고름 → raw accuracy는 YOLOv3가 좋지만, 33.3 FPS에서는 **YOLOv2**가 더 좋음 → YOLOv2 선정.

추가 기법:

- Merge Normalization(scale/bias를 필터에 합침), Selective Half → 20.5 FPS.
- **Image Size Reduction**(416×416 → 320×320 등): CNN 연산량이 이미지 크기에 크게 좌우 → 마지막 속도 향상 기법.
- (ETRI는 외부 인터페이스 없는 TX2 **모듈**만 써서 에너지를 1/3로 줄여 별도 트랙 우승.)

# 6. ResNet 경량화 실습 ⭐

데이터셋: **CIFAR-10** (32×32 RGB, 10 클래스, 학습 5만/테스트 1만). 모델: **ResNet-20**(weight 레이어 20개, 3 스테이지로 채널 16→32→64, 스테이지 첫 블록만 stride 2로 spatial 절반). PyTorch, epoch 3.

ResNet의 핵심: **residual(shortcut)** — 출력 = \(x + F(x)\). 잔차만 학습해 깊어도 vanishing gradient가 덜 발생.

총 4가지 경량화 실습:

| 기법 | 아이디어 | 결과(파라미터/정확도) |
|---|---|---|
| **Baseline** | ResNet-20 (3×3 conv 2개/블록) | 약 23만 파라미터, acc 약 0.62 |
| **1. Bottleneck** | 3×3 conv를 좁은 채널에서. 1×1로 압축 → 3×3 → 1×1로 복원 | 약 45,000 (스테이지당 약 5.5배 절감), acc 소폭 하락 |
| **2. Depthwise Separable** | spatial mixing(채널별 3×3, group=입력채널)과 channel mixing(1×1 pointwise)을 분리 | 약 36,000 (약 6.6배 절감), acc는 bottleneck보다 높음 |
| **3. Quantization (PTQ)** | FP32 weight를 INT8로. scale을 외부에서 줘야 함 | 파라미터 수 동일, **size만 감소**(약 294). Calibration 미적용 시 acc 약 0.15, 적용 시 약 0.29 |
| **4. Pruning (Magnitude)** | weight 절대값 하위 50%를 0으로 (sparse) | 0으로만 만들고 계산은 그대로 → **파라미터 수/size 동일**, acc 약 0.51 |

핵심 포인트:

- 표준 conv는 spatial mixing + channel mixing을 동시에 한다. Depthwise separable은 이를 분리해 \(K \cdot K \cdot C \to K \cdot K \cdot C + C \cdot N\)으로 분해.
- **Quantization의 어려움**: INT8(−128~127)은 지수부가 없어 scale을 외부에서 줘야 함. scale을 잘못 잡으면 정확도 붕괴.
- **Calibration**: weight는 고정이지만 activation은 입력마다 달라지므로, 데이터를 흘려 observer로 activation의 min/max 분포를 관측해 scale을 정함. INT8 양자화의 필수 단계. Calibration 유무가 정확도에 큰 영향.
- Pruning은 보통 이후 재학습(fine-tuning)을 하지만 실습에서는 생략.
- 실무는 4가지를 조합해 정확도/파라미터/레이턴시 trade-off를 하드웨어 메모리에 맞게 조정.

# 7. 병렬화(Parallelization) ⭐

## 7.1 배경

- Jetson TX2(CPU+GPU) → Xavier/Orin부터 NPU(DLA) 추가. 스마트폰도 GPU+NPU.
- 벤더 SDK(**TensorRT**)가 베이스라인 — original compiler 최적화(constant propagation, fusing, 순서 변경)는 accuracy 유지. FP16/INT8 지원.
- TensorRT는 기본적으로 inference를 GPU **또는** DLA 하나만 사용, pre/post-processing(이미지 리사이즈/NMS)은 CPU.

## 7.2 병렬화 단계 — 8배의 원리 ⭐

목표: CPU/GPU/MPU(DLA)를 최대한 활용(utilization 극대화).

| 단계 | 기법 | 이득 |
|---|---|---|
| 1 | **Double buffering + Pipelining** (CPU↔GPU 버퍼 2개로 중첩) | 약 2배 |
| 2 | **Multiple Context** (GPU에 여러 프로그램 동시 — GPU util은 보통 30~50%뿐) | 약 2배 (누적 4배) |
| 3 | **DLA + GPU 동시 사용** (네트워크를 나눠 별도 네트워크처럼 파이프라이닝, DLA0/DLA1/GPU 모두) | 약 2배 (누적 8배) |

- 추가: shared memory(임베디드는 CPU/GPU 메모리 공유 → 데이터 복사 제거), CPU 스레드 증가, batch execution.
- 8배가 가능한 핵심 이유: GPU/DLA에 일을 시켜도 **메모리 접근 동안 노는 시간**이 많아, hardware context switching으로 계속 일을 줄 수 있기 때문.
- 같은 병렬화를 **삼성 갤럭시폰**에서 그대로 적용 → 약 4~5배 향상(OS 스케줄러만 다루던 LSI 엔지니어가 하드웨어 엔진을 활용하지 않았던 것).

## 7.3 JEDI 프레임워크와 멀티 애플리케이션 ⭐

**JEDI**: Jetson-aware, TensorRT 기반 병렬화 프레임워크(공개). Application/batch size 등을 configuration하면 최적 병렬화를 찾아줌.

- DSE(Design Space Exploration): context 개수, 버퍼 개수, 파이프라인 단계/순서를 자동 탐색.
- DLA0/DLA1/GPU 3엔진에서 네트워크를 어떻게 쪼개는가? 옵션 비교:
  - **P&DA** (Partial Duplication + DLA): 2단계 파이프라인이 서로 다른 이미지를 처리하며 overlap → FP16에서 대부분 가장 좋음.
  - **E (GPU only)**: INT8로 바꾸면 DLA 안 쓰고 GPU에 컨텍스트를 많이 넣는 게 더 좋음(파이프라이닝 오버헤드 회피).
  - 파이프라인 단계가 많다고 좋은 게 아니다(balance가 어려워짐).

**멀티 애플리케이션 확장** ⭐:

- 피지컬 AI 시대엔 로봇/드론이 여러 딥러닝 응용을 동시에 돌림(하드웨어는 하나). 각 응용은 다른 제약(FPS/latency/throughput/메모리)을 가짐(예: 자율주행차의 전방 카메라는 빨라야, 후방은 덜).
- 풀이: 각 응용에 대해 **Pareto Optimal mapping** 곡선(자원 요구량 vs 1/속도)을 구하고, 이를 합쳐 GPU/DLA/CPU 용량 100%를 넘지 않으면서 제약을 만족하는 조합을 찾음.
- 탐색에는 **Genetic Algorithm**을 사용.
- **Pareto Optimal**: 여러 objective가 있을 때 모두 최적인 해는 없으므로, 각 objective에서 가장 좋은 해들의 집합.

# 8. 오후 핵심 정리 ⭐

- LPIRC 우승의 핵심은 새로운 기술이 아니라 **잘 알려진 기법을 어떤 순서로 어떻게 적용하느냐의 방법론(methodology)**과 하드웨어-소프트웨어 **co-design**이다.
- Object Detection은 **MAP**(precision-recall 곡선 면적)와 **IoU**로 평가하며, Two-stage(Faster R-CNN, 정확하나 느림)와 One-stage(YOLO/SSD, 빠르나 덜 정확함)로 나뉜다.
- 소프트웨어 최적화 절차는 **항상 bottleneck을 찾아 줄이는 것**: Pipelining → Tucker → OpenMP → FP16으로 2.5배, 이후 DVFS로 에너지 최적화.
- 2회차의 교훈: **소프트웨어 최적화 효과는 네트워크마다 다르므로, 목표 FPS(33.3) 지점에서 최적인 네트워크(YOLOv2)를 선정**해야 한다.
- ResNet 경량화 실습은 **Bottleneck**(약 5.5배), **Depthwise Separable**(약 6.6배, 정확도 더 유지), **Quantization(PTQ + Calibration)**(size 감소, calibration이 정확도 좌우), **Magnitude Pruning**(50% sparse, 단순 0화)을 다뤘다.
- 병렬화는 **Pipelining(2배) × Multiple Context(2배) × DLA+GPU(2배) = 8배**가 가능하며, 핵심은 메모리 접근 동안 노는 하드웨어에 계속 일을 줘 **utilization을 극대화**하는 것이다.
- **JEDI** 프레임워크로 단일 응용의 최적 병렬화를 DSE(P&DA 등)로 찾고, 멀티 애플리케이션은 각 응용의 **Pareto Optimal mapping**을 **Genetic Algorithm**으로 조합해 스케줄링한다.
