# 1. 왜 도메인 특화 가속기인가

## 1.1 무어의 법칙 둔화

과거에는 CPU/GPU 같은 범용 칩 아키텍처가 주류였다.

- 2014~2015년경부터 아카데믹 그룹들(중국 DianNao 시리즈 → Cambricon, MIT CNN 가속기)에서 딥러닝 가속기 연구가 발표되기 시작했다.
- Google TPU도 2013년에 시작되어 2016년 알파고 대국에서 TPU v1이 사용되었다.

가장 중요한 변화는 **트랜지스터 가격 스케일링의 정체**이다.

- 원래 공정 미세화의 핵심 이득은 단위 면적당 트랜지스터를 더 많이 집적해 **트랜지스터 가격이 싸지는 것**이었다.
- 그러나 선단공정 개발/생산 비용이 너무 커지면서 가격 스케일링이 거의 멈췄다.
- 미세화(3nm, 2nm)는 계속되지만 더 이상 싸지지도 빨라지지도 않는다.

## 1.2 Above-Silicon / DSA ⭐

트랜지스터가 더 싸지거나 빨라지지 않으면, 아키텍처 기술로 한계를 극복해야 한다.

- 이것이 **도메인 특화 컴퓨팅(Domain Specific Architecture, DSA)**이다.
- 특정 도메인에 커스터마이징된 아키텍처로 "마지막 한 방울까지 짜낸다".
- GPU도 원래 그래픽 렌더링용이라 AI에 불필요한 부분(렌더링, 물리 시뮬레이션 등)이 붙어 있다. 그런 것을 다 떼어내면 더 효율적이다.
- 사람들이 도메인마다 가속기를 만들면서 XPU(APU, BPU, CPU, DPU, ...)로 알파벳이 다 소진되었다는 농담이 있을 정도로 붐을 이루고 있다.

## 1.3 TPU 경제성과 Efficiency is Intelligence ⭐

Google의 전통적 역할 분담:

| 구분 | 대상 | 하드웨어 |
|---|---|---|
| 1P (First Party) | 자체 서비스(Gmail, YouTube, Drive) | TPU |
| 3P (Third Party) | 외부 클라우드 커스터머 | 주로 GPU |

- 최근에는 이 구분이 흐려지고 있다(Anthropic이 TPU 도입, 1P 서비스에도 GPU 사용).
- TPU 기반은 GPU 대비 TCO(Total Cost of Ownership) 기준으로 약 1/3~1/4 수준이라고 알려져 있다. 단, 소프트웨어 최적화/생태계가 부족한 "히든 코스트"가 존재한다.

**Efficiency is Intelligence**: 하드웨어는 항상 부족하므로, 기존 하드웨어(GPU 1장, HBM 1스택)를 누가 더 효율적으로 쓰느냐가 시스템의 지능을 결정한다. 결국 **누가 더 값싸고 질 좋은 토큰을 만드느냐**의 싸움이다.

- 이 때문에 하드웨어-소프트웨어를 수직통합(vertical integration)으로 최적화하는 것이 인더스트리의 핵심 화두가 되었다.
- 거의 모든 빅테크(MS, Meta, Amazon 등)가 자체 가속기를 만들며, 중요한 동기 중 하나는 **엔비디아 세금(NVIDIA tax)을 피하거나 협상 카드로 쓰는 것**이다.
- 하드웨어를 잘 만드는 것만큼 그 위의 소프트웨어 스택 능력이 중요하다.

## 1.4 DSA가 주는 구체적 이득 ⭐

1. **도메인 특화 병렬성(parallelism)**: 워크로드에 맞는 벡터 프로세싱/멀티스레딩 등을 선택할 수 있다.
2. **메모리 대역폭의 효과적 활용**: 범용 캐시 대신 정확한 access pattern을 알면 just-in-time으로 필요한 데이터만 불러올 수 있다.
3. **불필요한 정밀도(accuracy) 제거**: 범용 32/64bit 대신 16/8/4/2bit의 narrow datapath를 쓴다. 칩 면적·전력 효율이 좋아진다.
4. **도메인 특화 프로그래밍 모델**로 소프트웨어 엔지니어링 코스트 절감.

또한 모델/토큰 스케일링이 계속되어(테스트 타임 스케일링, KV cache, agent 간 통신) 더 많은 compute와 메모리 수요가 발생하고 있다.

# 2. Flexibility-Efficiency 스펙트럼 ⭐

AI 알고리즘을 돌리는 실리콘 옵션은 flexibility와 efficiency가 서로 상충하는 스펙트럼 위에 있다.

| 위치 | 아키텍처 | 특징 |
|---|---|---|
| 가장 Flexible / 가장 Inefficient | **CPU** | 범용 |
| Flexible | **GPU** | 범용적이나 효율 중간, 그래픽/HPC도 지원(고정밀도) |
| 중간 | **TPU / NPU** | 그래픽·고정밀도 제거, narrow datapath, 더 효율적·덜 유연 |
| 가장 Efficient / 가장 Inflexible | **Fixed-Function ASIC** | 단일 알고리즘만(예: 트랜스포머만) |

- AI 가속기 하드웨어의 역사는 AI 기술 자체만큼 오래되었다(1960년대 Mark 1 Perceptron). 과거엔 무어의 법칙이 잘 동작해 커스텀 하드웨어가 성공하지 못했다.
- 자율주행(Tesla FSD, Waymo SOC), 로보틱스 등도 중요한 커스텀 하드웨어 도메인이다.

# 3. GPU 아키텍처 ⭐

## 3.1 데이터 병렬성과 SIMD/Warp

GPU는 그래픽 처리를 위해 만들어져 **데이터 병렬성(data parallelism)**에 특화되었다.

- 수많은 픽셀을 독립적으로 병렬 처리할 수 있다(예: diffuse shader).
- 각 코어는 fetch → decode → execute → writeback을 수행하지만, 많은 코어가 **같은 프로그램**을 실행한다.

핵심 최적화: **front-end(fetch/decode) 공유**

- 8개(또는 32개) 코어를 하나의 그룹으로 묶어 front-end를 공유하고, back-end(ALU/context/register file)만 따로 둔다.
- GPU에서 32개를 하나의 그룹으로 처리하는 것을 **warp**라고 한다.
- 이것이 **SIMD(Single Instruction Multiple Data)** 처리이며, 하나의 명령어로 여러 데이터를 동시 처리한다.
- 이런 코어가 Streaming Multiprocessor(SM)이고, GPU는 SM 개수가 매우 많아 massively parallel하다.

## 3.2 Tensor Core와 Sparsity

- Volta(2016~7) 세대부터 행렬 연산 가속용 **Tensor Core**(MMA, Matrix Multiply-Accumulate 특화)를 추가했다.
- 최신 세대는 16/8/4bit 연산과 스케일링을 지원한다.
- **2:4 sparsity (zero-skipping)**: 연속된 4개 숫자 중 최대 2개가 0이면 그 0을 계산하지 않는다. 어떤 수에 0을 곱하면 0이므로 accuracy loss가 없다. 마케팅상 dense 1 PFLOP가 sparse로는 2 PFLOP에 해당한다고 주장한다.

## 3.3 세대 비교

| GPU | 세대/연도 | 특징 |
|---|---|---|
| H100 | Hopper, 2022 | FP32 1 PFLOP, FP16/8 더 높은 PFLOP, Transformer Engine |
| B300 | Blackwell Ultra | FP4 15 PFLOP, 288GB HBM3E, 8TB/s 대역폭, 파워 2배 |

# 4. TPU 아키텍처 (Hard DPU) ⭐

## 4.1 배경

- TPU v1은 2013년 기획 시작(스마트폰 음성 어시스턴트 수요 추산 결과 기존 데이터센터의 2~3배 capa가 필요 → inference 비용을 CPU 대비 10배 이하로 낮추는 design goal).
- 2015년 deploy, 2016년 공식 발표. Inference용 칩으로 integer 연산만 지원.

## 4.2 Systolic Array / MXU ⭐

TPU의 핵심은 **MXU(Matrix Multiply Unit)** — 256×256 FMA(Fused Multiply-Add) 배열이다.

동작(weight-stationary):

\[
y = Wx
\]

- Weight matrix 값을 미리 각 FMA 셀에 preloading한다.
- 입력 벡터/행렬 값이 한쪽에서 한 사이클씩 흘러들어간다.
- 매 사이클마다 (현재 weight × 막 들어온 입력) 곱셈 후 partial sum을 옆으로 넘기며 누적한다.
- 입력이 사선형으로 한 사이클씩 delay되어 들어오고, n 사이클 후 모든 출력값이 계산된다.

특징:

- Steady state에서 매 사이클 모든 FMA가 busy → **dense computation, throughput 최적화**.
- 모든 FMA가 **이웃하고만 통신**(글로벌 와이어 없음) → area/energy efficient.
- TPU는 v1부터 v8까지 발표되었고, 같은 세대 NVIDIA GPU와 스펙(공정/메모리)이 거의 유사하다.

## 4.3 Hard DPU vs Soft DPU

| 구분 | 정의 |
|---|---|
| **Hard DPU** | 실행 가능 연산/데이터 타입이 설계 시점(design time)에 고정. 예: TPU |
| **Soft DPU** | FPGA 기반, field에서 number format·연산을 커스터마이징 가능 |

- FPGA 기반 딥러닝은 현재 프로덕션에서는 거의 사용되지 않고 Hard DPU(NPU)로 넘어갔다. (FPGA는 룩업 테이블 기반이라 area 효율이 떨어짐)
- Microsoft Brainwave(FPGA, 기존 네트워크 패킷 처리용 FPGA 활용) → 현재는 자체 Maia 가속기로 전환.

## 4.4 Fixed-Function ASIC

- 트랜스포머만 돌리는 하드웨어를 만드는 스타트업이 있다(GPU 대비 token/watt 수십~100배 주장).
- Self-attention은 컨텍스트 길이 \(L\)의 제곱에 비례하는 complexity \(O(L^2)\)를 가져(모든 토큰 쌍의 pairwise attention score) 컨텍스트가 길어질수록 비중이 커진다.

# 5. AI 컴퓨팅의 챌린지 ⭐

## 5.1 데이터 무브먼트 비용 ⭐

데이터센터 파워의 대부분은 연산(FLOP)이 아니라 **데이터 이동**에 쓰인다(on-chip interconnect, chip-to-chip(PCIe/NVLink), server-to-server(CXL/InfiniBand)).

에너지 비교(8bit add 기준):

\[
\text{32bit DRAM read} \approx 20000 \times \text{8bit add}
\]

(약 640 vs 0.03 단위)

→ **데이터 무브먼트 최소화가 에너지 효율/성능의 가장 중요한 챌린지.**

해결 방향: **PIM/FIM(Processing/Function in Memory)**

- DRAM 칩 안에 가벼운 연산기를 넣어 SOC로 데이터를 옮기지 않고 로컬에서 처리한다.
- 삼성: HBM 기반 FIM, LPDDR 기반 FIM.
- 데이터센터는 PIM에 소극적(전력보다 그냥 발전소 짓는 게 낫다고 봄), 엣지/모바일은 출구가 없어 더 적극적.
- AX-DIMM: DRAM 칩이 아니라 DIMM 모듈의 버퍼칩에 compute를 넣음. arithmetic intensity가 낮은 워크로드(추천 모델, 필터링 성격)에 적합.

## 5.2 메모리 대역폭/용량 수요 ⭐

- HBM이 비싸고 DRAM cost scaling이 정체. Flash는 3D NAND/TLC/QLC로 아직 스케일링 중.
- **Near-Far Memory**: 빠른 HBM + 느린 CXL/LPDDR/HBF로 메모리를 계층화(bandwidth memory vs capacity memory).
- Agentic AI로 KV cache 수요가 폭증: single agent는 토큰 사용량 약 4배, multi-agent는 약 15배(Anthropic 리포트). 과거의 weight parameter 대신 **KV cache/context가 메모리 driver**가 됨.
- Tiered memory: NVLink C2C(Grace Hopper/Vera Rubin은 CPU-GPU 통합, 호스트 LPDDR을 capacity memory로), CXL(Meta의 cold page를 CXL 메모리로 내려보내는 연구).

## 5.3 Resource Underutilization과 Roofline ⭐

- 하드웨어의 **ridge point**(compute-to-memory bandwidth ratio)는 design time에 고정된다.
- 모델의 **arithmetic intensity**가 ridge point에 딱 떨어지면 compute와 memory를 동시에 saturate → 이상적.
- CNN은 compute-bound, NLP/Transformer는 메모리-bound(낮은 arithmetic intensity)인 경향.
- 미스매치 시 한쪽 리소스가 논다.

해결: **여러 모델을 섞어 co-scheduling**

- 메모리-intensive 모델(A)과 compute-intensive 모델(B)을 적절한 비율로 layer 단위로 섞어 돌리면 composite workload의 arithmetic intensity가 ridge point 근처에 떨어진다.
- 연구실의 **LayerWeaver**: DNN 멀티스레딩으로 PE flop와 memory throughput을 동시에 saturate.

## 5.4 지속가능성 ⭐

- CAPEX가 천문학적(Amazon 올해 약 300B$, Google 약 180B$).
- 에너지/전력이 bottleneck(미국 원전 재가동 등), carbon footprint 문제.
- Meta의 power breakdown: inference 약 70%, training 약 20%, R&D/운영 약 10%. Reasoning 모델 확산으로 inference 비중이 증가 추세.

# 6. RISC-V와 오픈소스 하드웨어 ⭐

## 6.1 오픈소스 하드웨어의 부상

- 과거 오픈소스는 소프트웨어(Linux) 패러다임이었으나, 최근 하드웨어(CAD 툴, IP 코어)에서도 확산.
- 기폭제가 된 것이 **RISC-V** — CPU 아키텍처 + 생태계(ARM의 AMBA/AXI 등에 대항하는 오픈 생태계).

## 6.2 RISC-V 철학

- UC Berkeley(Krste Asanović 교수 + Andrew Waterman, Yunsup Lee)에서 "3 months 프로젝트"로 시작(2010).
- ARM 라이선스가 비싸고(라이선스도 잘 안 해줌) MIPS 등 기존 오픈 코어는 제약이 많아 새로 만듦. **CPU계의 Linux**를 표방.
- License-free, royalty-free ISA. 비영리 **RISC-V International**(미중 텐션을 피해 스위스로 HQ 이전)이 ISA 스펙을 관리하고 ratify.
- 중국이 정부 레벨에서 적극 드라이브(미중 금수조치로 CAD 툴/IP 접근 제한 → 오픈소스로 전환).

## 6.3 RISC-V ISA 특징 ⭐

**Naming convention** 예: `RV64GC`

| 부분 | 의미 |
|---|---|
| RV | RISC-V prefix |
| 32 / 64 / 128 | 레지스터/주소공간 비트 폭(64bit가 가장 흔함) |
| 서픽스(IMAFDGC...) | 구현된 extension |

- 기본 명령어는 **47개**뿐 + 모듈식 extension으로 확장.
- I=Integer, M=Multiply, A=Atomic, F/D=Single/Double Precision Float, **G=IMAFD 묶음**, C=16bit Compressed Instruction(ARM Thumb과 유사, 임베디드 코드 사이즈 절감).
- 따라서 `RV64GC` = `RV64IMAFDC`.
- 레지스터: 32개 integer 레지스터(x0~x31) + (F/D extension 시) 32개 float 레지스터. ABI도 표준으로 정의.
- 명령어 타입 6종(R/I/S/B/U/J), 일부 opcode space는 custom 명령어용으로 reserve(Non-standard extension).

## 6.4 Rocket Chip / Chisel / BOOM ⭐

**Rocket Chip**: parameterizable·modular SOC generator ("one tapeout a day" 모토)

- Reusable SOC 컴포넌트(메모리 프로토콜 컨버터, bus arbiter/crossbar, TileLink 버스).
- **Chisel**: 하드웨어 description language(SystemVerilog보다 high-level, 소프트웨어에 가까움). 호불호가 갈림.
- SOC 템플릿: **Tile**(CPU 코어 1개 + L1 캐시 + 버스 인터페이스) → 모이면 core complex → L2 bank, front bus(DMA), control bus, system bus(계층적).
- 클래스 선언처럼 config를 정의하면 RTL까지 생성.

코어 종류:

| 코어 | 특징 |
|---|---|
| **Rocket** | in-order, 7-stage pipeline, single-issue, 컨트롤 프로세서용, Linux 부팅 가능, ROCC 인터페이스 |
| **BOOM** | open-source out-of-order 고성능 코어, parameterizable, full RV64GC |
| 香山(Xiangshan) | 중국발, 오픈소스 RISC-V 중 가장 고성능 |

## 6.5 ROCC 인터페이스와 Gemmini ⭐

**ROCC**: 코프로세서(가속기)를 코어에 붙이는 인터페이스

- 통신은 단순: command queue + response queue. Command는 2개 레지스터 값 + opcode를 전달.
- **Decoupled**: 큐잉되어 준비되면 처리, 몇 사이클 내 응답 같은 약속 없이 loose하게 통신.
- ROCC accelerator는 메모리 access도 가능(garbage collection, mem copy 등).
- 코프로세서 통신 방법 2가지 비교:

| 방법 | 특징 |
|---|---|
| **MMIO (Memory-Mapped IO)** | 메모리 리전을 매핑, 표준화·드라이버 지원, 가볍게 구현 |
| **ROCC (Tightly-Coupled)** | direct queue로 low latency, 단 드라이버/opcode/프로토콜을 직접 만들어야 함 |

**Gemmini(잼민이)**: ROCC 기반 오픈소스 머신러닝 가속기

- TPU와 같은 systolic array 기반 풀스택 DNN 가속기. 원하는 대로 instantiate해 간단한 NPU를 만들 수 있음.
- 단점: 복잡한 activation function(softmax 등 native 미지원 연산)은 **CPU fallback**(Rocket Core로) → 성능이 바닥으로 떨어짐.

# 7. 오전 핵심 정리 ⭐

- 무어의 법칙(특히 트랜지스터 가격 스케일링)이 둔화되면서, 범용 컴퓨팅에서 **도메인 특화 컴퓨팅(DSA)**으로 무게중심이 이동했다.
- DSA의 이득은 도메인 특화 병렬성, 메모리 대역폭 활용, narrow datapath(number format), 프로그래밍 모델 커스터마이징이다.
- 하드웨어는 **Flexibility-Efficiency 스펙트럼**(CPU → GPU → TPU/NPU → Fixed-Function ASIC)으로 정렬되며, 더 효율적일수록 덜 유연하다.
- **GPU**는 SIMD/warp로 데이터 병렬성을 활용하고 front-end를 공유하며, Tensor Core와 2:4 sparsity로 딥러닝을 가속한다.
- **TPU**는 256×256 systolic array(MXU)에서 weight-stationary 방식으로 행렬곱을 계산하며, 이웃하고만 통신해 area/energy 효율적이고 throughput에 최적화되어 있다.
- AI 컴퓨팅의 핵심 챌린지는 데이터 무브먼트 비용(8bit add 대비 32bit DRAM read 약 2만 배), 메모리 대역폭/용량 수요(KV cache), resource underutilization(Roofline ridge point 미스매치), 지속가능성이다.
- **RISC-V**는 license-free 오픈소스 ISA(47개 기본 명령어 + 모듈식 extension, RV64GC 네이밍)로 오픈소스 하드웨어 생태계의 기폭제가 되었다.
- **Rocket Chip**(SOC generator) + **Chisel**(HDL) + **BOOM**(고성능 코어) + **ROCC**(코프로세서 인터페이스) + **Gemmini**(systolic array 기반 오픈소스 NPU)로 커스텀 SOC를 파라미터만으로 설계할 수 있다.
