## 1. 음성 AI의 핵심 난제 — Alignment Problem

### 1.1 Alignment 정의
- **Alignment(정렬)** = 가변적인 입력과 출력의 **대응 관계**.
- 음성 AI에서 입출력은 모두 **가변 길이(variable length)**.
  - 입력 음성 : 같은 텍스트라도 빠르게/천천히 말하기에 따라 길이 변동.
  - 출력 텍스트 : 문장 길이 자체가 가변.

### 1.2 다양한 음성 Task와 공통 문제
| Task | 입력 | 출력 |
|------|------|------|
| 음성인식 (ASR) | Speech | Text |
| 음성합성 (TTS) | Text | Speech (Mel Spec) |
| 음성번역 (S2ST) | 한국어 음성 | 영어 음성 |
| 보이스 클로닝 | 짧은 reference 음성 + 텍스트 | 그 사람 목소리의 음성 |
| 언어 감지, 음성 변조 등 | (다양) | (다양) |

> 🔥 **(시험 출제 / 핵심)** **거의 모든 음성 Task가 입출력 길이가 가변적이며, 이들 사이의 대응관계 = Alignment**. 이걸 모델링하는 것이 음성 AI의 처음이자 끝.

### 1.3 예시 — "안녕하세요"
- 누군가는 "안녕하세요"를 짧게, 누군가는 길게 말함.
- "안" 이 음성에서 몇 초부터 몇 초인지, "녕"은 어디인지 **그 정보가 학습 데이터에 주어지지 않음**.
- 단순한 분류(Classification)나 CNN/RNN/FC layer로는 풀리지 않음.

### 1.4 보이스 클로닝 사례 (트럼프 예시)
- 짧은 reference 음성만으로 한국어 합성 가능.
- 같은 모델·같은 텍스트로 두 번 생성해도 **톤, 속도, 길이, 추임새가 매번 다름** → 가변성·alignment 변동.

## 2. Alignment 학습이 어려운 6가지 이유

> ⭐ **시험 출제 포인트 — 음성이 어려운 이유 6가지**

| # | 이유 | 설명 |
|---|------|------|
| 1 | **말 속도(Speed) 가변** | 같은 사람·같은 텍스트라도 빠르게/느리게 말하면 데이터가 변함 |
| 2 | **Boundary 모호** | 사람은 "안 녕 하 세 요"를 딱딱 분절하지 않고 부드럽게 연음 → 경계가 모호한 구간 존재 |
| 3 | **Speaker(화자) 다양성** | 같은 말도 화자마다 다르게 발음. 한 명으로 학습할 수 없음 |
| 4 | **동음이의어 / 발음 동일** | 예: "아이스크림(빙수)" vs "아이 스크림(아이가 외침)", 영어 동음이의어. 음성으론 같지만 의미 다름 |
| 5 | **배경 노이즈** | 카페 소음 등이 끼면 음성인식 성능 급락 (사람도 마찬가지) |
| 6 | **Hallucination, 기타** | 다양한 추가 변수 |

> 🔥 핵심 메시지 : "음성인식은 단순히 음성→텍스트 매핑처럼 보이지만, **그 사이의 alignment를 배우는 것이 본질** 이고 매우 어렵다."

### 2.1 가장 큰 문제 — 학습 데이터에 alignment 정보가 없다
- 학습 데이터셋 = (음성 통째, 텍스트 통째) 쌍.
- 음성 안에서 "안"이 몇 초~몇 초인지 등의 **frame-level 정보는 주어지지 않음**.
- 그래서 **알고리즘이 alignment를 스스로 찾아내야 함**.

## 3. 음성인식의 역사 (간략)

| 연도 | 접근 |
|------|------|
| 1950년대 | Bell Labs, Audrey 시스템. 한 사람 목소리로 0~9 분류. **Pattern Matching** 기반 |
| 1980~2000년대 초 | 통계적 방법 (GMM-HMM 등) |
| 2010년대 초 | DNN(Deep Neural Network) 적용 시작 |
| 2015년 ~ 현재 | **End-to-End Neural Network** 가 alignment까지 통째로 학습 |

### 3.1 초기 패턴 인식 방식 (1950s ~ 2014)
1. 학습 데이터에서 알파벳 별로 대응되는 Mel Spec 구간을 **모두 모음**.
2. 알파벳별 Mel Spec 패턴을 학습 (Gaussian Mixture Model 등).
3. Inference 시 입력 음성의 각 프레임이 어느 알파벳과 비슷한지 매칭.

> ⚠️ **모순점** : 패턴 매칭을 하려면 "각 알파벳이 음성의 어디에 있는지" alignment를 알아야 하는데, 학습 데이터에는 그 정보가 없다.

---

## 4. EM 알고리즘적 접근 — "찍고 학습 반복"

### 4.1 핵심 아이디어 (닭이냐 달걀이냐 문제)
1. **Step 1** : Alignment를 **일단 균등하게 찍는다** (예: 텍스트 3글자 → 음성을 3등분).
2. **Step 2** : 그 추정 alignment를 기반으로 **패턴 학습** 진행.
3. **Step 3** : 학습된 패턴으로 alignment를 **재추정** (수정).
4. **Step 4** : Step 2-3을 **수렴할 때까지 반복**.

이 절차의 학명 = **EM (Expectation-Maximization) 알고리즘**.

### 4.2 토이 예시 (시험 출제 가능)
- 학습 데이터 : Mel Spec 길이 12, 텍스트 "배고파" (3글자).
- Step 1 (균등 분할) : "배" 4프레임, "고" 4프레임, "파" 4프레임으로 가정.
- Step 2 (패턴 매칭) : 각 글자가 어떤 Mel Spec과 매핑됐는지 본다.
  - "고" → 4개 프레임 모두 파란색 → "파란색 = 고" 라는 패턴 학습.
  - "배" → 빨강 3 + 파랑 1 (헷갈림).
  - "파" → 파랑 1 + 초록 3 (헷갈림).
- Step 3 (alignment 재추정) : "고는 파란색"이라 학습됐으니, "배" 영역의 파랑 프레임을 "고"로 재할당.
- Step 4 (재학습) : 이제 정렬이 정답에 가까워짐 → 패턴 매칭이 깨끗해짐.

### 4.3 한계
- **반복(Iterative) 학습** 이라 매우 느리다.
- 데이터·모델이 커지면 수렴까지 수백 번 반복 필요.
- 영어 외 언어로 확장 어려움.
- → 2015년 이후 End-to-End 방식으로 대체됨.

### 4.4 🔥 그러나 TTS에서는 여전히 사용 중
- **음성인식에서는 안 쓰지만, 음성합성(TTS)에서는 핵심 알고리즘**.
- Day 3 (5/14)에 다시 이 패턴이 등장 → 잘 기억해 둘 것.

> ⭐ **시험 출제 포인트** : **EM 방식 = "alignment 찍기 → 학습 → alignment 재추정"의 반복**. 이 알고리즘은 음성인식에서는 사라졌지만 **TTS에서는 현재까지도 과반 이상**이 이 방식을 쓴다.

## 5. 현대 End-to-End 음성인식과 Whisper

### 5.1 End-to-End 패러다임
- 더 이상 alignment를 별도로 찍어 반복 학습하지 않음.
- **하나의 신경망**이 음성과 텍스트의 관계 + alignment를 **동시에** 학습.
- Day 2에서 다룰 3가지 대표 알고리즘이 모두 이 패러다임. (현업에서도 활발히 사용)

### 5.2 Whisper 소개
- **OpenAI의 오픈소스 음성인식 모델** (2022년 발표, 2021년 학습).
- 학습 데이터 **68만 시간** (당시 기준 10배 규모. 대부분 유튜브에서 수집된 것으로 추정).
- **99개 언어** 지원, 단순 ASR 외에도 **번역(translation), 언어 감지(language detection)** 까지 한 모델로 처리.
- 모델 크기 : tiny (1GB) ~ large (10GB VRAM 필요). large는 tiny 대비 ~32배 느림.
- 연구자/회사 모두 여전히 활발히 사용. medium·large가 주력.

### 5.3 Whisper의 한계 — Hallucination
- 정답을 뱉은 후 **갑자기 자기 스스로 소설을 쓰기 시작** 하는 현상.
- 해결책 1 : 데이터 무한 증가 (최근 알리바바는 **3000만 시간**까지 사용).
- 해결책 2 : Initial Prompt 등 외부 hint 제공 (실습에서 다룸).

## 6. 실습 1 (Session 5) — Audio 처리 기초

> 강사 : 이동욱 (서울대 인공지능대학원)

### 6.1 환경 셋업
- Google Colab + T4 GPU 필요 (런타임 → 유형 변경 → T4).
- `torch`, `torchaudio` 등 임포트.

### 6.2 Sample Rate 표준값의 유래 (재미 있는 배경)
| Sample Rate | 유래 |
|-------------|------|
| **44.1kHz** | 인간 가청주파수 ~20kHz × 2(나이키스트) + 필터 마진 = 44.1kHz. 음악·고품질 표준 |
| **24kHz** | 가청 2만 × 약간 여유 |
| **16kHz** | 음성 처리 표준 (저주파면 충분) |
| **8kHz** | 전화 통화 (Latency 우선, 품질 trade-off) |

### 6.3 Torchaudio로 오디오 다루기
```
waveform, sample_rate = torchaudio.load(path)
# waveform : [-1, 1] 범위의 1D float 배열
# sample_rate : 16000 등
```
- `display.Audio(...)` 로 Colab 상에서 재생 가능.
- 메타데이터 : shape, sample_rate, channels(mono/stereo), duration, dtype, min/max amplitude.

### 6.4 Resampling
- `torchaudio` 내장 함수 사용.
- 16kHz → 8kHz로 down-sampling 시 음질 먹먹해짐(전화 음질).
- 고화질 → 저화질은 가능, 역은 정보 손실되어 비가역적.

### 6.5 오디오 가공 — 1D 배열이므로 자유롭게
| 연산 | 효과 |
|------|------|
| **Trim** (앞부분만 자르기) | "안녕하세요" → "하세요" 정도만 남음 |
| **Concat** (이어붙이기) | "안녕하세요 안녕하세요" 두 번 반복 |
| **Volume × 10** | 진폭 10배. 다만 [-1,1] 범위를 넘으면 **clipping 발생 → 지지직 잡음** |
| **Volume × 0.1** | 작은 목소리 (clipping 없음) |

### 6.6 STFT → Linear Spectrogram
- 파라미터 : `n_fft=1024`, `hop_length=256`, `power=2`(magnitude squared).
- 출력 shape 예시 : `(513, 84)` — 513=주파수 빈, 84=시간 프레임.
- `n_fft` 작을수록 **시간 해상도 ↑**, 클수록 **주파수 해상도 ↑**. Trade-off.

### 6.7 Decibel (dB) 색상 표현
- Spectrogram의 색상 = 에너지의 **dB(로그 스케일)** 값.
- 사람의 라우드니스 인식이 로그 스케일이기 때문.
- 진하면 큰 에너지, 검으면 작은 에너지.

### 6.8 Mel Spectrogram
- `n_mels=80` (TTS) / `n_mels=128` (Whisper).
- 저주파는 dense, 고주파는 sparse하게 평균.
- 등간격이 아닌 비등간격 분포.
- 정보량이 풍부하고 사람·AI 모두 해석 용이.

### 6.9 Mel Spec으로 알 수 있는 정보
- **남자 / 여자 / 아이** 구분 : F0(가장 낮은 진한 선)의 위치.
  - 남자 : F0 낮음 (간격 좁음)
  - 여자 : F0 중간
  - 아이 : F0 높음 (간격 넓음)
- **모음 / 자음 구분** : 포먼트 위치와 모양.
- **음악의 음정 변화** : 도레미파솔솔파미레도 → 선이 위로 갔다 아래로 내려옴.
- **악기 음색** : 피아노 vs 플루트 등 하모닉 분포 다름.

## 7. 실습 2 (Session 6) — Whisper in Action

### 7.1 Whisper 모델 로드
```python
import whisper
model = whisper.load_model("base")  # tiny, base, small, medium, large
```
- 실습에서는 GPU 제약으로 **base** 모델 사용. 시간 여유 있으면 large로 비교 권장.

### 7.2 Whisper 내부 동작
1. 오디오 입력 → `whisper.log_mel_spectrogram()` 으로 **로그 멜 스펙트로그램** 추출 (자체 내장).
2. Encoder (Transformer) → 인코딩.
3. Decoder (Transformer) → 텍스트 토큰 생성.
4. 시작 시 **언어 special token** (예: `<|en|>`, `<|ko|>`)이 먼저 나옴 → 언어 감지.

### 7.3 기능 1 : Language Detection
```python
_, probs = model.detect_language(mel)
# probs : 각 언어 special token의 확률 분포
```
- 영어 음성 → "en" 96% 확률.
- 한국어 음성 → "ko" 높음.

### 7.4 기능 2 : Transcribe
```python
result = model.transcribe(audio_path, language="en")
result["text"]
```
- 고유명사(예: 김정은, 백악관)는 자주 잘못 표기 → **base 모델 한계**. medium·large로 키우면 개선.
- 추임새("어", "음") 등은 transcript 데이터에 적게 들어가 잘 못 잡음.

### 7.5 기능 3 : Resampling 안 하면 망가진다 (실무 함정)
> 🔥 **(시험 출제 / 실무 핵심)** Whisper는 **16kHz로 학습** 됐기 때문에, 44.1kHz 원본을 그대로 넣으면 **완전히 망가진 출력** 이 나온다. 항상 resample 후 입력해야 한다.

```python
waveform, sr = torchaudio.load(path)  # sr = 44100
waveform_16k = resample(waveform, sr, 16000)  # 반드시 변환
```

이는 Whisper에 한정되지 않음. 모든 오디오 모델(TTS 포함)이 자기 학습 sample rate를 강요한다.

### 7.6 기능 4 : Decoding 파라미터 (LLM처럼 작동)
| 파라미터 | 의미 |
|----------|------|
| **temperature** | 0이면 항상 1등 토큰 (deterministic). 높일수록 랜덤 샘플링 |
| **beam_size** | 동시에 유지할 후보 스트림 개수 (메모리 ↑, 품질 ↑) |
| **patience** | beam search를 얼마나 오래 유지할지 |

- temperature=0 → 매번 같은 결과.
- temperature=0.6 → 매 실행마다 결과 다름.
- beam_size 키우면 더 그럴듯한 결과 (단 메모리 비용).

### 7.7 기능 5 : Word-level Timestamp
- `model.transcribe(..., word_timestamps=True)`.
- 각 단어가 몇 초~몇 초인지 출력 → 자막 생성, 데이터 라벨링에 활용.
- 정확도 높지만 약간의 오차는 있음 (1.84초 → 1.86초처럼 살짝 빗나감).

### 7.8 🔥 기능 6 : Initial Prompt — Hallucination 줄이기
> ⭐ **시험 출제 / 실무 핵심**

```python
result = model.transcribe(audio_path, initial_prompt="김정은 북한 백악관 미국 대통령 트럼프 친서")
```

- LLM 구조이므로 **prefix를 미리 제공** 하면 모델이 이를 참고하여 생성.
- 효과 :
  - "백학관" → "백악관" 으로 정정.
  - "Kim Jong UN" → "Kim Jong EUN" 으로 고유명사 정정.
- 실무 응용 (키오스크 예시) :
  - 메뉴 이름("후파뽕 커리" 같은 특수 용어)을 prompt에 넣어두면 hallucination 급감.

### 7.9 기능 7 : Translation
```python
result = model.transcribe(audio_path, task="translate")  # task="transcribe"가 기본값
```
- 한국어 음성 → 영어 텍스트로 번역.
- 시퀀스-투-시퀀스 모델이므로 자연스럽게 지원.

## 8. 실습 3 — Audio Quality Judgment 모델 만들기

### 8.1 목적
- 컴퓨터가 어떤 음성이 좋은 음성인지 자동으로 판단할 수 있는 **Audio Judgment** 프로그램 제작.
- 두 가지 메트릭 사용.

### 8.2 메트릭 1 : WER (Word Error Rate)
- 정답 텍스트(GT)와 모델 transcribe 텍스트 비교.
- 룰베이스(인공지능 아님).
- 공식 :
  ```
  WER = (Substitutions + Deletions + Insertions) / Total Words in Reference
  ```
- 예시 : Reference="The cat sat on the mat" (6 words), Hypothesis="The cat the mat"
  - Deletion 2 → WER = 2/6 ≈ 0.33.
- **작을수록 좋음** (0이 완벽).

### 8.3 메트릭 2 : UTMOS (1점 ~ 5점)
- **AI 모델 기반** (일본 유명 랩 발표).
- 음성을 입력하면 **자연스러움 점수** 출력.
- 1점 : 로보틱·노이즈, 5점 : 매우 자연스러움.
- 사람들의 점수 데이터로 학습된 회귀 모델.
- 랜덤성 있음 → 보통 **5번 평균** 사용.

### 8.4 합성 데이터 생성 (TTS 모델 두 개)
| 모델 | 시기 | 품질 |
|------|------|------|
| **Silero TTS** | 2021년 | 옛 모델, 품질 낮음 |
| **Kokoro TTS** | 최근 | 작지만 좋은 모델 |

- 30개 텍스트 × 다양한 화자/모델로 ~100개 샘플 합성.

### 8.5 Normalization (필수!)
- WER 계산 전 텍스트 정규화 :
  - 모든 대문자 → 소문자.
  - 다중 공백 → 단일 공백.
  - 구두점 제거.
- 이걸 안 하면 단순 case 차이로 WER이 부풀려짐.

### 8.6 Filtering 로직
```python
filtered = df[(df.wer < WER_THRESHOLD) & (df.utmos > UTMOS_THRESHOLD)]
```
- 예시 임계값 : WER < 0.05, UTMOS > 3.5.
- 임계값을 더 조이면(예: UTMOS > 4) 필터를 통과하는 샘플 수가 급감.

### 8.7 종합 Score 정의
```
score = -100 × WER + 20 × UTMOS  (예시)
```
- WER은 작을수록 좋으니 음수 부호.
- UTMOS는 클수록 좋으니 양수 부호.
- 점수 순으로 정렬해 best/worst 샘플 비교 가능.

### 8.8 결과 분석
- **Kokoro 샘플** : WER 낮음 + UTMOS 높음 → 합성 잘했고 품질도 좋음.
- **Silero 샘플** : WER 가끔 튀고 UTMOS 낮음 → 노이즈·부자연스러움.
- → 좋은 모델 vs 옛 모델의 차이를 자동으로 구분 가능.

### 8.9 활용 시나리오
- 합성 후 자동 필터링으로 **고품질 데이터셋 자동 큐레이션**.
- 사람의 manual 평가 비용 절감.
- 데이터 증강 파이프라인의 마지막 단계로 사용.

## 9. 핵심 정리 (한 페이지 요약)

### 이론 핵심
1. **Alignment Problem** = 음성 AI의 본질적 난제. 입출력 길이 모두 가변, 그 사이의 대응 관계를 학습 데이터에서 찾아야 함.
2. **음성이 어려운 6가지 이유** : 말 속도, 경계 모호, 화자 다양성, 동음이의어, 노이즈, 기타.
3. **EM 알고리즘** (찍고-학습-재추정 반복) : 옛 음성인식의 표준. **현재 TTS 합성에서는 여전히 핵심**.
4. **End-to-End** : 2015년 이후 음성인식의 주류. 한 신경망이 alignment까지 학습.
5. **Whisper** : OpenAI의 99개 언어 ASR 모델, 68만 시간 학습. ASR + 번역 + 언어 감지 통합. Hallucination이 한계.

### 실무 핵심 (시험 출제 가능 ⭐)
| 항목 | 핵심 |
|------|------|
| **Sample Rate Mismatch** | Whisper=16kHz 학습. 44.1kHz를 그대로 넣으면 망가짐. **반드시 resample** |
| **Initial Prompt** | LLM 기반이므로 prefix로 hint 제공 → hallucination·고유명사 오류 급감 (예: 메뉴 이름) |
| **Word Timestamp** | `word_timestamps=True` 로 단어별 시간 추출 |
| **Decoding** | temperature, beam_size, patience로 품질·다양성 조절 |
| **Audio Quality Judgment** | WER (룰베이스) + UTMOS (AI) 결합으로 자동 필터링 |

### 강의 중 주요 비유
