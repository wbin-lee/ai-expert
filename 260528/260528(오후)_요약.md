# 1. Chipyard / Gemmini 개요 ⭐

## 1.1 철학

**Chipyard**: RTL 소스 코드 라이브러리. 검증된 아키텍처를 기반으로 필요한 블록만 추가해 커스텀 SOC를 설계.

- 모든 RTL을 직접 짜는 대신, 이미 만들어진 것을 쓰고 필요한 모듈만 수정.
- 4단계 시뮬레이션(RTL 로직 테스트 → FPGA 시뮬레이션 가속 → 프로토타이핑 → tape-out 전 검증)을 **하나의 통합 프레임워크**에서 지원 → 엔지니어링 코스트 절감, 오픈소스.

**Gemmini(잼민이)**: Chipyard에서 가속기를 담당하는 프로젝트(하드웨어 가속기 이름이자 풀스택 인프라 이름).

- Systolic array 기반 matrix multiplication 가속기(TPU/GPU Tensor Core처럼).
- **ROCC accelerator**: RISC-V custom instruction(안 쓰는 비트로 확장한 ROCC)으로 코어(Rocket)와 통신.
- **Passive 가속기**: 코어(CPU)가 ROCC command로 연산을 시키면 수행하고 대기. 스스로 명령어를 fetch하지 않음.
- **Flexible Hardware Templates**: RTL을 직접 짜지 않고 파라미터(config)만 설정하면 하드웨어가 빌드됨.

## 1.2 Gemmini 하드웨어 구성 ⭐

| 구성 요소 | 설정 가능 항목 |
|---|---|
| **Systolic Array** | dataflow(weight-stationary / output-stationary), PE → tile → array의 row/column 차원, data type(integer/float/custom), bit width |
| **Special Function Unit (SFU)** | GEMM 외 Transpose, Matrix-Scalar Multiplication, ReLU 등 추가 |
| **Scratchpad memory** | weight/input 저장(SRAM). capacity/bank 수/cache policy/single·dual port |
| **Accumulator memory** | 연산 결과 누적(SRAM) |
| **Global memory** | DRAM / shared L2 cache / L3 cache |

Weight-stationary 동작: weight는 CPU→scratchpad→array에 상주, input은 scratchpad에서 올라가 연산 후 accumulator를 거쳐 scratchpad에 저장.

## 1.3 소프트웨어 스택 ⭐

- **Non-standard ISA extension** = ROCC command. 종류: config(가속기 설정), data movement, execution instruction.
- C 코드 작성 시 ROCC command를 직접 입력하기 어려우므로 **매크로**로 제공.
- **프로그래밍 모델**: CPU가 C 바이너리를 실행하다가 ROCC instruction을 만나면 opcode를 확인해 어느 코프로세서(Gemmini)로 dispatch할지 결정. CPU는 일을 시켜놓고 자기 일을 하다가 **fence** 구간에서 sync(완료 확인/대기).
- Instruction 타입: **Load**(global→scratchpad), **Execution**(scratchpad에서 읽어 연산 후 다시 저장), **Store**(Gemmini→global memory).

# 2. Config 파일과 시뮬레이터 ⭐

## 2.1 4개의 Scala config 파일

| 파일 | 역할 |
|---|---|
| **gemmini-default-config.scala** | 전형적 systolic array config(input/accumulator type, bit width, array size 등). 16×16 기본 |
| **gemmini-custom-config.scala** | baseline/high-performance inference config, training config 등. 53번째 라인에서 실제 빌드할 config 지정 |
| **CPU config** | Gemmini는 코프로세서라 단독 불가 → CPU 필수. Rocket(가벼움)/BOOM(무거움), big/medium/small(코어 개수) |
| **SOC config** | 실제 빌드할 것을 모음(Gemmini custom config + L2 + CPU config) |

권장: default는 두고 custom config에 추가해 빌드.

## 2.2 시뮬레이터 선택 ⭐

| 시뮬레이터 | cycle-accurate | 특징 |
|---|---|---|
| **Spike** | ✗ (functional만) | 가장 빠름, 대략적 cycle 표시(예: CPU 200만 cycle → Gemmini 98 cycle) |
| **Verilator** | ✓ | Verilog용, 빌드·실행 느림, waveform 확인 가능 |
| **MIDAS** | ✓ | Verilator와 유사하나 DRAM을 realistic하게 모델링(DRAM config 지정 가능) |
| **FireSim** | ✓ | 위 단점 모두 극복 — Verilator/MIDAS 시뮬레이션을 **FPGA에 올려 가속**. 빌드는 느리나 실행이 빠르고, Linux를 올려 작은 컴퓨터처럼 사용 가능 |

- Verilator/MIDAS는 빌드가 너무 오래 걸려 실습 내 미리 빌드한 체크포인트 사용. Spike만 직접 빌드.
- 빌드 흐름: 하드웨어 config 변경 → 시뮬레이터 빌드(`build-spike.sh` 등) → 소프트웨어(ROCC test C 코드, `template.c`/`tiled-matmul`) 빌드 → `run-spike.sh <binary>` 실행.
- 생성된 Verilog는 `generated-src/.../top.v`에서 확인 가능.

# 3. 실습1: Complex Number 데이터 타입 구현 ⭐

`arithmetic.scala`(Chisel)에서 새 데이터 타입/연산을 정의. 예시로 `UnsignedInt`의 곱셈(`self * T`), 덧셈, MAC(`m1*m2` 합)가 정의되어 있다.

**과제**: Complex Number의 덧셈과 MAC를 완성(곱셈은 예시로 제공). 변수 3개: bit width \(W\), real, imaginary(각각 signed int).

- 복소수 곱셈 \((a+bi)(c+di)\):

\[
\text{real} = ac - bd, \quad \text{imag} = ad + bc
\]

- 덧셈(더 간단): \(W\)는 두 피연산자의 max, real끼리·imag끼리 합.
- MAC: 이미 정의된 곱셈/덧셈을 이용해 `self + m1*m2`.

완성 후 config를 complex config로 변경 → 시뮬레이터/테스트 코드 재빌드 → 체크포인트로 실행 확인.

# 5. TVM 컴파일러 스택 ⭐

후반부(조교 정종헌): 실제 DNN 모델을 가속기 위에서 돌리는 **컴파일러 스택**.

## 5.1 TVM 개요

**TVM (Tensor Virtual Machine)**: 가상화 기능을 가진 컴파일러.

- 탄생 배경: FPGA 가속기를 바꿀 때마다 컴파일러도 바꿔야 하는 소모적 작업을 자동화.
- 다양한 프레임워크(TensorFlow/PyTorch/Keras)의 모델을 다양한 백엔드(CPU/GPU/TPU/커스텀 NPU)에서 돌 수 있는 executable로 컴파일.

## 5.2 컴파일 과정(lowering) ⭐

| 단계 | 설명 |
|---|---|
| **Relay** (IR) | ML 프레임워크의 그래프를 내부 IR로 변환 |
| **TE** (Tensor Expression) | 개별 operator를 어떻게 연산하는지 정의(예: A·B=C matmul). 그래프 수준 high-level 최적화 |
| **AutoTVM / Auto-scheduler** | search space에서 최적 연산 방식을 자동 탐색 |
| **TIR** (Tensor IR) | low-level 백엔드에 specific한 IR. 타깃 하드웨어 스펙(예: NVIDIA Tensor Core)에 따라 최적화 → 최종 executable |

- (Disclaimer: 최신 TVM은 Pynq 보드를 더 이상 지원 안 함 → 이전 버전 유지. high-level은 유사, detail은 다를 수 있음.)

# 6. VTA와 FPGA 실습 ⭐

**VTA (Versatile Tensor Accelerator)**: 실습에서 타깃으로 쓰는 커스텀 가속기.

- FPGA **Pynq 보드**(3대 공유, pincu 0/1/2)에 bitstream이 미리 올라가 있음. 보드에서 서버가 돌아 request를 받아 연산 후 결과 반환.
- 아키텍처: **Load → Compute → Store 3단계 파이프라인**(Gemmini와 유사). 내부 SRAM 버퍼에 DRAM→SRAM으로 옮겨 연산 후 저장. FPGA 가속이 CPU보다 훨씬 빠름.

## 6.1 실습 A: ResNet end-to-end

- ResNet으로 이미지 분류를 VTA에서 실행. weight/graph는 도커에 파일로 준비됨, 스크립트 실행으로 Top-5 결과 확인.
- 보드 IP 환경변수 설정(`VTA_PYNQ_RPC_HOST`, 본인 번호 % 3). 보드 공유라 serialize되어 한 명씩 처리 → 대기.

## 6.2 실습 B: CPU matmul 최적화 ⭐

문제: 텐서 A(128×16K), B(16K×256) matmul을 CPU에서 수행(INT8 — VTA 데이터 타입과 맞춤). Jupyter notebook 셀 단위 실행.

- placeholder로 연산 그래프 정의(메모리 할당 아님), `lower`로 컴파일 결과(TIR, C의 for loop 형태) 확인, helper function으로 빌드한 executable 실행 + numpy로 정확성 verify + 시간 측정.

최적화를 하나씩 적용하며 실행시간 단축:

| 단계 | 기법 | 아이디어 |
|---|---|---|
| 1 | **Blocking / Tiling** | K 방향으로 하나씩 읽던 것을 cache-friendly하게 |
| 2 | **Vectorize (Factorization)** | SIMD(인텔 AVX 등)로 factor 연산을 한 번에 — 축을 명시 |
| 3 | **Loop Permutation (reorder)** | 6-nest loop의 순회 순서를 cache-friendly하게 변경 |
| 4 | **Parallelization** | matmul을 여러 스레드에 나눠 동시 실행(`parallel` 구문) |

- 각 단계마다 baseline 대비 몇 배 빨라지는지 확인. 이 수동 과정을 자동화한 것이 **AutoTVM**(사용자가 정의한 template의 search space를 탐색).

## 6.3 실습 C: VTA matmul 최적화 ⭐

같은 128×16K, 16K×256 matmul을 VTA의 tensor core engine으로. opt0~opt3 4개 파일.

VTA config:

- input buffer 32KB, weight buffer 256KB, accumulator 128KB(log 숫자로 설정).
- tensor core: input tile 1×16, weight tile 16×16, output tile 1×16(batch=1이라 사실상 GEMV).
- accumulator는 32bit(곱셈은 INT8) → 타일 개수 역산(input 2K, weight 1K, accumulator 2K).
- CPU와 달리 DRAM→SRAM을 사용자가 명시적으로 **DMA load/store** 코드로 작성. SRAM 텐서임을 선언(AB buff)하고 DMA 시점을 compute_at으로 지정.

| 파일 | 최적화 | 아이디어 |
|---|---|---|
| **opt0** | Baseline | 성능 기준선. SRAM 버퍼를 비효율적으로 사용(DMA 너무 자주 — input 128 tile, weight 16 tile) |
| **opt1** | **Weight-stationary Tiling** | n_factor/o_factor로 split해 한 번에 많은 데이터를 DMA(DMA는 OS system call 오버헤드 큼). SRAM 버퍼를 최대로 채울 때까지 factor를 키워 DMA 횟수↓ |
| **opt2** | **Pipeline Parallelism (Double Buffering)** | Load와 Compute를 두 스레드로 overlap(3단계 파이프라인 활용) |
| **opt3** | AutoTVM 모사 | 7개 config를 미리 준비해 돌려보며 최적 config 탐색(GOPS, peak 대비 utilization 개선 확인) |

- 최종 셀에서 각 최적화 단계별 실행시간/baseline 대비 배수를 한눈에 비교. 내부 Load/Compute/Store 동작 비율을 VTA cycle 시뮬레이터로 visualize(opt1은 weight 재사용, opt2는 compute-load overlap).

# 7. 오후 핵심 정리 ⭐

- **Chipyard/Gemmini**는 RTL을 직접 짜지 않고 **파라미터(config)만으로** systolic array 커스텀 가속기를 설계·빌드하는 통합 프레임워크이며, Gemmini는 ROCC로 Rocket 코어와 통신하는 passive 가속기다.
- Gemmini 하드웨어(systolic array dataflow/PE·tile·array 차원/data type, scratchpad·accumulator·global memory)와 소프트웨어(ROCC ISA: config/load/execution/store, fence sync)는 **4개의 Scala config 파일**(default/custom/CPU/SOC)로 설정한다.
- 시뮬레이터는 **Spike**(functional, 빠름) → **Verilator/MIDAS**(cycle-accurate, 느림, MIDAS는 realistic DRAM) → **FireSim**(FPGA 가속, Linux 부팅)으로 정확도·속도 트레이드오프가 있다.
- 실습1은 Chisel로 **Complex Number 데이터 타입**(곱셈 \(ac-bd, ad+bc\)/덧셈/MAC)을 구현했고, 실습2는 systolic array를 **16×16→32×32**로 키워 더 적은 cycle을 확인했다.
- **TVM**은 Relay→TE→AutoTVM→TIR로 lowering하며 다양한 프레임워크 모델을 다양한 백엔드 executable로 컴파일하는 가상화 컴파일러 스택이다.
- **VTA**(FPGA Pynq) 실습에서, CPU matmul은 **Blocking/Vectorize/Loop Permutation/Parallelization**으로, VTA matmul은 **weight-stationary tiling(DMA 횟수↓)·double buffering pipelining**으로 최적화하며, 이 수동 탐색을 자동화한 것이 **AutoTVM**이다.
