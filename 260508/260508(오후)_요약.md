## 1. ⭐⭐ Procrustes — SVD로 닫힌 해

### 1.1 ⭐ 핵심
- ⭐ Orthogonal Procrustes 문제는 ⭐ **SVD (Singular Value Decomposition)** 로 ⭐ **닫힌 해**.
- Gradient Descent 같은 근사 X.
- ⭐ Linear Algebra 라이브러리 한 줄로 해결.

### 1.2 ⭐⭐ Less is More 실험
- Dictionary size 변화시키며 P@1 측정:
  - 2000 → 5000: ⭐ **증가**.
  - 5000 → 6000: ⭐ **오히려 감소**.
- 이유: ⭐ ⭐ **빈도 낮은 단어의 임베딩 품질이 낮아 노이즈 작용**.
- ⭐ "양적으로 반드시 많다고 좋은 것 아님 — 도움 되는 쌍이 많아야".

### 1.3 ⭐ 어족 비교 (실습)
- 핀란드어 = 영어와 어족 다름 → P@1 ⭐ **낮음**.
- 스페인어 = 영어와 비교적 가까움 → P@1 ⭐ **높음**.
- ⭐ ⭐ ⭐ **구조적 유사성 = 문화적 유사성 = 임베딩 alignment 잘 됨**.
- ⭐ 결국 임베딩은 ⭐ **코퍼스 안에서 무엇과 같이 자주 언급되는가** 기준이라 문화권 영향.

### 1.4 ⭐⭐ 시험 출제 포인트 — LLM이 다국어 잘하는 이유
- ⭐ ⭐ ⭐ **임베딩 레이어가 암묵적으로 같은 일을 함**.
- Claude / ChatGPT는 ⭐ **별도 행렬 곱 없이도 다국어 잘함**.
- 모든 언어 토큰 (한자 / 한글 / 알파벳)이 ⭐ **유니코드 / 바이트 단위로 토큰화**.
- 임베딩 레이어 거치고 나면 ⭐ **다른 언어라도 비슷한 곳에 모임**.

### 1.5 ⭐ 한계
- ⭐ 자주 나오는 언어가 더 잘 됨 (편향).
- ⭐ ⭐ ⭐ **에이전트가 한국어 vs 영어 시킬 때 성능 다름**.
- ⭐ ⭐ "커서 같은 툴도 한국어로 시키면 ⭐ **tool calling 안 한다거나** 행동 변화".

## 2. ⭐⭐ Multilingual RAG 실습

### 2.1 ⭐ 시나리오
- ⭐ 질문: 한국어 / 검색 대상: 영어 + 다른 언어 섞임.
- ⭐ **읽고 한국어로 답할 수 있는가**?
- 가장 단순한 방법: 모든 문서를 모든 언어로 번역해놓기 → 너무 비쌈.

### 2.2 ⭐ QRels 데이터셋
- ⭐ MS MARCO 같은 형식: query ID + relevant document ID 페어.
- ⭐ Sparse하게 표시 (irrelevant는 그냥 없음).
- ⭐ **편의상 QRels** — 사람이 단 relevance annotation.

### 2.3 ⭐⭐ Recall@K / MRR 정의
- ⭐ **Recall@K (검색)**: 검색해온 K개 중 ⭐ **하나라도 relevant한 게 있나** (히트).
  - 진짜 정답이 매우 적을 때 (수백만 문서 중 2개만 relevant) 사용.
  - ⭐ Missing gold 문제 — extensive annotation 못해서 진짜 relevant인데 빠진 것 있음.
- ⭐ **MRR (Mean Reciprocal Rank)**:
  - ⭐ **1등에 relevant 있으면 1점, 2등에 있으면 1/2, 3등에 있으면 1/3**.
  - ⭐ 가장 잘 찾은 거 하나의 rank 역수.
  - ⭐ "상호 관세에 나오는 reciprocal".

### 2.4 ⭐ MRR@K 구현
```python
for i, doc_id in enumerate(retrieved_top_k, start=1):
    if doc_id in relevant_set:
        return 1.0 / i
return 0.0  # K개 안에 없으면 0점
```

### 2.5 ⭐⭐ BM25 vs Dense 비교 — 다국어
- ⭐ **BM25 (Sparse)**: 한국어 질문 → ⭐ **lexical match만 → 영어 문서 못 가져옴**.
- ⭐ **Dense Multilingual**: ⭐ **애플 ↔ 사과가 비슷한 공간** → 영어 / 스페인어도 가져옴.
- ⭐ ⭐ ⭐ "전통음식 → traditional food로 번역 → 영어 BM25" 같은 방법도 가능.

### 2.6 ⭐ 번역 Baseline
- ⭐ 한국어 query를 영어로 번역 → 영어 문서에 BM25.
- ⭐ Dense보다 검색 lexical 미스가 더 많지만 (Recall ↓), MRR은 높을 수 있음.

### 2.7 ⭐ 다양한 변형
- 영어 문서 보고 ⭐ **바로 한국어로 답** vs 영어로 답 후 ⭐ **한국어로 번역**.
- ⭐ ⭐ **극도로 low-resource 언어**일 때 cross-lingual의 진짜 가치 부각.

### 2.8 ⭐ Generator도 다국어 처리
- 작은 LM (Qwen 1.5B)도 ⭐ **다국어 prompt 그냥 받음**.
- ⭐ ⭐ "Zagreb"를 "Zagreber"라고 잘못 읽는 등 ⭐ **작은 모델은 웃긴 에러** 포함.
- 자체적으로 citation 달기도 함.

### 2.9 ⭐ Hard Negative
- Anchor + Positive + Negative 구조 (Contrastive Learning).
- ⭐ ⭐ **Hard Negative**: 실제론 negative인데 모델이 positive보다 높게 평가한 것.
- 실패 케이스 5%, 성공 95%.

## 3. ⭐⭐ Code Generation — 도입

### 3.1 ⭐ 다양한 형태
- ⭐ **자연어 → 코드**: "버블소트 해줘".
- **코드 → 코드**: 절반 작성된 코드 자동완성.
- **코드 → 다른 언어 코드**: Python → Rust (안전성 보장).
- ⭐ ⭐ **Input/Output 페어 → 코드**: 예시만 보고 함수 유추 (Excel Flash Fill).
- **UI 스크린샷 → 코드**: 디자인 → 웹/앱 코드 자동 생성.

### 3.2 ⭐⭐ 시험 출제 포인트 — 자연어 vs 코드의 차이
- ⭐ **자연어**: 문법 약간 어긋나도 통함.
- ⭐ **코드**: ⭐ ⭐ **세미콜론 하나만 빠져도 컴파일 X**.
- ⭐ 더 엄격한 grammar 요구 (단점).
- ⭐ **장점**: ⭐ ⭐ ⭐ **실행 가능** — 컴파일 + execute → 결과로 평가.
- ⭐ Parse tree / dependency graph / 컴파일러 정보 활용 가능.

### 3.3 ⭐⭐ Execution-based Metric — 학생 영어 평가 비유
- ⭐ ⭐ ⭐ ⭐ 강사 비유:
  - 예전: "발음 굴리네 / 헐리우드 영어 같네" — ⭐ **레퍼런스와 similarity**.
  - 요즘 강사 평가법: ⭐ ⭐ ⭐ **"학생한테 아메리카노 시럽 빼고 사오라 시켰을 때 그대로 가져오면 100점"**.
  - 발음이 후졌든 율동을 하든 ⭐ **task 정확도가 핵심**.
- ⭐ 이게 ⭐ **Execution-based Metric**.

### 3.4 ⭐ Test-time Scaling
- ⭐ ⭐ 코드 생성 엔진: ⭐ **한 번에 greedy 안 하고 K개 생성 → 컴파일 / 테스트 → 통과하는 것만 선택**.
- ⭐ ⭐ ⭐ **Pass@K**: K개 만들었을 때 그 중 ⭐ **하나라도 통과하는** 비율.
- ⭐ "신문지상 개발자 능가 모델"이 Pass@K로 보통 평가.

### 3.5 ⭐ Verifier — 최근 핫한 연구
- ⭐ Verifier도 함께 생성해야 함 — 사람이 일일이 테스트케이스 못 만듦.
- ⭐ 100개 생성 + verifier 통과 → 한 번에 잘하는 척 가능.
- Verifier가 ⭐ **어디가 틀린 것 같은지 의견 주면** → debugging처럼 수정.

### 3.6 ⭐ Communication-based / Incremental
- ⭐ 사람도 디버깅 / 리퀘스트 / 수정 반복.
- ⭐ ⭐ **PR 단위 / 코드베이스 단위**로 점점 발전.

## 4. ⭐⭐ Parallel Corpus & Pre-training

### 4.1 ⭐⭐ Parallel Corpus
- ⭐ "Parallel Corpus" = 같은 의미의 두 언어 쌍.
- ⭐ 입국심사 폼 — 성명/Name, 주소/Address — ⭐ **자연 마이닝 가능**.
- ⭐ ⭐ **최초의 Parallel Corpus = 성경** — 사람들이 목숨 걸고 여러 언어로 번역.
- ⭐ ⭐ ⭐ **코드 세계의 성경 = GitHub** — 자연어 주석 + 코드 + before/after PR.

### 4.2 ⭐ The Stack
- ⭐ ⭐ 2.4 TB 자연어-코드 쌍.
- ⭐ 라이선스 이슈 / 저질 코드 필터링 후 공개.

### 4.3 ⭐ Comparable Corpus
- ⭐ 완전 동일 의미는 아니지만 유사 내용 다루는 쌍.
- ⭐ Wikipedia "Seoul National University" 영어/한국어 페이지 — Parallel은 아니어도 ⭐ **comparable**.

### 4.4 ⭐⭐ Post-training의 중요성
- ⭐ **Pre-training** 후 ⭐ **Post-training** 단계가 매우 중요.
- ⭐ ⭐ **Instruction Tuning**:
  - "다음을 번역하시오: ____" 식의 ⭐ **시킬 일을 빈칸 채우기로 표현**.
  - ⭐ 강사 비유: "예스/노만 답하는 무당이 있는데 ⭐ **프롬프팅 잘하면 원하는 일 시킬 수 있다**".
- ⭐ ⭐ **Continual Pre-training** + 강화학습.
- ⭐ ⭐ 텍스트북에 안 나옴 — 모델 출시 보도자료에만 나옴.

### 4.5 ⭐⭐ Mixing Ratio
- ⭐ ⭐ ⭐ 코드 ↑ 비중 → 코드 잘하는 모델 / 수학 ↑ 비중 → 수학 잘함.
- ⭐ ⭐ ⭐ **왕도 없음 — 그냥 믹스 바꿔가며 실험**.
- Standard practice.

## 5. ⭐⭐ BLEU와 그 한계

### 5.1 ⭐ BLEU 정의
- ⭐ N-gram overlap 측정.
- 1-gram, 2-gram, 3-gram이 사람 reference와 얼마나 겹치는가.
- ⭐ 1-gram만 보면 ⭐ **백오브워드처럼 순서 망쳐도 통과**.
- ⭐ 2-gram, 3-gram 보면 더 엄격.

### 5.2 ⭐⭐ 한계 — 자연어
- ⭐ ⭐ **뜻 같은데 BLEU 0점**일 수 있음.
- 강사 어록: ⭐ ⭐ "내 학생 영어 평가 — 발음 따라하기보다 task 수행이 진짜".

### 5.3 ⭐⭐ 한계 — 코드는 더 심함
- ⭐ Variable 이름 x → y로 바꿔도 ⭐ **완벽히 같은 코드**.
- ⭐ for loop vs while loop — 같은 일을 함.
- ⭐ ⭐ 텍스트만 비교하면 큰 차이처럼 보임.

### 5.4 ⭐ CodeBLEU
- ⭐ Variable name 같은 거 버리고 ⭐ **구조적 비교**.
- ⭐ Parse tree / dependency graph 비교.
- ⭐ 컴파일러 정보 활용.

## 6. ⭐⭐ CodeBERT

### 6.1 ⭐⭐ 컨셉
- ⭐ Microsoft Research Asia (강사 협력).
- ⭐ ⭐ ⭐ **자연어 ↔ 프로그래밍 언어를 BERT로 alignment**:
  - NL: "Find the maximum value and return it".
  - PL: 해당 Python 코드.
  - ⭐ ⭐ **두 입력을 BERT에 동시에 — 같은 뜻 가까운 벡터 되도록 훈련**.

### 6.2 ⭐ 가능한 Task
- ⭐ Code search (NL → 해당 code 검색).
- ⭐ Code Classification.
- ⭐ Pretrained representation으로 다양한 코드 task.

### 6.3 ⭐ 다국어 BERT와 같은 패턴
- ⭐ ⭐ Parallel 쌍으로 빈칸 채우기 → 같은 뜻 representation 비슷한 곳.
- ⭐ 한↔영 alignment와 ⭐ **완전 동일한 발상**.

### 6.4 ⭐ Pre-training에 포함된 데이터 문제 — Sasmsung 일화
- ⭐ ⭐ 강사 본인 페이퍼 사례:
  - ⭐ Rust는 low-resource (사내).
  - Claude Code의 Rust 점수: 50 (다른 언어 90).
  - 강사 팀 결과: 70 달성.
  - ⭐ ⭐ ⭐ 리뷰어 질문: ⭐ "잘 푼 Rust가 너네 ⭐ **pre-training에 들어간 거 아니냐**?"
- ⭐ **모든 모델의 pre-training 데이터를 알 수 없으니 영원한 의심**.

- ⭐ 다국어 코드 — R / Racket(거의 멸종 직전) 등 ⭐ **low-resource 언어**.

### 8.4 ⭐⭐ SWE-bench
- ⭐ ⭐ ⭐ **PR 단위 / 전체 코드베이스 다루기**.

- ⭐ **현실적 개발자 니즈와 일치**.

### 8.5 ⭐ UI/UX 평가
- ⭐ 사람이 써본 후 평가 / 또는 사람 시뮬레이트하는 에이전트.
- "네이버 쇼핑 사이트 만들어줘" → 원하는 쇼핑 빠르게 하는지.

## 10. ⭐⭐ Code Generation의 강력한 응용

### 10.1 ⭐⭐ Retrieval-Augmented Code Generation
- ⭐ ⭐ 사람은 ⭐ **빈 메모패드에서 코드 짜지 않음**.
- ⭐ Stack Overflow / 과거 코드 보고 copy + paste 후 수정.
- ⭐ ⭐ ⭐ **RAG가 자연스럽게 도움 — 비슷한 코드 가져와서 문맥에 추가**.

### 10.2 ⭐⭐ InterCode — Execution Feedback Loop
- ⭐ Try Again (가장 소극적 — 처음부터 다시).
- ⭐ Chain of Thought — 순서도 먼저 그리고 부분 테스트.
- ⭐ ⭐ **Test 실패 부분만 부분 수정** — 인크리멘털 개발 모방.

### 10.3 ⭐ Excel Flash Fill
- ⭐ ⭐ Microsoft Research의 ⭐ **승진 직행 기능**:
  - First Name 첫 자 + Last Name → S황.
  - 몇 가지 예시만 주면 ⭐ **유추해서 함수 생성** → 나머지 자동 채움.
- ⭐ Input/Output 쌍에서 코드 추론.

### 10.4 ⭐⭐ Pseudocode — 자연어와 코드 사이의 Pivot
- ⭐ ⭐ 강사 본인 연구:
  - ⭐ ⭐ ⭐ **Pseudocode = 한국어↔일본어처럼 자연어↔코드 사이의 중간 언어**.
  - Retrieval시 ⭐ 코드 외에 pseudocode도 저장 → 검색 범위 ↑.
- ⭐ ⭐ 최근 논문:
  - 일반 코드 = 실행 가능 = ⭐ **execution 이득**.
  - Pseudocode = ⭐ **LLM에게 "상상 실행" 시킬 수 있음**.
  - ⭐ **두 이득이 상보적** → 같이 쓰면 좋음.

## 11. ⭐⭐ Chain-of-Thought as Code — 수학 추론

### 11.1 ⭐ 컨셉
- ⭐ 일반 CoT: "다섯 개 있고 2 × 3이니까 11"이라는 ⭐ **자연어 풀이 과정**.
- ⭐ ⭐ 강사 발견: ⭐ ⭐ ⭐ **풀이 과정을 코드로 쓰면 더 잘 됨**.
  ```python
  tennis_balls = 5
  bought_balls = 2 * 3
  answer = tennis_balls + bought_balls
  ```
- ⭐ 이유: 개발자 마인드 / 변수 정의 강제 / 엄밀한 reasoning.

### 11.2 ⭐ Reasoning 언어 자체에 대한 연구
- ⭐ CoT가 ⭐ **꼭 영어일 필요 없음** — 중국말 / 벡터 / 코드.
- ⭐ ⭐ "사람이 보기에 맞아야 한다고 누가 정했나" — 모델에게 좋으면 됨.
- 중국 모델은 ⭐ 중국말로 CoT 하는 경우 多.

### 11.3 ⭐ 코드 CoT의 우위
- 변수 / 빠진 정보 확인 / 엄밀한 reasoning.
- ⭐ 일종의 ⭐ "상상 컴파일".

## 12. ⭐⭐ Theorem Proving — Lean / Coq

### 12.1 ⭐ 동기
- ⭐ ⭐ 수학 증명은 보통 ⭐ **자연어로 작성** (논문, 책).
- ⭐ ⭐ Terence Tao (필즈상): "이렇게 할 경우 ⭐ **검증에 몇 달**".
- ⭐ P=NP 같은 증명 → ⭐ ⭐ 커뮤니티가 토론 후 ⭐ **검증 실패 결론** 내리기도.

### 12.2 ⭐⭐ Lean / Coq의 컨셉
- ⭐ ⭐ ⭐ 증명을 ⭐ **자연어가 아닌 코드로**.
- ⭐ 작성은 더 어렵지만 ⭐ **검증은 기계가 한 방에**.
- ⭐ ⭐ ⭐ ⭐ **AI + 인간 협업의 새로운 모델**:
  - GitHub처럼 ⭐ **proof들을 오픈소스로 공유**.
  - P=NP를 ⭐ **100명이 조각조각 분담**.
  - 조립 완성 후 ⭐ **돌려보면 검증 완료**.

### 12.3 ⭐ AutoFormalization
- ⭐ ⭐ 자연어로 이미 작성된 proof → ⭐ **자동으로 Lean/Coq 코드로**.
- ⭐ 사람들 몇 개월 검증 작업 단축.

### 12.4 ⭐ 강사 메시지
- ⭐ ⭐ ⭐ "코드 자체가 중요하다기보다 ⭐ **검증 가능한 추론**이 핵심".
- ⭐ ⭐ "AI의 추론도 인간의 추론도 개선될 수 있는 흐름".

## 14. ⭐⭐ NL → SQL (Text-to-SQL)

### 14.1 ⭐ 동기
- ⭐ ⭐ 회사 대시보드: "4월 5일부터 왜 하향세?"
- ⭐ 답은 문서 X / ⭐ **데이터 테이블에 있음**.
- ⭐ ⭐ SQL = 데이터베이스가 알아듣는 언어.
- ⭐ ⭐ ⭐ **목표: 자연어 질문 → SQL → 데이터 fetch → 답**.

### 14.2 ⭐⭐ SQL 핵심 문법
| 키워드 | 역할 |
|--------|------|
| **FROM** | 어떤 테이블 |
| **SELECT** | 어떤 컬럼 가져올지 |
| **WHERE** | row 조건 |
| **DISTINCT** | 중복 제거 (set) |
| **GROUP BY** | 기준 컬럼으로 그룹핑 |
| **COUNT / SUM / AVG** | 그룹 대표 통계 |
| **JOIN** | 두 테이블 합치기 |

### 14.3 ⭐⭐ Declarative Language
- ⭐ ⭐ "나는 이걸 원해"라고 말만 하면 ⭐ **DB가 알아서 처리**.
- ⭐ Procedural과 달리 ⭐ **순서를 명시하지 않음**.
- ⭐ "원해라는 것만 번역하면 되니까 ⭐ **일반 코드보다 쉬운 면도 있음**".

### 14.4 ⭐ 예시 — 기본 SELECT
```sql
SELECT url FROM keyword WHERE word = 'Paris'
```
- ⭐ Paris 단어 들어있는 페이지의 url 가져오기.

### 14.5 ⭐ COUNT / SUM 예시
```sql
SELECT COUNT(*) FROM keyword WHERE word = 'Paris'  -- 3개
SELECT SUM(freq) FROM keyword WHERE word = 'Paris' -- 5+1+2 = 8
```

### 14.6 ⭐⭐ GROUP BY 예시
```sql
SELECT url, COUNT(*) FROM hyperlink GROUP BY url
```
- ⭐ ⭐ ⭐ "팀 그러면 팀끼리 모여 앉음" 비유:
  - 같은 url끼리 모임.
  - 각 그룹에 대해 ⭐ **하나의 대표 값**.
  - 평균 나이 / 평균 근속연수 / 인원 수 등.

### 14.7 ⭐⭐⭐ JOIN — 가장 어려운 부분
- ⭐ 데이터베이스는 ⭐ **정보를 일부러 쪼개서 저장**.
- 강사 비유: ⭐ ⭐ ⭐ "황승원이 CS과 교수 / CS과는 300동":
  - 한 데 모으면 답하기 편하지만.
  - ⭐ ⭐ CS과가 ⭐ **400동으로 이사 가면** → 모든 교수 페이지 일일이 수정.
  - ⭐ 따로 저장 → ⭐ **여기 한 곳만 고치면 됨**.
- ⭐ ⭐ 답하려면 ⭐ **여러 테이블 JOIN** 필요 → ⭐ **multi-hop 추론**.

### 14.8 ⭐⭐ Cartesian Product (Cross Join)
- ⭐ ⭐ 두 테이블 JOIN시 ⭐ **조건 없으면 모든 쌍 생성**.
- 강사 비유: ⭐ ⭐ ⭐ "소개팅 — 모든 페어가 1:1 데이트 해야 됨".
- ⭐ ⭐ 보통 ⭐ **WHERE 조건** 줘서 필터링 (같은 전공 끼리만 등).

### 14.9 ⭐ JOIN 예시
```sql
SELECT hyperlink.url FROM hyperlink, keyword
WHERE keyword.word = 'Bogota' AND hyperlink.link = keyword.url
```
- ⭐ Bogota 단어 있는 페이지로 링크하는 페이지 url 찾기.
- ⭐ 같은 컬럼 이름 (`url`) 충돌 방지 → ⭐ **테이블명.컬럼명**으로 명시.

### 14.10 ⭐ NL → SQL의 미래
- ⭐ ⭐ ⭐ 회사 / 산업 데이터베이스 질의 자동화.
- ⭐ ⭐ Multi-turn (질문 → 답 → 다음 질문 → ...) 으로 ⭐ **대화형 분석**.
- ⭐ RAG와 결합 — 검색 + 데이터베이스 통합 답변.
