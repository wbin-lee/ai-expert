> 강의 주제: **텍스트로부터 KG 만들기 + KG의 모든 활용 + 다국어 NLP의 패러다임** — 황승원 교수님 + 조교진.

## 1. ⭐⭐ 텍스트 → Knowledge Graph 파이프라인 (실습)

### 1.1 ⭐ 큰 그림
- 줄글로 주어진 시퀀스 → 구조화된 데이터 → ⭐ **그래프**.
- 어제까지 빌드업한 모든 것 (POS / NER / Relation Extraction)을 합쳐서 ⭐ **end-to-end 파이프라인** 구현.
- ⭐ **GPT-4 mini를 블랙박스 LLM**으로 활용 (구조화된 output 받아서).

### 1.2 ⭐ Chunking (청킹)
- 긴 문서는 LLM에 한 번에 못 넣고, 들어가도 ⭐ **Lost in the Middle** 발생.
- ⭐ Substring으로 자르고 약간의 ⭐ **Overlap** 둠 → 의미 단절 방지.
- 200 토큰 chunk + 일부 overlap.
- ⭐ **검색 가치 측면**: 한 chunk가 일관성 있게 한 주제 다뤄야 retrieve 의미 있음.

### 1.3 ⭐⭐ Triple Extraction — NER + RE 동시 수행
- ⭐ "트리플"이라는 단어는 강의에서 처음 등장 → ⭐ **두 entity + 그 사이 relation = 3 elements**.
- ⭐ **이전에는 NER 모듈, RE 모듈 따로 정의 / HMM도 따로 짜야** 했는데:
  - ⭐ **LLM 블랙박스 + structured output**으로 ⭐ **두 함수를 하나로 합침**.
- 프롬프트로 정의된 함수의 강력한 예시.
- 184개 트리플 추출 (Bill Gates → Microsoft co-founder 등).

### 1.4 ⭐ Open-set Information Extraction
- ⭐ Relation 종류를 ⭐ **미리 정해주지 않음** → LLM이 알아서 판단.
- 예: `co-founded`, `founded in`, `became billionaire in` 등 자유 생성.
- ⭐ ⭐ ⭐ **노멀라이제이션이 trivial하지 않다는 게 본질적 한계**:
  - "co-founded" vs "founded together" 같은 동의 표현.
  - Variant 너무 많음.
- 작은 코퍼스라 일단 OK, 방대한 스케일에서는 ⭐ **클러스터링 + centroid 표준 이름** 같은 방법 필요.

### 1.5 ⭐ Surface Form Normalization
- "Bill Gates" / "William Henry Gates III" / "William Gates" — 같은 사람.
- ⭐ 공백 제거, apostrophe 제거, ⭐ **alias dictionary** 등 간단 처리.
- 프로덕션에서는 임베딩 클러스터링 + 표준 이름 결정 필요.

## 2. ⭐ NetworkX 기반 그래프 시각화

### 2.1 ⭐ 트리플 = 그래프
- ⭐ 엔티티 = **노드**, 릴레이션 = **엣지**.
- ⭐ 트리플 모으면 그래프 — 우리는 이미 그래프 만든 셈.

### 2.2 ⭐ Degree (차수)
- ⭐ **한 노드에 연결된 엣지 개수** = degree.
- ⭐ Degree 높은 사람 = "인싸" — 일론 머스크가 빌 게이츠를 살짝 능가.

### 2.3 ⭐ Multi-hop 경로 추적
- ⭐ 한 다리 관계 = Relation Extraction 본연.
- ⭐ **여러 다리 = Multi-hop**.
- 예시 ⭐ ⭐: 팀쿡 → (CEO of) → 애플 → (founded by) → 스티브 잡스 → (founded) → NeXT → 같은 사람 → (acquired) → 픽사.
  - ⭐ "사돈의 팔촌" 식 관계 추적.
- ⭐ 줄글로 보면 팀쿡과 픽사 같이 언급되는 paragraph 찾기 어려움 → ⭐ **그래프 노드/엣지 따라가면 가능**.

### 2.4 한계 — Surface Form 미정합
- "Jeff Bezos"와 "Jeffrey Preston Bezos"가 ⭐ **다른 노드로 분리**.
- → 임베딩으로 비슷함을 정량화하고 싶음.

## 3. ⭐⭐ TransE — Knowledge Graph Embedding

### 3.1 ⭐ 동기
- 심볼은 binary match (스트링 같으면 같은 것 / 다르면 다른 것).
- ⭐ **비슷한 정도를 숫자로 표현**하고 싶음 → ⭐ **벡터(임베딩)**.
- 트리플 (H, R, T)도 임베딩 가능.

### 3.2 ⭐⭐ 핵심 아이디어
- ⭐ **head + relation ≈ tail**.
- ⭐ Word2Vec의 "king - man + woman = queen"과 같은 발상.
- 장점:
  - ⭐ tail 빠진 (H, R, ?) → 예측 가능.
  - ⭐ relation 빠진 (H, ?, T) → tail - head로 추정.
  - ⭐ analogy: "Apple : Steve Jobs = Microsoft : ?" — 유사도 기반.

### 3.3 ⭐ Score Function & Margin Loss
```
score(h, r, t) = -|| h + r - t ||
loss = max(0, margin - score(positive) + score(negative))
```
- ⭐ Score는 (h+r)이 t와 가까울수록 높음.
- ⭐ ⭐ **Margin Loss**: positive와 negative의 score 차이가 ⭐ **margin만큼 벌어지길 원함**.
- ⭐ Negative는 그냥 ⭐ **랜덤하게 엉뚱한 entity** 골라서 만듦.

### 3.4 ⭐ 학습 결과
- 200 epoch, margin loss 가파르게 0으로 수렴.
- ⭐ 2D PCA 시각화:
  - "Jeff Bezos" ↔ "Jeffrey Preston Bezos" — surface form 다른데 ⭐ **같은 공간**.
  - "Larry Page" ↔ "Sergey Brin" + Google — 셋이 가까운 위치.
  - Bill Gates ↔ William Henry Gates + Microsoft.

### 3.5 ⭐ Tail Prediction 응용
- (Elon Musk, CEO_of, ?) → Tesla, SpaceX, Neural Link 순으로 나옴.
- ⭐ **score function** 이용해서 모든 후보 vector와의 거리 계산 → top-K.

## 4. ⭐⭐ Graph RAG

### 4.1 ⭐ 그래프가 주는 새로운 자료
- 노드 degree.
- ⭐ **Eigenvector Centrality** — 노드의 영향력 (네트워크 분석은 한 학기 수업).
- ⭐ **Community Detection** — 그래프를 의미 있는 덩어리로 쪼갬.

### 4.2 ⭐ Community Detection (Louvain Algorithm)
- ⭐ 어떤 노드가 어떤 노드와 ⭐ **얼마나 연결되어 있느냐 기준**으로 덩어리 형성.
- 결과:
  - "Google People" 클러스터 (Sergey Brin + Larry Page + Sundar Pichai + 알파벳).
  - "Microsoft 영역" (사티아 나델라 + 빌 게이츠 + 관련인).
  - "Apple 영역".
  - 일론 머스크는 ⭐ **혼자 빛나는 태양처럼** 둥둥.

### 4.3 ⭐⭐ Community Summary — Verbalize
- ⭐ 노드와 엣지를 ⭐ **자연어로 verbalize** → LLM에 줘서 짧은 요약.
- Verbalize = **말로 표현** (verb = 동사).
- ⭐ 구조화된 데이터 → 텍스트 → LLM에 전달 가능한 형태.

### 4.4 ⭐ Vanilla RAG vs Graph RAG
| 방식 | 특징 |
|------|------|
| **Vanilla RAG** | 청크 검색 → 그대로 LLM에 |
| **Graph RAG (Local)** | ⭐ **하나의 커뮤니티** 서머리만 사용 |
| **Graph RAG (Global)** | ⭐ **모든 커뮤니티** 별도로 답 → 합치기 (Map-Reduce) |

### 4.5 ⭐⭐ Map-Reduce 패턴
- 각 커뮤니티에 대해 독립적으로 query 수행.
- ⭐ **분산 처리 가능** (서로 다른 서버 병렬).
- 마지막에 partial answer들 합쳐서 최종 답.

### 4.6 ⭐ Graph RAG의 차별화 — 매우 중요
- ⭐ ⭐ ⭐ **일반 RAG로 도저히 할 수 없는 것**:
  - 컨텍스트 윈도우에 안 들어가는 ⭐ **전체 코퍼스에 대한 질문**.
  - 어찌 우겨넣어도 ⭐ **Lost in the Middle** 발생.
- ⭐ Graph RAG는 ⭐ **분산 질의로 광범위 처리 가능**.
- 예: "이 25GB Wikipedia에서 인물 공통점은?" — Graph RAG만 가능.

### 4.7 ⭐ 비교 결과
- **Local 답**: 트리플 그래프에 ⭐ **그라운드된 건조한 답**.
- **Global 답**: AI / 페이지 랭크 등 ⭐ **다른 커뮤니티 정보 포함**.
- **Vanilla 답**: Paul Allen 등장 — ⭐ **그래프엔 없는 LLM 자체 지식**으로 답한 것 (hallucination 가능성).

### 4.8 ⭐ 실무 적용
- ⭐ "5년치 정치면 뉴스 → 정치인 관계도" 같은 작업 자동화.
- 의약 분야: 아스피린 + 같이 먹으면 안 되는 성분 + 그 성분 포함 음식 → ⭐ **multi-hop 인과**.
- ⭐ 한 문단에 모든 인과가 등장하지 않을 때 그래프 RAG의 위력.

### 4.9 ⭐ 한계 — 희망편이라는 것
- 코퍼스 사이즈가 작고 GPT-4 mini라는 ⭐ **치트키** 사용한 결과.
- 인터넷 전체에 돌리는 건 곤란.
- ⭐ **KG 퀄리티가 받쳐줘야** — Triple 몇 개 이상하게 들어가면 multi-hop 자체가 왜곡.

## 5. ⭐⭐ Multilingual NLP — 왜 중요한가

### 5.1 ⭐ High vs Low Resource Languages
- ⭐ 영어 = 대표적 **High Resource**.
- 사라져가는 아프리카 부족 언어 = **Low Resource**.
- ⭐ 한국어는 ⭐ **Medium Resource** — 논문 내면 리뷰어들이 "너네 low resource 아님" 함.

### 5.2 ⭐ 번역 데이터의 본질적 희소성
- ⭐ ⭐ 번역 annotation은 ⭐ **두 언어 모두 native 수준**인 사람이 필요.
- 외국어 대학 통번역 대학원이 그래서 존재.
- ⭐ Low-resource 일수록 ⭐ **bilingual annotator도 함께 부족** → ⭐ **격차 가속화**.

### 5.3 ⭐ 95% 위협론
- ⭐ 95%의 언어는 LM을 통한 사용이 ⭐ **거의 안 될 것**.
- 정보 격차 체감: 지도 앱 — 영어/한글로 켜면 ⭐ **레스토랑 이름 / 리뷰 안 나옴**.

### 5.4 ⭐ 강의자 본인 페이퍼 — 식당 이름 번역
- ⭐ "낙성대 맛집" 같은 ⭐ **테일 entity**:
  - 이름 자체 번역도 어려움.
  - "Let it be"를 "그냥 놔둬"로 번역할지 그대로 둘지.
- 고유명사 번역엔 ⭐ **소셜 레터클** 봐야 → 어려움.

### 5.5 ⭐⭐ 그래프가 멀티링구얼에도 도움
- ⭐ N-gram = window 안만 / RNN, CNN = 한 문장 / ⭐ **그래프 = 한 단계 더 멀리**.
- ⭐ "황승원과 서울대학교가 한 문장 안에 한 번도 등장하지 않아도" 그래프로 관계 파악.
- ⭐ 다른 SNU 교수님과 나도 같은 직장이라는 관계 추론 가능 → ⭐ **K-hop 추론으로 확장**.
- 테일 entity의 번역도 그래프로 모이면 가능.

### 5.6 ⭐ 문화 차이
- 한국의 존댓말 / 문화적 금기 — ⭐ **번역에 잘 반영 안 됨**.
- 이런 거 연구하시는 분 多.

## 6. ⭐⭐ Cross-Lingual vs Multilingual

### 6.1 ⭐ 시험 출제 포인트 — 두 패러다임 차이
| 패러다임 | 특징 |
|----------|------|
| **Cross-Lingual** | ⭐ **주종 관계** — 영어 모델이 핵심, 한국어를 영어로 인코드 → 영어 모델 통과 → 한글 디코드 |
| **Multilingual** | ⭐ **양쪽 다 잘함** — 한국어/영어 둘 다 처리 가능한 모델 |

### 6.2 ⭐ Cross-Lingual의 장점
- 영어 모델이 강력한 task가 있을 때 ⭐ **언어 장벽을 건너서** 활용.
- 모든 task (QA, 리뷰 분석, NER 등)에 시도 가능.

## 7. ⭐⭐ Cross-Lingual Embedding — 별자리 비유

### 7.1 ⭐⭐ 시험 출제 포인트 — 별자리 비유
- ⭐ ⭐ ⭐ **영어 임베딩(초록색) / 한국어 임베딩(보라색) — 둘은 서로 다른 공간**.
- ⭐ 별자리 비유:
  - 큰곰자리 / 작은곰자리.
  - 오스트리아 남반구에서 보는 별자리는 다르지만.
  - ⭐ **수학적 트랜스포메이션 (회전·대칭)으로 일치시킬 수 있음**.

### 7.2 ⭐ 핵심 아이디어
- ⭐ ⭐ **same meaning words → same position** 되도록 트랜스포메이션 함수 학습.
- ⭐ 별자리 비유의 수학적 의미 = ⭐ **Linear Transformation Matrix W**.
- 한국어로 말한 것 → 한국어 벡터 → ⭐ **W 곱하면 영어 벡터** → 영어 모델로 처리 → 결과 다시 W^-1으로 한글로.

### 7.3 ⭐ 두 문장 요약 — 강사 어록
1. ⭐ ⭐ 별자리 맞추는 거가 **꽤 잘 된다**.
2. ⭐ ⭐ **양 언어 잘하는 사람 (슈퍼비전) 없어도 잘 될 때가 있다**.

### 7.4 ⭐⭐ 멀티모달도 같은 아이디어
- 영어 임베딩 ↔ 프랑스어 임베딩 align = 번역기.
- ⭐ **영어 임베딩 ↔ 이미지 임베딩 align** = 이미지/텍스트 검색.
- ⭐ "결국 representation alignment 문제".

## 8. ⭐⭐ BLI — Bilingual Lexicon Induction

### 8.1 ⭐ 컨셉
- ⭐ "두 언어 사이의 ⭐ **어휘 사전을 추론**" 한다.
- Source 단어 → 벡터 공간에서 가장 가까운 Target 단어.
- ⭐ ⭐ ⭐ **사전이 공짜로 생긴다는 의미**.

### 8.2 ⭐ 평가 — Precision@K
- **Precision@1**: cat에서 가장 가까운 게 진짜 "고양이"인가? 빡센 것.
- **Precision@5**: 가까운 5개 중 "고양이"가 있나?
- ⭐ **MRR (Mean Reciprocal Rank)**: 정답 문서의 rank 역수.

### 8.3 ⭐ MUSE 프로젝트 (Facebook → Meta)
- ⭐ **Multilingual Unsupervised And Supervised Embeddings**.
- 30개 언어에 대해 ⭐ **같은 임베딩 공간 매핑**.
- 우리는 영어 / 스페인어 / 핀란드어 사용.

### 8.4 ⭐ 실습 결과
- 100개 단어 쌍 중 25개 맞춤 (영-스페인 only).
- ⭐ ⭐ **30개 언어 다 고려해 만든 거라 두 언어만 보면 P@1 낮음** — alignment 필요.

## 9. ⭐⭐ Orthogonal Procrustes — 회전·대칭으로 별자리 맞추기

### 9.1 ⭐ Procrustes 신화
- 그리스 신화의 ⭐ **도적 프로크루스테스**.
- 자기 침대에 사람을 눕혀서 ⭐ **큰 사람은 잘라내고 / 작은 사람은 늘림**.
- 영미권 표현 "Procrustes 침대" = 답정너.

### 9.2 ⭐⭐ 수학적 정의
```
min ||AΩ - B||_F  subject to Ω^T Ω = I
```
- ⭐ A = source 임베딩 매트릭스, B = target.
- ⭐ Ω = ⭐ **Orthogonal matrix** (회전 또는 대칭만).
- ⭐ Frobenius Norm — 행렬 모든 원소 제곱합 루트.

### 9.3 ⭐⭐ Orthogonal 제약의 의미
- Ω^T Ω = I 조건 = ⭐ **회전 또는 대칭 변환만 허용**.
- ⭐ ⭐ ⭐ 보존되는 것:
  - **벡터 크기**.
  - **두 벡터의 거리**.
  - **두 벡터의 각도**.
- ⭐ 즉 **별자리 안에서의 위치 관계 훼손 X**.
- 프로크루스테스보다 젠틀 — 자르거나 늘리지 않음.

### 9.4 ⭐ 두 가지 모티베이션 (시험 출제 포인트)
1. ⭐ **Source의 vector → Target과 비슷해지게** (cross-lingual alignment).
2. ⭐ **원래 source 안에서의 일관성 유지** (intra-language relation 보존).
- → Orthogonal 제약이 두 가지를 동시에 충족.

### 9.5 ⭐ SVD로 풀기
- ⭐ Procrustes 문제는 ⭐ **Singular Value Decomposition으로 닫힌 형태로 풀림**.
- Gradient descent 같은 근사 X → ⭐ **행렬 연산으로 한 방에**.

## 10. ⭐ Seed-based Bootstrapping (Less is More)

### 10.1 ⭐ 컨셉
- 작은 영-한 사전 (cat-고양이, dog-개)으로 시작.
- ⭐ 그 두 점이 일치하게 회전.
- 회전했더니 university-대학교 자연 발견 → 사전에 추가.
- ⭐ 다시 회전 → 더 많은 페어 발견 → ⭐ **Snowball식**.

### 10.2 ⭐ 어제 본 Snowball / Bootstrapping과 동일
- ⭐ Self-Learning / 패턴↔트리플 mutual recursion.
- ⭐ Seed가 너무 많으면 오히려 안 좋음 — ⭐ **노이즈 포함될 수 있음**.

### 10.3 ⭐ 실습 발견
- 사이즈 2000 → 5000으로 늘리면 P@1 ↑.
- 5000 → 6000은 오히려 ↓ — ⭐ **인베딩 품질 낮은 빈도 낮은 단어가 노이즈**.

## 11. ⭐⭐ Unsupervised — Seed = 0도 가능

### 11.1 ⭐ Isomorphism Assumption
- ⭐ ⭐ ⭐ **두 언어 임베딩 공간이 ⭐ approximately isomorphic** 하다는 가정.
- ⭐ 한국어와 영어 화자가 ⭐ **같은 태양계(세계)를 보고** 있으니까 별자리 비슷.
- 개, 고양이, 말 같은 단어는 어느 언어에도 있고 그 관계가 비슷.

### 11.2 ⭐ 자연 시드의 존재
- ⭐ 강사가 페이퍼 읽고 깨달은 점:
  - ⭐ ⭐ **아라비아 숫자 1, 2, 3** — 어느 나라나 동일.
  - ⭐ ⭐ **글로벌라이제이션의 부산물** — "Microsoft" 같은 단어가 한글에도 등장.
  - → ⭐ "공기 중에 시드가 있다".

### 11.3 ⭐⭐ 어족이 가까울수록 잘 됨
- ⭐ **어떤 페어에 대해서는 unsupervised가 supervised 뺨치게 잘함**.
- ⭐ 스페인어와 카탈란 (사실상 거의 같은 언어).
- 스페인어 ↔ 포르투갈어 — 매우 가까움.
- ⭐ 한국어와 일본어 — 어순 / 단어 유래 비슷.
- ⭐ 한국어 ↔ 터키어 같은 멀리 있는 페어는 어려움.

### 11.4 ⭐ 한국어 → 일본어가 ⭐ ⭐ 슈퍼비전이 공짜로 많은 이유
- ⭐ ⭐ ⭐ 수많은 ⭐ **애니덕후들의 자막** — 자발적으로 캡션 단 것들.

### 11.5 ⭐ Pivot Language
- 한국어 → 프랑스어 직접 번역보다 ⭐ **한국어 → 일본어 → 프랑스어**가 더 잘 될 수 있음.
- 두 언어 가깝거나 리소스 풍부한 중간 언어 활용.
- ⭐ ⭐ "전청조의 언어처럼 한·영 섞어 쓰는 ⭐ **기괴한 코드 스위칭**도 ⭐ pivot 언어로 활용 가능".

## 12. ⭐ 실습 — MUSE 임베딩 + Procrustes

### 12.1 단어 레벨 cosine similarity
- 같은 언어 안: king-queen 가까움 / dog-computer 멀음.

### 12.2 ⭐ 2D PCA로 시각화
- 영어와 스페인어 단어 쌍이 ⭐ **벌써 어느 정도 align**되어 있는 것 확인.
- Dog ↔ perro / house ↔ casa.

### 12.3 ⭐⭐ Procrustes 적용
- Linear algebra 라이브러리로 ⭐ **alignment matrix 한 방에 계산**.
- 적용 후 P@1 향상 확인.

### 12.4 ⭐ 핀란드어 — 어족 차이 검증
- 핀란드어는 영어와 어족이 다름.
- ⭐ 같은 30개 언어 임베딩에서 영-핀란드 P@1이 영-스페인보다 낮게 나옴.
- ⭐ "구조적 유사성이 있는 언어는 alignment 잘 됨".

## 13. ⭐ LLM이 이미 암묵적으로 하고 있는 것

### 13.1 컨셉
- ⭐ ⭐ Claude / ChatGPT는 ⭐ **따로 행렬 곱하지 않아도** 다국어 잘함.
- 이유: ⭐ **방대한 코퍼스 학습으로 임베딩 레이어가 자동 mapping**.
- 모든 언어 토큰 (한자/한글/알파벳)이 유니코드 / 바이트 단위.
- ⭐ 임베딩 레이어 거치고 나면 ⭐ **서로 다른 언어라도 비슷한 곳에 모임**.

### 13.2 ⭐ 한계
- ⭐ 자주 나오는 언어 vs 안 자주 나오는 언어 차이.
- ⭐ ⭐ **에이전트도 한국어 vs 영어 시킬 때 성능 다름**.
- 커서 같은 툴도 ⭐ **한국어로 시키면 tool calling 안 한다거나** 발생.

### 13.3 ⭐⭐ 본질은 같음
- ⭐ "결국 우리가 하는 것 = 행렬 곱하기".
- ⭐ Procrustes도 행렬 / 임베딩 레이어도 행렬.
- ⭐ ⭐ **암묵적으로 같은 일을 LLM이 하는 중**.

### 14.4 ⭐ 강사 어록
- "NER 모듈, RE 모듈 따로 짜야 했는데 LLM 블랙박스로 두 함수를 하나로."
- "코파운디드와 파운디드 투게더 같은 variant 노멀라이제이션이 trivial하지 않다."
- "팀쿡과 픽사가 같이 등장하는 패러그래프를 찾기 어려운데, 그래프로는 노드 따라가면 된다."
- "TransE의 발상은 head + relation ≈ tail — Word2Vec king-man+woman=queen과 같은 원리."
- "일론 머스크는 혼자 빛나는 태양처럼 거대한 노드들을 거느리고 둥둥 떠 있다."
- "Graph RAG로는 25GB Wikipedia 전체에 대한 인물 공통점 질문이 가능 — 일반 RAG로는 절대 불가."
- "Vanilla RAG는 Lost in the Middle 때문에 컨텍스트 우겨넣어도 답이 안 나온다."
- "지금은 희망편이다 — 코퍼스 작고 GPT-4 mini라는 치트키 썼기 때문에."
- "황승원과 서울대학교가 한 문장 안에 한 번도 등장하지 않더라도 그래프 K-hop으로 관계가 표현된다."
- "Cross-lingual은 주종 관계 — 영어 모델을 통해서 한국어 문제를 푼다."
- "별자리만 맞출 수 있다면 한국어 → 한국어 벡터 → W로 회전 → 영어 벡터 → 영어 모델 → 결과 → W^-1로 한글 — too good to be true 아닌가요?"
- "양 언어 잘하는 사람 없이도 잘 되더라 — 두 문장 요약입니다."
- "프로크루스테스는 답정너 — 자르고 늘리고. 우리는 회전·대칭만 허용해서 별자리 안 망가뜨림."
- "정답을 몰라도 두 별자리가 같은 태양계를 보고 있으면 맞출 수 있다."
- "공기 중에 시드가 있다 — 1, 2, 3 / Microsoft 같은 단어들이 자연 시드 역할."
- "한↔일이 잘 되는 이유 중 하나가 애니덕후들의 자막 — 슈퍼비전이 공짜로 많이 있다."
- "전청조의 언어처럼 기괴한 언어도 pivot으로 활용할 수 있다."
- "Claude / ChatGPT가 따로 행렬 곱하지 않아도 다국어를 잘하는 건 임베딩 레이어가 자동으로 alignment 하고 있기 때문 — 결국 같은 일을 암묵적으로 하고 있다."
- "한국어로 시키면 tool calling 안 한다거나 — LLM의 다국어 한계가 에이전트에서도 드러난다."
