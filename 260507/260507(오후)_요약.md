## 1. ⭐⭐ Viterbi 알고리즘 구현

### 1.1 ⭐ 알고리즘의 취지 리캡
- 주어진 단어 시퀀스에 대해 **확률이 가장 높은 hidden state(품사) 시퀀스**를 찾는 것.
- 전수조사는 exponential → ⭐ **Markov property** 덕분에 한 단계 이전 결과만 활용해서 줄임.

### 1.2 ⭐ 핵심 점화식
```
V(j, s) = max_{s'} [ V(j-1, s') × A(s', s) ] × B(s, w_j)
```
- `V(j, s)`: 포지션 j에서 태그 s로 끝나는 최대 확률.
- `A(s', s)`: ⭐ **Transition probability** — 이전 태그 s' → 현재 태그 s.
- `B(s, w_j)`: ⭐ **Emission probability** (=Output probability) — 태그 s일 때 단어 w_j가 나올 확률.
- 모든 가능한 s'를 다 넣어보면서 max를 구함.

### 1.3 ⭐ 초기화 (Initialization)
- 맨 처음 단계에는 ⭐ **start라는 특수 태그**가 있고, 거기서 첫 단어가 나오는 조건부 확률을 계산.
- 첫 단계는 backpointer가 None (이전 선택이 없음).
- 로그 확률로 계산 → 곱셈이 덧셈으로 바뀜.

### 1.4 ⭐ Recursion 구현
- 가능한 모든 previous 태그를 for문으로 순회.
- `V[i-1][prev_tag] + log(A[prev_tag][cur_tag]) + log(B[cur_tag][w_i])`를 계산.
- 기존 best보다 크면 갱신, backpointer에 `prev_tag` 저장.

### 1.5 ⭐ Backpointer로 시퀀스 복원
- 모든 포지션을 다 채운 후 **맨 오른쪽에서 시작**.
- backpointer 따라가며 ⭐ **꼬리 밟기 식으로 best path 복원**.
- `best_path.insert(0, prev_tag)`로 앞쪽으로 붙임.

### 1.6 ⭐ 결과 — 92.31% Accuracy
- 로그를 씌우니까 음수 값으로 나옴.
- determiner → adjective → noun 같은 시퀀스 출력.
- ⭐ **앞 태그 하나만 봐도 92.31%**. 앞 태그 2개를 보면 (higher order Markov) 정확도 더 ↑.

## 2. ⭐ Greedy Decoding

### 2.1 컨셉
- 다른 경우는 고려하지 않고 **그 다음에 가능한 선택지 중 가장 확률 높은 것만** 따라감.
- 테이블 사용 X — 12개 정도 가능한 태그만 돌면서 transition × emission 계산.
- ⭐ **Optimal 보장은 안 됨, but 빠름.**

### 2.2 ⭐ 비유 — "돌에 새기기"
- 한 번 결정하면 되돌리지 않음.
- Viterbi와 항상 같은 결과를 보장하지 않음.

### 2.3 시간
- 0.319 밀리초 — 매우 빠름.
- Viterbi 대비 ~6배 빠름.

## 3. ⭐ Beam Search

### 3.1 컨셉
- Greedy의 확장 — ⭐ **K개의 경로를 유지**하면서 진행.
- 한 스텝마다 각 빔에서 펼치고, 펼친 candidate 중 ⭐ **상위 K개만 보존** (정렬 후 자르기).
- 맨 마지막에 최종 score 가장 높은 거 선택.

### 3.2 ⭐ K=1이면 Greedy
- Greedy는 Beam Search의 **K=1 특수 케이스**.
- K가 커질수록 Viterbi에 가까워짐.

### 3.3 구현 핵심
```python
beam = [(log_prob, tag_sequence, last_tag), ...]
# 각 스텝마다 candidates 만들고
beam = sorted(candidates, key=lambda x: x[0], reverse=True)[:beam_width]
```
- 람다 `key=lambda x: x[0]`: 튜플의 첫 번째(log probability) 기준 정렬.

### 3.4 ⭐ 시험 출제 포인트 — LLM 디코딩에도 동일 적용
- ⭐ LLM에서 조건부 확률로 단어들을 뽑는 게 **Greedy decoding**.
- ⭐ 당장 확률이 높지만 **이후 말문이 막히는 경우** → Beam Search가 더 좋을 수 있음.
- ⭐ **에이전틱 RAG**에서 디코딩 방식이 매우 중요 → Greedy / Beam 익혀두면 큰 도움.
- 빔서치는 LLM에서 그대로 쓰는 알고리즘 → ⭐ "이미 익숙해도 코드로 써보면 감회가 새롭다."

### 3.5 비용 vs 정확성 (비례)
| 방법 | 시간 | 정확성 |
|------|------|--------|
| **Greedy** | 0.319ms (가장 빠름) | 낮음 |
| **Beam Search** | 중간 | K 클수록 ↑ |
| **Viterbi** | Greedy의 ~6배 | ⭐ Optimal |

### 3.6 ⭐ 주의 — Viterbi가 동작하는 이유
- ⭐ **태그 셋이 한정 (12개)** 이라서 DP가 현실적.
- 만약 태그가 vocabulary 크기 (2만 개)면 → ⭐ **Viterbi도 현실적으로 불가**.
- POS는 한정적이라 OK.

## 4. ⭐⭐ Relation Extraction — Knowledge Graph로의 다리

### 4.1 ⭐ 시험 출제 포인트 — Triple과 이진 관계 가정
- 텍스트에서 ⭐ **(Head, Relation, Tail) 트리플** 추출.
- 삼각관계, 사각관계도 ⭐ **이진 관계 여러 개로 분해** 가능 → "모든 관계는 이진".
- 예: 
  - IBM과 1911 → `(IBM, founding_year, 1911)`
  - IBM과 New York → `(IBM, location, New York)`
  - IBM의 원래 이름 → `(IBM, alias, Computing Tabulating Recording)`

### 4.2 ⭐ Wikification와 Knowledge Graph
- ⭐ **Wikification**: 텍스트의 엔티티를 Wikipedia URL로 link.
- ⭐ **Infobox**: Wikipedia 오른쪽의 박스. 각 엔티티에 대한 관계 테이블.
- ⭐ Infobox = "민주적으로 제작된 거대한 Relation Dictionary".
- ⭐ **DBpedia**: Infobox를 JSON 트리플로 변환한 데이터셋 (1B+ triples).

### 4.3 ⭐ Multi-hop QA — Knowledge Graph가 필요한 이유
- ⭐ "ET에 나왔던 배우의 손녀는 누구인가?" 같은 질문 → **세 개의 지식을 이어붙여야 함**.
- BM25 / Sparse / Dense로는 ⭐ **한 문서에 다 없으면 답 불가**.
- 트리플들을 미리 모아두면 ⭐ **각 트리플을 찾아 이어붙이기**로 답 가능.
- 옛날 IBM Watson 퀴즈쇼 / AI 스피커 / CD가 **단답형 트리플 기반**으로 동작했음.

## 5. ⭐ ACE 데이터셋 (Relation Extraction 1.0)

### 5.1 컨셉
- ⭐ 사람들이 모여서 **에노테이트한 정답지** 만듦.
- 사람-장소 (physical location), 사람-소속, 사람-회사 (founded), 사람-대학 (graduated), 회사 부모자식 관계, 사람-사람 friend 관계 등.

### 5.2 ⭐ 데이터 만들기 = 머신러닝 가능해짐
- ⭐ "정답지를 100배 더 만들면 그걸로 훈련 가능" → 데이터 → 모델.
- 의학 도메인: 의사들이 모여 약과 증상의 관계 (cure, prevent, cause, treat) 정의.
- 반도체 도메인: 어떤 물질이 어떤 걸 일으키는지 등 ⭐ **내가 뽑아내고 싶은 Relation을 먼저 정의**.

### 5.3 ⭐ Apple-Steve Jobs 예시
- 둘 사이 관계는 무궁무진 (founder / employee / hater / ...).
- ⭐ **사이에 낀 단어 "co-founder"** → founding relation.
- 단어가 어떤 개체를 수식하고 있는지도 단서.

## 6. ⭐⭐ Hearst Pattern (Rule-based, is-a 관계)

### 6.1 ⭐ 발견자 & 동기
- ⭐ **Marti Hearst** (UC Berkeley NLP 교수).
- ⭐ 사람들이 ⭐ **자기 머릿속 개념 상하관계를 말로 표현한다는 발견**.
- 줍줍하면 ⭐ **세상에 있는 Linnaeus 식 분류도를 공짜로 얻을 수 있음**.

### 6.2 ⭐ 주요 패턴
```
NP such as NP            → 뒤 NP가 상위, 앞 NP들이 하위
NP including NP          → 뒤가 상위
NP, especially NP        → 뒤가 상위
NP and other NP          → 뒤가 상위
NP or other NP           → 뒤가 상위
```

### 6.3 ⭐ 예시 — 처음 보는 단어도 알 수 있음
- "Korean Universities such as Seoul National University"
  - SNU가 Korean University의 일종이라는 걸 ⭐ **모르는 단어도 추론 가능**.
- ⭐ "Gelidium은 algae(잇기)의 일종" — 생물학 용어 몰라도 such as로 추론.
- "Samsung, Hynix 같은 반도체 회사들" → 삼성, 하이닉스가 반도체 회사 카테고리.

### 6.4 ⭐ 강사 본인 경험
- 마이크로소프트 연구소와 함께 ⭐ **Wikipedia / 웹 코퍼스에서 전체 개념도를 만듦**.
- 어떤 개체가 어떤 개념에 속하는지 ⭐ **확률 분포** 만들어 → 다양한 NLP task에 유용.

## 7. ⭐ Feature-based Supervised Relation Extraction (2.0)

### 7.1 컨셉
- Hearst Pattern은 is-a 관계만. **다양한 관계 (founder, location, works_for, ...)** 는 다른 방법 필요.
- ⭐ 사람이 annotation으로 ⭐ **각 테이블에 100개 정도 채워줌**.
- 머신은 ⭐ 본 적 없는 101번째 페어에서 ⭐ **클래시피케이션**.

### 7.2 ⭐ 시험 출제 포인트 — 중요 피처들
1. ⭐ **두 argument 사이 단어들** — "founded", "is located in" 같은 동사가 결정적.
2. ⭐ **엔티티를 둘러싼 단어들**.
3. ⭐ **Entity Type**: (Person, Organization) → 가능한 관계 100개 → 3-4개로 좁힘.
4. ⭐ **POS**: 엔티티는 명사구, 사이의 동사가 관계.
5. **Parse tree의 수식 관계** (있다면 더 좋음).
6. **대문자 / 호칭** — "campaign", "남자인 듯" 같은 추가 정보.

### 7.3 ⭐ Precision-Recall Trade-off
- ⭐ "is located in" 패턴만 쓰면 → ⭐ **High precision, Low recall** (동의어 표현 많음).
- "works for SNU" vs "is an employee of SNU" → 표현 무수히 많음 → 패턴만으로는 다 못 잡음.

### 7.4 ⭐ 한계
- ⭐ 트레이닝셋 많이 필요.
- 개체와 표현이 계속 ⭐ **새로 생겨남** → 1.0/2.0의 본질적 한계.

## 8. ⭐⭐ Semi-Supervised / Bootstrapping / Snowball

### 8.1 ⭐ Snowball — Columbia 박사학위 연구
- ⭐ 두 분 (지금 NLP 유명 교수) — 졸업 연구가 ⭐ Snowball 알고리즘.
- ⭐ **눈덩이 굴리기 비유**: 패턴 하나로 시작 → 굴려서 산더미만한 눈사람.

### 8.2 ⭐ 핵심 아이디어 — 패턴 ↔ 트리플 상호 강화 (Mutual Recursion)
```
Seed Pattern → 트리플 몇 개 추출
   ↓
새 트리플로 코퍼스 재검색 → 새 패턴 발견
   ↓
새 패턴으로 다시 더 많은 트리플 추출
   ↓
반복
```
- ⭐ "if A is true then B is true" — 마치 ⭐ **Mutual Recursion**.

### 8.3 ⭐ 예시 1 — Location
- Seed: `is located in` 패턴 1개만 주입.
- → (Seoul National Univ, Seoul), (Samsung, Suwon) 추출.
- → 다시 코퍼스에서 ⭐ "Samsung, Suwon" 같은 인 패턴 발견.
- → "연세대, 신촌" 추가됨.
- → 이걸로 다시 새 패턴 발견 → 반복.

### 8.4 ⭐ 예시 2 — Mark Twain의 묘지
- Seed: `Mark Twain is buried in Elmira, NY` (high precision).
- → 묘지 트리플 몇 개 추출.
- → 코퍼스에서 "the grave of X is in Y", "X's final resting place is Y" 같은 새 패턴 발견.
- → ⭐ 패턴이 ⭐ **양질의 패턴 리스트와 트리플 리스트로 동시에 완성**.

### 8.5 ⭐ 시험 출제 포인트 — 용어 정리
| 용어 | 의미 |
|------|------|
| **Bootstrapping** | 컴퓨터 부팅처럼 작은 코드(seed)로 시작해 점점 커짐 |
| **Distant Supervision** | 시드 몇 개만 주고 distantly 슈퍼바이즈 |
| **Semi-Supervised** | Supervised와 Unsupervised의 중간 |
| **Unsupervised (Open IE)** | ⭐ Seed가 0 — 시드 zero 케이스 |

### 8.6 ⭐ 비유 — "마을의 돌멩이 수프"
- 마을에 돌멩이 넣고 끓이기 시작 → 누군가 후추, 소금 넣고 → 결국 수프 완성.
- ⭐ "약간 너무 좋은 세상에서만 일어나는 일 같은데, Relation에서는 잘 됨".

### 8.7 ⭐⭐ 노이즈 폭주 위험 — 매우 중요
- ⭐ **한 번 이상한 패턴이 들어오면 걷잡을 수 없게 됨**.
- 예: 휴게실에서 쉬고 있던 사람을 ⭐ "묘지에 묻혀있다"로 잘못 분류 → 그 사이의 모든 패턴이 묘지 관계로 오염됨.
- ⭐ Recall이 올라가다가 ⭐ **품질이 급추락**.
- ⭐ Verify를 계속 하는 것이 ⭐ **굉장히 중요**.

### 8.8 ⭐ 실무 — Silver Annotation
- 사람이 한 100개 = Gold.
- LLM이 시드 보고 만든 1000개 = ⭐ **Silver Annotation**.
- ⭐ "사람의 수제 + 기계의 실버 합쳐서 평범한 방법론이 됨".

## 9. ⭐ Open Information Extraction (Unsupervised)

### 9.1 컨셉
- ⭐ **Seed = 0**. 어떤 페어들이 "co-founder"라는 개념으로 자꾸 등장하면 → 그 자체로 ⭐ **co-founder 테이블을 새로 만듦**.
- ⭐ "Closed (정해진 180개 relation) vs Open (자유롭게 발견)".

### 9.2 ⭐ 동기
- ⭐ 새로운 관계가 ⭐ **어제 발명될 수도 있음** (예: 강사가 삼성에서 강의하는 관계를 "카레"라고 부르기로 했다면).
- ⭐ AI2 / UDub의 초기 방법론 — 굉장히 야심찬 어프로치.

### 9.3 ⭐ 한계 — Precision/Recall 정의 불가
- 슈퍼비전이 없으니 ⭐ **정답이 무엇인지조차 모름**.
- "5천 개 나라 중 3천 개 찾았다 = recall 60%" 같은 평가 자체가 ⭐ **불가능**.
- → ⭐ 어떻게 approximation으로 평가할지가 연구 과제.

## 11. ⭐⭐ Knowledge Graph Embedding

### 11.1 ⭐ 컨셉 — TransE류
- 각 엔티티 / 관계를 벡터로 임베딩.
- ⭐ **head + relation ≈ tail** 이 되도록 학습.
- ⭐ Word2Vec의 ⭐ "king - man + woman = queen" 과 같은 원리.

### 11.2 ⭐ Positive / Negative Sampling
- ⭐ **Positive**: (France, capital, Paris) — 실제 KB에 있음.
- ⭐ **Negative**: (France, capital, Seoul) — 말도 안 되는 것.
- ⭐ Positive는 가깝게, Negative는 멀게 → Loss로 학습.

### 11.3 ⭐ Word2Vec과의 비교
- Word2Vec: ⭐ **윈도우 안 단어 의미** (좁은 의미적 이웃).
- KG Embedding: ⭐ **Relation 기반의 의미적 윈도우** (멀리 있어도 관계 있으면 이웃).
- ⭐ "윈도우의 개념이 다름" → ⭐ **합쳐 쓰면 도움**.

### 11.4 ⭐ BERT Fine-tuning for Relation
- ⭐ **CLS + entity1 + SEP + entity2** → classify into N relations.
- ⭐ Pre-trained BERT를 Relation Extraction용으로 fine-tuning.
- 매우 표준적인 방법.

## 12. ⭐ KG로 NLP 보강하는 3가지 방법

### 12.1 ⭐ Input/Embedding Level Concatenation
- 입력에 ⭐ **BERT/ELMo 임베딩 + KG 임베딩**을 concatenate.
- "황승원"이라는 input에 ⭐ **KG에서 관계를 맺는 정보 기반 임베딩**도 추가.
- → 성능 향상.

### 12.2 ⭐ Retrofitting / Embedding Debugging
- ⭐ Word2Vec 임베딩의 위치를 ⭐ **KG에 맞게 조정**.
- 예: "프랑스 수도는 파리"가 KB에 있는데 임베딩이 잘못되어 있으면 → post-hoc transform.
- ⭐ **편향 제거**도 같은 방식: "여자는 프로그래머가 될 수 없다" 같은 편향을 retrofitting으로 제거.

### 12.3 ⭐ Decoding 시 KG 참조
- Beam Search 디코딩 중 ⭐ **KG와 호환되는지 확인**, 부적합하면 확률 조정.

### 12.4 ⭐ 핵심 정리
- ⭐ **Symbolic Ontology** (KB) vs ⭐ **Parametric Knowledge** (LM) — 장단점 다름.
- ⭐ Input / Embedding / Decoding 모든 단계에서 합칠 수 있음.

## 13. ⭐ LM as KB — 논쟁

### 13.1 컨셉
- ⭐ "KB 왜 만드냐? ChatGPT 안에 이미 다 있다"고 주장한 분.
- ChatGPT에 "Mark Twain의 묘지는?" 물어봐서 "Elmira" 나오면 → ⭐ **언어모델 자체를 KB로 간주**.

### 13.2 ⭐ 한계 — Hallucination
- ⭐ 잘 알려진 사람만 잘 답하고, 비슷한 이름이면 ⭐ **프랑스 이름 같으면 프랑스에서 태어났다고 hallucination**.
- 90점 받는 줄 알았는데 ⭐ **우연히 안 맞도록 문제 내면 급추락**.

### 13.3 ⭐ Prompt Optimization
- ⭐ **Auto-prompt**: 정보를 잘 뽑아내는 프롬프트를 자동 튜닝.
- ⭐ 더 나아가서 ⭐ **사람이 이해할 필요 없는 벡터 프롬프트** (continuous prompt) 사용.
- ⭐ "이 답이 틀리면 우리 할머니가 총 맞아" 같은 압박 프롬프트도 유명한 예제.

### 13.4 ⭐ 시험 출제 포인트 — Symbolic vs Parametric
- ⭐ **Symbolic Ontology**: 정확, 해석 가능, 명확.
- ⭐ **Parametric Knowledge** (LM): 풍부, 유연, but hallucination.
- ⭐ 둘을 잘 합치는 게 연구의 핵심.

## 14. ⭐ Relation Extraction 실습

### 14.1 ⭐ Research Question으로 정리한 NLP 파이프라인
| Task | Research Question |
|------|-------------------|
| **Tokenization** | "어떻게 쪼개야 돼?" |
| **POS Tagging** | "이 단어들의 역할이 뭐야?" |
| **Named Entity Recognition** | "얘 이름 알아야 돼?" |
| **Relation Extraction** | ⭐ **"얘네 무슨 사이야?"** |
| **Knowledge Graph** | ⭐ **모아서 관계도 그리기** |

### 14.2 ⭐ Hearst Pattern 실습 (Regex)
```python
# NP such as NP, NP, ..., NP
pattern1 = r'([A-Z][a-z]+\s?[a-z]*)\s+such as\s+([A-Za-z\s,]+)'
# NP, NP, and other NP / or other NP
pattern2 = r'([A-Z][a-z]+(?:,\s+[A-Z][a-z]+)*)\s+(?:and|or)\s+other\s+([A-Z][a-z]+)'
```
- ⭐ `(?:...)` ⭐ **Non-capturing group** — 매칭은 되지만 캡쳐 안 됨.
- 사용 이유: `and|or`를 묶고 싶지만 결과로 뽑지는 않을 때.

### 14.3 ⭐ Regex의 가치 — 매우 중요
- ⭐ "LLM이 다 되는 세상이지만 Regex는 여전히 중요".
- ⭐ **1PB 데이터에서 Relation Extraction**해야 하면? — LLM은 너무 비쌈, BERT도 부담.
- ⭐ ⭐ ⭐ **현실적 선택: Regex로 충분히 처리 가능한 패턴**.
- ⭐ ⭐ "프리시전 높고, 해석 가능성 높고, 데이터 학습 불필요".

### 14.4 ⭐ Hearst Pattern의 한계
- is-a 관계만 잡음.
- 다양한 관계 (founder, location, ...) 는 ⭐ **다른 방법 필요**.

## 15. ⭐ Feature-based Supervised Relation Extraction 실습

### 15.1 예시 데이터
- "Bill Gates founded Microsoft in 1975" → (Bill Gates, founder, Microsoft).
- 32개 샘플 짜리 작은 데이터셋.

### 15.2 ⭐ 핵심 피처 — Between Words
```python
def get_words_between(entity1, entity2, sentence):
    idx1 = sentence.find(entity1)
    idx2 = sentence.find(entity2)
    if idx1 == -1 or idx2 == -1:
        return []
    start = min(idx1 + len(entity1), idx2 + len(entity2))
    end = max(idx1, idx2)
    return sentence[start:end].split()
```
- ⭐ `find()`가 -1이면 둘 다 등장 안 함 → 빈 리스트.
- ⭐ 등장 인덱스로 사이 단어 추출.

### 15.3 ⭐ 타입 페어 피처
- spaCy NER로 두 엔티티 type 추출 → ⭐ **(PERSON, ORGANIZATION) 같은 type pair**.
- ⭐ 이것만으로도 가능한 관계가 ⭐ **상당히 좁혀짐**.

### 15.4 ⭐ 학습
- ⭐ **Logistic Regression** (또는 SVM) — 간단하게 import해서 사용.
- 피처: type pair, between word, etc.
- ⭐ **Binary feature vector (0/1)**.

### 15.5 결과
- 전체 accuracy 62%, ⭐ **employee 관계는 굉장히 잘 맞춤**.
- ⭐ **해석 가능 + 적은 데이터로 학습** — 좋은 피처 덕분.

### 15.6 ⭐ 한계
- ⭐ "엔지니어의 장인정신에 의존" — 피처 만들기 어려움.
- ⭐ 자연어의 ⭐ **무궁무진한 변형** 포착 어려움.

## 16. ⭐⭐ Language Model 기반 Relation Extraction 실습

### 16.1 ⭐ 핵심 아이디어
- ⭐ Language Model의 정의 = ⭐ **자연어 시퀀스에 확률을 할당하는 함수**.
- 이 성질을 이용해 Classification 풀기.

### 16.2 ⭐ Entity Marker 추가
```python
# "Bill Gates founded Microsoft" → 
# "[E1] Bill Gates [/E1] founded [E2] Microsoft [/E2]"
```
- ⭐ 토큰 단위로 처리하기 때문에 ⭐ **어디부터 어디까지가 엔티티인지** 명시 필요.
- ⭐ **Special tokens**으로 시작/끝 표시 (HTML 태그처럼).

### 16.3 ⭐ 예측 방법
```python
sentence = "[E1] Bill Gates [/E1] founded [E2] Microsoft [/E2]."
for relation in ["founder of company", "employee of company", 
                 "company located in place"]:
    full_text = sentence + " The relation between E1 and E2 is " + relation
    prob = LM(full_text)  # 시퀀스 전체의 확률
# 확률 가장 높은 relation 선택
```
- ⭐ **자유로운 자연어로 relation 정의 가능** — "founder of company", "founder of corporation" 둘 다 가능.
- ⭐ 미리 심볼로 정의해 가둘 필요 없음.

### 16.4 ⭐⭐ 시험 출제 포인트 — 진짜 확률 vs API 토큰 confidence
- ⭐ ChatGPT API로 sentiment classification 시킬 때 confidence 숫자 받는 것과 ⭐ **질적으로 다름**.
- ⭐ ⭐ ⭐ **ChatGPT가 "0.884" 토큰을 뱉으면 → 그건 그냥 ⭐ "0.884"라는 토큰을 선택했을 뿐 ⭐ 수학적 계산 X**.
- ⭐ 본 실습의 확률 = ⭐ **LM이 실제로 Assign한 (softmax 거친) 확률**.
- ⭐ 이게 진짜 confidence — 비교 가능, 의미 있음.

### 16.5 ⭐ Top-K 결과
- 검색 때처럼 Top-K 뽑아서 sort.
- ⭐ Elon Musk와 SpaceX → ⭐ "founder of company" (가장 높음), CEO, employed by 순.
- ⭐ Softmax로 합 1로 정규화됨.

### 16.6 ⭐ 단점
- ⭐ **계산 비용 높음**: 1PB 데이터엔 부담.
- ⭐ **해석 가능성 낮음**: "근거가 뭐냐?" → "그냥 LM이 그렇게 말함".
- Pre-training + Fine-tuning 필요.

### 16.7 ⭐ Relation Extraction 비용 스펙트럼
| 방법 | 비용 | 정확도 |
|------|------|--------|
| **Hearst Pattern (Regex)** | 가장 저렴 | 좁은 범위 |
| **Feature-based + Logistic Regression** | 중간 | 적은 데이터 OK |
| **LM-based (BERT 등)** | 비쌈 | 풍부한 표현 |
| **ChatGPT API** | 가장 비쌈 | 가장 강력 |

## 17. ⭐ Knowledge Graph & Graph RAG 맛보기

### 17.1 ⭐ 트리플 → 그래프
- ⭐ **트리플 = 그래프의 가장 작은 단위** (Head, Relation, Tail).
- 트리플 모으면 그래프.
- ⭐ Multi-hop 추론: A의 아들이 B, B의 아들이 C → A의 손자가 C.

### 17.2 ⭐ Relation Embedding
- ⭐ **벡터로 만들면 cosine similarity로 비슷한 관계 찾기 가능**.
- "Steve Jobs ↔ Apple 같은 관계와 비슷한 트리플 찾아줘" → ⭐ **string match 안 되어도 의미 비슷한 거 검색 가능**.

### 17.3 ⭐⭐ Graph RAG — 일반 RAG와의 차이
- ⭐ **일반 RAG**: 문서를 chunk로 잘라 retrieve → 이어붙여 LLM에 줌.
- ⭐ **Graph RAG**: 문서들이 연결 관계를 가지고 있으니까 ⭐ **클러스터 형성** → 클러스터 별로 요약.

### 17.4 ⭐ Graph RAG의 차별화 — 매우 중요
- ⭐ "내가 가진 25GB Wikipedia 전체에 대해 ⭐ **인물 공통점** 질문 가능".
- ⭐ ⭐ ⭐ **일반 RAG로는 불가**: 컨텍스트에 다 안 들어감.
- ⭐ 분산된 클러스터에 ⭐ **분산 질의 → 넓은 범위 처리** 가능.

### 17.5 ⭐ 예시
- "Bill Gates와 Jeff Bezos가 무슨 관계?" → ⭐ **둘 다 centibillionaire 상태 달성**이라는 접점 발견.
- ⭐ **굉장히 멀리 떨어진 노드들끼리도 어떤 경로로 이어지는지** 찾기 가능.
