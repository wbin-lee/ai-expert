# 260507 오전 강의 요약 — ColBERT 구현 · Sparse+Dense 결합 · RAG · RankGPT · POS Tagging · NER · IOB · HMM

> 오전 흐름: **ColBERT 실제 구현 (Token-wise embedding · max → sum) → Score Normalization (Min-Max) + α-blending으로 Sparse+Dense 결합 → LangChain RAG 파이프라인 (Chunking / Overlap / Extractive Reader) → ⭐ RankGPT (LLM Listwise Re-ranking) → Pointwise vs Pairwise vs Listwise + Sliding Window + Noise Model 보정 → Query Augmentation / Rank Fusion → ⭐ Knowledge Graph로의 진입 — POS Tagging의 중요성 → POS 1.0 (Rule) / 2.0 (Weight) / 3.0 (Prompt) → Closed vs Open Class → Penn Treebank → 단어 ambiguity (back / around / that) → Word Shape feature → Named Entity Recognition → Wikification → IOB Tagging → ⭐ Hidden Markov Model (Output + Transition Probability) → Greedy / Beam / Viterbi → 실습 (Brown Corpus + NLTK + spaCy NER + IOB + HMM 학습)**
> 강의 주제: **검색의 마무리(Re-ranking) + 자연어의 구조 이해(POS · NER · HMM)** — 황승원 교수님.

## 1. ⭐ ColBERT 구현 — Late Interaction의 코드 레벨

### 1.1 개념 리캡
- Bi-Encoder처럼 Query와 Document를 **각각 따로 임베딩**.
- 단, 단일 벡터로 압축 X — **토큰별 contextualized embedding 유지**.
- ⭐ **Pairwise similarity matrix** 계산 → Doc 차원에 max → Query 차원에 sum.
- "Murder vs Assassination" 같은 ⭐ **Soft Match** 가능.

### 1.2 구현
- Query embedding: `(Q_tokens, dim)`, Document embedding: `(D_tokens, dim)`.
- ⭐ Matrix multiplication 한 번으로: `Q @ D.T` → `(Q_tokens, D_tokens)` 시밀러리티.
- ⭐ 차원별 처리:
  - `sim.max(dim=doc)` → Query 토큰별 best match.
  - `result.sum()` → 최종 score.
- 384차원 모델 사용 / Normalize는 학습 시 unit norm으로 들어감.

### 1.3 ⭐ 주의
- ⭐ ColBERT 점수 계산 방식은 **임베딩 모델 학습 단계부터 그렇게 학습**되어야 최적.
- Bi-Encoder로 학습된 모델에 ColBERT 적용 → mismatch는 있지만 동작은 함.

## 2. ⭐ Sparse + Dense Score Fusion

### 2.1 ⭐ Score Normalization 필요성
- BM25 score: 2점, 1점, 0점대... **bound가 없음**.
- Cosine similarity: ⭐ **-1 ~ 1로 bound**.
- 둘을 그대로 합치면 스케일 차이로 왜곡 → ⭐ **min-max normalize**해서 둘 다 [0, 1] 범위로.

### 2.2 ⭐ Min-Max Normalization
```
normalized = (raw - min) / (max - min)
```
- 화씨 → 섭씨 변환과 같은 식.
- ⭐ Broadcasting (NumPy/PyTorch) — scalar와 array 자동 element-wise 처리.
- `torch.zeros_like` / `np.ones_like` / `clone()` 등 shape 보존 유틸.

### 2.3 ⭐ α-Blending
```
combined = α × dense_score + (1-α) × sparse_score
```
- 두 점수 모두 0~1 범위 → 합쳐도 0~1.
- α=0.5면 1:1 혼합.

### 2.4 ⭐ 더 좋은 방법 — Rank Fusion
- Score normalize는 정보 손실 위험.
- ⭐ **Rank만 합치기** — 어떤 점수든 순위는 동일하게 매겨짐.
- 외부 검색엔진(블랙박스)은 score 정보 없이 rank만 줄 수 있음 → rank fusion이 더 robust.

## 3. ⭐ RAG 파이프라인 (LangChain)

### 3.1 컴포넌트
1. **Embedding Encoder**: query/document를 벡터로.
2. **Index**: Document 벡터 저장 (FAISS 등).
3. **Retriever**: top-K 검색.
4. **(Optional) Re-ranker**: Cross-Encoder 등.
5. **Generator**: LLM이 retrieved context로 답 생성.

### 3.2 ⭐ Chunking — 긴 문서 처리
- 긴 문서는 검색 단위로 너무 큼 → ⭐ **chunk로 잘라서 각 chunk를 별도 문서처럼 취급**.
- ⭐ **Trade-off**:
  - **길게 자름** → 컨텍스트 보존 ↑, but 토큰 한계 / 모델 처리 부담.
  - **짧게 자름** → 같은 토큰 버젯에 더 많은 chunk 가능, but 의미 파편화.
- ⭐ **Overlap**: chunk 경계가 의미를 자르지 않도록 100 토큰 정도 겹치게 설정 (예: 0-700, 600-1300, 1200-1900...).

### 3.3 ⭐ LangChain의 Chain 개념
- Prompt 정의 → LLM 호출 → 검색 → Parse → 다음 단계로 전달.
- ⭐ 일직선 = Chain / 분기 = Graph.
- 마우스로 드래그하는 그래픽 도구도 등장.

### 3.4 RAG 실습 — 미국 대통령 연설문
- Justice Ketanji Brown Jackson 같은 query.
- 정확한 이름이 chunk에 잘려도 **Justice 같은 관련 단어 매칭**으로 fetching.

## 4. ⭐ RankGPT — LLM Listwise Re-ranking

### 4.1 컨셉
- 기존 Cross-Encoder는 ⭐ **Pointwise**: Q + D 각각 점수 → 전역 순위 자동.
- ⭐ **RankGPT**: LLM에게 다중 passages를 한꺼번에 주고 **순서를 다시 매겨라** 요청.
- 인스트럭션 + Query + 다중 Passages → LLM이 새로운 ranking 출력.

### 4.2 Pointwise vs Pairwise vs Listwise
| 방식 | 입력 | 특징 |
|------|------|------|
| **Pointwise** | (Q, D) 한 쌍 → 점수 | 전역 순위 자동, but 절대 점수 매기기 어려움 |
| **Pairwise** | (Q, D1, D2) → A vs B | 비교가 더 쉬움 |
| **Listwise** | (Q, [D1...Dk]) → 순서 | 동시 비교 가능, but ⭐ 한계 多 |

### 4.3 ⭐ Listwise의 장점 — 비유
- ⭐ "발표 평가 비유": 발표 하나 듣고 90점 vs 다른 팀과 같이 보고 비교가 훨씬 쉽다.
- 사람도 절대 점수보다 비교가 더 정확.

### 4.4 ⭐ Listwise의 한계 — 두 가지
1. **Scaling 불가**: 100개를 한 번에 ranking 불가능 (LLM 컨텍스트 + 신뢰도).
   - ⭐ **Sliding Window**: 5개씩 끊어서 → 토너먼트식 / 아래에서 위로 / 다양한 전략.
2. ⭐ **순서에 따라 결과가 바뀜**:
   - D1~D5 vs D5~D1로 순서 바꿔서 줘도 답이 다름.
   - **Lost in the Middle / Recency Bias의 발현**.

### 4.5 ⭐ Noise Model로 보정 (Spelling Correction 비유)
- 오타 수정에서 "키보드 인접 키 실수 확률 모델링"과 동일 발상.
- LLM의 listwise 출력을 **noise channel을 통과한 결과**로 해석.
- 진짜 좋은 순서가 있는데, 입력 순서에 따라 output에 noise 추가됨.
- ⭐ 여러 입력 순서로 시도 → noise 평균 → **진짜 ranking 추정**.

## 5. ⭐ Query Augmentation / Multi-hop

### 5.1 컨셉
- 답에 필요한 재료가 3개인데 Retriever가 2개만 찾았다면 → ⭐ **나머지 1개를 위한 새 query 생성**.
- 발견한 정보를 기반으로 다시 검색 → Multi-hop retrieval.
- HyDE도 같은 맥락 — query를 hypothetical doc으로 변환.

### 5.2 의사 피드백 (Pseudo-Relevance Feedback)
- 사람의 클릭이 없을 때 — Top-K를 일단 relevant로 가정해서 Augmentation에 활용.

## 6. ⭐⭐ Knowledge Graph 추출 — 자연어의 숨은 구조

### 6.1 동기
- 텍스트는 시퀀스지만 ⭐ **숨은 구조**가 있음 (주어, 동사, 명사, 형용사).
- 토큰 prediction에는 명시적 구조 학습 X — but 구조를 적극적으로 학습하면 다양한 task에 큰 도움.
- ⭐ **특히 Knowledge Graph / Entity Extraction에 결정적**.

### 6.2 결정적 예시
- "**승원 is teaching at SNU**":
  - "승원" = 개체 (Person), 다양한 surface form (황승원 교수, Prof. Hwang).
  - "SNU" = 개체 (Location), 서울대 / 국립서울대 / Seoul National Univ. — ⭐ **모두 같은 grounded entity**.
  - "teaching" = 동사 → **Relation**.
- ⭐ 이걸 그래프로: `[승원] —teaching at→ [SNU]`.

### 6.3 ⭐ POS의 중요성
- ⭐ **개체는 거의 항상 명사구** → 명사구를 알면 개체 위치 예측 ↑.
- ⭐ **관계는 동사** → POS 정보가 Knowledge Extraction에 sine qua non.
- 데이터 부족한 도메인 (반도체 등)에서 ⭐ **구조 정보가 데이터 부족을 대체**.

## 7. ⭐ POS (Part-of-Speech) Tagging

### 7.1 ⭐ 1.0 / 2.0 / 3.0 패러다임
| 버전 | 방법 | 예시 |
|------|------|------|
| **1.0** | Python + Rule-based | `if word.endswith('ly')`: 부사 / regex로 대명사 매칭 |
| **2.0** | Weight (학습) | 텍스트 → 태그 시퀀스 라벨 데이터로 RNN/HMM 학습 |
| **3.0** | Prompting | "POS-tag this sentence: I love you" → ChatGPT 답 |

### 7.2 ⭐ Closed vs Open Class
| Closed | 거의 새 단어 X | 대명사 / 관사 / 전치사 |
|--------|---------------|----------------------|
| **Open** | 신조어 多 | 명사 / 동사 / 형용사 / 부사 |
- Closed Class는 룰베이스로 거의 100% 가능.
- Open Class는 신조어로 인해 룰만으로는 한계.

### 7.3 ⭐ POS Tagging의 어려움 — 단어의 다의성
- "back": 명사 (등) / 형용사 (뒷~) / 부사 (다시 돌이키다) / 동사 (밀어주다, 지지하다).
- "around" / "that": 다양한 품사로 사용.
- ⭐ "lead" — 발음까지 다름 (리드 vs 납) → POS 알아야 TTS 정확.
- ⭐ **특정 단어만 어렵고 대부분은 쉽다** — ambiguity는 소수에 집중.

### 7.4 ⭐ Penn Treebank
- 펜실베니아 대학에서 만든 **POS 태깅 코퍼스 — NLP 표준 리소스**.
- 단어-태그 쌍을 언어학자가 어노테이트.
- ⭐ NN, NNP, VB, VBG, JJ, RB, IN, DT 등 **표준 태그 컨벤션**의 출처.

### 7.5 ⭐ Ambiguity 통계
- ⭐ **토큰 기준**: 60% 단일 태그 / 40% 모호.
- ⭐ **타입(Unique word) 기준**: ~89% 단일 태그.
- 자주 쓰이는 단어일수록 다의성 多 (back, run, light, well 같은 단어들).

### 7.6 ⭐ POS의 정확도 패러다임
- ⭐ "단어별 가장 흔한 태그" 베이스라인만으로도 **90%+ 달성**.
- ⭐ 하지만 **ambiguous 단어로 한정**하면 50%대로 폭락.
- 즉 진짜 어려운 건 **소수의 모호 단어**.

### 7.7 ⭐ Word Features (피처 설계)
- **Capitalization**: SNU, Seoul → 명사구/개체일 확률 ↑.
- **접미사**: -ly → 부사, -ive → 형용사, -ing → 현재분사.
- **접두사**: un- → 형용사 가능성 ↑.
- **숫자 + 명사**: "80-anniversary" 형태 → 형용사.
- **하이픈, 슬래시, 특수문자**: 형태로 의미 추정.

## 8. ⭐ Named Entity Recognition (NER)

### 8.1 컨셉
- 단어들을 **사람 / 장소 / 조직 / 시간 / 숫자** 등으로 태깅.
- ⭐ Knowledge Graph 추출의 **첫 번째 단계**.

### 8.2 ⭐ Coarse vs Fine-grained
- **Coarse**: 4-5개 클래스 (Person / Location / Organization / Date).
- **Fine-grained**: ⭐ **Wikification** — 위키피디아의 10만 개 페이지 중 어디로 link.

### 8.3 ⭐ Wikification
- 평범한 텍스트 → **Hyperlinked text**.
- ⭐ "Andrew Wilkie" → 정확한 위키 URL로 link.
- 어려움: **Disambiguation** — 같은 이름 여러 엔티티 (Super Mario 게임 vs 영화 vs 애니).
- ⭐ **Overconfidence 문제**: 위키 페이지가 없는데도 억지로 link하려는 경향 → "link하지 않는 결정"도 중요.

### 8.4 사용처
- 캘린더 자동 입력 (전화번호 → 전화 link, 주소 → 지도, 날짜 → 캘린더).
- 의학 — 약품명 / 성분명 / 부작용 관계 자동 추출.
- 정치 뉴스 — 인물 / 정당 / 관계 그래프 자동 생성.

## 9. ⭐⭐ IOB Encoding — 다단어 엔티티 처리

### 9.1 문제
- POS tagging: I love you → PRON, VERB, PRON (단어별로 깔끔).
- ⭐ NER에서 "Shen Guofang"가 등장하면:
  - 한 사람의 이름인지? 두 사람인지? **단일 태그로는 구별 불가**.

### 9.2 ⭐ IOB / BIO 인코딩
- ⭐ HTML의 start/end tag와 비슷한 발상.
- **B (Beginning)**: 새 entity 시작.
- **I (Inside)**: 같은 entity 계속.
- **O (Outside)**: entity 아님.

### 9.3 예시
```
Andrew    Wilkie     supported     Labor     ...
B-PERSON  I-PERSON   O             B-ORG     ...
```
- Andrew + Wilkie = 한 사람 (B 다음 I).
- John, Smith → B-PERSON, B-PERSON이면 두 사람.

### 9.4 ⭐ 응용 — 모든 NLP 문제를 태깅으로
- **중국어 띄어쓰기**: B, I, I, B, I, B, B로 어절 경계 표시.
- **QA span 추출**: 어디부터 답인지 BIO로 태깅.
- ⭐ "수많은 NLP 문제가 사실상 sequence tagging".

## 10. ⭐⭐ Hidden Markov Model (HMM)

### 10.1 ⭐ 이름의 두 가지 의미
- **Markov**: ⭐ **직전 상태만 알면** 다음 상태 결정 가능 (Markov property).
- **Hidden**: ⭐ 우리가 보고 싶은 **품사는 숨겨져 있고 단어만 관찰 가능**.

### 10.2 ⭐ 두 종류의 확률
| 확률 | 의미 |
|------|------|
| **Output Probability** | P(word \| tag) — 명사일 때 SNU가 나올 확률 |
| **Transition Probability** | P(tag_t \| tag_{t-1}) — 명사 다음 동사가 올 확률 |

### 10.3 ⭐ 학습 — 매우 단순
- Penn Treebank 같은 ⭐ **태깅된 코퍼스에서 카운트**.
- Transition: tag pair 빈도 / 전체.
- Output: (tag, word) 빈도 / tag 빈도.
- ⭐ **Add-one smoothing** (확률 0 방지).

### 10.4 ⭐ Inference — 시퀀스 디코딩 3가지 방법
| 방법 | 특징 | Trade-off |
|------|------|----------|
| **Greedy** | 매 단계에서 최선 선택, 돌에 새김 | 빠름, suboptimal |
| **Beam Search** | K개 후보 유지 → 끝까지 가서 선택 | 중간, K에 따라 누락 가능 |
| **Viterbi (DP)** | ⭐ **Dynamic Programming**, 모든 경우 최적해 | 느림, **반드시 best** |

### 10.5 ⭐ Viterbi의 직관
- 모든 시퀀스 조합은 12^10 (12 태그, 10 토큰) → 너무 많음.
- ⭐ **Markov property** 덕분에 직전 단계 최적해만 기억하면 됨.
- ⭐ Edit Distance DP와 동일 발상.

## 11. 실습 — Brown Corpus + NLTK + spaCy

### 11.1 환경
- **Brown Corpus** (1961, 65년 전 신문 기사) — 태깅된 대형 코퍼스.
- **NLTK** `pos_tag` — 사전 학습된 POS tagger.
- **spaCy** `nlp(text).ents` — Named Entity Recognition.

### 11.2 단계
1. NLTK로 word_tokenize + pos_tag.
2. Brown Corpus 통계 — 카테고리별 / 태그 분포.
3. ⭐ **다중 태그 단어 찾기** — back, run, light, well 발견.
4. **Word Shape feature** 구현 (대문자/숫자/특수문자 패턴).
5. spaCy로 NER 실행 → Apple INC / Steve Jobs / San Francisco 등.
6. ⭐ **IOB encoding** 변환 함수 구현.
7. ⭐ **HMM 학습** — Conditional Probability counts (start tag 포함).
8. Add-one smoothing.
9. Brown Corpus tagged sentences로 transition / output probability 추출.
10. 다음 시간 — Viterbi 알고리즘 구현 (점심 후).

### 11.3 발견
- 대명사 다음 동사 확률 매우 높음.
- 관사 다음 명사 / 형용사 확률 패턴 확인.
- 끝 3글자만 봐도 POS 잘 맞음 (룰베이스의 위력).

## 12. ⭐ 한 페이지 요약

### 12.1 오전 핵심 포인트

| 주제 | 핵심 |
|------|------|
| **ColBERT 구현** | Token-wise embedding → pairwise sim → max(doc) → sum(query) |
| **Score Fusion** | Min-Max normalize 후 α-blending / 또는 Rank Fusion |
| **Chunking** | 길이 trade-off / Overlap 100 토큰 정도 |
| **RankGPT** | LLM으로 listwise re-ranking |
| **Pointwise vs Pairwise vs Listwise** | 비교가 절대 점수보다 쉬움 |
| **Listwise 한계** | Sliding Window 필요 / 순서에 따라 답 변동 / Noise Model 보정 |
| **Knowledge Graph 추출** | 개체 → 명사구 / 관계 → 동사 → POS가 결정적 |
| **POS 1.0/2.0/3.0** | Rule / Weight / Prompt |
| **Closed vs Open Class** | 대명사 closed / 명사 open |
| **Penn Treebank** | NN/VB/JJ/RB 표준 태그의 출처 |
| **POS 베이스라인** | 90%+ 쉽게, 모호 단어로 한정시 50%대 |
| **Word Shape** | 대문자 / 접미사 / 숫자 패턴이 강력 |
| **NER** | 4-5 클래스 (Person/Loc/Org/Date) |
| **Wikification** | 위키피디아 10만 페이지로 fine-grained linking |
| **IOB Encoding** | B/I/O로 다단어 엔티티 경계 표시 |
| **HMM** | Output P(word\|tag) + Transition P(tag\|tag) |
| **Inference** | Greedy / Beam / ⭐ Viterbi (DP, optimal) |

### 12.2 전체 구조 도식
```
[검색 마무리]
ColBERT 구현 (token-wise → max → sum)
   ↓
Score Fusion (Normalize + α-blend / Rank Fusion)
   ↓
RAG 파이프라인 (Chunking + Overlap + LangChain)
   ↓
RankGPT (LLM Listwise Re-ranking)
   ↓ Listwise 한계
Sliding Window / Noise Model 보정

[Knowledge Graph로 진입]
"승원 is teaching at SNU"
   ↓ 개체 = 명사구 / 관계 = 동사
POS Tagging의 결정적 중요성
   ↓
1.0 (Rule) / 2.0 (Weight) / 3.0 (Prompt)
   ↓
Closed (대명사) vs Open (명사) Class
   ↓
Penn Treebank (NN/VB/JJ/RB...)
   ↓
Ambiguity: 토큰 60%/40% — 타입 89%/11%
   ↓
Word Shape Features (Capital / 접미사 / 숫자)
   ↓
NER (Person/Loc/Org/Date) → Wikification
   ↓
IOB Encoding (B/I/O로 다단어 엔티티)
   ↓
[Hidden Markov Model]
Output Probability + Transition Probability
   ↓ Inference
Greedy → Beam → Viterbi (DP, optimal)
```

### 12.3 비유 / 직관 모음

| 개념 | 비유 / 직관 |
|------|-----------|
| **ColBERT** | "토큰별 인베딩 다 살려놓고 query 토큰 best match × 합산" |
| **Score Normalize 비유** | "화씨 → 섭씨 변환 — 32 빼고 180으로 나누고 100 곱하기" |
| **Rank Fusion** | "어떤 점수든 순위는 동일 — 정보 손실 없음" |
| **Chunking Overlap** | "0-700, 600-1300 — 100 토큰 겹쳐서 의미 보존" |
| **Listwise = 발표 평가** | "절대 점수보다 비교가 쉽다 — 발표 듣고 90점은 어렵다" |
| **Sliding Window** | "100개를 5개씩 — 토너먼트 / 아래에서 위로" |
| **Noise Model** | "오타 수정 비유 — 진짜 ranking이 있고 LLM 출력은 noise channel" |
| **POS의 결정성** | "개체 = 명사구 / 관계 = 동사 — Knowledge Graph의 sine qua non" |
| **POS Closed vs Open** | "대명사는 새로 발명 안 됨 / 명사는 매일 만들어짐 (유퀴즈)" |
| **POS 다의성** | "back = 등 / 뒷~ / 다시 / 밀어주다" |
| **lead vs read 발음** | "리드(이끌다) vs 레드(납) — POS 알아야 TTS 정확" |
| **Ambiguity 분포** | "쉬운 단어는 쉽고 / 어려운 단어는 극악" |
| **Word Shape** | "SNU 같이 대문자만 → 거의 100% 명사구" |
| **NER → Wikification** | "Andrew Wilkie → 위키 URL로 link / Super Mario는 disambiguation" |
| **IOB 비유** | "HTML start/end tag — Begin person → Inside person" |
| **HMM Hidden** | "단어는 보이는데 품사는 숨어있다" |
| **HMM Markov** | "직전만 알면 다음 결정 가능" |
| **Greedy → Beam → Viterbi** | "돌에 새기기 / 숙제함에 K개 / 모든 경우 DP optimal" |

### 12.4 강사 어록
- "Murder vs Assassination 같은 비슷한 매치를 ColBERT로 soft하게 잡을 수 있다."
- "BM25는 0~1 bound가 없으니 Cosine과 합치려면 normalize 필수."
- "Rank로 합치는 게 사실 더 robust하다 — 어떤 점수든 순위는 같으니까."
- "발표 평가 비유 — 절대 점수보다 비교가 사람 입장에서도 쉽다."
- "100개를 한 번에 ranking할 수 없으니 Sliding Window — 토너먼트 / 아래에서 위로."
- "LLM listwise는 입력 순서에 따라 답이 바뀐다 — Noise Model로 보정."
- "구조를 적극적으로 배우는 게 토큰 prediction에는 없지만 Knowledge Graph에는 결정적이다."
- "개체는 거의 항상 명사구 / 관계는 동사 — POS의 sine qua non."
- "POS Tagging은 쉬운 단어는 너무 쉽고 어려운 단어는 극악이다."
- "lead라고 할지 read라고 할지 결정해야 음성을 정확히 생성할 수 있다."
- "Andrew Wilkie 같은 위키 페이지가 없는 사람은 link하지 않는 결정이 더 어렵다."
- "Overconfidence 문제 — 모르겠는데 자꾸 어디 link하려고 한다."
- "수많은 NLP 문제가 사실상 sequence tagging으로 풀린다 — POS / NER / 중국어 띄어쓰기 / QA span."
- "HMM의 이름에 모든 게 들어있다 — Markov는 직전만 보고 / Hidden은 품사가 숨어있다."
- "조건부 확률, 결국 우리는 항상 조건부 확률로 돌아온다."
- "Greedy는 돌에 새기고, Beam은 숙제함에 넣어두고, Viterbi는 모든 가능성을 DP로 본다."
